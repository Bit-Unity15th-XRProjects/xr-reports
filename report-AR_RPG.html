프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 개발하고 있으며, 현재 몬스터 스폰(MonsterSpawner)·전투(BattleManager)·레벨업(LevelManager)·인벤토리(InventoryController)의 핵심 게임 루프가 실제로 동작하는 프로토타입 단계까지 도달했다. 3×3 OSM 타일 맵(MapGridGenerator) 위에서 몬스터가 등급별 확률로 스폰되고, 어디티브 씬 로딩으로 배틀에 진입하여 클릭 데미지·타이머·드롭 시스템이 작동하며, 승리/패배 분기까지 완성되어 있다. 그러나 프로젝트 제목의 핵심인 GPS 이동이 주석 처리되어 터치 드래그로 대체된 상태이고, SaveManager·SoundManager가 완전히 부재하여 데이터 영속성과 게임 몰입감이 크게 부족하다. 가장 시급한 것은 GPS 기능 복원과 최소한의 저장 시스템 구현이며, origin/LimChaesung 브랜치의 VFX 작업을 main에 머지하고 팀 전원이 남은 기간에 집중적으로 마무리 작업에 돌입해야 한다.

권고사항 (Top 3)

1. SaveManager 즉시 구현 — PlayerPrefs 기반 최소 버전
게임의 가장 기본적인 기능인 진행 상황 저장이 없으면 시연 시에도 매번 처음부터 시작해야 함. MapDataManager.cs에 JSON 직렬화 유틸이 이미 있으므로 이를 활용 가능
  1. SaveManager.cs 생성: PlayerPrefs에 JSON 직렬화된 SaveData(level, exp, inventory items) 저장/로드
  2. LevelManager에 Save/Load 메서드 추가, AddExp() 후 자동 저장
  3. InventoryController에 아이템 목록 직렬화 추가, AddItem()/ToggleEquip() 후 자동 저장
  4. 앱 시작 시(MapGridGenerator.Start 또는 별도 GameManager) SaveManager.Load() 호출
2. GPS 이동 기능 복원 및 하이브리드 입력 시스템 구현
프로젝트 제목이 'AR 위치기반 몬스터 RPG'이므로 GPS 연동이 핵심 평가 요소. 현재 터치 드래그만 동작하는 상태에서는 '위치기반'이라 할 수 없음
  1. GpsMapPlayerMover.Update()에서 #else 블록을 수정: UpdateGps()를 주석 해제하여 GPS 위치를 _targetPosition에 반영
  2. HandleTouchDrag()는 GPS 오프셋에 대한 미세 조정으로 병행 유지
  3. MapGridGenerator.cs:54의 폴백 좌표 오타(7→37) 수정
  4. GPS 정확도가 낮을 때(accuracy > 30m) UI에 경고 표시 추가
3. BattleManager 리팩토링 — 책임 분리
403줄의 BattleManager가 HP 계산, 아이템 생성, 드롭 확률, 씬 전환, UI 제어를 모두 담당하여 유지보수와 버그 추적이 어려움. 300줄 이하 권장 기준도 초과
  1. BattleRewardService.cs 분리: RollWeaponDrop(), CreateRandomItem(), GetWeaponAttackByGrade() 이동
  2. BattleSceneTransition.cs 분리: ReturnToField(), ReturnToFieldWithoutReward()의 카메라/씬 복원 로직 이동
  3. BattleManager는 전투 루프(Update, TryHitMonster, TimeOutDefeat)와 상태 관리만 담당하도록 축소

코드 리뷰

BattleManager.cs개선필요
403줄로 프로젝트 내 가장 큰 파일. HP 스케일링(GetScaledMaxHp), 데미지 처리(TryHitMonster), 아이템 생성(CreateRandomItem), 드롭 확률(RollWeaponDrop), 씬 정리(ReturnToField/ReturnToFieldWithoutReward), 승리 UI 연동(VictoryData)까지 모든 전투 관련 로직을 담당. GetRewardItem()은 null 반환하는 데드 코드. ReturnToField()와 ReturnToFieldWithoutReward()에서 MonsterSpawner.RestoreInGameSceneRoots() 호출이 중복될 수 있으며, TimeOutDefeat()에서 RestoreInGameSceneRoots() 호출 후 ReturnToFieldWithoutReward()에서 다시 호출. 366줄에서 LevelManager.Instance.AddExp() null 체크 없이 호출하여 크래시 가능.
MonsterSpawner.cs양호
깔끔한 스폰/디스폰 로직. sqrMagnitude로 거리 비교 최적화, 역순 순회로 리스트 제거 안전 처리, 레벨별 확률 테이블 구현이 잘 됨. EnterBattle()에서 additive 씬 로딩과 카메라 전환, HideInGameSceneRootsExcept()로 씬 오브젝트 관리가 체계적. 다만 주석 처리된 이전 코드(128~131줄, 141~165줄)가 남아있어 정리 필요.
Monster.cs개선필요
331줄로 경계선. Wander/Flee AI는 잘 동작하나, ApplyVisualByGradeType()이 140줄~241줄(약 100줄)로 하나의 메서드에 스프라이트 로딩, SpriteRenderer 생성, Material 생성, 셰이더 폴백, 캡슐 메시 숨기기까지 모두 포함. 이 메서드를 분리해야 함. Epic Slime 하드코딩 오프셋(148~152줄)은 데이터 드리븐으로 변경 권장. Resources.Load를 매번 호출하며 캐싱이 없어 스프라이트 중복 로딩 발생.
BattleSelectUI.cs양호
399줄로 길지만 UI 코드 특성상 합리적. 0.3초 주기 리스트 갱신, EventSystem.IsPointerOverGameObject()로 UI 클릭 필터링, 슬라이드 애니메이션 코루틴, 배틀 전후 패널 상태 복원 로직이 잘 구현됨. 다만 RefreshList()에서 매번 모든 리스트 아이템을 Destroy/Instantiate하는 것은 비효율적이며 오브젝트 풀링 고려 필요.
GpsMapPlayerMover.cs개선필요
GPS→월드 좌표 변환(ConvertGpsToWorld, GetMeterOffset)이 정확한 수식으로 구현되어 있고, 에디터 WASD 목업도 잘 작동. 그러나 실제 GPS 업데이트(UpdateGps)가 주석 처리되어 있고(131줄), 안드로이드에서는 HandleTouchDrag()만 사용 — 즉 위치기반 게임인데 GPS가 비활성화 상태. _isReady와 _isInitialized 두 플래그의 역할이 불명확하고 SyncWithMap()에서 둘 다 true로 설정. RecenterOriginToCurrentLocation()은 에디터에서만 동작하고 안드로이드 빌드에서는 빈 메서드.
InventoryController.cs개선필요
Singleton 중복 방어(19~21줄)와 OnDestroy 정리(28~31줄)는 잘 됨. 그러나 NUnit.Framework.Interfaces 네임스페이스를 using하고 있어(1줄) 테스트 프레임워크 의존이 프로덕션 코드에 유입됨. 슬롯이 가득 찼을 때 AddItem()이 아이템을 버리고 Debug.Log만 남기는데, 유저에게 피드백이 없음. 변수명 오타: OutPanul(Panul→Panel). 인벤토리 영속성(저장/로드)이 전혀 없어 앱 재시작 시 모든 아이템 소실.
LevelManager.cs개선필요
42줄로 매우 간결하지만 핵심 문제 존재. AddExp()에서 ExpUIController.Instance.UpdateExpUi()를 null 체크 없이 호출(30줄)하여, ExpUIController가 없는 씬에서 NullReferenceException 발생. 레벨/경험치 영속성 없음. currentExp가 float인데 BattleManager에서 int로 넘기는 등 타입 불일치. Awake()에서 기존 Instance가 있을 때 새 인스턴스를 무시하지만 Destroy하지 않아 중복 컴포넌트 잔류 가능.
DefeatController.cs양호
103줄로 간결하며 안전한 폴백 처리가 돋보임. BattleManager.Instance가 null인 경우(BattleScene이 이미 언로드된 경우)에 대한 방어 코드(68~91줄)가 잘 작성됨. 씬 유효성 검사(battleScene.IsValid() && battleScene.isLoaded) 후 언로드, 카메라 복원 로직이 체계적. Update()에서 클릭 감지로 맵 복귀하는 UX도 적절.

아키텍처 평가

수동 Singleton 패턴이 프로젝트 전반에 사용됨(BattleManager, LevelManager, InventoryController, VictoryController, DefeatController, ExpUIController — 총 6개). 그러나 구현이 일관되지 않아, InventoryController만 중복 방어(Instance != null && Instance != this)를 하고 나머지는 무조건 덮어씌움. 씬 간 데이터 전달에 BattleSelectionContext(static 필드)와 MonsterSpawner의 static 필드(FieldPaused, FieldCameraRef, ArCameraRef)를 사용하는데, 이는 간단하지만 전역 상태에 과도하게 의존하여 디버깅이 어려움. Assembly Definition으로 모듈 분리(ARRPG.Inventory, ARRPG.Level 등)는 학생 프로젝트치고 우수한 설계 판단이나, BattleManager가 3개 네임스페이스를 직접 참조하여 사실상 의존 그래프의 중심점 역할. 어디티브 씬 로딩(InGame + BattleScene)과 카메라 뎁스 레이어링을 활용한 씬 전환 구조는 잘 설계되었으나, 씬 루트 오브젝트를 수동으로 숨기는 HideInGameSceneRootsExcept() 방식은 새 오브젝트 추가 시 누락 위험이 있음.

5
BattleManager.cs가 403줄로 CLAUDE.md 권장(300줄)을 초과하며, HP 스케일링·드롭·아이템 생성·씬 전환을 모두 담당하는 God Object 경향. GetRewardItem()이 null만 반환하는 데드 코드로 남아있고, CreateRandomItem()에서 ScriptableObject.CreateInstance<Item>()을 런타임에 반복 생성하여 메모리 누수 가능성 있음. 반면 MonsterSpawner.DespawnDistantMonsters()의 역순 순회, sqrMagnitude 활용 등 성능을 고려한 좋은 패턴도 확인됨.
설계/구조
5
모든 매니저 클래스가 public static Instance 패턴(수동 Singleton)을 사용하나, BattleManager.Awake()에서 중복 검사 없이 Instance = this로 덮어씌움(InventoryController만 중복 방어 구현). 씬 간 통신이 MonsterSpawner의 static 필드(FieldCameraRef, ArCameraRef, FieldPaused)와 BattleSelectionContext의 static 필드에 과도하게 의존하여 결합도가 높음. Assembly Definition으로 네임스페이스 분리는 잘 했으나, BattleManager가 ARRPG.Inventory, ARRPG.Level, ARRPG.Victory를 모두 직접 참조하여 단방향 의존이 아닌 중심점 역할을 함.
