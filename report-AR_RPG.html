프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 개발하고 있으며, 핵심 게임 루프(필드 탐색→몬스터 조우→전투→보상)의 기본 골격은 구현된 상태이다. BattleManager의 레벨 스케일링 시스템, MonsterSpawner의 등급 확률 스폰, Monster의 배회/도주 AI, GpsMapPlayerMover의 GPS 좌표 변환 등 AR RPG의 핵심 시스템들이 실제로 동작하는 코드로 존재한다. 그러나 SoundManager가 완전히 부재하고 오디오 파일이 단 하나도 없어 게임이 무음 상태이며, 플레이어 레벨/인벤토리를 저장하는 시스템이 없어 앱 종료 시 모든 진행이 초기화된다. 가장 시급한 것은 SaveManager와 SoundManager의 구현이며, 그다음으로 origin/LimChaesung 브랜치(GPS/맵 코드)의 main 머지와 싱글톤 중복 인스턴스 버그 수정이 필요하다. 현재 진행률은 약 46%로, 남은 2주 안에 저장/사운드/폴리시를 집중하면 발표 가능한 수준에 도달할 수 있다.

리스크 알림

SoundManager 완전 부재 및 오디오 에셋 0개
AR RPG에서 전투 타격음, 몬스터 등장/도주 효과음, 승리/패배 BGM은 게임 경험의 핵심. 현재 오디오 파일이 단 하나도 없고 SoundManager 클래스도 없어서, 발표 시 무음 상태로 시연하게 됨. 사운드 추가는 에셋 확보부터 매니저 구현, 각 시스템 연동까지 시간이 걸림.
  • SoundManager 싱글톤 생성 (BGM/SFX AudioSource 분리)
  • 최소 5개 효과음 확보: 몬스터 타격, 승리, 패배, 아이템 획득, UI 클릭
  • BattleManager.TryHitMonster(), VictoryData(), TimeOutDefeat()에서 SoundManager 호출 추가
플레이어 진행 데이터 저장/로드 시스템 부재
현재 LevelManager의 레벨/경험치, InventoryController의 아이템이 앱 종료 시 모두 초기화됨. MapDataManager는 맵 데이터만 저장하고 플레이어 데이터는 처리하지 않음. AR 위치기반 RPG는 반복 플레이가 핵심인데, 진행이 저장되지 않으면 게임의 의미가 없음.
  • SaveManager 클래스 생성: PlayerPrefs 또는 JSON으로 레벨, 경험치, 인벤토리 직렬화
  • LevelManager.LevelUp()과 InventoryController.AddItem() 후 자동 저장 호출
  • 앱 시작 시 SaveManager.Load()로 데이터 복원
LevelManager/VictoryController 싱글톤 중복 인스턴스 버그
BattleScene이 Additive 로딩될 때 두 씬에 각각 매니저가 있으면 Instance가 덮어씌워짐. LevelManager의 경우 레벨과 경험치가 초기화되는 치명적 버그로 이어짐. 실제 플레이 테스트에서 '레벨이 갑자기 1로 돌아가는' 현상으로 나타날 수 있음.
  • 모든 싱글톤 Awake()에 중복 방지 추가: if (Instance != null && Instance != this) { Destroy(gameObject); return; }
  • DontDestroyOnLoad 적용 여부 검토
  • BattleScene에는 매니저 프리팹을 배치하지 않도록 씬 구성 정리
origin/LimChaesung 브랜치 머지 안 됨 (ahead 10, behind 7)
GPS/맵 관련 핵심 코드를 담당하는 브랜치가 main과 10커밋 차이나며 미머지 상태. 마지막 활동이 3/20으로 6일 전이고 behind 7이라 충돌 가능성 높음. 이 브랜치의 코드가 합쳐지지 않으면 최종 빌드에서 맵/GPS 기능이 불완전할 수 있음.
  • 즉시 main으로 rebase 또는 merge 시도
  • 충돌 발생 시 팀 전체가 모여서 해결
  • 머지 후 Android 실기기에서 GPS 동작 테스트

권고사항 (Top 3)

1. SaveManager + SoundManager 즉시 구현
이 두 시스템이 없으면 게임으로서 최소 기능을 충족하지 못함. 저장 없이는 반복 플레이 불가, 사운드 없이는 발표 시 임팩트 부족. 나머지 시스템은 이미 동작하므로 이 두 개만 추가하면 완성도가 크게 올라감.
  1. SaveManager: PlayerPrefs 기반으로 {level, exp, inventory[]} JSON 직렬화 구현 (1일)
  2. SoundManager: AudioSource 2개(BGM/SFX), PlaySFX()/PlayBGM() 메서드, 무료 에셋(freesound.org) 활용 (1일)
  3. 기존 BattleManager, LevelManager, InventoryController에서 Save/Sound 호출 연동 (0.5일)
2. 싱글톤 패턴 통일 및 안전장치 추가
현재 5개 이상의 싱글톤이 각각 다른 방식으로 구현되어 있고, 일부는 중복 인스턴스 방지가 없어 Additive 씬 로딩 시 버그 발생. 통일된 베이스 클래스를 만들면 모든 매니저에 일관된 안전장치를 적용할 수 있음.
  1. SingletonBase<T> 제네릭 베이스 클래스 작성 (중복 방지 + DontDestroyOnLoad 옵션)
  2. BattleManager, LevelManager, InventoryController, VictoryController, DefeatController를 SingletonBase 상속으로 전환
  3. 각 씬에서 매니저 프리팹 중복 배치 여부 점검
3. 300줄 초과 파일 분리 리팩터링
BattleManager(403줄), BattleSelectUI(399줄), GpsMapPlayerMover(341줄), Monster(331줄) 모두 300줄을 초과하여 가독성과 유지보수성이 떨어짐. 특히 BattleManager는 전투 로직, 보상 계산, UI 업데이트, 씬 관리가 한 클래스에 혼재.
  1. BattleManager에서 보상 로직을 BattleRewardCalculator로, 스케일링을 MonsterStatScaler로 분리
  2. GpsMapPlayerMover에서 디버그 필드/메서드를 GpsDebugOverlay로 분리
  3. Monster에서 스프라이트 로딩을 MonsterSpriteLoader 유틸리티로 분리

코드 리뷰

BattleManager.cs개선필요
403줄로 300줄 권장치를 크게 초과. HP/XP 스케일링(GetScaledMaxHp, GetScaledXpReward)과 무기 드롭(RollWeaponDrop) 로직은 잘 구현됨. 그러나 GetRewardItem()이 null만 반환하는 스텁으로 남아있고, Awake()에서 Instance = this를 중복 체크 없이 덮어씀. TryHitMonster()에서 InventoryController.Instance.EquippedWeapon의 attackPower 접근 시 EquippedWeapon이 null일 경우 NullReferenceException 위험. hpScalePerPlayerLevel(0.12f)이 SerializeField인데 GetMaxHpByGrade()에서 동일 값을 하드코딩하여 불일치 가능.
MonsterSpawner.cs개선필요
레벨 기반 등급 확률 시스템(UpdateProbabilityValues)과 거리 기반 디스폰(DespawnDistantMonsters)은 핵심 게임플레이에 잘 기여함. EnterBattle()에서 Additive 씬 로딩과 HideInGameSceneRootsExcept()로 씬 전환을 처리한 구조도 합리적. 그러나 static 필드(FieldPaused, FieldCameraRef, ArCameraRef, _hiddenInGameSceneRoots)가 4개나 되어 전역 상태 오염이 심하고, 주석 처리된 데드코드가 남아있음.
Monster.cs개선필요
331줄로 초과. Init()→ApplyVisualByGradeType() 초기화 체인, Wander()/Flee() AI, LoadMonsterSprite() 리소스 로딩 등 몬스터 동작이 온전히 구현됨. GetSpriteScaleForGradeAndType()에서 등급×타입별 스케일을 TypeSpriteScales 구조체로 관리한 점은 좋으나, Epic Slime에 대한 하드코딩된 오프셋 특수처리가 존재. colorNormal/colorRare/colorEpic이 BattleUI.cs에도 동일하게 정의되어 있어 색상 상수가 중복됨.
GpsMapPlayerMover.cs개선필요
341줄로 초과. ConvertGpsToWorld()의 Haversine 좌표 변환과 GPS 필터링(최소 이동 임계, 정확도 체크)은 위치기반 게임의 핵심으로 잘 구현됨. UpdateEditorMock()으로 에디터 테스트 지원도 적절. 그러나 디버그용 필드가 10개 이상(_gpsStatus, _lastFilterReason, _gpsUpdateCount 등)으로 클래스가 비대하고, _isInitialized와 _isReady 두 개의 초기화 플래그로 상태 관리가 복잡. 네임스페이스가 ARRPG.LimChaeSung으로 개인 이름 사용.
BattleSelectUI.cs개선필요
399줄로 초과. OnStartBattle()에서 BattleSelectionContext 설정 후 MonsterSpawner.EnterBattle() 호출하는 흐름이 명확. GetKoreanFont()에서 OS별 한글 폰트 로딩(Malgun Gothic/AppleGothic)과 TMP/Legacy Text 양쪽 지원은 실용적. 슬라이딩 애니메이션 AnimateListSlide() 코루틴도 잘 구현됨. 그러나 UI 구성과 게임 로직이 한 파일에 혼재되어 있고, 리스트 갱신을 0.3초 간격 폴링으로 처리하는 것은 이벤트 기반보다 비효율적.
LevelManager.cs심각
AddExp()/LevelUp() 경험치 루프는 동작하나, Awake()에서 Instance == null만 체크하고 중복 인스턴스 Destroy를 하지 않아 Additive 씬 로딩 시 다중 인스턴스 생성 버그 위험. maxExp 증가율 1.2배가 하드코딩. ExpUIController.Instance.UpdateExpUi()를 직접 호출하여 UI와 강하게 결합됨.
DeviceCameraBackground.cs개선필요
WebCamTexture 기반 AR 카메라 배경 구현으로, SetupUI()에서 Canvas/RawImage/AspectRatioFitter를 코드로 생성하고, ApplyVideoTransform()에서 카메라 회전 보정을 처리. 기능적으로는 동작하나 주석의 한글이 전부 인코딩 깨짐(mojibake)으로 가독성 심각하게 저하. UNITY_EDITOR 분기로 에디터 실행 방지는 적절.
InventoryController.cs개선필요
AddItem()에서 빈 슬롯 탐색 후 아이템 할당, ToggleEquip()으로 장착/해제 토글, LINQ OrderBy로 슬롯 정렬 등 인벤토리 기본 기능 구현됨. 그러나 인벤토리 가득 찼을 때 아이템 추가 시 로그만 출력하고 사용자에게 피드백 없음. GetSlotNumber()가 이름 문자열의 숫자 파싱에 의존하여 네이밍 규칙 변경 시 취약.

아키텍처 평가

프로젝트는 Singleton 패턴을 핵심 아키텍처로 채택하여 BattleManager, LevelManager, InventoryController, VictoryController, DefeatController 등 5개 이상의 매니저를 운영함. 씬 간 데이터 전달은 BattleSelectionContext라는 정적 클래스를 사용하는데, Unity의 Additive 씬 로딩 환경에서는 합리적인 선택임. 그러나 MonsterSpawner의 static 필드 4개(FieldPaused, FieldCameraRef 등)로 전역 상태를 관리하는 것은 씬 전환 순서에 민감한 버그를 유발할 수 있음. 매니저 간 결합도가 높아 BattleManager→LevelManager→ExpUIController→InventoryController로 직접 참조 체인이 형성되어 있으며, 이벤트 시스템이나 중재자 패턴 없이 직접 메서드 호출에 의존함. GPS→MapGridGenerator→GpsMapPlayerMover→MonsterSpawner 체인은 초기화 순서 의존성이 강해 코루틴 타이밍에 민감함.

심각도파일설명
CRITICALBattleManager.csTryHitMonster()에서 InventoryController.Instance?.EquippedWeapon의 attackPower에 접근할 때 EquippedWeapon이 null이면 NullReferenceException 발생. 무기 미장착 상태에서 전투 시 크래시 위험.
CRITICALLevelManager.csAwake()에서 Instance != null일 때 Destroy(gameObject)를 하지 않음. BattleScene이 Additive 로딩될 때 LevelManager가 중복 생성되면 이전 인스턴스의 레벨/경험치 데이터가 덮어씌워질 수 있음.
HIGHBattleManager.csGetMaxHpByGrade()에서 scalePerLevel을 0.12f로 하드코딩했으나, 클래스의 SerializeField hpScalePerPlayerLevel도 0.12f. Inspector에서 값을 변경해도 static 메서드에는 반영되지 않아 밸런스 불일치 발생.
HIGHSpawnProbUI.csUpdate()에서 매 프레임 FindObjectOfType<MonsterSpawner>() 호출. FindObjectOfType은 씬 전체를 탐색하는 O(n) 연산으로, 모바일 AR 환경에서 프레임 드롭 유발 가능.
HIGHMonsterSpawner.csFieldPaused, FieldCameraRef, ArCameraRef, _hiddenInGameSceneRoots 등 4개 static 필드로 전역 상태 관리. 씬 전환 순서가 꼬이면 static 상태가 오염되어 필드 복귀 시 오브젝트가 영구 숨겨질 수 있음.
MEDIUMBattleManager.csGetRewardItem() 메서드가 항상 null을 반환하는 스텁 상태. 전투 보상 아이템 시스템이 미완성이나, 무기 드롭은 별도 RollWeaponDrop()으로 동작하므로 혼란 유발.
MEDIUMDeviceCameraBackground.csSetupUI(), OnEnable(), Start() 등 주요 메서드의 한글 주석이 전부 인코딩 깨짐(예: 'Canvas ���� �� ����'). 파일 인코딩이 UTF-8이 아닌 것으로 추정되며 협업 시 코드 이해 불가.
MEDIUMVictoryController.csToggleVictory() 메서드가 항상 UI를 비활성화하는 로직으로 구현됨. 토글이라면 현재 상태의 반대로 전환해야 하나 실질적으로 Hide만 수행.
MEDIUMMonster.cscolorNormal(빨강), colorRare(보라), colorEpic(금색) 색상 상수가 BattleUI.cs에도 동일하게 정의됨. 한쪽만 수정하면 UI와 몬스터 색상이 불일치.
LOWGpsMapPlayerMover.cs네임스페이스가 ARRPG.LimChaeSung으로 팀원 개인 이름 사용. 코드 소유권과 무관하게 기능 기반 네임스페이스(예: ARRPG.Map 또는 ARRPG.GPS)로 변경 필요.
