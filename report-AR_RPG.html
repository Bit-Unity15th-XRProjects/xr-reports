프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 만들고 있으며, GPS 기반 필드 이동(GpsMapPlayerMover), 몬스터 스폰/선택(MonsterSpawner, BattleSelectUI), 턴제 전투(BattleManager), 레벨 성장(LevelManager), 인벤토리(InventoryController)까지 핵심 게임 루프의 골격이 연결되어 있다. 특히 BattleManager의 등급별 HP/XP 스케일링과 MonsterSpawner의 레벨 기반 스폰 확률 시스템은 게임 디자인이 코드로 구현된 수준이 높다. 그러나 SaveManager와 SoundManager가 완전히 부재하여 게임 진행 저장과 오디오가 전혀 없고, GetRewardItem() 스텁과 VictoryController의 모순 로직으로 승리 보상 흐름이 끊겨 있다. 가장 시급한 것은 SaveManager 구현으로, 이것 없이는 AR RPG의 핵심인 '탐험하며 성장하는 경험'이 앱 종료와 함께 사라지며, 이어서 origin/LimChaesung 브랜치의 즉각 머지와 오디오 시스템 추가가 시연 완성도를 위해 필요하다.

  • 팀 규칙으로 커밋당 변경 500줄 이하, PR당 변경 1000줄 이하 제한 설정

Singleton 남용으로 인한 높은 결합도
BattleManager, LevelManager, InventoryController, VictoryController, DefeatController, ExpUIController 등 6개 이상이 모두 Singleton. Instance 직접 참조로 클래스 간 강결합이 형성되어 한 시스템 변경이 다른 시스템에 연쇄적 영향을 미침
  • 단기: 각 Singleton 접근 전 null 체크를 반드시 추가하여 NullReferenceException 방지
  • 중기: 이벤트 시스템(C# event 또는 UnityEvent) 도입하여 BattleManager→LevelManager 같은 직접 호출을 이벤트 구독으로 전환
  • 장기: ServiceLocator 패턴이나 ScriptableObject 기반 이벤트 채널로 의존성 역전 적용

권고사항 (Top 3)

1. SaveManager 구현 및 데이터 영속화 (최우선)
현재 레벨, 경험치, 인벤토리가 앱 종료 시 모두 초기화됨. AR 위치기반 RPG의 핵심 가치인 '성장'이 유지되지 않으면 게임으로서 기능하지 못함
  1. SaveData 클래스 정의: playerLevel, currentExp, maxExp, inventoryItems[], equippedSlotIndex
  2. SaveManager.cs 생성: Save()는 JsonUtility.ToJson → File.WriteAllText(persistentDataPath + '/save.json'), Load()는 역순
  3. LevelManager.Awake()에서 SaveManager.Load() 호출하여 레벨/경험치 복원
  4. InventoryController.Awake()에서 저장된 아이템 복원
  5. OnApplicationPause(true)와 전투 종료 시 자동 Save() 호출
2. BattleManager/Monster 대형 파일 리팩토링
BattleManager(403줄), Monster(331줄)이 파일 크기 제한을 초과하고 ApplyVisualByGradeType() 같은 90줄 메서드가 존재. 3명이 동시 작업하는 팀에서 대형 파일은 머지 충돌의 주요 원인
  1. BattleManager에서 HP/XP 스케일링 로직을 MonsterStats 유틸리티 클래스로 분리
  2. BattleManager에서 아이템 드롭/생성 로직을 ItemDropService로 분리
  3. Monster.ApplyVisualByGradeType()을 MonsterVisualHandler 컴포넌트로 분리하여 50줄 이하로 유지
  4. 각 분리 시 기존 테스트가 없으므로 분리 전후 InGame 씬에서 수동 동작 확인 필수
3. 스테일 브랜치 정리 및 브랜치 전략 수립
origin/Feat.Monster(28커밋 behind), origin/Feat_Inventory(23 behind), origin/feat/ingame(28 behind) 등 3개 브랜치가 main과 크게 벌어져 사실상 사용 불가 상태. 브랜치 난립은 팀의 작업 현황 파악을 어렵게 함
  1. origin/Feat.Monster, origin/Feat_Inventory, origin/feat/ingame의 변경사항 중 main에 미반영된 것이 있는지 확인
  2. 필요한 변경이 없으면 해당 브랜치 삭제
  3. 앞으로 브랜치 명명 규칙 통일(feat/기능명) 및 1주 이상 미활동 브랜치 자동 경고 설정
  4. PR 기반 머지 워크플로우를 팀 규칙으로 확립

코드 리뷰

BattleManager.cs개선필요
전투의 핵심 로직(HP/XP 등급별 스케일링, 무기 드롭 확률, 제한시간 패배)이 실제로 동작하는 코드로 잘 구현됨. GetScaledMaxHp()의 공식(1+(lv-1)×scalePerLevel)과 switch expression 활용이 깔끔함. 그러나 403줄로 300줄 제한 초과, GetRewardItem()이 null 반환 스텁으로 보상 시스템이 미완성, CreateRandomItem()에서 ScriptableObject.CreateInstance<Item>()으로 런타임 생성하는 것은 메모리 누수 위험(Destroy 호출 없음). hpScalePerPlayerLevel 필드(0.12f)와 GetMaxHpByGrade의 하드코딩된 0.12f가 중복되어 불일치 가능성 존재.
Monster.cs개선필요
3등급×3타입 시스템, TypeSpriteScales 구조체로 등급/타입별 스프라이트 크기 관리가 체계적. LoadMonsterSprite()의 Resources 경로 규칙('Monsters/{Type}/{Grade}-{Type}')이 명확함. 그러나 ApplyVisualByGradeType()이 90줄 이상의 거대 메서드로 스프라이트 로드/적용/메쉬 색상/배틀 분기를 모두 처리하여 단일 책임 원칙 위반. 도주 로직에서 fleeDirection 정규화 시 영벡터 체크가 없어 NaN 발생 가능. 331줄로 파일 크기 제한 초과.
MonsterSpawner.cs개선필요
레벨 기반 스폰 확률 시스템(UpdateProbabilityValues)이 3단계(lv<10/lv<30/lv≥30)로 잘 설계됨. EnterBattle()에서 InGame 씬 루트 오브젝트 숨기기/복원(HideInGameSceneRootsExcept/RestoreInGameSceneRoots)으로 씬 전환을 처리하는 방식이 실용적. 그러나 128-165줄에 주석 처리된 미사용 코드 블록이 있어 정리 필요. LevelManager.Instance 접근 시 null 체크 없음(90줄) — LevelManager가 없는 씬에서 NullReferenceException 발생.
GpsMapPlayerMover.cs개선필요
Haversine 공식 기반 GPS→월드 좌표 변환(ConvertGpsToWorld, GetMeterOffset)이 수학적으로 정확하게 구현됨. GPS 노이즈 필터링(_minMoveThresholdMeters=0.3m), 정확도 체크(_maxAcceptedHorizontalAccuracy=50m), 에디터 목업 키보드 이동이 실용적. 그러나 _isInitialized 플래그가 InitializeWithMap()에서만 설정되어 Start() 코루틴과 경쟁 조건 가능성 존재. 위치 동기화 메서드가 3개(ForceUpdatePosition, InitializeWithMap, SyncWithMap)로 API가 혼란스럽고, 341줄로 파일 크기 제한 초과.
BattleSelectUI.cs개선필요
몬스터 리스트 UI, 토글 슬라이드 애니메이션, 등급별 텍스트 색상 적용이 완성도 높게 구현됨. GetKoreanFont()에서 플랫폼별 한글 폰트 대응(Windows: Malgun Gothic, Mac: Apple SD Gothic Neo, Android: NotoSansCJK)이 세심함. 그러나 399줄로 제한 초과, ApplyListLayoutSettings()가 빈 메서드(스텁), 레이캐스트 카메라 기본값으로 Camera.main을 매번 호출하는 것은 비효율적(캐싱 필요).
InventoryController.cs개선필요
AddItem/SelectSlot/ToggleEquip의 인벤토리 핵심 플로우가 구현되어 아이템 추가→선택→장착이 동작함. EquippedWeapon 프로퍼티로 BattleManager에서 장착 무기 공격력을 참조하는 연동이 잘 됨. 그러나 한글 주석의 인코딩이 깨져 있고, 인벤토리 가득 찰 때 AddItem()이 조용히 실패(사용자 피드백 없음), OnConfirmUseOrEquip()에서 selectedSlot null 검증 부재.
DeviceCameraBackground.cs개선필요
WebCamTexture를 이용한 디바이스 카메라 배경 표시, Canvas/RawImage 자동 생성, 회전/미러링 보정(ApplyVideoTransform)이 구현됨. 에디터에서 카메라를 시작하지 않는 #if UNITY_EDITOR 분기가 적절함. 그러나 주석의 한글이 완전히 깨져 읽을 수 없는 상태(SetupUI 라인 67-68 등). Ensure() 정적 메서드에서 FindFirstObjectByType 사용은 Unity 2023+ 전용이라 하위 호환성 문제 가능.
LevelManager.cs심각
42줄의 매우 단순한 구현으로 AddExp()와 LevelUp()만 존재. maxExp 성장률이 1.2배 고정이라 레벨 50에서 약 900만 XP 필요 — 밸런스 미검증. 가장 심각한 문제는 저장/로드 기능이 전혀 없어서 앱을 종료하면 레벨/경험치가 모두 초기화됨. AR RPG에서 진행 상황 유지는 필수 기능이므로 PlayerPrefs나 JSON 직렬화를 즉시 구현해야 함.

아키텍처 평가

Singleton 패턴이 프로젝트의 지배적 아키텍처로, BattleManager.Instance, LevelManager.Instance, InventoryController.Instance 등 6개 이상의 매니저가 이 패턴을 사용. 씬 간 데이터 전달에는 BattleSelectionContext 정적 클래스를 사용하는데, 이는 간단하지만 데이터 유효성 검증이 없어 잘못된 상태가 전파될 수 있음. 네임스페이스(ARRPG.Inventory, ARRPG.Level 등)와 Assembly Definition(9개)으로 모듈 분리를 시도한 점은 학생 프로젝트로서 우수하나, 실제 클래스 간 참조는 Instance 직접 접근으로 모듈 경계를 무시함. MonsterSpawner가 전투 씬 전환 시 InGame 씬의 루트 오브젝트를 직접 숨기는 방식(HideInGameSceneRootsExcept)은 Additive Scene Loading의 실용적 활용이나, 정적 리스트에 의존하여 멀티씬 구조가 복잡해질수록 관리가 어려워질 것임. 전체적으로 MVC나 MVP 같은 상위 아키텍처 패턴 없이 MonoBehaviour에 로직이 집중된 전형적인 Unity 초보 구조.

심각도파일설명
CRITICALBattleManager.csGetRewardItem() 메서드(285줄)가 항상 null을 반환하는 스텁 상태. 전투 승리 후 보상 아이템이 실제로 지급되지 않아 핵심 게임 루프가 끊김
CRITICALLevelManager.cs레벨/경험치 데이터의 저장/로드 기능이 전혀 없음. 앱 재시작 시 currentLevel=1, currentExp=0으로 초기화되어 플레이어 진행 상황이 모두 소실됨
HIGHBattleManager.csCreateRandomItem()에서 ScriptableObject.CreateInstance<Item>()으로 런타임 아이템 생성 후 Destroy()를 호출하지 않아 메모리 누수 발생 가능. 인벤토리에서 아이템 제거 시에도 ScriptableObject 인스턴스가 해제되지 않음
HIGHVictoryController.csToggleVictory() 메서드가 VictoryUI.SetActive(false)만 수행하여 '토글'이 아닌 '항상 비활성화'로 동작. Start()에서도 비활성화하므로 승리 UI가 표시될 경로가 BattleManager.VictoryData()에서의 직접 SetActive(true) 호출뿐
HIGHMonsterSpawner.csUpdateProbabilityValues()에서 LevelManager.Instance를 null 체크 없이 직접 접근(90줄). LevelManager가 초기화되지 않은 씬에서 NullReferenceException 발생
MEDIUMMonster.csApplyVisualByGradeType() 메서드가 90줄 이상으로 스프라이트 로드, 적용, 메쉬 색상 변경, 배틀/필드 분기를 모두 처리. 단일 책임 원칙 위반이며 디버깅/유지보수가 어려움
MEDIUMSpawnProbUI.csUpdate()에서 매 프레임 FindObjectOfType<MonsterSpawner>()를 호출하여 O(n) 탐색 반복. Start()에서 캐싱해야 함
MEDIUMDeviceCameraBackground.csSetupUI() 등 다수 메서드의 한글 주석이 인코딩 깨짐(UTF-8 BOM 미적용 또는 EUC-KR 혼용 추정). 코드 가독성 심각하게 저하
MEDIUMItem.csgrade 필드가 int 타입으로 선언되어 MonsterGrade enum과 타입이 불일치. 잘못된 정수값 할당 시 런타임 에러 없이 의도하지 않은 동작 발생 가능
LOWInventorySlotUI.cs메서드명 오타: 'DestoryItemData()' → 올바른 이름은 'DestroyItemData()'. 팀 내 코드 검색 시 혼란 유발
