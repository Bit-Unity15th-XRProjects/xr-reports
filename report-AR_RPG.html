프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 3인(kimin, KSJ, Lachesis)으로 개발 중이며, 현재 필드 맵(OpenStreetMap 타일) 위에서 몬스터 스폰→선택→배틀 씬 전환→클릭 전투→승리/패배→필드 복귀의 핵심 게임 루프가 동작하는 상태까지 구현했다. MonsterSpawner의 레벨별 등급 확률, BattleManager의 HP/XP 스케일링, InventoryController의 무기 장착 시스템 등 RPG 핵심 메카닉이 코드 수준에서 연동되어 있다. 그러나 SaveManager가 완전히 부재하여 앱 종료 시 모든 진행이 사라지고, GpsMapPlayerMover가 실제 GPS 대신 터치 드래그를 사용하여 '위치기반'이라는 프로젝트 핵심 컨셉이 미구현 상태이다. SoundManager도 없어 전투와 UI에 오디오 피드백이 전혀 없다. 가장 시급한 것은 SaveManager 구현과 GPS 이동 로직 복구이며, BattleManager의 403줄 단일 파일 리팩토링과 싱글톤 null 안전성 확보가 그 다음 우선순위이다.

리스크 알림

SaveManager 완전 부재 — 게임 진행 데이터 유실
LevelManager.currentLevel, InventoryController의 아이템, 경험치 등 모든 게임 상태가 메모리에만 존재. 앱 종료 시 모든 진행이 초기화됨. GuestAuthManager가 PlayerPrefs로 게스트 ID만 저장하고, 실제 게임 데이터는 어디에도 영속화되지 않음. AR RPG에서 레벨과 아이템이 저장되지 않으면 플레이 동기가 없어짐.
  • PlayerPrefs 또는 JSON 파일 기반 SaveManager 클래스 생성
  • LevelManager(레벨, 경험치)와 InventoryController(아이템 목록) 데이터를 직렬화하는 SaveData 구조체 정의
  • 씬 전환 시, 앱 일시정지(OnApplicationPause) 시 자동 저장 호출
  • Start()에서 저장 데이터 로드하여 상태 복원
GPS 이동이 실제로는 터치 드래그로 대체됨
GpsMapPlayerMover.Update() line 131에서 모바일 빌드 시 HandleTouchDrag()만 호출하고 GPS 데이터를 사용하지 않음. Start()에서 GPS를 초기화하지만 Update에서 읽지 않아 '위치기반' RPG의 핵심 기능이 미작동. MapGridGenerator는 GPS로 초기 위치만 잡고, 이후 이동은 터치 드래그에 의존.
  • UpdateGps() 메서드를 구현하여 Input.location.lastData로 실시간 GPS 좌표 읽기
  • ConvertGpsToWorld()로 변환한 좌표를 _targetPosition에 반영
  • HandleTouchDrag()는 GPS 보조 수단으로만 유지하거나 디버그 모드로 전환
LevelManager/ExpUIController 싱글톤 null 참조로 인한 런타임 크래시
BattleManager.ReturnToField() line 366, MonsterSpawner.UpdateProbabilityValues() line 90, LevelManager.AddExp() line 30에서 각각 다른 싱글톤을 null 체크 없이 호출. Additive 씬 로딩 구조에서 싱글톤 초기화 순서가 보장되지 않아 실행 순서에 따라 NullReferenceException 발생 가능.
  • 모든 싱글톤 접근 전에 null 체크 추가 (특히 BattleManager.ReturnToField line 366)
  • LevelManager와 ExpUIController를 DontDestroyOnLoad로 전환하거나, 씬 로드 시 존재 보장
  • 또는 ServiceLocator 패턴으로 싱글톤 접근을 중앙화
SoundManager 부재 — 오디오 피드백 전무
오디오 파일 0개, SoundManager 클래스 없음. 전투 타격, 몬스터 처치, 레벨업, UI 클릭 등 모든 상호작용에 소리가 없어 게임 경험이 크게 저하됨. AR 환경에서 시각적 피드백이 제한적이므로 청각 피드백이 더욱 중요.
  • AudioClip 참조와 PlayOneShot 기반의 간단한 SoundManager 싱글톤 생성
  • 전투 타격, 몬스터 처치, 레벨업, 아이템 획득 4가지 핵심 사운드 추가
  • 무료 에셋(freesound.org 등)에서 효과음 확보
대형 커밋과 브랜치 미머지로 인한 병합 충돌 위험

권고사항 (Top 3)

1. SaveManager 즉시 구현
게임 진행 저장이 없으면 테스트도, 시연도, 실사용도 불가능. 현재 LevelManager와 InventoryController의 데이터 구조가 단순하여 빠르게 구현 가능.
  1. [Serializable] SaveData 클래스 생성: int level, float exp, float maxExp, List<ItemData> items
  2. SaveManager에 Save()/Load() 메서드 구현 (JsonUtility + Application.persistentDataPath)
  3. LevelManager.Awake()에서 Load(), BattleManager.ReturnToField()에서 Save() 호출
  4. OnApplicationPause(true)에서 자동 저장
2. BattleManager 리팩토링 — 책임 분리
403줄의 단일 클래스가 전투 로직, 아이템 생성, UI 참조, 씬 전환을 모두 담당하여 유지보수와 버그 추적이 어려움. 팀원 3명이 동시 작업 시 머지 충돌의 주요 원인이 됨.
  1. 아이템 생성 로직(CreateRandomItem, RollWeaponDrop, GetWeaponAttackByGrade)을 ItemFactory 클래스로 분리
  2. HP/XP 스케일링 static 메서드들을 MonsterStatCalculator 유틸리티로 이동
  3. 타이머 UI 갱신 로직을 BattleUI로 통합 (현재 BattleManager.Update와 BattleUI.Update에서 이중 관리)
3. 실제 GPS 이동 로직 복구 및 테스트
AR 위치기반 RPG의 차별점인 GPS 이동이 터치 드래그로 대체되어 있어 프로젝트 컨셉과 불일치. 발표/시연 시 '위치기반'을 증명할 수 없음.
  1. GpsMapPlayerMover에 UpdateGps() 메서드 구현: Input.location.lastData에서 위경도 읽기 → ConvertGpsToWorld로 변환 → 노이즈 필터링 → _targetPosition 갱신
  2. Update()의 모바일 분기에서 HandleTouchDrag() 대신 UpdateGps() 호출
  3. 캠퍼스/건물 주변에서 실제 디바이스 테스트 수행

GPS→월드좌표 변환(ConvertGpsToWorld, GetMeterOffset)과 에디터 WASD 목업이 잘 구현됨. 그러나 실제 모바일에서는 GPS 대신 HandleTouchDrag()(line 131)로 터치 드래그 이동을 사용하여 '위치기반' RPG의 핵심인 실제 GPS 이동이 비활성화됨. OpenStreetMap 타일 좌표 계산(GetTileCoord, GetSubTilePos)이 정확한 Mercator 투영 공식 사용. 3×3 그리드 생성 후 GPS 좌표로 플레이어 정확한 위치 배치, _playerMover.SyncWithMap() 연동이 잘 됨. GPS 실패 시 fallback 좌표 제공. 단 fallback 좌표(line 54)가 lat=7.494920으로 의도된 37.494920과 다른 것으로 보이는 오타 의심.

399줄로 BattleManager와 함께 가장 큰 파일. 몬스터 리스트 UI, 토글 슬라이드 애니메이션, 레거시 Text/TMP 듀얼 지원, 등급별 색상 등 기능이 많음. RefreshList()에서 매 0.3초마다 기존 리스트 아이템을 모두 Destroy하고 Instantiate로 재생성하여 GC 부담. GetKoreanFont()에서 OS별 폰트 하드코딩(Malgun Gothic/AppleGothic)은 실용적이나 Android에서의 처리가 불명확.

아키텍처 평가

전체적으로 Singleton 패턴에 강하게 의존(BattleManager, LevelManager, InventoryController, VictoryController, DefeatController, ExpUIController 총 6개)하며, 씬 간 데이터 전달은 BattleSelectionContext static 클래스와 MonsterSpawner의 static 필드(FieldPaused, FieldCameraRef, ArCameraRef)로 처리함. Additive 씬 로딩(InGame + BattleScene)과 루트 오브젝트 숨김/복원(HideInGameSceneRootsExcept/RestoreInGameSceneRoots)으로 씬 전환을 구현한 것은 독창적이나, 복원 실패 시 UI가 영구히 숨겨지는 위험이 있음. 네임스페이스(ARRPG.Inventory, ARRPG.Level, ARRPG.Victory, ARRPG.Map, ARRPG.LimChaeSung, GamePlay.Data)와 asmdef 분리는 좋으나, BattleManager가 Inventory/Level/Victory를 직접 참조하고 MonsterSpawner가 Level을 참조하여 의존 방향이 복잡함. ScriptableObject(Item)을 데이터 컨테이너로 사용하되 런타임에 CreateInstance로 생성하는 혼합 패턴은 메모리 관리에 주의 필요.
