프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 만들고 있으며, 현재 GPS 기반 맵 이동(GpsMapPlayerMover + MapGridGenerator), 필드 몬스터 스폰/배회(MonsterSpawner + Monster), 전투 시스템(BattleManager + BattleSelectUI), 인벤토리(InventoryController), 레벨 시스템(LevelManager)까지 핵심 게임 루프가 동작하는 상태입니다. 전투 흐름은 필드에서 몬스터 선택 → Additive 씬 로딩 → 클릭 공격 → 제한시간 내 처치 → 무기 드롭/XP 획득 → 맵 복귀로 완성되어 있고, 레벨에 따른 몬스터 등급 확률 조정과 HP 스케일링도 구현되어 있습니다. 그러나 SaveManager가 전혀 없어 앱 종료 시 모든 진행도가 소실되고, SoundManager와 오디오 에셋이 0개이며, ARFoundation 없이 WebCamTexture만 사용하여 실질적 AR 기능이 카메라 배경 표시에 그칩니다. 가장 시급한 것은 SaveManager 구현과 LimChaesung 브랜치의 VFX/ShaderGraph 작업 병합이며, 이 두 가지만 해결해도 완성도가 체감적으로 크게 올라갈 것입니다.

HP 스케일링(GetScaledMaxHp), XP 보상 계산(GetScaledXpReward), 무기 드롭 확률(RollWeaponDrop) 등 핵심 전투 로직이 실제 동작하는 수준으로 구현됨. 그러나 GetRewardItem()이 null을 반환하는 죽은 코드이고, CreateRandomItem()에서 ScriptableObject.CreateInstance<Item>()으로 런타임 에셋을 생성하는 것은 메모리 누수 위험. Awake()에서 Instance = this만 하고 중복 인스턴스 체크 없음. 전투 결과 처리(ReturnToField), 타이머 시스템, UI 업데이트, 무기 생성, 씬 전환을 한 클래스가 모두 담당하여 단일 책임 원칙 위반.
MonsterSpawner.cs양호
285줄로 적절한 크기. UpdateProbabilityValues()의 레벨 구간별 확률 시스템이 깔끔하게 구현됨. EnterBattle()에서 Additive 씬 로딩, 카메라 전환, 필드 오브젝트 숨기기를 순차적으로 처리하는 흐름이 명확함. HideInGameSceneRootsExcept()로 배틀 중 필드 오브젝트를 비활성화하는 접근은 실용적이나 static List<GameObject>에 의존하여 멀티씬 상황에서 문제 가능. GetMonstersInSelectRange()가 매 호출마다 new List를 생성하는 것은 GC 부담.
Monster.cs개선필요
331줄로 300줄 초과. MonsterGrade/MonsterType enum 정의, 배회 AI(Wander/PickNewWanderTarget), 도주 행동, 스프라이트 동적 로드(LoadMonsterSprite)가 모두 한 파일에 있어 과도하게 비대함. LoadMonsterSprite()에서 Resources.Load 실패 시 Texture2D 폴백 로직이 있는 것은 좋으나, 스프라이트 캐싱이 없어 동일 몬스터 타입을 반복 로드할 때 비효율적. TypeSpriteScales 구조체를 Inspector에서 등급×타입별로 설정하는 방식은 유지보수에 불리함.
BattleSelectUI.cs개선필요
399줄로 300줄 제한 크게 초과. 몬스터 리스트 UI, 슬라이드 애니메이션, 레이캐스트 선택, 아웃라인 관리, 한국어 폰트 처리를 한 클래스가 모두 담당. 슬라이드 애니메이션(AnimateListSlide)은 코루틴으로 잘 구현됨. GetKoreanFont()에서 플랫폼별 OS 폰트를 정적 캐싱하는 것은 실용적이나, Legacy Text와 TMP를 동시 지원하는 코드가 복잡도를 높임. RefreshList()에서 매번 Instantiate/Destroy하는 방식은 Object Pooling으로 개선 필요.
GpsMapPlayerMover.cs양호
341줄이지만 GPS 처리의 복잡성을 감안하면 납득 가능한 수준. ConvertGpsToWorld()와 GetMeterOffset()의 Web Mercator 좌표 변환이 수학적으로 정확하고, _minMoveThresholdMeters(0.3m)로 GPS 노이즈 필터링, _maxAcceptedHorizontalAccuracy(50m)로 저정확도 데이터 차단 등 실무적 GPS 처리가 잘 되어 있음. 에디터 모의 이동(UpdateEditorMock)과 실제 GPS 입력을 #if UNITY_EDITOR로 분리한 것도 적절. 다만 private 필드가 20개 이상으로 상태 관리가 복잡하고, 디버그용 필드(_gpsUpdateCount 등)가 릴리즈에도 남아있음.
InventoryController.cs개선필요
154줄로 크기는 적절하나, Singleton 초기화에 중복 보호 없음. AddItem()에서 빈 슬롯을 순회하며 아이템 추가하는 로직은 동작하지만, 인벤토리 데이터가 메모리에만 존재하여 앱 종료 시 모든 아이템이 소실됨. EquippedWeapon 프로퍼티로 장착 무기의 attackPower를 BattleManager에 전달하는 연동은 구현됨. RefreshSlots()가 매 변경마다 전체 슬롯을 갱신하는 것은 소규모에서는 문제없으나 확장성 부족.
LevelManager.cs개선필요
42줄의 최소 구현. AddExp()와 LevelUp() 두 메서드만으로 동작하며, 레벨업 시 maxExp를 1.2배로 증가시키는 로직은 기능적으로 문제없음. 그러나 레벨/경험치 데이터가 영속화되지 않아 앱 재시작 시 레벨 1로 초기화됨. 레벨업 이벤트 시스템(event/delegate)이 없어, 레벨 변경 시 MonsterSpawner, BattleManager 등이 직접 LevelManager.Instance.currentLevel을 폴링해야 함.
DeviceCameraBackground.cs개선필요
AR 카메라 배경 표시를 WebCamTexture로 구현. SetupUI()에서 Canvas/RawImage/AspectRatioFitter를 코드로 동적 생성하는 방식은 씬 의존성을 줄이지만, 주석이 UTF-8 인코딩 깨짐으로 가독성이 심각하게 저하됨. CoStartCamera() 코루틴에서 카메라 초기화, 권한 요청, 폴백 처리를 잘 다루고 있으나, ARFoundation 대신 WebCamTexture를 사용하여 AR 추적(Plane Detection, Image Tracking 등)이 불가능함.

아키텍처 평가

Singleton 패턴을 BattleManager, LevelManager, InventoryController, ExpUIController, VictoryController, DefeatController 총 6곳에서 사용하지만, 모두 보호 없는 Instance = this 할당으로 Additive 씬 로딩 환경에서 중복 인스턴스 위험이 있습니다. 씬 간 데이터 전달은 BattleSelectionContext static 클래스와 MonsterSpawner의 static 필드(FieldPaused, FieldCameraRef)로 처리하는데, 이는 씬 재로드 시 stale 참조 문제를 야기합니다. 네임스페이스는 ARRPG.Inventory, ARRPG.Level, ARRPG.Victory, ARRPG.LimChaeSung으로 부분 적용되어 있으나, 핵심 전투 스크립트 15개(BattleManager, Monster, MonsterSpawner 등)가 글로벌 네임스페이스에 있어 일관성이 없습니다. Additive 씬 로딩으로 필드↔배틀 전환을 구현한 것은 AR 게임에 적합한 좋은 설계이며, HideInGameSceneRootsExcept()로 필드 오브젝트를 숨기는 접근은 실용적이나, SceneManager.MoveGameObjectToScene() 없이 static 리스트로 관리하여 복잡도가 높습니다.

심각도파일설명
CRITICAL전체 프로젝트SaveManager가 완전히 부재하여 인벤토리 아이템, 플레이어 레벨/경험치, 게임 진행도가 앱 종료 시 모두 소실됨. LevelManager.currentLevel과 InventoryController의 슬롯 데이터가 메모리에만 존재.
HIGHBattleManager.csGetRewardItem() 메서드가 항상 null을 반환하는 죽은 코드. CreateRandomItem()이 실제 아이템 생성을 담당하지만 GetRewardItem()이 호출되는 경로가 있다면 보상 없이 전투가 종료될 수 있음.
HIGHBattleManager.csCreateRandomItem()에서 ScriptableObject.CreateInstance<Item>()으로 런타임에 ScriptableObject를 생성하면 Unity 에디터 외부에서 메모리 관리가 불명확하고, 인벤토리에서 제거 시 Destroy()를 명시적으로 호출하지 않으면 메모리 누수 발생.
HIGH전체 프로젝트SoundManager/AudioManager가 없어 전투 타격음, 몬스터 등장음, 배경음악, UI 효과음이 전무함. audio 에셋도 0개로 게임 체감 품질에 큰 영향.
MEDIUMBattleManager.csAwake()에서 Instance = this만 할당하고 기존 인스턴스 체크(if(Instance != null && Instance != this) Destroy(gameObject)) 없음. Additive 씬 로딩 시 BattleManager가 중복 생성되면 Instance가 덮어씌워져 이전 참조가 유실됨.
MEDIUMMonsterSpawner.csGetMonstersInSelectRange()에서 매 호출마다 new List<Monster>()를 생성하여 GC Allocation 발생. BattleSelectUI.RefreshList()가 Update에서 주기적으로 호출하면 GC 스파이크 유발 가능.
MEDIUMMonster.csLoadMonsterSprite()에서 Resources.Load<Sprite>() 결과를 캐싱하지 않아, 동일 등급/타입의 몬스터가 여러 마리 스폰될 때마다 동일 스프라이트를 반복 로드. Dictionary<string, Sprite> 캐시 필요.
MEDIUMDeviceCameraBackground.cs한글 주석이 UTF-8 인코딩 깨짐으로 '// 1. Canvas ���� �� ����' 형태로 표시되어 코드 가독성 심각 저하. 파일 인코딩을 UTF-8 BOM으로 재저장 필요.
MEDIUMMonsterSpawner.csstatic 필드 FieldPaused, FieldCameraRef, ArCameraRef, _hiddenInGameSceneRoots가 씬 전환 후에도 이전 값을 유지하여, 비정상 종료나 씬 재로드 시 stale 참조로 NullReferenceException 발생 가능.
MEDIUMBattleSelectUI.csRefreshList()에서 리스트 아이템을 매번 Instantiate/Destroy하여 빈번한 GC 발생. 몬스터가 20마리까지 스폰 가능하므로, Object Pooling 패턴으로 재사용해야 모바일 성능 확보 가능.
