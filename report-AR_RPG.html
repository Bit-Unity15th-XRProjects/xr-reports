프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 만들고 있으며, BattleManager 중심의 전투 시스템(클릭 데미지, 레벨 스케일링 HP/XP, 등급별 무기 드롭), MonsterSpawner의 레벨 기반 등급 확률 스폰, 6슬롯 인벤토리/장착 시스템, 경험치/레벨업 시스템까지 핵심 게임 루프의 약 60%를 구현했다. 그러나 프로젝트명의 핵심인 'AR 위치기반' 기능이 사실상 미구현 상태로, GpsMapPlayerMover의 GPS 업데이트 루프가 터치 드래그로 대체되어 있고 MapGridGenerator에는 위도 오타 버그가 있다. SaveManager와 SoundManager가 완전히 부재하여 데이터 영속성과 오디오 피드백이 전무하며, 모든 Singleton이 DontDestroyOnLoad 없이 씬 전환 시 소멸 위험에 노출되어 있다. 가장 시급한 것은 GPS 최소 동작 구현과 MapGridGenerator 위도 버그 수정이며, 그 다음으로 PlayerPrefs 기반 간이 저장과 기본 사운드 추가가 발표 준비에 필수적이다. 팀 전원이 10일 이상 비활성 상태인 점도 마감 리스크로, 즉시 작업 재개와 LimChaesung 브랜치 병합이 필요하다.

권고사항 (Top 3)

1. 최소 동작 가능한 GPS 루프 구현
'AR 위치기반'이 프로젝트의 핵심 차별점인데 현재 GPS가 전혀 동작하지 않음. 발표에서 실제 위치 연동을 보여주지 못하면 프로젝트 정체성이 흔들림.
  1. GpsMapPlayerMover.Start()에서 Input.location.Start() 후 while 루프로 lastData 읽기 구현
  2. 읽은 좌표를 ConvertGpsToWorld()로 변환하여 _targetPosition 업데이트
  3. MapGridGenerator.cs 위도 오타 수정 (7→37)
  4. 에디터에서는 기존 터치 모드 유지하되, 디바이스 빌드 시 GPS 모드로 자동 전환
2. PlayerPrefs 기반 간이 저장 시스템 구축
현재 앱 재시작 시 모든 진행 데이터가 유실됨. 시연 중 크래시나 재시작 시 처음부터 다시 플레이해야 하는 리스크가 높음.
  1. SaveManager 스크립트 생성: PlayerPrefs.SetInt로 레벨, 경험치 저장
  2. LevelManager.LevelUp()과 BattleManager.ReturnToField()에서 자동 저장 호출
  3. LevelManager.Awake()에서 저장된 값 로드
  4. 인벤토리는 JSON 직렬화 후 PlayerPrefs.SetString으로 저장
3. Singleton 생명주기 안정화 (DontDestroyOnLoad 적용)
BattleManager가 배틀씬에서 생성되고 필드로 복귀 시 파괴되는데, LevelManager/InventoryController도 동일한 문제를 가짐. 씬 전환 시 NullReferenceException이 발생할 수 있어 시연 중 크래시 위험.
  1. LevelManager, InventoryController의 Awake()에 DontDestroyOnLoad(gameObject) 추가
  2. 중복 Instance 생성 시 새 인스턴스를 Destroy하도록 가드 코드 추가
  3. BattleManager는 배틀씬 전용이므로 DontDestroyOnLoad 불필요, 대신 Instance null 체크를 호출 측에 추가

코드 리뷰

BattleManager.cs개선필요
403줄로 파일 크기 권장치(300줄)를 초과함. 핵심 전투 루프(TryHitMonster→HP감소→Victory/Defeat 분기)는 잘 동작하며, GetScaledMaxHp()의 레벨 스케일링 공식(baseHp×(1+(lv-1)×0.12))도 적절함. 그러나 GetRewardItem()이 항상 null을 반환하는 미완성 스텁이고, ReturnToField()에서 LevelManager.Instance.AddExp()를 null 체크 없이 호출하여 LevelManager가 없는 씬에서 NullReferenceException 발생 위험. CreateRandomItem()에서 ScriptableObject.CreateInstance<Item>()으로 런타임 에셋을 생성하는데, 이는 메모리 누수 가능성이 있어 Destroy 처리가 필요함.
Monster.cs개선필요
ApplyVisualByGradeType() 메서드가 100줄 이상으로 스프라이트 로딩, 머티리얼 생성, 메시 렌더러 색상 적용, 스케일 조정을 모두 처리하여 단일 책임 원칙 위반. Shader.Find()를 4단계 폴백(URP/Unlit→Unlit/Texture→Sprites/Default→Legacy)으로 하는 것은 렌더 파이프라인 호환성 문제를 임시 처리한 것. Epic Slime에 대한 spriteOffset 하드코딩(line 148)은 확장성이 없음. 배회(Wander) AI는 기본적이지만 동작하며, Init()→ApplyVisualByGradeType()→Despawn() 생명주기는 명확함.
MonsterSpawner.cs개선필요
레벨 기반 몬스터 등급 확률 시스템(UpdateProbabilityValues)은 3단계(Lv<10/10-29/30+)로 잘 설계됨. EnterBattle()에서 SceneManager.LoadSceneAsync(Additive) + HideInGameSceneRootsExcept()로 배틀 씬 전환을 처리하는 구조는 적절함. 하지만 주석 처리된 옛 코드가 30줄 이상 남아있고(lines 128-165), LevelManager.Instance.currentLevel 접근 시 null 체크가 없어 LevelManager 초기화 전에 스폰이 시작되면 크래시 발생. FieldPaused 정적 플래그로 필드/배틀 상태를 관리하는 것은 다중 배틀 시나리오에서 문제가 될 수 있음.
GpsMapPlayerMover.cs심각
341줄의 대형 스크립트지만 실제 GPS 기능은 미구현 상태. Start() 코루틴에서 Input.location 초기화 후 실제 위치 업데이트 루프로 진입하지 않고, Update()에서 UNITY_EDITOR일 때 UpdateEditorMock(), 그 외에는 HandleTouchDrag()만 호출하여 GPS를 터치 드래그로 완전 대체함. RecenterOriginToCurrentLocation()도 #if UNITY_EDITOR 블록만 있고 디바이스용 구현이 없음. ConvertGpsToWorld()와 GetMeterOffset() 등 좌표 변환 함수는 올바르게 구현되어 있으나 실제로 호출되는 경로가 없음. AR 위치기반 RPG의 핵심 기능인 GPS가 동작하지 않는 것은 치명적.
InventoryController.cs개선필요
ARRPG.Inventory 네임스페이스로 적절히 분리됨. 6슬롯 인벤토리, 장착/해제, BattleManager에서 EquippedWeapon.attackPower 참조하여 데미지 보너스 적용까지 연동됨. 하지만 변수명 'OutPanul'(OutPanel 오타), GetSlotNumber()에서 GameObject 이름 파싱으로 슬롯 순서를 결정하는 것이 취약하고, 인벤토리 가득 참 시 Debug.Log만 출력하고 플레이어에게 UI 피드백이 없음. Singleton 패턴에 DontDestroyOnLoad가 없어 씬 전환 시 인벤토리 데이터 유실 위험.
BattleSelectUI.cs개선필요
399줄로 파일 크기 권장치 초과. 몬스터 리스트 UI, 레이캐스트 클릭 선택, 슬라이드 패널 애니메이션 등 풍부한 UI 기능이 구현됨. GetKoreanFont()에서 OS별 폰트(Malgun Gothic/AppleGothic/NotoSansCJK)를 정적 캐시하는 것은 동작하지만 확장성이 없음. ScrollRect를 런타임에 AddComponent로 추가하는 SetupScrollRect()는 Inspector 설정과 충돌 가능성 있음. RefreshList()가 0.3초 간격으로 전체 리스트를 재생성하는데, 풀링 없이 매번 Instantiate/Destroy하여 GC 압박 유발.
MapGridGenerator.cs심각
Android 빌드 시 GPS 폴백 좌표에 위도 오타 존재: lat = 7.494920f (정상: 37.494920f). 이 버그로 인해 실제 디바이스에서 대서양 한가운데의 타일을 로드하게 됨. 3×3 타일 그리드 생성과 OpenStreetMap 타일 다운로드 로직 자체는 동작하나, 줌 레벨 15 하드코딩, 에러 핸들링 부재, GPS 실패 시 폴백 전략 없음. GpsMapPlayerMover.SyncWithMap() 호출로 플레이어 위치를 맵에 동기화하는 연동은 구현되어 있음.
DeviceCameraBackground.cs개선필요
sealed 클래스와 Ensure() 팩토리 메서드 패턴은 좋은 설계. 하지만 SetupUI()가 Awake()와 Start() 양쪽에서 호출되어 Canvas/RawImage가 이중 생성될 수 있음. WebCamTexture에 대해 Destroy()를 호출하는데(CoStartCamera line 134), WebCamTexture는 UnityEngine.Object가 아닌 Texture 파생이라 Stop()+null 할당이 올바른 정리 방법. #if UNITY_EDITOR에서 OnEnable()이 즉시 return하여 에디터에서 UI 미구성 상태로 남을 수 있음.

아키텍처 평가

전체 아키텍처가 Singleton 패턴에 강하게 의존함(BattleManager.Instance, LevelManager.Instance, InventoryController.Instance, VictoryController.Instance, DefeatController.Instance). 씬 간 데이터 전달은 BattleSelectionContext라는 정적 클래스를 통해 이루어지는데, 이는 간단하지만 초기화 순서 보장이 없어 취약함. MonsterSpawner가 SceneManager.LoadSceneAsync(Additive)와 HideInGameSceneRootsExcept()로 필드↔배틀 씬 전환을 관리하는 구조는 학생 프로젝트로서 적절한 수준. 네임스페이스는 ARRPG.Inventory, ARRPG.Level, ARRPG.Victory, ARRPG.Map, ARRPG.LimChaeSung으로 기능별 분리를 시도했으나, 가장 많이 수정되는 핵심 클래스들(BattleManager, Monster, MonsterSpawner)이 네임스페이스 없이 전역 스코프에 있어 일관성이 부족함. Observer/Event 패턴이 전혀 사용되지 않아 시스템 간 결합도가 높고, HP 변경→UI 갱신이 Update() 폴링 방식으로 구현되어 있음.

코드 이슈 (10건)

심각도파일설명
CRITICALMapGridGenerator.csAndroid GPS 폴백 좌표에서 위도가 7.494920f로 설정됨 (정상값: 37.494920f). 실제 디바이스 빌드 시 대서양 한가운데 좌표로 맵 타일을 로드하게 되어 게임이 정상 동작하지 않음.
CRITICALGpsMapPlayerMover.csStart() 코루틴에서 GPS 위치 업데이트 루프가 구현되지 않았고, Update()에서 HandleTouchDrag()로 대체됨. AR 위치기반 게임의 핵심인 실시간 GPS 추적이 전혀 동작하지 않아 프로토타입 시연 시 위치 연동 불가.
HIGHBattleManager.csReturnToField() line에서 LevelManager.Instance.AddExp()를 null 체크 없이 호출. LevelManager가 Singleton이지만 DontDestroyOnLoad 미적용이라, 배틀씬에서 필드로 복귀 시 Instance가 null일 수 있어 NullReferenceException 발생 가능.
HIGHLevelManager.csAwake()에서 Instance 할당만 하고 DontDestroyOnLoad() 미호출. 씬 전환(배틀→필드) 시 LevelManager가 파괴되어 레벨/경험치 데이터가 유실됨. MonsterSpawner.UpdateProbabilityValues()에서도 이 Instance를 참조하므로 연쇄 크래시 가능.
MEDIUMBattleManager.csGetRewardItem() 메서드가 항상 null을 반환하는 미완성 스텁. 현재 보상 시스템이 RollWeaponDrop()→CreateRandomItem() 경로로 우회 구현되어 있어 즉시 크래시는 아니지만, 이 메서드를 참조하는 코드가 추가되면 문제 발생.
MEDIUMDeviceCameraBackground.csSetupUI()가 Awake()와 Start()에서 이중 호출됨. Canvas, RawImage, AspectRatioFitter가 두 번 생성되어 렌더링 오버헤드 및 UI 레이어 충돌 가능. Start()의 SetupUI() 호출을 제거해야 함.
MEDIUMMonster.csApplyVisualByGradeType()에서 Shader.Find()를 4단계 폴백으로 호출하는데, URP 프로젝트에서 'Unlit/Texture' 셰이더가 빌드에 포함되지 않으면 null이 반환되고, 최종 폴백 'Legacy Shaders/Diffuse'도 URP에서는 누락될 수 있어 몬스터가 분홍색(missing shader)으로 렌더링될 수 있음.
MEDIUMSpawnProbUI.csUpdate()에서 FindObjectOfType<MonsterSpawner>()를 매 프레임 호출. FindObjectOfType은 O(n) 탐색이므로 성능 저하 유발. Start()에서 캐싱하여 참조를 유지해야 함.
LOWInventoryController.cs변수명 'OutPanul'은 'OutPanel'의 오타. 기능에는 영향 없으나 코드 가독성과 유지보수성을 저하시킴. Inspector에서 할당하므로 런타임 에러는 아님.
LOWBattleSelectUI.csRefreshList()에서 리스트 아이템을 매번 Destroy 후 Instantiate로 재생성. 0.3초 간격으로 호출되어 GC Alloc이 누적됨. 오브젝트 풀링 또는 기존 아이템 재사용 패턴으로 변경 권장.
