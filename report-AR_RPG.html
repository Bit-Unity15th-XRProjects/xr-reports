프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 만들고 있으며, 전투 시스템(BattleManager)과 몬스터 스폰 시스템(MonsterSpawner)이 실제로 동작하는 핵심 게임 루프를 완성한 상태임. 필드에서 몬스터를 터치하여 선택하고, 배틀씬으로 전환하여 터치 공격으로 처치하면 경험치와 무기를 획득하는 플로우가 BattleSelectUI→MonsterSpawner.EnterBattle()→BattleManager→LevelManager.AddExp()→InventoryController.AddItem()으로 연결되어 있음. 그러나 가장 치명적인 문제는 SaveManager가 완전히 부재하여 모든 진행 데이터가 앱 종료 시 소실된다는 점이며, SoundManager도 없어 게임이 완전 무음 상태임. GPS 기반 맵 시스템(GpsMapPlayerMover + MapGridGenerator)은 좌표 변환 로직은 구현되었으나 실제 GPS와 맵 타일의 실시간 연동이 하드코딩된 좌표로 인해 불완전함. 가장 시급한 것은 SaveManager 구현과 LimChaesung 브랜치(ahead 10) 병합이며, 이 두 가지가 해결되지 않으면 발표 시 게임의 핵심 가치인 '성장하는 AR RPG'를 보여줄 수 없음.

리스크 알림

SaveManager가 없어 레벨, 경험치, 인벤토리 아이템이 앱 종료 시 모두 초기화됨. AR RPG에서 성장과 수집이 핵심 루프인데, 진행이 저장되지 않으면 게임의 존재 의미가 없음. 발표/시연 중에도 앱이 백그라운드로 가면 데이터가 날아갈 수 있음
  • PlayerPrefs 또는 JsonUtility + Application.persistentDataPath로 SaveManager 구현
  • LevelManager.currentLevel/currentExp, InventoryController.items 리스트를 직렬화
  • 씬 전환 시, 앱 일시정지 시(OnApplicationPause) 자동 저장 호출
  • 앱 시작 시 LoadData() → LevelManager/InventoryController에 복원

  • BattleManager.TryHitMonster()와 ReturnToField()에 효과음 재생 호출 추가

권고사항 (Top 3)

1. SaveManager 즉시 구현 — 최우선 과제
현재 모든 진행 데이터(레벨, 경험치, 인벤토리)가 메모리에만 존재. 게임의 핵심 루프(전투→성장→수집)가 저장 없이는 의미가 없고, 시연 시에도 앱 재시작마다 초기 상태로 돌아감
  1. SaveData 클래스 정의: int level, float exp, List<ItemData> items
  2. SaveManager 싱글톤: Save()는 JsonUtility.ToJson → File.WriteAllText(persistentDataPath)
  3. Load()는 File.ReadAllText → JsonUtility.FromJson → LevelManager/InventoryController에 주입
  4. OnApplicationPause(true)와 씬 전환 시 자동 Save() 호출
  5. 앱 시작 시 SaveManager.Load() → 데이터 존재하면 복원, 없으면 기본값
2. BattleManager 책임 분리 리팩토링
403줄의 God Object가 전투/아이템/UI/씬 관리를 모두 담당하여 버그 수정과 기능 추가가 매우 어려움. 팀원 3명이 동시에 이 파일을 수정할 확률이 높아 Git 충돌도 빈발할 것
  1. BattleCalculator static 클래스로 HP/XP 스케일링 공식 분리 (GetScaledMaxHp, GetScaledXpReward 등)
  2. ItemFactory 클래스로 CreateRandomItem(), RollWeaponDrop() 분리
  3. BattleTimerUI 컴포넌트로 타이머 관련 UI 업데이트 로직 분리
  4. BattleManager는 전투 상태 관리와 흐름 제어만 담당하도록 축소
3. 최소한의 SoundManager + 핵심 효과음 5개 추가
오디오가 전혀 없는 상태에서 게임 완성도가 크게 떨어짐. AR 게임에서 몬스터 발견 알림음과 전투 효과음은 사용자 경험의 핵심이며, 최소한의 구현으로 큰 효과를 얻을 수 있음
  1. SoundManager 싱글톤: AudioSource 2개 (BGM + SFX), PlaySFX(AudioClip), PlayBGM(AudioClip)
  2. 무료 효과음 5개 확보: 타격(hit), 승리(victory), 패배(defeat), 레벨업(levelup), 몬스터등장(spawn)
  3. BattleManager.TryHitMonster()에 hit SFX, ReturnToField()에 victory SFX 추가
  4. MonsterSpawner에서 범위 내 몬스터 최초 발견 시 spawn SFX 재생

코드 리뷰

BattleManager.cs개선필요
전투의 핵심 루프가 실제로 동작하는 가장 중요한 파일. GetScaledMaxHp()와 GetScaledXpReward()의 레벨 스케일링 공식이 깔끔하고, RollWeaponDrop()의 등급별 확률 시스템도 잘 설계됨. TryHitMonster()에서 레이캐스트 기반 터치 공격→HP 감소→승리/패배 분기가 완성됨. 그러나 403줄로 SRP 위반이 심각함: 전투 로직, HP/XP 계산, 아이템 생성(CreateRandomItem), UI 타이머 업데이트, 씬 전환을 모두 한 클래스가 담당. GetRewardItem()이 null 리턴 스텁으로 남아있어 보상 시스템이 미완성. hpScalePerPlayerLevel(0.12f)이 SerializeField이면서 GetMaxHpByGrade()에서 동일한 0.12f를 하드코딩하여 불일치 가능성 존재.
MonsterSpawner.cs양호
레벨 기반 몬스터 등급 확률 시스템(UpdateProbabilityValues)이 3단계로 잘 설계됨. EnterBattle()에서 FieldPaused 설정→InGame 씬 루트 숨기기→BattleScene additive 로드→BattleSelectionContext에 선택 정보 저장하는 플로우가 완결됨. HideInGameSceneRootsExcept()와 RestoreInGameSceneRoots()의 씬 관리 로직이 견고함. 다만 static 필드(FieldPaused, FieldCameraRef, ArCameraRef, _hiddenInGameSceneRoots)가 4개나 되어 전역 상태 관리가 취약하고, 멀티씬 환경에서 경쟁 조건 가능성 있음.
Monster.cs양호
등급(Normal/Rare/Epic)×타입(Slime/Beast/Golem) 매트릭스 기반 비주얼 시스템이 잘 구현됨. ApplyVisualByGradeType()에서 스프라이트 유무, 배틀/필드 상태, showSpriteInField 플래그를 복합적으로 판단하여 렌더링 모드를 전환하는 로직이 정교함. Wander AI(wanderSpeed, retarget interval, arrive distance)와 Flee 메카닉(10초 후 도주)으로 필드에서의 몬스터 행동이 자연스러움. GetSpriteScaleForGradeAndType()에서 스케일 값이 0이면 0.25f 기본값을 사용하는 방어 코드도 적절함.
BattleSelectUI.cs개선필요
399줄로 Monster.cs보다 길며, UI 생성+애니메이션+입력처리+몬스터 선택을 모두 담당. AnimateListSlide()의 Lerp 기반 슬라이딩 애니메이션과 등급별 텍스트 컬러링은 잘 구현됨. GetKoreanFont()에서 플랫폼별 OS 폰트 로딩(Windows: Malgun Gothic, Mac: AppleGothic, Android: NotoSansCJK)은 실용적인 해결책. 그러나 UI 로직과 게임플레이 로직(레이캐스트 몬스터 선택, 아웃라인 상태 관리)이 혼재되어 있고, OnStartBattle()에서 spawner.EnterBattle() 호출 전 null 체크만으로 에러 처리가 부족함.
GpsMapPlayerMover.cs개선필요
GPS 좌표→월드 좌표 변환(ConvertGpsToWorld)과 Haversine 기반 미터 오프셋 계산(GetMeterOffset)이 정확하게 구현됨. 에디터 모의 이동(UpdateEditorMock, WASD)과 실제 GPS 모드의 이중 구조가 잘 설계됨. 위치 보간(Lerp)과 회전 보간으로 부드러운 캐릭터 이동 구현. 그러나 _isReady와 _isInitialized 두 개의 초기화 플래그가 혼란을 야기하고, Start() 코루틴에서 GPS 서비스 초기화 실패 시 복구 로직이 불명확. 341줄로 GPS 처리, 입력 처리, 애니메이션, 맵 동기화를 모두 한 클래스에서 담당하여 분리가 필요함.
InventoryController.cs개선필요
싱글톤 기반 인벤토리 관리자로 AddItem()/RemoveItem()/EquipOrUnequip() 기본 기능이 동작함. EquippedWeapon 프로퍼티를 통해 BattleManager에서 장착 무기의 공격력 보너스를 참조하는 연동이 구현됨. RefreshUI()에서 슬롯별 아이콘 업데이트도 동작. 그러나 인코딩 깨진 한글 주석이 다수 존재하여 유지보수성이 떨어지고, 가장 치명적으로 데이터 영속성이 전혀 없어 앱 종료 시 모든 아이템이 소실됨.
DeviceCameraBackground.cs양호
WebCamTexture 기반 AR 배경 카메라를 자체 구현한 파일. Ensure() 팩토리 메서드로 씬에 하나만 존재하도록 보장하고, Canvas+RawImage+AspectRatioFitter 조합으로 카메라 피드를 화면에 맞춤. ApplyVideoTransform()에서 디바이스 회전 보정과 전면/후면 카메라 미러링 처리가 정확함. OnDisable()/OnDestroy()에서 WebCamTexture 정리가 깔끔하게 구현됨. 에디터에서는 카메라를 시작하지 않는 #if UNITY_EDITOR 가드도 적절함. 다만 주석의 인코딩 깨짐 문제가 있음.
LevelManager.cs양호
42줄의 간결한 레벨 시스템. maxExp *= 1.2f의 지수적 성장 공식이 적절하고, AddExp()에서 while 루프로 연속 레벨업을 처리하는 로직이 올바름. ExpUIController 연동으로 UI 자동 갱신도 동작. 다만 레벨/경험치 데이터가 메모리에만 존재하여 SaveManager 없이는 무의미한 진행 시스템이 됨.

아키텍처 평가

Singleton 패턴을 광범위하게 사용하여(BattleManager, LevelManager, InventoryController, VictoryController, DefeatController, ExpUIController 등 7개 이상) 전역 접근성을 확보했지만, 이로 인해 클래스 간 직접 참조가 만연하고 의존성 방향이 불명확함. 씬 구조는 InGame(필드)+BattleScene(전투) additive 로딩으로 적절하나, 씬 간 통신이 BattleSelectionContext static 클래스와 MonsterSpawner의 static 필드 4개에 의존하여 상태 관리가 취약함. 데이터 흐름은 Monster→BattleSelectionContext→BattleManager→LevelManager/InventoryController로 단방향이어서 기본적인 흐름은 명확하지만, 이벤트 시스템 없이 null 체크 기반 호출로 구현되어 있어 새 시스템 추가 시 기존 코드 수정이 불가피한 OCP 위반 구조임. namespace 분리(ARRPG.Inventory, ARRPG.Level 등)는 잘 되어 있으나, 핵심 Script 폴더에 BattleManager, Monster, MonsterSpawner 등이 namespace 없이 혼재하는 것은 개선 필요.

기반 구축 4/5점
Unity 프로젝트 구조가 잘 잡혀 있고, asmdef(9개)로 어셈블리 분리, namespace(ARRPG.Inventory, ARRPG.Level, ARRPG.Victory, ARRPG.LimChaeSung) 사용, ScriptableObject 기반 아이템 데이터 설계가 되어 있음. 씬은 InGame(필드) + BattleScene(전투) 구조로 additive 로딩 사용. AR Foundation 대신 WebCamTexture 기반 자체 AR 배경 구현. 다만 통합 GameManager가 없고, 프로젝트 전체를 관장하는 진입점이 부재하여 1점 감점.
핵심 시스템 16/25점
BattleManager: HP/XP 레벨 스케일링(baseHp × (1+(lv-1)×0.12)), 제한시간 전투, 무기 드롭 확률 시스템(Normal 30%/Rare 15%/Epic 5%), 레이캐스트 기반 터치 공격 등 실제 동작하는 전투 루프 완성. MonsterSpawner: 레벨 기반 등급 확률(lv<10: 90/8/2, lv10-30: 60/30/10, lv30+: 30/40/30), 거리 기반 디스폰, 필드 일시정지 구현됨. Monster: Wander AI + Flee 메카닉 + 등급/타입별 스프라이트 로딩(Resources/Monsters/{Type}/{Grade}-{Type}). LevelManager: 지수적 경험치 요구량(×1.2/레벨). InventoryController: 아이템 추가/장착 토글 동작하지만 영속성 없음. 반면 GetRewardItem()이 null 리턴 스텁이고, VictoryController는 UI 토글만 수행하며 실제 승리 로직이 빈약함. SaveManager와 SoundManager는 완전히 부재.
통합 13/25점
게임 루프가 실제로 존재함: 필드에서 몬스터 스폰 → BattleSelectUI로 범위 내 몬스터 선택 → BattleSelectionContext에 등급/타입 저장 → SceneManager.LoadSceneAsync(battleSceneName, Additive) → BattleManager에서 전투 → 승리 시 XP 부여(LevelManager.Instance.AddExp) + 무기 드롭(InventoryController.Instance.AddItem) → ReturnToField()에서 RestoreInGameSceneRoots() 호출하여 필드 복귀. 씬 전환 시 MonsterSpawner.HideInGameSceneRootsExcept()로 InGame 씬 루트 오브젝트를 비활성화하고, 전투 종료 시 복원하는 구조가 완성됨. 다만 시스템 간 통신이 static 필드(FieldPaused, FieldCameraRef, BattleSelectionContext)와 싱글톤 null 체크에 의존하여 결합도가 높고 깨지기 쉬움. GPS→맵→몬스터 스폰의 위치 연동은 MapGridGenerator에서 하드코딩된 좌표(37.494927, 127.0280)를 사용하여 실제 GPS 연동이 불완전.
콘텐츠 8/20점
몬스터 스프라이트가 Resources/Monsters/ 하위에 3타입×3등급=9종 구비(Slime, Beast, Golem 폴더 각 6파일). 프리팹 5개, 머티리얼 4개, 애니메이션 2개 존재. Monster.cs에서 LoadMonsterSprite()를 통해 실제 로딩하고 spriteQuad에 적용하는 로직 확인. 이미지 15장, prefab 10개로 비주얼 에셋이 어느 정도 갖춰짐. 그러나 오디오 파일 0개(SoundManager 부재), FBX 모델 0개(3D 몬스터 없이 2D 스프라이트만 사용), 셰이더 0개(ShaderGraph 노드는 커밋 메시지에 있으나 shader 파일 미확인). 애니메이션도 2개뿐이라 인게임 피드백이 부족할 것으로 보임.
폴리싱 6/20점
BattleUI에서 HP바 동적 컬러링, 타이머 UI(텍스트+슬라이더+원형 게이지) 등 UI 요소가 구현됨. BattleSelectUI의 슬라이딩 패널 애니메이션(Lerp 기반 0.25초)과 등급별 텍스트 컬러링은 완성도 높음. MonsterSelectableOutline에서 프로시저럴 외곽선+바닥 링 시각화도 잘 구현됨. 반면 SaveManager가 없어 앱 재시작 시 모든 진행 데이터 소실. DeviceCameraBackground.cs에 인코딩 깨진 한글 주석 다수 존재(유니코드 미스매치). InventoryController에도 동일한 인코딩 문제. GetRewardItem()이 null 리턴하는 스텁 상태로 남아있고, MapLoader.cs의 Overpass API 연동이 주석 처리된 상태. 에러 처리는 Debug.LogWarning 수준에 머물러 있음.
