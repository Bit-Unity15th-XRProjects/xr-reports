프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 개발하고 있으며, GPS 기반 필드 이동(GpsMapPlayerMover), 몬스터 스폰 및 등급 시스템(MonsterSpawner, Monster), 턴제 전투(BattleManager), 레벨링(LevelManager), 인벤토리(InventoryController), 승리 처리(VictoryController)까지 핵심 게임 루프의 대부분을 구현한 상태다. 특히 GPS→월드 좌표 변환, 레벨 기반 몬스터 등급 확률 조정, HP/XP 레벨 스케일링 등 게임 밸런싱 로직이 실질적으로 동작하며, BattleSelectUI의 슬라이드 패널과 MonsterSelectableOutline의 바닥 링 등 UX도 고려되어 있다. 그러나 SaveManager가 없어 게임 진행이 저장되지 않고, SoundManager와 오디오 에셋이 전무하며, GetRewardItem()이 null을 반환하여 전투→보상→성장 루프가 끊겨 있다. 가장 시급한 것은 SaveManager 구현, LimChaesung 브랜치의 VFX 작업 병합, 그리고 보상 시스템 연결로, 이 세 가지가 해결되면 동작하는 프로토타입으로서 시연이 가능한 수준이다.

리스크 알림

SaveManager 부재로 게임 진행 상태 유실
현재 레벨, 경험치, 인벤토리 아이템 등 플레이 데이터를 저장하는 시스템이 전혀 없음. 앱 종료 시 모든 진행이 초기화되어 시연 중 앱이 꺼지면 처음부터 다시 시작해야 함. 발표 시연에서 치명적.
  • PlayerPrefs 기반 간단한 SaveManager 구현 (레벨, XP, 인벤토리 JSON 직렬화)
  • LevelManager와 InventoryController에 Save/Load 호출 추가
  • 앱 시작 시 자동 로드, 레벨업/아이템 획득 시 자동 저장
LimChaesung 브랜치 미병합 (VFX/이펙트 작업 10커밋)
origin/LimChaesung 브랜치가 main 대비 10커밋 ahead, 7커밋 behind로 상당한 분기 상태. ShaderGraph, VFX 작업이 포함되어 있으나 main에 미병합. 마지막 활동이 10일 전이고 충돌 해결이 필요할 수 있어 마감 직전 병합 시 큰 혼란 발생 가능.
  • 즉시 LimChaesung 브랜치를 main에 rebase 또는 merge 시도
  • 충돌 발생 시 해당 팀원(Lachesis)과 함께 해결
  • 병합 후 BattleScene에서 VFX가 정상 동작하는지 확인
오디오 에셋 0개, SoundManager 부재
게임에 소리가 전혀 없으면 시연 시 매우 밋밋한 인상을 줌. AR RPG에서 전투 효과음, 배경음악은 기본적인 기대 요소. 에셋 준비와 구현 모두 필요하여 시간이 많이 소요됨.
  • 무료 효과음 에셋(전투 타격, 몬스터 등장, 승리 팡파레) 3-5개 확보
  • 간단한 AudioManager 싱글턴 구현 (AudioSource + PlayOneShot)
  • BattleManager의 데미지 처리와 승리/패배 시점에 효과음 재생 추가
GetRewardItem() 스텁으로 보상 시스템 미완성
BattleManager.GetRewardItem()이 null을 반환하여 전투 승리 후 아이템 보상이 실제로 지급되지 않을 수 있음. CreateRandomItem()은 구현되어 있으나 GetRewardItem()과 연결되지 않아 보상 루프가 끊김.
  • GetRewardItem()에서 RollWeaponDrop() 결과에 따라 CreateRandomItem() 호출하도록 연결
  • InventoryController에 아이템 추가 로직 확인 및 연동
  • 배틀 승리 → 아이템 드롭 → 인벤토리 추가 플로우 전체 테스트
모든 팀원 9-12일간 비활동 상태
마지막 커밋이 3월 20일이고 현재 3월 30일. 마감이 임박한 상황에서 10일간 커밋이 없다는 것은 작업이 멈췄거나 로컬에서만 작업 중일 수 있음. 미푸시 작업이 있다면 충돌 위험, 없다면 진행 정체.
  • 팀원 각자 로컬 변경사항 즉시 커밋 및 푸시
  • 남은 작업 우선순위 정리 (SaveManager > 오디오 > VFX 병합)
  • 매일 1회 이상 커밋하는 규칙 설정

권고사항 (Top 3)

1. PlayerPrefs 기반 최소 저장 시스템 긴급 구현
저장 없이는 시연이 불안정하고, 레벨/인벤토리 초기화 시 게임 흐름을 보여줄 수 없음. PlayerPrefs는 가장 빠르게 구현할 수 있는 저장 방식.
  1. SaveManager.cs 생성: PlayerPrefs.SetInt로 레벨/XP 저장, JsonUtility로 인벤토리 직렬화
  2. LevelManager에 SaveLevel()/LoadLevel() 추가, 레벨업 시 자동 저장
  3. InventoryController에 SaveInventory()/LoadInventory() 추가
  4. 게임 시작 시(Awake) 자동 로드, OnApplicationPause(true)에서 자동 저장
2. 보상 루프 완성 (GetRewardItem → 인벤토리 연동)
전투→보상→성장 루프는 RPG의 핵심. 현재 전투는 되지만 보상이 null로 반환되어 성장 동기가 없음.
  1. BattleManager.GetRewardItem()에서 RollWeaponDrop() + CreateRandomItem() 호출 연결
  2. VictoryData()에서 GetRewardItem() 결과를 InventoryController에 추가
  3. 승리 UI에 획득 아이템 표시 (아이콘 + 등급 색상)
3. 최소 효과음 3개 추가로 시연 품질 향상
무음 게임은 시연 시 완성도가 크게 떨어져 보임. 무료 에셋 3개만으로도 체감 품질이 크게 향상됨.
  1. freesound.org 등에서 전투 타격, 몬스터 등장, 승리 효과음 다운로드
  2. AudioManager 싱글턴 생성 (public AudioClip[] + PlayOneShot)
  3. BattleManager 데미지 시, 몬스터 스폰 시, 승리 시 재생 호출 추가

코드 리뷰

BattleManager.cs개선필요
핵심 전투 관리 클래스로 HP 레벨 스케일링(GetScaledMaxHp), XP 보상 계산(GetScaledXpReward), 제한시간 타이머, 무기 드롭 확률(RollWeaponDrop) 등 실질적인 게임 로직이 구현되어 있음. 그러나 403줄로 권장 초과이며, GetRewardItem()이 null을 반환하는 스텁 상태. CreateRandomItem()에서 ScriptableObject.CreateInstance<Item>()으로 런타임 아이템 생성은 동작하지만 메모리 관리 주의 필요. GetBaseMaxHpByGrade → GetScaledMaxHp → GetMaxHpByGrade로 이어지는 3단계 래퍼 체인이 과도하며, hpScalePerPlayerLevel 인스턴스 필드와 GetMaxHpByGrade의 하드코딩된 0.12f가 불일치할 위험 있음. Singleton의 Awake()에서 중복 인스턴스 체크가 없음.
Monster.cs양호
등급(Normal/Rare/Epic)과 타입(Slime/Beast/Golem) 조합으로 몬스터 다양성을 구현. Init()에서 초기화, ApplyVisualByGradeType()에서 Resources.Load로 스프라이트 적용, 배회 AI(wanderSpeed, wanderRetargetInterval)와 도주(fleeAfterSeconds) 행동이 구현됨. RequireComponent(typeof(Collider))로 컴포넌트 의존성 명시한 점이 좋음. 다만 ApplyGradeColor()가 ApplyVisualByGradeType()을 단순 호출만 하여 메서드명이 오해를 유발. TypeSpriteScales 구조체로 타입별 크기 조절이 Inspector에서 가능하도록 설계됨.
MonsterSpawner.cs양호
UpdateProbabilityValues()에서 레벨 구간별(10 미만, 10 이상 등) 등급 출현 확률을 조정하고, DecideGradeByLevel()에서 Random.Range로 등급 결정하는 로직이 잘 구현됨. EnterBattle()에서 battleSceneName 빈값 체크 후 LoadSceneAsync로 배틀 진입. HideInGameSceneRootsExcept()로 씬 전환 시 필드 오브젝트를 숨기고 RestoreInGameSceneRoots()로 복원하는 패턴이 안정적. static FieldPaused와 FieldCameraRef로 전역 상태를 관리하는 것은 간결하지만 테스트 시 상태 오염 가능성 있음.
GpsMapPlayerMover.cs양호
GPS 좌표를 월드 좌표로 변환하는 ConvertGpsToWorld()와 GetMeterOffset()이 지구 반경 상수(6378137.0)를 사용해 정확히 구현됨. _minMoveThresholdMeters(0.3m)로 GPS 노이즈 필터링, _positionLerpSpeed로 부드러운 이동, 에디터에서 WASD 목업 이동(UpdateEditorMock) 등 실용적 기능이 갖춰짐. 네임스페이스(ARRPG.LimChaeSung)를 사용하여 코드 정리도 양호. 341줄로 권장 초과이지만 GPS 처리 특성상 설정값이 많아 불가피한 면이 있음.
BattleSelectUI.cs개선필요
399줄로 권장 초과. 몬스터 리스트 생성, 슬라이드 애니메이션, 등급별 텍스트 색상, 한국어 폰트 폴백(GetKoreanFont에서 OS별 분기) 등 풍부한 UI 기능을 구현. 그러나 UI 로직, 애니메이션, 폰트 관리, 이벤트 처리가 한 클래스에 집중되어 있어 분리가 필요. _cachedKoreanFont를 static으로 캐싱한 점은 좋으나 플랫폼별 폰트 로직이 UI 클래스에 있는 것은 부적절.
DeviceCameraBackground.cs양호
모바일 카메라를 AR 배경으로 사용하는 클래스. #if UNITY_EDITOR 가드로 에디터에서 카메라 시작을 방지하고, OnDisable/OnDestroy에서 StopCameraInternal()로 WebCamTexture를 정리하여 메모리 누수를 방지. Ensure() 정적 메서드로 싱글턴 생성 패턴 구현. 주석이 인코딩 깨져서 보이지만(UTF-8 문제로 추정) 로직 자체는 안정적.
MonsterSelectableOutline.cs양호
몬스터 선택 가시성을 위한 외곽선과 바닥 링을 구현. 탑다운 뷰에서도 선택 가능 몬스터가 보이도록 groundRing을 그리는 아이디어가 좋음. FieldPaused 상태와 씬 이름으로 배틀 중 비활성화 처리. SetVisible()로 선택/미선택 상태 전환. outlineRenderQueue 설정으로 렌더링 순서 제어가 Inspector에서 가능.

아키텍처 평가

Singleton 패턴을 BattleManager, LevelManager, VictoryController에 사용하여 전역 접근을 구현했으나, Awake()에서 중복 체크나 DontDestroyOnLoad 없이 Instance = this만 할당하는 불완전한 구현. MonsterSpawner는 static 필드(FieldPaused, FieldCameraRef)로 필드/배틀 상태를 관리하며, BattleSelectionContext 정적 클래스로 씬 간 데이터 전달. 씬 구조는 InGame(필드) ↔ BattleScene(전투)로 분리되어 있고 LoadSceneAsync + Additive 방식 사용. 데이터 모델은 ScriptableObject(Item)와 enum(MonsterGrade, MonsterType)으로 구성. 전체적으로 학생 프로젝트로서 실용적인 구조이나, BattleManager가 전투·아이템·UI·씬전환을 모두 담당하는 God Object 경향이 있고, 네임스페이스 사용이 일부 모듈에만 적용되어 일관성이 부족함.

단계별 진행률 (AI 평가)

기반 구축 4/5점
Unity 프로젝트 구조가 잘 갖춰져 있음. Scripts 폴더가 Core, Data, Inventory, Level, Map, Victory 등 기능별로 분리되어 있고, 씬 8개(InGame, BattleScene 등), 프리팹 5개, ScriptableObject 4개가 존재. 네임스페이스도 ARRPG.Inventory, ARRPG.Level, ARRPG.Victory, ARRPG.LimChaeSung 등 부분적으로 사용 중. asmdef 파일 9개로 어셈블리 분리도 시도함. 다만 일부 핵심 스크립트(BattleManager, Monster 등)는 네임스페이스 없이 글로벌 스코프에 있어 일관성이 부족함.
핵심 시스템 20/25점
구현된 핵심 시스템: (1) BattleManager - HP/XP 레벨 스케일링, 제한시간 타이머, 무기 드롭 확률, 승리/패배 분기가 실제 로직으로 구현됨. (2) Monster - 등급(Normal/Rare/Epic)·타입(Slime/Beast/Golem) 시스템, 배회 AI, 도주 행동, Resources 기반 스프라이트 로딩이 동작함. (3) MonsterSpawner - 레벨 기반 등급 확률 조정, 범위 기반 스폰/디스폰, 배틀 씬 전환 로직이 완성됨. (4) GpsMapPlayerMover - GPS 좌표→월드 좌표 변환, 노이즈 필터링, 에디터 목업이 구현됨. (5) BattleSelectUI - 몬스터 리스트, 슬라이드 패널, 등급별 색상 표시. (6) InventoryController - 핫파일에 5회 변경, 6개 파일 존재하나 코드 미제공으로 상세 확인 불가. (7) LevelManager - BattleManager와 MonsterSpawner에서 .Instance.currentLevel로 참조됨. (8) VictoryController - BattleManager에서 VictoryController.Instance.VictoryUI로 호출됨. 다만 SaveManager와 SoundManager는 코드에서 참조되지 않으며, GetRewardItem()이 null을 반환하는 스텁 상태.

심각도파일설명
HIGHBattleManager.csGetRewardItem()이 항상 null을 반환하는 스텁 메서드. 배틀 승리 시 보상 아이템 로직이 실제로 동작하지 않을 가능성 있음.
HIGHBattleManager.cshpScalePerPlayerLevel 필드(SerializeField, 기본값 0.12f)와 GetMaxHpByGrade() 내부의 하드코딩된 0.12f가 별도로 존재. Inspector에서 값을 변경해도 정적 메서드에는 반영되지 않아 의도와 다른 HP 스케일링이 적용될 수 있음.
MEDIUMBattleManager.csAwake()에서 Instance = this만 할당하고 기존 인스턴스 중복 체크가 없음. 씬 재로드 시 여러 인스턴스가 생기거나, 이전 인스턴스의 참조가 남아 NullReferenceException 발생 가능.
MEDIUMBattleManager.csCreateRandomItem()에서 ScriptableObject.CreateInstance<Item>()으로 런타임에 에셋을 생성하지만 Destroy 호출이 보이지 않음. 반복적인 아이템 생성 시 메모리 누수 가능.

MEDIUMMonsterSpawner.csstatic FieldPaused, FieldCameraRef, ArCameraRef 등 전역 상태가 명시적 초기화 없이 사용됨. 배틀에서 복귀 시 RestoreInGameSceneRoots()가 호출되지 않으면 필드 오브젝트가 영구 숨김 상태로 남을 수 있음.
