프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 만들고 있으며, 핵심 게임플레이 루프(맵 이동→몬스터 발견→전투 진입→클릭 공격→승리/패배→보상)까지 실제로 동작하는 수준까지 구현했다. BattleManager와 MonsterSpawner를 중심으로 전투 시스템이 가장 완성도가 높고, GpsMapPlayerMover의 GPS 연동도 에디터/실기기 양쪽을 고려한 세심한 구현이다. 그러나 SaveManager가 완전히 부재하여 앱 종료 시 모든 진행이 소실되고, SoundManager도 없어 게임이 무음 상태이며, AR Foundation 대신 WebCamTexture를 사용하는 임시 구현이라 실제 AR 경험(평면 인식, 오클루전 등)이 없다. Singleton 간 null 체크 부재로 씬 전환 시 크래시 위험이 높고, BattleManager가 403줄로 비대하며 VFX/이펙트 작업이 미병합 브랜치에 고립되어 있다. 가장 시급한 것은 SaveManager 구현과 Singleton null 안전성 확보이며, 이 두 가지가 해결되어야 '플레이 가능한 프로토타입'에서 '완성된 게임'으로 넘어갈 수 있다.

리스크 알림

게임 진행 데이터 저장 기능 완전 부재
LevelManager의 레벨/경험치, InventoryController의 장비 데이터가 메모리에만 존재하고 PlayerPrefs나 파일로 저장되지 않음. 앱을 종료하거나 크래시가 발생하면 모든 진행이 소실됨. AR 위치기반 게임에서 이동하며 모은 진행 상태가 날아가면 사용자 이탈로 직결됨
  • SaveManager 클래스를 만들어 레벨/경험치/인벤토리를 JSON으로 직렬화
  • Application.persistentDataPath에 저장/로드 구현
  • 씬 전환 시, 배틀 승리 시, 앱 OnApplicationPause에서 자동 저장 호출
  • LevelManager와 InventoryController에 DontDestroyOnLoad 적용
Singleton null 참조로 인한 런타임 크래시 다수
LevelManager.Instance, ExpUIController.Instance, BattleManager.Instance 등 6개 이상의 Singleton이 null 체크 없이 상호 참조하고 있음. 씬 전환 시 Destroy 순서에 따라 NullReferenceException이 발생하며, 특히 Additive 씬 로딩 구조에서 타이밍 문제가 빈번함
  • 모든 Singleton 접근 지점에 null 체크 추가(특히 LevelManager.cs:30, BattleManager.cs:366, MonsterSpawner.cs:90)
  • 핵심 매니저(LevelManager, InventoryController)에 DontDestroyOnLoad 적용
  • OnDestroy에서 Instance = null 패턴을 모든 Singleton에 통일
LimChaesung 브랜치 병합 지연 (VFX/이펙트 미통합)
origin/LimChaesung 브랜치가 main 대비 10 ahead / 7 behind로 상당한 분기가 발생함. ShaderGraph, VFX 작업이 이 브랜치에만 있어 main에 비주얼 효과가 전혀 없는 상태. 병합이 늦어질수록 충돌이 심화됨
  • 이번 주 내로 rebase 또는 merge 시도
  • 충돌 파일(주로 .unity 씬 파일) 목록을 미리 확인
  • 씬 파일 충돌은 한 명이 작업 후 다른 사람이 반영하는 방식으로 해결

  1. SaveData 클래스 정의 (level, exp, inventory items)
  2. JsonUtility.ToJson/FromJson으로 직렬화
  3. Application.persistentDataPath + '/save.json'에 저장
  4. LevelManager.AddExp(), BattleManager.ReturnToField() 후 자동 저장 호출
  5. 앱 시작 시 SaveManager.Load()로 복구
2. BattleManager 클래스 분리 리팩토링
403줄에 HP계산, XP계산, UI갱신, 아이템생성, 씬관리가 혼재. 버그 수정이나 기능 추가 시 서로 영향을 줘서 개발 속도가 저하됨
  1. BattleCalculator (static): HP/XP/드롭 확률 계산 분리
  2. BattleRewardHandler: 아이템 생성, 경험치 부여, VictoryData 로직 분리
  3. BattleManager는 전투 흐름 제어(Start→Update→End)만 담당하도록 축소
3. 팀 내 Git 워크플로우 개선
커밋 평균 1032줄 변경, 4883줄짜리 단일 커밋 존재, 주석 처리된 코드 커밋, 미병합 브랜치 방치 등 Git 사용이 미숙함. 충돌과 코드 유실 위험이 높음
  1. 커밋은 500줄 이하로 제한, 기능 단위로 분리
  2. 주석 처리 코드는 커밋 전에 삭제 (Git 히스토리에 남으므로 복구 가능)
  3. 주 2회 이상 main 브랜치에 병합하여 분기 최소화
  4. Unity 씬 파일(.unity) 동시 편집 금지 규칙 수립

코드 리뷰

BattleManager.cs개선필요
전투 시스템의 핵심 로직(HP 스케일링, 타이머, 레이캐스트 전투, 무기 드롭, 경험치 보상)이 실제로 동작하는 완성된 구현임. 그러나 403줄로 CLAUDE.md의 300줄 권장을 초과하며, HP 계산/XP 계산/UI 업데이트/씬 관리/아이템 생성이 하나의 클래스에 모두 들어있어 SRP 위반. GetRewardItem()(285행)이 항상 null을 반환하는 죽은 코드이고, ReturnToField()(349행)에서 LevelManager.Instance.AddExp() 호출 시 null 체크가 없어 LevelManager가 없는 씬에서 NullReferenceException 발생 가능. CreateRandomItem()(318행)에서 ScriptableObject.CreateInstance<Item>()으로 런타임에 SO를 생성하는데, 이는 메모리 누수 원인이 될 수 있음(Destroy 미호출).
MonsterSpawner.cs양호
스폰/디스폰 타이머, 레벨 기반 등급 확률, 배틀 씬 전환(Additive), 인게임 씬 루트 오브젝트 숨기기/복구 로직이 깔끔하게 구현됨. DespawnDistantMonsters()에서 sqrMagnitude를 사용한 성능 최적화가 적절함. 다만 주석 처리된 이전 코드(128-131행, 141-165행)가 남아있어 정리 필요. UpdateProbabilityValues()(88행)에서 LevelManager.Instance null 체크 없이 접근하여 런타임 에러 위험.
Monster.cs개선필요
Wander/Flee AI, 등급별 비주얼, 스프라이트 로딩이 잘 구현됨. 그러나 ApplyVisualByGradeType()(138-241행)이 100줄 이상의 거대 메서드로, SpriteRenderer 경로/Material 경로/Quad 경로를 모두 처리하면서 가독성이 매우 떨어짐. Epic Slime에 대한 하드코딩된 오프셋(149행: -0.35f)은 매직 넘버로, Inspector에서 설정 가능하게 해야 함. Resources.Load를 매번 호출하는데 캐싱이 없어 동일 스프라이트를 반복 로드할 수 있음.
BattleSelectUI.cs개선필요
몬스터 선택 UI, 리스트 슬라이드 애니메이션, 레이캐스트 클릭 선택이 잘 동작함. RefreshList()에서 매 0.3초마다 기존 리스트를 전부 Destroy하고 다시 Instantiate하는 것은 GC 압력이 큼(Object Pooling 권장). 399줄로 파일 크기 제한 초과. GetKoreanFont()의 플랫폼별 분기 처리는 좋으나, Android에서 'NanumGothic'이 없을 경우 fallback이 LegacyRuntime.ttf뿐이라 한글이 깨질 수 있음.
GpsMapPlayerMover.cs양호
GPS→월드 좌표 변환, 에디터 모킹(WASD), GPS 노이즈 필터링(_minMoveThresholdMeters), 이동 상태 감지, 부드러운 위치/회전 보간이 전문적으로 구현됨. 341줄이지만 GPS 시스템의 복잡성을 감안하면 적절한 수준. WGS84 좌표계 변환(GetMeterOffset)이 정확하고, _maxAcceptedHorizontalAccuracy로 GPS 점프 방지도 잘 되어있음.
InventoryController.cs개선필요
아이템 추가/장착/해제 기본 로직은 동작하나, using NUnit.Framework.Interfaces(1행)가 프로덕션 코드에 포함되어 있어 빌드 시 문제 발생 가능. 변수명 'OutPanul'은 'OutPanel'의 오타. 슬롯이 꽉 찼을 때 AddItem()이 단순히 Debug.Log만 하고 사용자에게 피드백을 주지 않음. 아이템 삭제/버리기 기능이 구현되어 있지 않아 인벤토리가 가득 차면 게임 진행에 영향.
LevelManager.cs개선필요
42줄의 매우 단순한 레벨 시스템. AddExp()에서 ExpUIController.Instance(30행)를 null 체크 없이 호출하여, ExpUIController가 없는 씬(BattleScene 등)에서 NullReferenceException 발생. 레벨/경험치 데이터가 메모리에만 존재하고 저장되지 않아 앱 종료 시 모든 진행이 소실됨. DontDestroyOnLoad가 없어 씬 전환 시 파괴될 수 있음.
DefeatController.cs양호
패배 UI 표시/숨기기, BattleManager 부재 시 안전한 fallback 처리(73행: 씬 유효성 확인 후 언로드), 카메라 복구까지 방어적 코딩이 잘 되어있음. 104줄로 적절한 크기. Update()에서 매 프레임 DefeatUI.activeInHierarchy를 체크하는 것은 약간 비효율적이나 심각하지 않음.

아키텍처 평가

전체 아키텍처는 Unity의 전형적인 MonoBehaviour + Singleton 패턴 기반. BattleManager, LevelManager, InventoryController, VictoryController, DefeatController, ExpUIController 총 6개의 Singleton이 존재하며, 이들이 서로를 Instance로 직접 참조하는 강결합 구조. 씬 간 데이터 전달은 BattleSelectionContext(순수 static 클래스)를 사용하는데, 이는 가볍고 적절한 선택임. 씬 전환은 Additive Loading(BattleScene)을 사용하며, HideInGameSceneRootsExcept/RestoreInGameSceneRoots로 기존 씬을 토글하는 방식인데, 이 패턴은 작동하지만 확장성이 낮고 경쟁 상태(race condition)에 취약함. 네임스페이스를 ARRPG.Inventory, ARRPG.Level, ARRPG.Map, ARRPG.Victory, ARRPG.LimChaeSung으로 분리한 것은 좋으나, 가장 핵심인 BattleManager, Monster, MonsterSpawner가 글로벌 네임스페이스에 있어 설계 의도가 불완전함. 이벤트 시스템(Observer 패턴)이 전혀 사용되지 않아 모든 통신이 직접 참조로 이루어지고 있음.
