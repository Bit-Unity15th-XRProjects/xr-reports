프로젝트 현황 요약

이 팀은 AR 위치기반 몬스터 RPG를 개발하고 있으며, GPS 기반 맵 이동, 몬스터 스폰/배틀, 레벨링, 인벤토리의 핵심 게임 루프를 구현했습니다. BattleManager의 HP/XP 레벨 스케일링, MonsterSpawner의 등급 확률 시스템, GpsMapPlayerMover의 Haversine 기반 좌표 변환 등 기술적으로 의미 있는 로직들이 작동하고 있습니다. 그러나 GPS 시스템이 실제 안드로이드 기기에서 정상 동작할지 검증되지 않았고(_isInitialized 플래그 문제, 안드로이드 GPS 코드 경로 부재), MapGridGenerator의 폴백 위도 오타(7.49→37.49)와 같은 치명적 버그가 존재합니다. SaveManager와 SoundManager가 완전히 부재하여 게임 저장과 오디오가 없으며, 11일간 커밋이 없는 상태로 마감이 다가오고 있습니다. 가장 시급한 것은 GPS 플로우 수정과 실기기 테스트, 그리고 PlayerPrefs 기반 최소 저장 시스템 구현입니다.

리스크 알림

GPS 시스템이 실제 안드로이드 기기에서 동작하지 않을 가능성
GpsMapPlayerMover의 _isInitialized 플래그 문제와 안드로이드 빌드에서 GPS 위치 업데이트 코드 경로 부재로, 위치기반 AR RPG의 핵심 기능인 GPS 이동이 실기기에서 작동하지 않을 수 있습니다. MapGridGenerator의 위도 오타(7.49 vs 37.49)도 GPS 실패 시 완전히 잘못된 지도를 표시하게 만듭니다.
  • GpsMapPlayerMover.cs에서 SyncWithMap() 호출 후 _isInitialized = true가 정상 설정되는지 디버그 로그로 확인
  • 안드로이드 #else 블록에 Input.location.lastData 기반 실제 GPS 업데이트 로직 추가
  • MapGridGenerator.cs의 폴백 위도를 37.494920f로 수정
  • 실제 안드로이드 기기에서 GPS 연동 테스트 수행
11일간 커밋 없음 - 마감 전 개발 중단 위험
마지막 커밋이 2026-03-20이고 현재 4월 1일로 11일간 활동이 없습니다. 3명의 팀원 모두 비활성 상태이며, LimChaesung 브랜치에 병합되지 않은 10개 커밋이 있어 코드가 분산되어 있습니다. SaveManager, SoundManager 등 필수 시스템이 미구현 상태에서 개발이 멈춘 것은 제출 준비에 심각한 위험입니다.
  • 팀 미팅을 즉시 잡고 남은 작업 목록과 역할 분담 재정리
  • LimChaesung 브랜치의 10개 커밋을 main에 병합하여 코드 통합
  • SaveManager와 SoundManager를 최소 기능으로 구현 (PlayerPrefs 기반 저장, AudioSource 기반 효과음)
Shader.Find() null 반환으로 인한 런타임 크래시
MonsterSelectableOutline.cs와 Monster.cs에서 Shader.Find()의 반환값을 검증하지 않고 new Material()에 전달합니다. URP 프로젝트에서 빌드 시 사용하지 않는 셰이더가 스트리핑되면 모든 셰이더가 null을 반환할 수 있어, 몬스터 렌더링 시 앱이 크래시합니다.
  • Shader.Find() 후 null 체크를 추가하고, null일 경우 기본 머티리얼 사용하도록 폴백 구현
  • Project Settings > Graphics에서 Always Included Shaders에 'Unlit/Color'와 'Sprites/Default' 추가
게임 진행 저장 기능 부재
LevelManager의 currentLevel/currentExp, InventoryController의 아이템 데이터가 메모리에만 존재하여 앱 종료 시 모든 진행이 초기화됩니다. 시연 중 앱이 백그라운드로 가거나 크래시하면 처음부터 다시 시작해야 합니다.
  • PlayerPrefs 기반 간단한 SaveManager 구현 (레벨, 경험치, 인벤토리 JSON 직렬화)
  • OnApplicationPause()와 OnApplicationQuit()에서 자동 저장 호출
  • 앱 시작 시 저장 데이터 로드
오디오 에셋 완전 부재
오디오 파일 0개, SoundManager 미구현으로 게임에 소리가 전혀 없습니다. AR RPG 시연에서 전투 효과음, BGM, UI 사운드가 없으면 프레젠테이션 품질이 크게 저하됩니다.
  • 무료 에셋(freesound.org, Unity Asset Store 무료)에서 전투 타격음, 승리 팬파레, 배경음 최소 3개 확보
  • 간단한 SoundManager 싱글톤 구현 (AudioSource.PlayOneShot 기반)
  • BattleManager의 TryHitMonster()와 VictoryData()에서 효과음 재생 호출 추가

권고사항 (Top 3)

1. SaveManager 최소 구현으로 게임 진행 보존
시연 중 앱 재시작 시 레벨 1부터 다시 시작해야 하는 것은 치명적입니다. PlayerPrefs 기반 간단한 저장으로 핵심 데이터만 보존하면 시연 안정성이 크게 향상됩니다.
  1. SaveManager.cs 싱글톤 생성 - PlayerPrefs.SetInt('Level'), PlayerPrefs.SetFloat('Exp') 사용
  2. 인벤토리는 JsonUtility.ToJson()으로 직렬화하여 PlayerPrefs.SetString('Inventory')에 저장
  3. LevelManager.Awake()에서 저장 데이터 로드, LevelManager.LevelUp()에서 자동 저장
  4. OnApplicationPause(true)에서 SaveManager.Save() 호출
2. GPS 초기화 플로우 수정 및 실기기 테스트
위치기반 AR RPG의 핵심인 GPS 이동이 실기기에서 동작하지 않으면 프로젝트의 존재 의의가 없어집니다. 에디터에서만 테스트하고 있는 현재 상태는 위험합니다.
  1. GpsMapPlayerMover.cs의 Update() #else 블록에 Input.location.lastData 기반 위치 업데이트 코드 추가
  2. MapGridGenerator에서 SyncWithMap() 호출 전 GpsMapPlayerMover.Start() 코루틴 완료를 보장하는 초기화 순서 정리
  3. MapGridGenerator.cs 폴백 위도 7.494920f → 37.494920f 수정
  4. 안드로이드 실기기에서 GPS 연동 E2E 테스트 수행
3. SoundManager 최소 구현으로 시연 품질 향상
소리가 전혀 없는 게임은 시연 시 몰입감이 크게 떨어집니다. 최소 3개 효과음(타격, 승리, UI 클릭)만 추가해도 프레젠테이션 품질이 눈에 띄게 개선됩니다.
  1. SoundManager.cs 싱글톤 생성 - public AudioClip hitSfx, victorySfx, buttonSfx 필드와 AudioSource.PlayOneShot() 메서드
  2. 무료 효과음 3-5개를 Resources/Audio에 배치
  3. BattleManager.TryHitMonster()에서 타격 시 SoundManager.Instance.PlayHit() 호출
  4. DontDestroyOnLoad으로 씬 전환 시에도 유지

코드 리뷰

BattleManager.cs개선필요
핵심 전투 시스템의 중심 클래스로 403줄에 달해 권장 300줄을 초과합니다. GetScaledMaxHp(), GetScaledXpReward() 등 레벨 스케일링 로직은 수학적으로 정확하게 구현되어 있고, RollWeaponDrop()의 등급별 확률 시스템도 잘 동작합니다. 그러나 GetRewardItem()이 항상 null을 반환하는 데드코드이고, CreateRandomItem()에서 grade를 int로 받아 MonsterGrade enum과 타입 불일치가 발생합니다. VictoryController.Instance의 ItemText/ExpText를 직접 수정하는 강결합이 있으며, _victoryWeaponDropped 필드의 로직이 불완전합니다. TimeOutDefeat()→ReturnToFieldWithoutReward() 호출 체인이 혼란스럽습니다.
MonsterSpawner.cs개선필요
몬스터 스폰/디스폰, 레벨 기반 등급 확률, Additive 씬 로딩을 통한 전투 진입을 잘 구현했습니다. DecideGradeByLevel()의 확률 테이블(Lv<10: 90/8/2, Lv<30: 60/30/10, Lv30+: 30/40/30)이 명확합니다. 그러나 정적 변수 FieldPaused, FieldCameraRef, ArCameraRef가 씬 언로드 시 정리되지 않아 상태 누수 위험이 있습니다. HideInGameSceneRootsExcept()가 모든 루트 GameObject를 숨기므로 UI Canvas나 필수 오브젝트까지 숨길 수 있는 위험이 있습니다. 주석 처리된 코드(128-131줄)가 정리되지 않았습니다.
GpsMapPlayerMover.cs심각
GPS→월드좌표 변환의 Haversine 공식(EarthRadius=6378137.0)과 노이즈 필터링(_minMoveThresholdMeters=0.3) 구현은 기술적으로 우수합니다. 에디터 WASD 모의이동도 잘 동작합니다. 그러나 치명적 문제가 있습니다: _isInitialized가 InitializeWithMap() 또는 SyncWithMap()에서만 true로 설정되는데, 이 메서드들이 호출되지 않으면 Update()가 아예 실행되지 않습니다. 또한 실제 안드로이드 빌드에서 #else 블록이 HandleTouchDrag()만 호출하고 실제 GPS 위치 업데이트 로직이 없어, AR 위치기반 게임의 핵심 기능이 모바일에서 작동하지 않을 수 있습니다.
Monster.cs개선필요
등급×타입 시스템(3등급×3타입=9종), 배회AI(wanderSpeed=0.5, 재탐색 2.5초 간격), 도주 로직(10초 후 fleeSpeed=2.5로 도주)이 잘 구현되어 있습니다. LoadMonsterSprite()의 Resources 경로 규칙('Monsters/{Type}/{Grade}-{Type}')도 체계적입니다. 그러나 스프라이트 로딩 실패 시 릴리스 빌드에서 경고가 출력되지 않고(#if UNITY_EDITOR 내부에만 Debug.LogWarning), Shader.Find()가 null을 반환할 때 Material 생성이 크래시할 수 있습니다. Epic Slime에 대한 하드코딩된 오프셋(-0.35, 0, 0)은 문서화가 필요합니다.
BattleSelectUI.cs개선필요
399줄로 파일 크기 제한을 크게 초과합니다. 몬스터 리스트 UI 동적 생성, 슬라이드 애니메이션(코루틴), 등급별 텍스트 색상, 한국어 폰트 폴백(Malgun Gothic/AppleGothic) 등 UI 기능이 풍부합니다. RefreshList()가 0.3초마다 호출되며 매번 GameObject를 생성/삭제하는데, 오브젝트 풀링을 사용하면 성능이 개선됩니다. listSlidePanel과 rightPanel 사이의 전환 로직이 여러 곳에 분산되어 복잡합니다.
InventoryController.cs개선필요
슬롯 기반 인벤토리와 장착 시스템이 기본적으로 동작합니다. GetSlotNumber()로 슬롯 정렬, ToggleEquip()으로 장착/해제 토글이 구현되어 있습니다. 그러나 인벤토리가 가득 찼을 때 아이템이 조용히 사라지는 문제(Debug.Log만 출력), AddItem()에서 iconImageComponent null 체크 누락, Additive 씬에서 Singleton이 제대로 동작하지 않을 수 있는 문제가 있습니다. 아이템 삭제(쓰래기통) 기능은 커밋 메시지로 보아 구현되어 있으나 코드에서 확인 필요합니다.
MapGridGenerator.cs심각
OSM Slippy Map 타일 좌표 계산과 3×3 그리드 생성 로직은 기술적으로 올바르게 구현되어 있습니다. 그러나 GPS 실패 시 폴백 위도가 7.494920f로 되어 있어 37.494920f의 오타입니다. 이 경우 아프리카 지역의 지도 타일을 로드하게 됩니다. 또한 GpsMapPlayerMover.SyncWithMap() 호출 시 해당 컴포넌트의 Start() 코루틴이 완료되지 않았을 수 있는 레이스 컨디션이 존재합니다.
VictoryController.cs심각
30줄의 매우 단순한 클래스로 VictoryUI, ItemText, ExpText 참조만 보유합니다. ToggleVictory()가 이름과 달리 항상 SetActive(false)만 호출하여 토글 기능이 없는 버그가 있습니다. BattleManager가 이 클래스의 public 필드를 직접 수정하는 구조로, 캡슐화가 전혀 없습니다. 승리 화면에서 필드 복귀 버튼이나 확인 다이얼로그가 없어 UX가 불완전합니다.

아키텍처 평가

프로젝트는 Singleton 패턴을 주요 설계 패턴으로 사용합니다(BattleManager, LevelManager, InventoryController, VictoryController, ExpUIController). 네임스페이스(ARRPG.Inventory, ARRPG.Level, ARRPG.Victory, ARRPG.LimChaeSung, ARRPG.Map)와 asmdef(9개)로 모듈을 분리한 것은 학생 프로젝트로서 우수한 구조입니다. 씬 전환은 Additive 로딩 방식을 채택하여 InGame 씬 위에 BattleScene을 겹쳐 로드하고, MonsterSpawner.HideInGameSceneRootsExcept()로 필드 오브젝트를 숨기는 방식인데, 이는 창의적이나 정적 상태 관리가 취약합니다. 데이터 흐름은 BattleSelectionContext 정적 클래스를 통해 씬 간 전달되며, BattleManager가 VictoryController의 내부 UI 필드를 직접 수정하는 등 강결합이 존재합니다. Observer/Event 패턴 없이 직접 참조로 통신하여, 시스템 확장이나 독립 테스트가 어려운 구조입니다.

구현된 핵심 시스템: (1) BattleManager - 몬스터 HP 레벨 스케일링, 클릭 데미지, 제한시간 전투, 무기 드롭 확률 시스템이 실제 동작하는 로직으로 구현됨. (2) Monster - 등급(Normal/Rare/Epic)×타입(Slime/Beast/Golem) 시스템, 배회AI, 도주 로직, Resources 기반 스프라이트 로딩까지 완성. (3) MonsterSpawner - 레벨 기반 등급 확률 테이블(Lv10 미만 90/8/2%, Lv30+ 30/40/30%), 자동 스폰/디스폰 구현. (4) LevelManager - 경험치 누적, 레벨업(maxExp × 1.2 스케일링) 동작. (5) InventoryController - 슬롯 기반 아이템 관리, 장착/해제 토글 구현. (6) GpsMapPlayerMover - GPS→월드좌표 변환(Haversine 공식), 에디터 모의이동, 노이즈 필터링 구현. (7) MapGridGenerator - OSM 타일 3×3 그리드 생성, Slippy Map 좌표계 사용. 그러나 SaveManager와 SoundManager는 완전히 부재하며, VictoryController는 UI 홀더 수준의 스텁에 가까움.
통합 15/25점

ShaderGraph VFX 작업 진행 중(최근 커밋 '[Quote] making vfx'). 그러나 오디오 에셋 0개, 3D 모델(fbx) 0개, 커스텀 셰이더 0개로 프레젠테이션 품질이 부족. BGM/SFX 없는 AR 게임은 시연 시 임팩트가 크게 떨어짐.

CRITICALMapGridGenerator.csGPS 실패 시 폴백 위도가 7.494920f로 설정되어 있음. 서울 좌표 37.494920f의 오타로, 이 값이 사용되면 아프리카 지역(적도 부근)의 지도 타일을 로드하게 됨
CRITICALGpsMapPlayerMover.cs_isInitialized 변수가 InitializeWithMap() 또는 SyncWithMap() 외부에서 true로 설정되지 않아, 이 메서드들이 호출되기 전까지 Update()의 모든 로직이 실행되지 않음. 실제 안드로이드에서 GPS 위치 업데이트 코드 경로가 없어 위치기반 기능이 동작하지 않을 수 있음
CRITICALVictoryController.csToggleVictory() 메서드가 VictoryUI.SetActive(false)만 호출하여 토글이 아닌 항상 비활성화만 수행함. VictoryUI.SetActive(!VictoryUI.activeSelf)로 수정 필요
HIGHMonsterSelectableOutline.csCreateOutlineMaterial()에서 Shader.Find()가 4개 셰이더 모두 null을 반환할 경우 new Material(null)이 호출되어 NullReferenceException 발생. URP 프로젝트에서 Legacy 셰이더가 스트리핑될 수 있음

MEDIUMBattleManager.csGetRewardItem() 메서드가 항상 null을 반환하는 데드코드. CreateRandomItem()에서 grade를 int로 받아 MonsterGrade enum과 타입 불일치 발생
MEDIUMInventoryController.csAddItem()에서 인벤토리가 가득 찼을 때 아이템이 조용히 사라짐(Debug.Log만 출력). 유저에게 피드백 없이 보상 아이템이 소실될 수 있음
MEDIUMMonsterSpawner.cs정적 변수 FieldPaused, FieldCameraRef, ArCameraRef가 씬 언로드 후에도 이전 값을 유지하여 재진입 시 잘못된 상태로 시작될 수 있음
