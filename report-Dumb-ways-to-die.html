프로젝트 현황 요약

이 팀은 'Dumb Ways to Die' 컨셉의 VR 블랙코미디 데스게임을 만들고 있으며, '죽음이 곧 성공'이라는 독특한 게임 디자인을 채택했습니다. 현재 코어 프레임워크(GameManager, VRRigBuilder, GameBootstrap)와 사망 연출 시스템(DeathCameraEffect, PhysicsKnockback, 8종의 PostProcess 이펙트)이 높은 완성도로 구현되어 있습니다. StageCommonSystem의 유혹 팝업, PlayerInteractionBlocker의 선택적 차단, 3종 햅틱 모드 등 VR 특화 기능도 세심합니다. 가장 부족한 부분은 콘텐츠 에셋(오디오 4개, 애니메이션 1개)과 동기식 씬 로딩으로 인한 VR 멀미 위험입니다. StageScaffolder 에디터 도구로 새 스테이지 추가가 자동화되어 있어 확장성은 좋지만, Shark 스테이지(4929줄 변경)의 병합과 Quest 2 성능 최적화가 남은 가장 시급한 과제입니다.
!

리스크 알림

Quest 2 성능 최적화 미검증
GasTankFlameEffect의 다중 VFX 스폰, DeathOverlayRenderer의 실시간 RGB 분리, 8종의 PostProcess Volume 스태킹 등이 Quest 2의 제한된 GPU에서 72fps를 유지할 수 있는지 검증되지 않았습니다. VR에서 프레임 드롭은 곧 멀미입니다.
  • Unity Profiler로 각 사망 이펙트의 GPU 시간 측정
  • GasTankFlameEffect의 spawnCount를 Quest 2용으로 제한 (최대 5개)
  • PostProcess Volume의 priority를 정리하고 동시 활성 Volume 수를 2개 이하로 제한
  • OVR Metrics Tool로 실제 디바이스에서 프레임 타이밍 확인
동기식 씬 로딩으로 인한 VR 멀미
GameManager.LoadStage(), LoadLobby(), LoadTitle()이 모두 SceneManager.LoadScene()을 동기식으로 호출합니다. Quest 2에서 씬 로딩 중 렌더링이 멈추면 검은 화면이 수 초간 지속되어 심한 멀미를 유발합니다.
  • LoadSceneAsync로 전환하고 passthrough 또는 로딩 화면 구현
  • 씬 전환 시 페이드 아웃→로딩→페이드 인 시퀀스 추가
  • VRCanvasAdapter에 로딩 UI 프리팹 연동
Shark 스테이지 브랜치 병합 리스크
origin/stage/shark 브랜치가 main에서 분기되어 49개 파일, 4929줄 변경이 있습니다. Suimono 에셋, PostProcess 설정, 새로운 AI 시스템 등이 포함되어 있어 병합 시 충돌과 프로젝트 설정 꼬임이 예상됩니다.
  • main을 먼저 정리하고 shark 브랜치에 rebase
  • ProjectSettings와 EditorBuildSettings.asset 충돌 수동 해결 준비
  • 병합 후 전체 스테이지 플레이 테스트
콘텐츠 에셋 부족 (오디오/애니메이션)
오디오 4개, 애니메이션 1개로 5개 이상의 스테이지를 지원하기에 크게 부족합니다. 사망 이펙트 코드는 8종이 구현되어 있지만 실제 재생할 사운드가 없으면 몰입감이 크게 떨어집니다.
  • 스테이지별 필수 사운드 목록 작성 (폭발, 전기, 먹기, BGM 등)
  • 무료 에셋 또는 자체 제작으로 최소 스테이지당 3개 사운드 확보
  • BGMManager에 스테이지별 배경음악 할당
Hot File 충돌 위험
Mona12 SDF.asset(24회), EditorBuildSettings.asset(21회), Stage1.unity(20회) 등이 매우 자주 변경되고 있습니다. 바이너리 파일(.asset, .unity)은 Git 머지가 불가능하여 동시 작업 시 항상 한 쪽 변경이 손실됩니다.
  • Scene 파일 동시 편집 금지 규칙 수립
  • EditorBuildSettings.asset는 StageScaffolder 에디터 도구로만 수정
  • Font SDF 에셋은 한 명만 담당하도록 역할 분담

권고사항 (Top 3)

1. 비동기 씬 로딩 + 전환 연출 구현
VR에서 동기식 씬 로딩은 멀미의 직접적 원인입니다. GameManager에 이미 LoadTitle/LoadLobby/LoadStage 메서드가 있으므로 이를 비동기로 전환하면 됩니다.
  1. GameManager의 SceneManager.LoadScene을 LoadSceneAsync로 변경
  2. AsyncOperation.allowSceneActivation을 활용한 로딩 진행률 관리
  3. VRCanvasAdapter에 페이드 인/아웃 UI 캔버스 추가
  4. 씬 전환 시 VR passthrough 또는 검은색 페이드 적용
2. 글로벌 네임스페이스 파일 정리
StageHapticPlayer, ResultScreenSystem, LazyFollowUI, VRRigBuilder 등이 네임스페이스 없이 선언되어 있어 프로젝트가 커지면 이름 충돌이 발생합니다. 이미 DumbWays.Gameplay, DumbWays.UI 네임스페이스가 정의되어 있으므로 통일만 하면 됩니다.
  1. StageHapticPlayer → DumbWays.Gameplay 네임스페이스 추가
  2. ResultScreenSystem → DumbWays.Gameplay.StageEnd 네임스페이스 추가
  3. LazyFollowUI → DumbWays.UI 네임스페이스 추가
  4. VRRigBuilder, GameManager, GameBootstrap → DumbWays.Core 네임스페이스 추가
  5. asmdef의 rootNamespace 설정 확인
3. 스테이지별 사운드 에셋 우선 확보
StageSFXPlayer, StageVFXPlayer, StageHapticPlayer 시스템이 잘 구현되어 있지만 실제 오디오가 4개뿐입니다. 코드 인프라는 준비되어 있으므로 에셋만 추가하면 즉시 몰입감이 향상됩니다.
  1. 스테이지별 필수 사운드 목록 작성: 폭발(GasTank), 전자레인지 윙(Microwave), 씹는 소리(Eat), 불타는 소리(Fire), 상어(Shark)
  2. freesound.org 등에서 CC0 사운드 수집 또는 Unity Asset Store 무료 팩 활용
  3. 각 스테이지 StageSFXPlayer Inspector에 클립 할당
  4. BGMManager에 로비/타이틀 배경음악 추가

코드 리뷰

DeathCameraEffect.cs양호
사망 시퀀스의 핵심 오케스트레이터로, Normal Death(슬로모션→시체고정→트래킹해제→카메라상승→결과패널)와 Knockback Death(물리시뮬→수평복원→상승) 두 경로를 코루틴으로 깔끔하게 분기합니다. DeathBodyController, PhysicsKnockback, HeadTrackingController를 컴포지션으로 조합하는 설계가 우수합니다. WaitForSecondsRealtime 사용으로 슬로모션 중에도 정확한 타이밍을 보장합니다. OnCameraRiseComplete 이벤트로 VFX 타이밍을 외부에 위임하는 점도 좋습니다.
PhysicsKnockback.cs양호
폭발 넉백의 물리 시뮬레이션을 안전하게 관리합니다. maxDisplacementRadius(2.5m), maxVerticalSpeed(3m/s), maxSpeed(6m/s) 등 다중 안전장치가 VR 멀미를 방지합니다. KnockbackHandler가 프레임마다 ClampSpeed, ClampDisplacement, EnforceWallBounds를 실행하여 플레이어가 벽을 뚫거나 맵 밖으로 날아가는 것을 방지합니다. GasTankDeathTrigger.OnExplosionStarted와 MicrowaveDeathTrigger.OnExplosionStarted에 자동 구독하는 설계가 깔끔합니다.
VRRigBuilder.cs개선필요
XR Origin 자동 생성과 영구 EventSystem 관리가 잘 되어 있고, OnRigReady의 sticky 이벤트 패턴(이미 준비됐으면 즉시 호출)이 레이스 컨디션을 방지합니다. 하지만 XRRigPrefabPath가 'Assets/Samples/XR Interaction Toolkit/3.3.0/...'으로 하드코딩되어 있어 XRI 버전 업그레이드 시 깨질 수 있습니다. Resources.Load나 Addressable로 변경하는 것을 권장합니다.
StageCommonSystem.cs양호
스테이지의 중앙 허브로서 역할이 명확합니다. TriggerDeath()가 OnPlayerDeath 이벤트를 발행하고 GameManager에 결과를 기록합니다. 유혹 팝업 시스템(7초 간격, 3초 표시)이 게임 컨셉('죽음이 곧 성공')과 잘 맞습니다. PlayerInteractionBlocker와 연동하여 사망 후 상호작용을 차단하면서 UI는 보존하는 점이 좋습니다.
StageHapticPlayer.cs개선필요
3가지 햅틱 모드(SinglePulse, Pattern, Ramp)가 잘 설계되었고, HapticImpulsePlayer 자동 탐색 로직이 편리합니다. 하지만 네임스페이스가 없어 글로벌 스코프에 노출되어 있습니다. 또한 Play() 메서드에서 이전 코루틴을 자동 정지하지 않아 같은 index에 대해 중복 재생이 발생할 수 있습니다. activeCoroutine null 체크 후 StopCoroutine을 호출하는 가드가 필요합니다.
MicrowaveDeathEffect.cs양호
3단계(충전→섬광→연기) PostProcess 연출이 세밀하게 구현되어 있습니다. 각 Phase별 독립 Volume을 사용하여 블렌딩이 자연스럽고, MicrowaveDeathTrigger의 OnChargingStarted/OnExplosionStarted 이벤트에 구독/해제가 OnEnable/OnDisable에서 올바르게 처리됩니다. GasTankDeathEffect와 구조가 유사하여 공통 부분을 베이스 클래스로 추출할 수 있지만, 현재 수준에서도 충분히 읽기 좋습니다.
ResultScreenSystem.cs개선필요
사망 결과 패널 표시, 재시도/로비 버튼 연결이 잘 되어 있고, ApplyAlwaysOnTop()으로 ZTest Always를 적용하는 점이 VR에서 중요합니다. 하지만 네임스페이스가 없고, FindTMP()와 WireButton()에서 자식 오브젝트를 이름 문자열로 찾는 패턴(transform.Find('DeathMessageText'))이 오타에 취약합니다. SerializeField로 직접 참조하는 것이 더 안전합니다.
GasTankFlameEffect.cs개선필요
카메라 주변 불꽃 VFX 스폰 로직이 잘 구현되어 있으나, 바닥 Raycast(floorRayMaxDist=20f)에서 hit가 없을 때의 폴백 처리가 불명확합니다. 또한 spawnRadius=3f로 카메라 근접 스폰 시 VR 멀미 유발 가능성이 있습니다. 불꽃 수(spawnCount)가 많을 경우 Quest 2의 GPU 성능을 고려해야 합니다.

아키텍처 평가

Observer 패턴을 중심으로 한 이벤트 드리븐 아키텍처가 프로젝트 전반에 걸쳐 일관되게 적용되어 있습니다(StageCommonSystem.OnPlayerDeath → DeathCameraEffect → OnCameraRiseComplete → GasTankFlameEffect/DeathObjectHider 체인). VContainer를 통한 DI가 StageScope에서 DeathTrigger들을 자동 등록하여 새 스테이지 추가 시 보일러플레이트를 최소화합니다. Template Method 패턴이 BaseDeathEffect/BaseDeathOverlay에서 사용되어 8종의 사망 이펙트가 공통 라이프사이클(OnDeath→OnResultDimComplete)을 공유하면서 각자 고유한 PostProcess를 구성합니다. 싱글톤은 VRRigBuilder, GameManager, VRCanvasAdapter 3개로 절제되어 있고, GameBootstrap이 초기화 순서를 명시적으로 관리합니다. 개선점으로는 GasTankDeathEffect와 MicrowaveDeathEffect의 섬광→연기 시퀀스가 거의 동일하므로 ExplosionDeathEffectBase 추출을 고려할 수 있으며, 씬 로딩이 동기식인 점은 아키텍처적으로 가장 큰 약점입니다.

단계별 진행률 (AI 평가)

기반 구축 5/5점
VRRigBuilder(Singleton, DontDestroyOnLoad)가 XR Origin을 자동 생성하고, GameBootstrap이 RuntimeInitializeOnLoadMethod로 싱글톤 초기화 순서를 보장합니다. VRCanvasAdapter가 모든 Canvas를 WorldSpace로 자동 변환하며, 7개의 asmdef로 어셈블리가 깔끔하게 분리되어 있습니다. XR Interaction Toolkit 3.3.0+, VContainer DI, URP 파이프라인 등 인프라가 빈틈없이 세팅되어 있어 만점입니다.
핵심 시스템 20/25점
GameManager(씬 전환, 스테이지 진행, StageRecord[] 관리), StageCommonSystem(사망 트리거, 유혹 팝업), DeathCameraEffect(슬로모션→시체고정→카메라상승→결과패널 시퀀스 오케스트레이션), PhysicsKnockback(Rigidbody 부착→물리시뮬→안전클램프), ResultScreenSystem(사망메시지+재시도/로비 버튼), PlayerInteractionBlocker(사망시 이동·그랩 차단, UI 상호작용 보존), SaveManager(JSON 저장/로드) 등 핵심 시스템이 실제 동작하는 로직으로 구현되어 있습니다. 다만 독립적인 ScoreManager가 없고(GameManager.StageRecord[].cleared만 존재), 스테이지별 난이도 조절이나 타이머 시스템이 부재합니다.
통합 18/25점
VContainer의 StageScope가 모든 DeathTrigger를 자동 발견·등록하여 StageCommonSystem에 주입합니다. 이벤트 체인이 명확합니다: StageCommonSystem.OnPlayerDeath → DeathCameraEffect.Play() → OnCameraRiseComplete → GasTankFlameEffect/DeathObjectHider 등. GameManager가 Title→Lobby→Stage 씬 전환을 관리하고, GameBootstrap이 어떤 씬에서든 직접 실행을 지원합니다. XROriginSceneReset이 씬 전환시 Origin 위치를 초기화합니다. 다만 SceneManager.LoadScene이 동기식이라 로딩 화면이 없고, 스테이지 간 전환 연출이 부재합니다.
콘텐츠 14/20점
사망 이펙트 8종(Default, Bruise, Burn, Burn_Melt, Electrocution, GasTank, Microwave, Mushroom)이 각각 고유한 PostProcess 연출을 가지고 있습니다. StageSFXPlayer/StageVFXPlayer/StageHapticPlayer가 스테이지별 오디오·파티클·진동을 관리합니다. 에셋 현황: FBX 14개, 오디오 4개, 머티리얼 52개, 프리팹 30개, 텍스처 48개. 다만 오디오 파일이 4개뿐이고 애니메이션이 1개뿐이라 사운드와 애니메이션 콘텐츠가 부족합니다. 코드 이슈 (9건)

심각도파일설명
HIGHVRRigBuilder.csXR Rig 프리팹 경로가 'Assets/Samples/XR Interaction Toolkit/3.3.0/...'으로 하드코딩되어 있어, XRI 패키지 버전 업그레이드 시 경로가 변경되면 런타임에 프리팹 로드 실패로 XR Origin이 생성되지 않습니다.
HIGHGameManager.csSceneManager.LoadScene()이 동기식으로 호출되어 씬 전환 시 프레임 드롭이 발생합니다. Quest 2에서 VR 프레임 드롭은 멀미를 유발하므로 LoadSceneAsync + 로딩 화면이 필요합니다.
HIGH전체 프로젝트오디오 파일이 4개뿐입니다. VR 블랙코미디 데스게임에서 사망 이펙트별 고유 사운드, 배경음악, UI 사운드 등이 크게 부족합니다. StageSFXPlayer가 잘 구현되어 있지만 실제 오디오 에셋이 거의 없습니다.
MEDIUMStageHapticPlayer.cs, ResultScreenSystem.cs, LazyFollowUI.cs네임스페이스 없이 글로벌 스코프에 선언되어 있어 다른 패키지나 에셋과 이름 충돌 가능성이 있습니다. 프로젝트의 다른 파일들은 DumbWays.Gameplay, DumbWays.UI 등 네임스페이스를 사용 중입니다.
MEDIUMResultScreenSystem.csFindTMP(panel, 'DeathMessageText')와 WireButton(panel, 'RetryButton', ...) 등 transform.Find()로 자식을 문자열 검색하는 패턴이 반복됩니다. Inspector에서 이름을 변경하면 런타임에 null이 되어 패널이 작동하지 않습니다.
MEDIUMStageHapticPlayer.csPlay(int index) 메서드에서 동일 index의 이전 코루틴이 실행 중인지 확인하지 않아, 같은 햅틱을 빠르게 연속 호출하면 중복 재생됩니다. entry.activeCoroutine != null일 때 StopCoroutine 호출이 필요합니다.
MEDIUMHeadTrackingController.csTrackedPoseDriver 타입을 리플렉션(Type.GetType)으로 찾는데, Unity 버전에 따라 네임스페이스가 다를 수 있어 silent fail이 발생할 수 있습니다. 리플렉션 실패 시 Debug.LogWarning이 있긴 하지만, 사망 시퀀스 중 헤드트래킹이 해제되지 않으면 카메라가 정상적으로 상승하지 않습니다.
LOWPhysicsKnockback.csGasTankDeathTrigger.OnExplosionStarted와 MicrowaveDeathTrigger.OnExplosionStarted 두 이벤트에 동시 구독되어, 두 트리거가 동시에 발생하면 Apply()가 2회 호출될 수 있습니다. IsActive 플래그로 가드는 되어 있지만 명시적 방어가 더 안전합니다.
LOWDeathCameraEffect.cs사망 시퀀스가 시작되면 중단할 방법이 없습니다. 네트워크 멀티플레이어 확장이나 디버그 리셋 시 시퀀스를 강제 종료해야 할 수 있으므로 CancelSequence() 메서드가 필요합니다.
