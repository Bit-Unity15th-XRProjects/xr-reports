프로젝트 현황 요약

이 팀은 'Dumb Ways to Die' 컨셉의 VR 블랙코미디 데스게임을 개발하고 있으며, 현재 핵심 사망 시스템이 상당히 성숙한 단계에 도달했다. GameManager→StageCommonSystem→DeathTrigger→DeathEffect로 이어지는 이벤트 기반 사망 파이프라인이 완전히 동작하며, 7종의 사망 트리거(충돌, 잡기, 소켓, 먹기, 가스통 폭발, 곰덫, 전자레인지)와 8종의 시각 이펙트가 구현되어 있다. DeathCameraEffect의 슬로우모션→시체→카메라 상승 시네마틱은 전문적 수준이며, PhysicsKnockback의 물리 넉백 시뮬레이션까지 갖추고 있다. 가장 부족한 영역은 오디오 에셋(4개)과 ScoreManager(점수 체계 부재)로, 게임의 리플레이 가치와 VR 몰입감을 높이려면 이 두 가지가 가장 시급하다. 현재 stage/shark 브랜치에서 수중 환경(코스틱, 포그, 수면 셰이더)이 활발히 개발 중이며, 전자레인지 스테이지도 최근 커밋에서 몰입감 개선이 이뤄지고 있어 콘텐츠 확장이 빠르게 진행되고 있다.

리스크 알림

하드코딩된 에셋 경로로 인한 빌드 실패 위험
VRRigBuilder의 XRI 샘플 프리팹 경로와 GameManager의 StageRegistry 에셋 경로가 문자열로 하드코딩됨. XRI 버전 업그레이드 또는 폴더 구조 변경 시 에디터에서는 동작하지만 빌드에서 리그가 생성되지 않아 VR 조작 불가능 상태가 됨.
  • VRRigBuilder의 xrRigPrefab을 반드시 Inspector에서 할당하고 SerializeField 필드가 null이면 Application.Quit() 수준의 치명적 에러 로그 출력
  • GameManager의 StageRegistry를 Resources 폴더 또는 Addressables로 이동하여 런타임 로드 보장
  • 에디터 자동 탐색은 GUID 기반(AssetDatabase.GUIDToAssetPath)으로 교체
오디오 에셋 부족으로 VR 몰입감 저하
4개의 오디오 파일만 확인됨. VR에서 공간 음향은 몰입의 핵심 요소인데, 스테이지별 앰비언트, UI 효과음, 사망 효과음이 부족하면 데모 시 '미완성' 인상을 줄 수 있음. StageSFXPlayer가 구현되어 있으므로 에셋만 추가하면 즉시 활용 가능.
  • 각 스테이지별 최소 3개 효과음 확보: 환경음(앰비언트), 상호작용 피드백, 사망 효과음
  • StageSFXPlayer의 OnStageStart 트리거로 앰비언트 루프 설정
  • 폭발/전기/함정 등 스테이지 테마에 맞는 효과음 추가
싱글톤 4개의 초기화 순서 의존성
GameBootstrap에서 VRRigBuilder → GameManager → VRCanvasAdapter 순서가 강제되지만, 이 순서가 코드 주석으로만 문서화됨. 팀원이 새 싱글톤을 추가하거나 순서를 변경하면 연쇄 실패 발생 가능.
  • GameBootstrap에 초기화 순서를 enum 또는 배열로 명시적 선언
  • 각 싱글톤의 Awake()에 의존성 체크 추가 (예: VRRigBuilder.Instance != null assertion)
  • 싱글톤 생성을 하나의 Composition Root 클래스로 통합 검토
stage/shark 브랜치 미머지 (ahead 16, behind 6)
shark 스테이지가 16커밋 앞서 있고 6커밋 뒤처져 있음. 수중 효과(코스틱, 포그, 수면 셰이더 등) 작업이 main과 분리된 채 진행 중이어 머지 충돌 가능성이 높아지고 있음.
  • 이번 주 내 main을 stage/shark에 머지하여 behind 해소
  • 코드 리뷰 후 stage/shark → main PR 생성
  • 씬 파일(.unity) 충돌 방지를 위해 작업 전 최신 main pull 습관화
ScoreManager 부재로 게임 리플레이 동기 부족
현재 cleared boolean만 기록하여 '다시 하고 싶은' 동기가 없음. 데스게임 컨셉에서 사망 시간, 시도 횟수, 희귀 사망 방식 등의 메트릭이 있으면 리플레이 가치가 크게 올라감.
  • ScoreManager 클래스 생성: stageIndex별 bestTime, attempts, deathType 기록
  • SaveManager.SaveData에 스코어 필드 추가
  • ResultScreenSystem에 시간/시도 횟수 표시 UI 추가

권고사항 (Top 3)

1. 폭발 이벤트 구독을 인터페이스 기반으로 리팩토링
PhysicsKnockback이 GasTankDeathTrigger.OnExplosionStarted와 MicrowaveDeathTrigger.OnExplosionStarted에 각각 하드코딩 구독하고 있어, shark 스테이지 등 새 폭발형 트리거 추가 시 PhysicsKnockback 수정이 필수. 이는 확장에 닫혀 있는 설계.
  1. IExplosionSource 인터페이스 정의: static event Action OnExplosionStarted
  2. GasTankDeathTrigger, MicrowaveDeathTrigger가 IExplosionSource 구현
  3. 또는 StageCommonSystem에 OnExplosionTriggered 이벤트를 추가하고, 폭발 트리거들이 TriggerDeath() 전에 이 이벤트를 발행
  4. PhysicsKnockback은 단일 이벤트만 구독
2. 에셋 경로 하드코딩을 ScriptableObject 레지스트리로 교체
VRRigBuilder, GameManager, BGMManager에서 에셋 경로나 씬 이름을 문자열로 참조하고 있어 리팩토링이나 버전 업그레이드에 취약. 이미 StageRegistry ScriptableObject 패턴이 있으므로 확장하면 됨.
  1. GameSettings ScriptableObject 생성: titleSceneName, lobbySceneName, xrRigPrefab, bgmClip 등 중앙 관리
  2. GameManager, BGMManager, VRRigBuilder가 GameSettings를 참조하도록 변경
  3. 에디터 메뉴에서 GameSettings 유효성 검증 도구 추가
3. 머티리얼 인스턴스 풀링 및 정리 로직 추가
ResultScreenSystem.ApplyAlwaysOnTop()이 매 사망마다 Graphic/TMP 개수만큼 new Material을 생성하고 해제하지 않음. VR에서 반복 사망 시 메모리 증가가 프레임 드랍으로 이어질 수 있음.
  1. 생성된 Material 인스턴스를 List로 캐싱
  2. OnDisable() 또는 OnDestroy()에서 foreach Destroy(mat) 호출
  3. 또는 최초 1회만 머티리얼을 생성하고 이후 재사용하도록 캐싱 플래그 추가

GameManager.cs개선필요
씬 네비게이션(LoadTitle/LoadLobby/LoadStage/ReloadCurrentStage), 스테이지 해금 로직(IsStageUnlocked), 진행 상황 기록(RecordDeath/RecordClear)이 모두 동작하는 핵심 싱글톤. InitStageFromScene()으로 에디터에서 직접 스테이지 씬 테스트 가능한 점이 좋음. 문제점: ResolveAssets()에서 #if UNITY_EDITOR로 AssetDatabase.LoadAssetAtPath<StageRegistry>(StageRegistryPath)를 호출하는데, 에셋 경로가 하드코딩됨('Assets/_Game/Data/StageRegistry.asset'). 빌드 시에는 이 경로가 무시되므로 Inspector 미할당 시 stageRegistry가 null인 채로 런타임 진입 가능. Resources.Load 또는 Addressables로 대체 권장.
StageCommonSystem.cs양호
사망 시스템의 중앙 허브로 단일 책임 원칙을 잘 준수. TriggerDeath()에서 IsDead 가드 → 팝업 숨김 → OnPlayerDeath 이벤트 → RecordClear + RecordDeath 순서가 명확. 유혹 팝업 시스템(7초마다 랜덤 한국어 메시지 표시)이 게임 컨셉에 맞게 잘 구현됨. SuppressTemptation()으로 전자레인지 충전 중 팝업 차단하는 로직이 최근 diff에서 추가된 것도 좋은 설계. 사소한 개선: temptationMessages 배열이 비어있을 때의 방어 코드 추가 권장.
DeathCameraEffect.cs양호
NormalDeathSequence와 KnockbackDeathSequence 두 경로로 사망 시네마틱을 완전히 구현. 슬로우모션(timeScale 0.2) → DeathBodyController로 시체 고정 → HeadTrackingController.Disable() → DeathCameraAnimator.RiseAndRotate() → ResultScreenSystem.ShowAt() 순서가 정교하게 코루틴으로 조율됨. OnCameraRiseComplete 이벤트로 화염/연기 VFX 타이밍을 외부에서 동기화할 수 있는 점이 우수한 설계. 모든 코루틴이 Time.unscaledDeltaTime을 사용하여 슬로우모션 중에도 정상 동작.
MicrowaveDeathTrigger.cs개선필요
3단계 시퀀스(문 닫기 → 충전 → 폭발)가 잘 구현됨. 최근 diff에서 doorBlownOpenLocalEuler/doorBlownOpenDuration 추가, aftermathVFXPrefab으로 잔불 교체, _pot.SetActive(false)로 냄비 제거, SuppressTemptation() 호출 등 완성도 높은 개선이 이뤄짐. 문제점: (1) OnChargingProgress 이벤트가 정의되어 Invoke도 호출되지만 구독하는 클래스가 없어 데드 코드, (2) _pot.SetActive(false)만 하고 Destroy하지 않아 비활성 오브젝트가 씬에 잔류, (3) SpawnAftermath에서 explosionInstance를 Destroy한 뒤 aftermathVFXPrefab을 생성하는데 aftermathVFXPrefab이 null이 아닌지만 체크하고 aftermathDelay 동안 explosionInstance가 이미 파괴됐을 가능성 미고려.
EatDeathTrigger.cs양호
Idle→Eating→Done 상태 머신이 깔끔하게 구현됨. BeginEating()에서 XRGrab 강제 릴리즈 → Rigidbody 잠금 → 부모 해제 → 이벤트 발행 순서가 정확. EatAnimator/EatAudioPlayer/EatDebrisSpawner로 비주얼/오디오/파티클을 위임하여 SRP 준수. CancelEating()으로 먹기 중단 시 롤백 지원. MushroomDeathEffect가 OnEatingStarted/OnEatingCancelled를 구독하여 독버섯 환각 효과를 동기화하는 구조가 우아함.
VRRigBuilder.cs개선필요
XR 리그 프리팹 인스턴스화, 퍼시스턴트 EventSystem 생성, Sticky OnRigReady 이벤트 패턴이 모두 잘 설계됨. BuildRig()에서 씬의 기존 EventSystem을 비활성화한 후 새로 생성하는 순서가 정확. 문제점: XR 리그 프리팹 경로가 'Assets/Samples/XR Interaction Toolkit/3.3.0/...'으로 하드코딩되어 XRI 버전 업그레이드 시 반드시 깨짐. SerializeField로 Inspector 할당을 우선하되, 에디터 자동 탐색은 Guid 기반 AssetDatabase.GUIDToAssetPath로 대체해야 안전.
PhysicsKnockback.cs개선필요
KnockbackHandler와의 위임 구조, SimulateUntilSettled()의 속도 클램핑/경계 제한/벽 반사 물리가 잘 구현됨. OnComplete 이벤트로 DeathCameraEffect에 시뮬레이션 종료를 알려 넉백→카메라 상승을 자연스럽게 연결. 문제점: OnEnable/OnDisable에서 GasTankDeathTrigger.OnExplosionStarted와 MicrowaveDeathTrigger.OnExplosionStarted에 직접 구독하여, 새 폭발형 트리거 추가 시 이 파일도 수정해야 하는 OCP 위반. 공통 인터페이스(IExplosionSource) 또는 StageCommonSystem을 통한 중계로 개선 가능.
ResultScreenSystem.cs개선필요
ShowDeathResult()/ShowDeathResultImmediate()/ShowAt() 세 가지 표시 모드로 유연한 API 제공. ApplyAlwaysOnTop()으로 ZTest Always 머티리얼 오버라이드하여 기하 위에 항상 렌더링되는 점이 VR에 적합. 문제점: (1) FindTMP()와 WireButton()이 Transform.Find(childName)으로 UI 요소를 문자열 기반 탐색하여 프리팹 구조 변경 시 런타임 실패, (2) ApplyAlwaysOnTop()에서 매 호출마다 new Material(original)로 머티리얼 인스턴스를 생성하여 메모리 누수 가능성, (3) 캐싱 없이 매번 GetComponentsInChildren 호출.

아키텍처 평가

Observer 패턴이 아키텍처의 핵심으로, static event Action 기반의 이벤트 체인(OnPlayerDeath, OnExplosionStarted, OnCameraRiseComplete, OnEatingStarted 등)이 시스템 간 결합도를 효과적으로 낮춤. Template Method 패턴이 BaseDeathEffect/BaseDeathOverlay에서 잘 활용되어 8종의 사망 이펙트를 일관된 구조로 확장. VContainer를 활용한 DI가 StageScope → DeathTrigger 계층에 적용되어 있으나, DeathEffect 계층과 DeathCameraEffect는 여전히 FindFirstObjectByType과 싱글톤 직접 참조에 의존하여 DI 커버리지가 불균일. Composition 패턴이 DeathCameraEffect에서 두드러짐: DeathBodyController, HeadTrackingController, DeathCameraAnimator, PhysicsKnockback을 조합하여 복잡한 사망 시네마틱을 구성. 씬 전환은 단순한 SceneManager.LoadScene() 기반이며, GameBootstrap의 RuntimeInitializeOnLoadMethod로 어떤 씬에서든 Play 가능한 설계가 개발 효율을 높임.

심각도파일설명
HIGHVRRigBuilder.csXR 리그 프리팹 경로가 'Assets/Samples/XR Interaction Toolkit/3.3.0/Hands Interaction Demo/Prefabs/XR Origin Hands (XR Rig).prefab'로 하드코딩되어 있음. XRI 패키지 버전 업데이트(3.3.0 → 3.4.0 등) 시 경로가 변경되어 리그 생성 실패.
HIGHGameManager.csResolveAssets()가 #if UNITY_EDITOR 블록 안에서만 StageRegistry를 로드. 빌드에서 Inspector에 StageRegistry 미할당 시 stageRegistry가 null인 채로 동작하여 LoadStage() 등에서 NullReferenceException 발생 가능.
MEDIUMPhysicsKnockback.csOnEnable()에서 GasTankDeathTrigger.OnExplosionStarted와 MicrowaveDeathTrigger.OnExplosionStarted에 직접 구독. 새로운 폭발형 트리거 추가 시 이 파일을 반드시 수정해야 하는 Open-Closed 원칙 위반.
MEDIUMHeadTrackingController.cs매 사망마다 Type.GetType('Unity.InputSystem.XR.TrackedPoseDriver')로 리플렉션 호출. 타입 캐싱 없이 문자열 기반 탐색을 반복하여 불필요한 오버헤드 발생.
MEDIUMResultScreenSystem.csApplyAlwaysOnTop()에서 Graphic/TMP_Text마다 new Material(original)로 인스턴스 생성. 사망할 때마다 호출되면 머티리얼 인스턴스가 누적되어 메모리 누수 가능. OnDisable/OnDestroy에서 Destroy(material) 필요.
MEDIUMBGMManager.csOnSceneLoaded에서 scene.name == 'Title' || scene.name == 'Lobby' 문자열 비교로 분기. 씬 이름 오타나 변경 시 BGM 페이드가 동작하지 않으며, GameManager의 titleSceneName/lobbySceneName과 동기화되지 않음.
LOWMicrowaveDeathTrigger.csOnChargingProgress 이벤트가 정의되고 Invoke()도 호출되지만, 프로젝트 내 어떤 클래스도 구독하지 않는 데드 코드. MicrowaveDeathEffect는 OnChargingStarted만 구독.
LOWKnockbackHandler.csAttach()에서 _rb.mass = 50f가 하드코딩됨. 스테이지별로 넉백 느낌(가벼운 폭발 vs 무거운 충격)을 다르게 하려면 파라미터화 필요.
LOWMicrowaveDeathTrigger.cs_pot.SetActive(false)로 냄비를 숨기기만 하고 Destroy하지 않음. 비활성 오브젝트가 씬에 잔류하며, 씬 리로드 시까지 참조가 유지됨.
LOWDeathBodyController.csClampAboveFloor()에서 레이캐스트 파라미터(시작점 Vector3.up * 3f, 거리 10f, 바닥 오프셋 0.05f)가 매직 넘버로 하드코딩. 천장이 낮은 스테이지나 다층 구조에서 오동작 가능.
