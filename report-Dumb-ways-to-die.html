프로젝트 현황 요약

이 팀은 'Dumb Ways to Die' 컨셉의 VR 블랙코미디 데스게임을 개발하고 있으며, '죽는 것이 클리어'라는 독특한 게임 메카닉을 중심으로 아키텍처를 설계했다. 현재 상어(작살 발사→참치 흔들기→상어 유인→사망), 맨홀(뚜껑 제거→추락), 가스탱크(라이터→폭발→넉백), 전자레인지(금속 삽입→충전→폭발), 곰덫, 독버섯(먹기 상호작용) 등 최소 6개 스테이지의 고유 사망 메카닉이 구현되어 있고, 각 스테이지마다 전용 포스트프로세싱 DeathEffect와 시네마틱 DeathCameraEffect가 작동한다. VContainer DI, 이벤트 기반 아키텍처, ScriptableObject 데이터 분리, 다국어 지원(LocalizationManager)까지 갖춘 점은 학생 프로젝트로서 매우 인상적이다. 가장 시급한 것은 HarpoonGun의 null 가드 강화(발표 중 크래시 방지)와 feat-manhole 브랜치의 조기 머지, 그리고 전체 플로우(Title→Lobby→Stage→Death→Result→Lobby)의 통합 스모크 테스트다.

  • SetLayerRecursively 메서드 삭제 후 XR Origin 프리팹 자체에 Player 레이어를 미리 설정

권고사항 (Top 3)

1. HarpoonGun.cs 분할 및 null 가드 강화
550줄짜리 단일 파일은 유지보수가 어렵고, 초기화 실패 시 크래시 위험이 있음. 발표 안정성을 위해 최우선 처리 필요.
  1. AttachToRightHand()/Start()에서 필수 참조(rightHandAnchor, tunaTransform, gunTip, harpoonPrefab) null 체크 → 누락 시 LogError + enabled = false
  2. 가능하면 HarpoonProjectile(발사체 이동+히트판정)과 RopeVisualizer(LineRenderer)를 별도 클래스로 분리
  3. DebugFire/DebugHit ContextMenu로 XR 없이도 히트 시퀀스 테스트
2. 발표 전 전체 플로우 스모크 테스트
Title→Lobby→각 스테이지→사망→결과→Lobby 전체 루프를 한 번에 테스트하지 않으면 씬 전환 시 싱글톤 중복 생성, 이벤트 미해제, BGM 중첩 등 통합 버그가 발견되지 않음.
  1. Title 씬부터 시작해 모든 해금된 스테이지를 순서대로 플레이
  2. 각 스테이지에서 사망 후 Retry 1회 + Lobby 복귀 1회 테스트
  3. BGM이 중첩되지 않는지, 결과 패널이 정상 표시되는지, Save 파일이 올바르게 갱신되는지 확인
3. feat-manhole 브랜치 조기 머지
제출 직전 머지 충돌은 시간 낭비의 주범. 현재 2 behind 상태라 충돌 가능성 있음.
  1. git checkout feat-manhole && git rebase main으로 최신 main 기반으로 리베이스
  2. 충돌 해결 후 맨홀 스테이지 빌드 테스트
  3. main에 머지하고 feat-manhole 브랜치 삭제

HarpoonGun.cs개선필요
550줄로 프로젝트 300줄 제한을 크게 초과. 그러나 내부 로직은 견고함 — SafeLookRotation()의 up 벡터 예외처리, IsInsideTunaCylinder()의 실린더 히트테스트 수학, TunaGrabFilter로 그랩 허용/차단 제어 등 구현 완성도가 높음. 다만 AttachToRightHand()에서 rightHandAnchor를 못 찾으면 LogWarning만 하고 계속 진행하는데, 이후 Update에서 null 상태의 transform 접근 가능성 있음. Ready→Fired→Done 상태머신 + 로프 LineRenderer 시각화까지 한 파일에 있어 분리 권장.
StageCommonSystem.cs양호
스테이지 핵심 오케스트레이터로서 역할이 명확함. TriggerDeath()가 단일 사망 진입점이고, OnPlayerDeath 이벤트로 후속 시스템들과 깔끔하게 분리. 유혹 팝업 시스템이 LocalizationManager 연동 + fallback 배열로 이중 안전장치 구현. IsDead 플래그로 중복 사망 방지. popupInterval/popupDisplayTime 등 Inspector 튜닝 가능한 설계. 16회 변경된 핫 파일이지만 책임이 명확하게 유지됨.
GameManager.cs양호
싱글톤 + DontDestroyOnLoad로 전역 상태 관리, StageRecord[] 배열로 스테이지별 클리어 추적, LoadTitle/LoadLobby/LoadStage로 씬 전환 일원화. OnStageResultRecorded 이벤트로 BGMManager·StageSFXPlayer 등과 느슨하게 결합. RecordDeath()와 RecordClear() 분리가 명확하고, InitStageFromScene()으로 GameBootstrap 경유 직접 실행도 지원. IsStageUnlocked() 로직이 단순하지만 학생 프로젝트에 적합한 수준.
SharkAI.cs개선필요
Patrol→Chase→Attack 상태머신이 명확하고, Animator 파라미터(Speed, Attack) 연동, chaseWindupDelay로 연출 여유 제공 등 잘 구현됨. 그러나 FindPlayerTarget()에서 VRRigBuilder.Instance?.XRCamera ?? MicrowaveDeathTrigger.cs양호
소켓 삽입→문 닫힘→충전(3초, 프로그레스 이벤트)→폭발→여파 VFX→사망의 멀티페이즈 시퀀스가 코루틴으로 깔끔하게 구현됨. OnChargingProgress(float) 이벤트로 MicrowaveDeathEffect가 실시간 조명 연동 가능. 문 회전 애니메이션이 SmoothStep 보간으로 자연스러움. aftermathVFXPrefab + aftermathDelay로 폭발 후 잔여 효과까지 고려한 세밀한 설계.

아키텍처 평가

VContainer의 LifetimeScope(StageScope)를 활용한 의존성 주입이 아키텍처의 핵심으로, DeathTrigger들이 StageCommonSystem에 직접 의존하지 않고 [Inject]로 주입받아 결합도가 낮음. Observer 패턴이 프로젝트 전반에 걸쳐 일관되게 사용됨 — static event Action 패턴으로 StageCommonSystem.OnPlayerDeath, DeathTrigger별 커스텀 이벤트(OnSharkAttack, OnExplosionStarted 등), GameManager.OnStageResultRecorded가 시스템 간 통신을 담당. Singleton 패턴(GameManager, VRRigBuilder, BGMManager, LocalizationManager)이 DontDestroyOnLoad와 결합되어 씬 간 상태 유지를 처리하며, GameBootstrap이 RuntimeInitializeOnLoadMethod로 진입점을 통일. ScriptableObject(StageRegistry, LocalizationTable, StageTextData, UnderwaterEffectSettings)로 데이터와 로직이 분리되어 Inspector 기반 튜닝이 가능. 전체적으로 학생 프로젝트에서 보기 드문 수준의 아키텍처 성숙도를 보임.
