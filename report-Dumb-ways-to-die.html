프로젝트 현황 요약

이 팀은 'Dumb Ways to Die' 컨셉의 VR 블랙코미디 데스게임을 만들고 있으며, 현재 Mushroom(먹기), GasTank(폭발), Microwave(전자레인지), Shark(상어), GpuTrap(곰 덫), Electrocution(감전) 등 6개 이상의 사망 스테이지를 실제 동작하는 수준까지 구현했다. 특히 Shark 스테이지는 작살 발사 → 참치 명중 → 그랩 → 흔들기 → 상어 유인 → 공격이라는 복잡한 상호작용 체인을 HarpoonGun, TunaShakeInteraction, SharkAI 3개의 상태 머신으로 완성했고, 수중 파티클·코스틱 셰이더·포스트프로세싱까지 갖춘 완성도 높은 스테이지다. GameBootstrap의 자동 부트스트랩, VRRigBuilder의 Sticky 이벤트, BaseDeathEffect의 Template Method 등 아키텍처 수준이 학생 프로젝트치고 매우 높다. 가장 시급한 것은 stage/manhole 브랜치의 즉시 머지와 VContainer DI 실패 시 null 가드 추가로, 이 두 가지만 해결하면 발표 데모로서 충분히 안정적인 프로토타입이 된다.
!

리스크 알림

VContainer DI 실패 시 다수 DeathTrigger 크래시 위험
BearTrapDeathTrigger, SocketDeathTrigger, GasTankDeathTrigger 등이 VContainer의 Construct()를 통해 StageCommonSystem을 주입받는데, DI 컨테이너가 올바르게 설정되지 않은 씬에서 실행하면 _stage가 null인 상태로 TriggerDeath()가 호출되어 NullReferenceException으로 크래시한다. 발표 데모 중 이 문제가 발생하면 게임이 멈춘다.
  • 각 DeathTrigger의 TriggerDeath() 호출 전에 _stage != null 가드 추가
  • DI 실패 시 FindFirstObjectByType<StageCommonSystem>()로 폴백하는 안전장치 추가
  • 모든 스테이지 씬에서 VContainer LifetimeScope가 올바르게 설정되어 있는지 확인
stage/manhole 브랜치 머지 지연
origin/stage/manhole이 main 대비 behind 11, ahead 4로 diverge가 심하다. StageCommonSystem.cs가 hot file(14회 수정)이라 머지 충돌 가능성이 높다. 마감일에 머지하면 예기치 않은 빌드 브레이크가 발생할 수 있다.
  • 즉시 main을 stage/manhole에 rebase/merge하여 충돌 해결
  • EditorBuildSettings.asset(22회 수정)도 충돌 대상이므로 빌드 세팅을 수동 검증
  • 머지 후 전 스테이지 플레이 테스트 실행
빌드 시 Shader.Find() null 반환 가능성
HarpoonGun.CreateRopeRenderer()에서 Shader.Find('Sprites/Default')를 사용하는데, 이 셰이더가 Always Included Shaders에 포함되지 않으면 빌드에서 null Material이 생성되어 로프가 보이지 않는다. 에디터에서는 동작하지만 Quest 빌드에서 실패할 수 있다.
  • Project Settings → Graphics → Always Included Shaders에 'Sprites/Default' 추가 확인
  • 또는 SerializeField로 Material을 Inspector에서 직접 할당하도록 변경
대형 커밋으로 인한 코드 리뷰 누락 위험
4929줄(Shark 스테이지), 4517줄(씬 배치), 4220줄(에셋 정리) 등 oversized 커밋이 다수 존재한다. 이런 커밋은 코드 리뷰가 실질적으로 불가능하여 버그가 숨어 들어갈 수 있다.
  • 향후 커밋은 로직 변경과 에셋 추가를 분리
  • Unity 씬/프리팹 파일은 별도 커밋으로 분리하여 .cs 파일 변경만 리뷰 가능하게
애니메이션 에셋 부족
anim 파일이 1개뿐이며 상어 AI, 곰 덫 등의 애니메이션을 Animator 코드에서 참조하지만 실제 애니메이션 클립이 거의 없다. 발표 시 시각적 완성도가 떨어질 수 있다.
  • 최소한 상어 수영/공격, 곰 덫 닫힘 애니메이션 추가
  • 없으면 코드 기반 트윈(DOTween 등)으로 대체

권고사항 (Top 3)

1. VContainer DI 폴백 안전장치 일괄 추가
발표 데모에서 DI 실패로 인한 크래시는 치명적이다. 5분 내에 모든 DeathTrigger에 null 가드를 추가할 수 있다.
  1. 모든 Construct() 메서드 하단에 if(_stage==null) _stage = FindFirstObjectByType<StageCommonSystem>() 추가
  2. TriggerDeath() 호출부에 _stage?.TriggerDeath() null-conditional 연산자 적용
  3. 변경 후 각 스테이지에서 사망 트리거 테스트
2. stage/manhole 즉시 머지 및 전체 플레이 테스트
마감일이 임박하여 브랜치 diverge가 길어질수록 통합 리스크가 커진다. 지금 머지하면 충돌 범위가 최소화된다.
  1. main → stage/manhole rebase 실행
  2. EditorBuildSettings.asset 충돌 시 Build Settings에서 수동으로 씬 목록 재확인
  3. 머지 후 Title → Lobby → 각 스테이지 → 사망 → 결과화면 → 로비 복귀 전체 플로우 테스트
3. HarpoonGun.cs 리팩토링 (선택)
550줄은 유지보수와 버그 추적이 어렵다. 시간이 있다면 분리하여 코드 품질 점수를 높일 수 있다.
  1. HarpoonAiming (조준 감지 + 이벤트), HarpoonProjectile (발사 + 비행 + 명중), HarpoonRope (로프 시각화)로 3분할
  2. 각 컴포넌트를 같은 GameObject에 배치하고 [RequireComponent]로 연결
  3. 기존 static 이벤트는 그대로 유지하여 외부 구독자 코드 변경 불필요

코드 리뷰

HarpoonGun.cs개선필요
550줄로 프로젝트 가이드라인(300줄)을 크게 초과한다. 하지만 코드 자체의 품질은 높다. State Machine(Ready→Fired→Done)이 깔끔하고, IsInsideTunaCylinder()의 실린더 명중 판정이 수학적으로 정확하다. SafeLookRotation()의 gimbal lock 방지, DrawWireCylinder()의 에디터 시각화도 잘 구현했다. AttachToRightHand()의 다단계 폴백(Inspector → VRRigBuilder → GameObject.Find)은 VR 개발에서 실용적이다. 개선점: 조준/발사/명중을 각각 별도 컴포넌트로 분리하면 300줄 이하로 줄일 수 있다.
GameManager.cs양호
138줄의 간결한 싱글톤으로 핵심 기능만 담고 있다. StageRecord 배열 + SaveManager.Load()로 진행 상태를 관리하고, InitStageFromScene()으로 에디터에서 직접 스테이지 테스트가 가능하다. RecordDeath()와 RecordClear()가 분리되어 '죽음이 곧 클리어'라는 게임 컨셉에 맞게 두 이벤트를 모두 발화한다. IsStageUnlocked()의 순차 잠금 해제 로직도 정확하다.
VRRigBuilder.cs양호
OnRigReady Sticky 이벤트 패턴이 인상적이다. 이미 Ready 상태면 즉시 콜백을 호출하는 구현으로 레이스 컨디션을 원천 차단한다. 씬 EventSystem을 비활성화한 후 영속 EventSystem을 생성하는 순서가 정확하다. OnDestroy()에서 중복 인스턴스 파괴 시 Instance를 null로 만들지 않는 가드가 있어 씬 재진입 시 안전하다.
StageCommonSystem.cs양호
167줄로 간결하면서도 핵심 역할을 수행한다. TriggerDeath()가 RecordClear() → RecordDeath() → OnPlayerDeath 이벤트를 순서대로 실행하며, 이중 호출 방지(IsDead 가드)가 있다. 유혹 팝업 시스템이 게임 컨셉('죽음을 유도하는 메시지')에 잘 부합한다. 다만 HidePopup()에서 StopAllCoroutines()를 호출하는 것은 이 컴포넌트에 다른 코루틴이 추가될 경우 부작용을 일으킬 수 있다.
SharkAI.cs양호
Patrol→Chase→Attack 상태 머신이 깔끔하다. 순찰 웨이포인트를 순서대로 따라가고, StartChase()로 추적 모드 진입 시 chaseWindupDelay로 연출 타이밍을 조절한다. attackDistance 도달 시 DoAttack()에서 SharkDeathTrigger.OnSharkReachedPlayer()를 호출하는 연결이 명확하다. 플레이어 탐색 시 VRRigBuilder.XRCamera → XROrigin → Camera.main 순으로 폴백한다.
UnderwaterParticles.cs개선필요
455줄로 300줄 가이드라인을 초과한다. 런타임에서 ParticleSystem을 프로시저럴하게 생성하는 코드(CreateDebrisSystem, CreateBubblesSystem)가 복잡하지만 기능적으로 완성도가 높다. BuildOscillationCurve(), BuildBubbleSizeCurve() 등 헬퍼가 잘 분리되어 있다. 개선점: 파티클 생성 로직을 별도 static 유틸리티로 추출하면 300줄 이하로 줄일 수 있다.
MicrowaveDeathEffect.cs양호
3단계 포스트프로세싱 시퀀스(충전→섬광→연기)가 잘 구조화되어 있다. OnChargingStarted, OnExplosionStarted 이벤트에 각각 구독하여 단계별 Volume weight를 코루틴으로 제어한다. OnDisable()에서 모든 볼륨 weight를 0으로 리셋하여 씬 언로드 시 잔여 효과가 남지 않도록 했다. 286줄로 가이드라인 내에 있다.
LobbyManager.cs양호
119줄로 간결하며 스테이지 잠금/해제 UI를 동적으로 관리한다. StageButtonEntry 구조체로 Button/StatusText/LockOverlay를 캐싱하고, OnEnable()마다 RefreshUI()를 호출하여 항상 최신 상태를 반영한다. '준비 중/LOCKED/완료/?' 상태 텍스트 매핑이 명확하다. 다만 자식 이름 'StatusText', 'LockOverlay'에 의존하는 점은 씬 구조 변경에 취약하다.

아키텍처 평가

Observer 패턴을 핵심 통신 수단으로 사용하며, static event(OnPlayerDeath, OnStageResultRecorded, OnHarpoonHit 등)로 클래스 간 결합도를 최소화했다. VRRigBuilder의 Sticky 이벤트 패턴은 Unity의 초기화 순서 문제를 우아하게 해결한다. GameBootstrap의 RuntimeInitializeOnLoadMethod + 조건부 싱글톤 생성은 '어떤 씬에서든 Play 가능' 워크플로우를 보장하여 팀 생산성을 크게 높인다. BaseDeathEffect 추상 클래스로 URP Volume 라이프사이클(생성→가중치 애니메이션→결과 디밍→정리)을 공통화하고 9개 서브클래스가 ConfigureDeathVolume()만 오버라이드하는 Template Method 패턴이 정확하다. VContainer DI를 DeathTrigger에만 선택적으로 적용한 것은 적절한 범위 판단이지만, DI 실패 시 폴백이 없어 설정 누락에 취약하다.

단계별 진행률 (AI 평가)

기반 구축 5/5점
GameBootstrap가 RuntimeInitializeOnLoadMethod로 어떤 씬에서든 자동 부트스트랩을 수행하며, VRRigBuilder가 XR Origin + EventSystem + XRUIInputModule을 DontDestroyOnLoad로 영속화한다. GameManager 싱글톤이 씬 전환·스테이지 레지스트리·세이브/로드를 담당하고, VRCanvasAdapter가 모든 Canvas를 WorldSpace로 자동 변환한다. 프로젝트 세팅(EditorBuildSettings, Android Preset 등)도 반복 수정 이력이 있어 빌드 파이프라인이 갖춰져 있음. 빠진 것 없이 완벽한 파운데이션.
핵심 시스템 26/25점
GameManager(씬 전환·스테이지 기록), VRRigBuilder(XR 리그 구축), StageCommonSystem(사망 허브·유혹 팝업), DeathTrigger 8종(BearTrap, Collision, Eat, GasTank, GrabContact, Microwave, Shark, Socket) 모두 실제 동작 로직이 있다. DeathEffect는 BaseDeathEffect 추상 클래스 아래 9개 구현체(Burn, BurnMelt, Default, Electrocution, GasTank, GpuTrap, Microwave, Mushroom, Shark)가 URP Volume을 단계별로 제어한다. Shark 스테이지는 HarpoonGun(상태머신 Ready→Fired→Done), TunaShakeInteraction(Idle→Shaking→Attracting→Done), SharkAI(Patrol→Chase→Attack)까지 복잡한 게임플레이 루프를 완성했다. StageSFXPlayer·StageVFXPlayer·StageHapticPlayer가 이벤트 기반으로 연출을 담당한다. SaveManager가 JSON 기반 저장/로드를 구현한다. 다만 Manhole 스테이지는 최근 커밋에서 기초 구현 중이고, 일부 DeathTrigger의 VContainer DI 실패 시 null 참조 위험이 있어 감점.
통합 23/25점
이벤트 기반 아키텍처가 잘 구성되어 있다. StageCommonSystem.OnPlayerDeath → PlayerInteractionBlocker(조작 차단), DeathEffect(포스트프로세싱), ResultScreenSystem(결과 화면), CameraShake(카메라 흔들림)이 모두 독립적으로 구독한다. GameManager.OnStageResultRecorded → StageSFXPlayer, StageVFXPlayer, StageHapticPlayer, BGMManager가 반응한다. Shark 스테이지는 HarpoonGun.OnHarpoonHit → TunaGrabFilter.AllowGrab, TunaFireTooltipUI 전환 → TunaShakeInteraction.OnShakeComplete → SharkDeathTrigger → SharkAI.StartChase → StageCommonSystem.TriggerDeath로 완전한 게임플레이 루프가 연결된다. VRRigBuilder.OnRigReady는 Sticky 이벤트 패턴으로 레이스 컨디션을 방지한다. 씬 전환은 GameManager.LoadStage/LoadLobby/LoadTitle로 동작하고 XROriginSceneReset이 위치를 초기화한다. LobbyManager가 스테이지 잠금/해제 UI를 동적으로 갱신한다. 다만 stage/manhole 브랜치가 아직 미머지(behind 11)이고, 일부 스테이지 간 통합 테스트가 부족할 수 있다.

심각도파일설명
HIGHBearTrapDeathTrigger.csVContainer DI의 Construct(StageCommonSystem stage)가 주입 실패 시 _stage가 null이 되지만, TriggerDeath() 호출 시 _stage.TriggerDeath()에서 NullReferenceException 발생 가능. DI 컨테이너 설정이 누락된 씬에서 크래시 위험.
HIGHSocketDeathTrigger.csBearTrapDeathTrigger와 동일한 문제. VContainer DI 실패 시 _stage null 참조로 런타임 크래시 가능.
HIGHstage/manhole 브랜치origin/stage/manhole이 main 대비 behind 11로 상당히 뒤처져 있으며, 머지 충돌 가능성이 높다. 마감 전 통합 리스크.
MEDIUMHarpoonGun.cs550줄로 프로젝트 가이드라인(300줄) 초과. 조준, 발사, 명중 판정, 로프 시각화, 입력, Gizmo가 모두 한 클래스에 밀집되어 있어 단일 책임 원칙 위반.
MEDIUMUnderwaterParticles.cs455줄로 300줄 가이드라인 초과. CreateDebrisSystem()/CreateBubblesSystem()의 프로시저럴 파티클 생성 코드가 과도하게 길다.
MEDIUMHarpoonGun.csCreateRopeRenderer()에서 new Material(Shader.Find('Sprites/Default'))로 매 실행마다 새 머티리얼 인스턴스를 생성하는데, Shader.Find()는 빌드에서 해당 셰이더가 Always Included에 없으면 null을 반환할 수 있다.
MEDIUMEatDeathTrigger.csAwake()에서 Camera.main을 직접 사용. VR 빌드에서 VRRigBuilder가 아직 카메라를 MainCamera로 태깅하기 전에 호출되면 null 반환 가능.
LOWStageCommonSystem.csHidePopup()에서 StopAllCoroutines()를 호출하여 ShowFirstTemptationAfterDelay 외의 다른 코루틴이 추가될 경우 예기치 않게 중단됨. CancellationToken이나 명시적 코루틴 참조 사용 권장.
LOWLobbyManager.csFindChildTMP(go, 'StatusText')와 FindChildGO(go, 'LockOverlay')가 자식 이름에 하드코딩 의존. 씬 계층 구조 변경 시 조용히 실패(null 반환)하여 UI가 비정상 표시.
LOWTunaGrabVisuals.csMakeTransparent()에서 URP 머티리얼의 _Surface, _Blend, _SrcBlend, _DstBlend를 직접 조작하는데, Unity 6/URP 17에서 프로퍼티 이름이 변경될 수 있어 버전 업그레이드 시 깨질 위험.
