프로젝트 현황 요약

이 팀은 'Dumb Ways to Die' 컨셉의 VR 블랙코미디 데스게임을 만들고 있으며, 현재 핵심 게임 루프(스테이지 진입→유혹 메시지→사망 트리거→사망 이펙트→결과 화면→로비 복귀)가 완전히 동작하는 상태까지 구현했다. GameManager/VRRigBuilder/GameBootstrap 기반 인프라, 8종 이상의 DeathTrigger(EatDeathTrigger, SharkDeathTrigger, MicrowaveDeathTrigger 등), 10종 이상의 PostProcess 사망 이펙트(SharkDeathEffect, BurnDeathEffect 등), 유체이탈 카메라 시퀀스(DeathCameraEffect), 다국어 지원(LocalizationManager), 저장/로드(SaveManager) 등 프로토타입을 넘어선 수준의 완성도를 보인다. 가장 시급한 문제는 StageScope에 ManholeDeathTrigger DI 등록 누락과 BGMManager의 스테이지→로비 복귀 시 볼륨 0 버그로, 이 두 가지는 발표 전 반드시 수정해야 한다. 381커밋의 활발한 개발 이력과 마지막 날까지 디테일 추가/버그 수정이 이어지고 있어, 발표 준비도는 높은 편이다.
!

리스크 알림

ManholeDeathTrigger VContainer DI 미등록
StageScope.Configure()에 CollisionDeathTrigger/GrabContact/Socket/Eat/GasTank/BearTrap/Microwave/Shark는 등록되어 있지만 ManholeDeathTrigger가 빠져있음. Manhole 스테이지에서 _stage가 null로 남아 TriggerDeath() 호출이 실패하여 사망이 발동하지 않을 수 있음.
  • StageScope.Configure()에 foreach (var t in FindObjectsByType<ManholeDeathTrigger>(...)) builder.RegisterComponent(t); 추가
  • Manhole 스테이지에서 사망 트리거가 정상 동작하는지 플레이 테스트 확인
BGMManager 스테이지→로비 복귀 시 볼륨 0 문제
스테이지 진입 시 FadeTo(0f)로 볼륨이 0이 되는데, Title/Lobby 복귀 시 _bgmSource.isPlaying만 체크하고 볼륨 복원 없이 Play()만 호출. 결과적으로 로비에서 BGM이 무음으로 재생됨.
  • OnSceneLoaded()의 Title/Lobby 분기에서 FadeTo(bgmMaxVolume, fadeInDuration) 호출 복원
  • 스테이지→로비→스테이지 왕복 시 BGM 볼륨 정상 확인
EatDeathTrigger Camera.main null 위험
Awake()에서 _animator.Init(Camera.main)을 호출하는데, GameBootstrap 경유 시 VRRigBuilder가 MainCamera 태그를 설정하기 전에 EatDeathTrigger.Awake()가 실행될 수 있음. Camera.main이 null이면 먹기 애니메이션이 플레이어를 따라가지 못함.
  • EatDeathTrigger.Awake()에서 Camera.main 대신 Start()에서 VRRigBuilder.Instance?.XRCamera를 사용하도록 변경
  • 또는 VRRigBuilder.OnRigReady 콜백에서 Init 호출
발표 직전 feat-manhole 브랜치 미머지
feat-manhole 브랜치가 main 대비 1 ahead / 2 behind 상태로 남아있음. 맨홀 스테이지 관련 최신 변경이 main에 반영되지 않았을 수 있음.
  • feat-manhole 브랜치를 main에 머지하거나 리베이스
  • 머지 후 맨홀 스테이지 전체 플레이 테스트
대용량 커밋 패턴

권고사항 (Top 3)

1. ManholeDeathTrigger DI 등록 및 BGM 볼륨 버그 즉시 수정
두 버그 모두 발표 시연 중 발견될 수 있는 치명적/중요 이슈. 맨홀 스테이지에서 죽을 수 없거나 로비 BGM이 안 들리면 발표 품질에 직접 영향.
  1. StageScope.cs에 ManholeDeathTrigger 등록 추가
  2. BGMManager.OnSceneLoaded()의 Title/Lobby 분기에 FadeTo(bgmMaxVolume, fadeInDuration) 복원
  3. 모든 스테이지 순회 플레이 테스트로 사망→결과→로비→재진입 루프 검증
2. HarpoonGun 클래스 분할
550줄 단일 클래스는 버그 수정과 협업에 불리. 발표 전이라 리팩토링 우선순위는 낮지만, 향후 유지보수를 위해 권장.
  1. 히트 판정 로직을 HarpoonHitDetector로 분리
  2. 로프 시각화를 HarpoonRopeRenderer로 분리
  3. 입력 처리를 HarpoonInputHandler로 분리
3. 통합 플레이 테스트 체크리스트 작성
8종 이상의 사망 트리거, 다양한 스테이지, BGM/SFX/VFX/햅틱 연동 등 테스트해야 할 조합이 많음. 발표 전 체계적 테스트 없이는 미발견 버그 위험.
  1. 각 스테이지별 사망 트리거 동작 확인 체크리스트 작성
  2. 사망→결과→Retry/Lobby 전체 흐름 테스트
  3. Title→Lobby→Stage→Death→Result→Lobby 전체 루프 2회 이상 테스트
  4. VR 빌드(Quest) 실기기 테스트 최소 1회

코드 리뷰

GameManager.cs양호
139줄의 깔끔한 싱글톤. LoadStage/RecordDeath/RecordClear/IsStageUnlocked 등 핵심 API가 명확. StageRegistry null 체크, ResolveAssets()의 에디터 전용 자동 탐색, OnStageResultRecorded 이벤트 설계가 우수. 단 RecordClear()→RecordDeath() 순서로 이벤트가 2회 발생하는 구조는 구독자가 isDead 필터링을 잊으면 이중 실행 위험. 문서화가 잘 되어 있어 학생 프로젝트로서 모범적.
VRRigBuilder.cs개선필요
XR 리그 구축 로직이 체계적이고 OnRigReady sticky event 패턴이 인상적. EventSystem 충돌 방지를 위해 기존 EventSystem을 먼저 비활성화하는 순서 제어가 정교함. 단, SetLayerRecursively() 메서드(199-204줄)가 정의만 되고 한 번도 호출되지 않는 데드 코드. Player 레이어를 rigInstance.layer에만 설정하고 자식에는 적용하지 않는데, CharacterController 등 자식 오브젝트의 물리 충돌 레이어가 의도대로 동작하는지 확인 필요.
StageCommonSystem.cs양호
189줄로 사망 트리거와 유혹 팝업을 깔끔하게 관리. TriggerDeath()의 이중 호출 방지(IsDead 플래그), LocalizationManager 연동 폴백 패턴, SuppressTemptation() API가 잘 설계됨. 단 HidePopup()에서 StopAllCoroutines()를 호출하는데, ShowFirstTemptationAfterDelay 코루틴도 함께 중단될 수 있음. 특정 코루틴만 StopCoroutine()으로 중단하는 것이 안전.
DeathCameraEffect.cs양호
유체이탈 카메라 시퀀스가 정교하게 구현됨. NormalDeathSequence(슬로모→프리즈→헤드트래킹OFF→상승+회전)와 KnockbackDeathSequence(물리 넉백→수평 복원→상승) 두 경로를 깔끔히 분리. OnDestroy에서 Time.timeScale=1f 복원, HeadTrackingController/DeathCameraAnimator/DeathBodyController로 책임 분리가 우수. ResultScreenSystem.IsHandledByExternalEffect 플래그로 폴백 경로와 양립하는 설계가 인상적.
EatDeathTrigger.cs개선필요
상태 머신(Idle→Eating→Done) + XR Grab 제어가 잘 구현됨. VContainer [Inject]로 StageCommonSystem 주입, EatAnimator/EatAudioPlayer/EatDebrisSpawner로 책임 분리 우수. 단 Awake()에서 _animator.Init(Camera.main) 호출 시, GameBootstrap이 VRRigBuilder보다 먼저 카메라를 생성하지만 MainCamera 태그 설정 타이밍에 따라 Camera.main이 null일 수 있음. VRRigBuilder.Instance?.XRCamera 사용이 더 안전.
HarpoonGun.cs개선필요
작살 발사→실린더 히트 판정→참치 그랩 전환의 복잡한 게임플레이를 잘 구현했으나 550줄로 프로젝트 규칙(300줄)을 크게 초과. 발사/명중 판정/조준 감지/로프 시각화/입력 처리가 모두 한 클래스에 있어 분리 가능. Static 이벤트(OnAimAtTuna/OnHarpoonFired/OnHarpoonHit)와 Inspector 설정이 상세하고 ContextMenu 디버그 도구도 갖춤. SafeLookRotation과 DrawWireCylinder 유틸리티 메서드는 별도 클래스로 추출 가능.
BaseDeathEffect.cs양호
PostProcess Volume 기반 사망 이펙트의 견고한 추상 기반. ConfigureDeathVolume()/DeathDuration 추상 멤버로 서브클래스 확장 용이. AnimateWeight() 유틸리티의 SmoothStep 보간, EnsurePostProcessing()의 URP CameraData 자동 활성화, ResultDim() 코루틴의 타이밍 제어가 정교. OnDisable()에서 RestorePostProcessing() 호출로 정리도 완벽. 10종 이상의 서브클래스가 이 패턴을 일관되게 활용.
ResultScreenSystem.cs양호
사망 결과 UI를 AlwaysOnTop 렌더링(ZTest Always + sortingOrder 30000)으로 VR 환경에서 확실히 표시. ShowDeathResult/ShowDeathResultImmediate/ShowAt 세 가지 진입점으로 다양한 사망 시나리오 지원. WireButton()으로 Inspector 의존 없이 코드에서 버튼 자동 연결. _alwaysOnTopApplied 플래그로 Material 중복 생성 방지. DeathMessage 프로퍼티의 LocalizationManager 폴백 체인도 깔끔.

아키텍처 평가

Singleton 영속 레이어(GameManager/VRRigBuilder/BGMManager/LocalizationManager/VRCanvasAdapter)와 VContainer DI 스테이지 스코프(StageScope)를 조합한 이중 구조가 핵심. Observer 패턴(static event Action)을 광범위하게 활용하여 DeathTrigger→StageCommonSystem→DeathEffect/ResultScreen/PlayerInteractionBlocker 간 결합도를 최소화. GameBootstrap의 RuntimeInitializeOnLoadMethod + VRRigBuilder.OnRigReady sticky event 조합으로 초기화 순서 경쟁 조건을 우아하게 해결한 점이 인상적. Shark 서브시스템은 12개 클래스로 구성된 가장 복잡한 모듈이지만 HarpoonGun→OnHarpoonHit→TunaGrabFilter.AllowGrab→TunaShakeInteraction→OnShakeComplete→SharkDeathTrigger→SharkAI.StartChase 흐름이 이벤트로 느슨하게 연결됨. 개선점으로는 BaseDeathEffect 서브클래스 10종이 각각 별도 Volume을 생성하는 구조가 메모리 면에서 비효율적이나, 스테이지당 1-2개만 사용하므로 실질적 문제는 아님.
