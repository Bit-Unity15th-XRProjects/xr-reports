프로젝트 현황 요약

이 팀은 VR 블랙코미디 데스게임 'Dumb Ways to Die'를 만들고 있으며, 핵심 게임 루프(로비→스테이지→유혹→사망→결과)와 7종의 사망 트리거(가스통 폭발, 전자레인지, 독버섯 섭취, 곰 덫, 소켓 삽입, 충돌, 그랩 접촉)가 실제 동작하는 수준으로 구현되어 있다. DeathCameraEffect의 슬로모션→영혼 상승→시체 내려다보기 시퀀스와 8종 이상의 PostProcess 기반 사망 이펙트(화상, 감전, 폭발 섬광, 환각 등)가 VR 몰입감을 크게 높이고 있으며, VContainer DI와 Static Event 기반 아키텍처로 스테이지 추가가 용이한 구조를 갖추었다. 현재 Shark(상어) 스테이지의 수중 환경 셰이더 작업이 활발히 진행 중이다. 가장 시급한 것은 VR 동기 씬 로딩을 비동기로 전환하여 멀미를 방지하는 것과, 4개뿐인 오디오 에셋을 대폭 보강하여 데스게임의 코미디/공포 연출을 완성하는 것이다. 코드 품질과 아키텍처 설계력은 학생 팀 수준을 넘어서는 상당한 완성도를 보여주고 있다.

리스크 알림

VR 동기 씬 로딩으로 인한 멀미 유발
SceneManager.LoadScene()은 로딩 중 렌더링이 멈춰 VR에서 프레임 드롭→멀미를 직접 유발함. Meta Quest 스토어 제출 시 기술 검증(TRC)에서 거부 사유가 될 수 있음. 현재 Title→Lobby→Stage 전환마다 발생
  • SceneManager.LoadSceneAsync()로 전환
  • 페이드 아웃→로딩→페이드 인 시퀀스 구현 (OVRScreenFade 또는 커스텀)
  • LoadSceneAsync의 allowSceneActivation으로 로딩 완료 후 전환 제어
오디오 에셋 절대 부족 — 4개 파일로는 데스게임 몰입감 불가
StageSFXPlayer, EatAudioPlayer, GasTankFlameEffect 등 코드에서 AudioClip을 참조하지만 실제 에셋이 4개뿐. VR 데스게임에서 소리는 공포/코미디 연출의 핵심이며, 무음 사망은 긴장감을 완전히 파괴함
  • 각 스테이지별 최소 사망 효과음 1개 + 환경음 1개 확보 (무료 에셋: Freesound, Sonniss GDC)
  • 폭발, 전자레인지 웅웅, 먹는 소리, 곰 덫 등 스테이지 특화 SFX 우선 제작
  • BGMManager에 스테이지별 BGM 트랙 추가
Shark 브랜치 머지 지연 — 12커밋 ahead, 3 behind
origin/stage/shark가 main 대비 12커밋 앞서고 3커밋 뒤처져 있음. 수중 셰이더, 코스틱, 파티클 등 대량 에셋 변경이 포함되어 있어 머지 충돌 가능성 높음. .unity 씬 파일 충돌은 수동 해결이 거의 불가능
  • shark 브랜치를 main에 즉시 리베이스/머지
  • 씬 파일 충돌 방지를 위해 작업 씬 분리 (Testing_JH 사용 중이므로 유지)
  • 향후 씬 작업은 Prefab 기반으로 전환하여 씬 파일 변경 최소화
대형 커밋으로 인한 코드 리뷰 및 롤백 어려움
4929줄(Shark 기초 세팅), 4517줄(씬 오브젝트 배치) 등 대형 커밋이 빈번. 문제 발생 시 특정 변경만 롤백하기 어렵고, PR 리뷰가 사실상 불가능한 크기

권고사항 (Top 3)

1. 폭발 DeathEffect 공통 베이스 클래스 추출
GasTankDeathEffect와 MicrowaveDeathEffect의 섬광→해제→연기 패턴이 거의 동일하며, 향후 다른 폭발형 스테이지 추가 시 같은 코드가 또 복사될 것
  1. ExplosionDeathEffectBase 중간 클래스 생성 — flashAttack, bloomMax, releaseDuration, smoke 관련 필드와 Flash/Release/Smoke 코루틴 포함
  2. GasTankDeathEffect, MicrowaveDeathEffect가 이를 상속하고 충전 페이즈 등 고유 로직만 오버라이드
  3. ConfigureDeathVolume() 빈 오버라이드 대신 base 클래스의 Volume 생성 로직을 활용하도록 리팩토링
2. 비동기 씬 전환 + VR 페이드 시스템 구현
VR에서 동기 씬 로딩은 프레임 드롭→멀미의 직접적 원인이며, Meta Quest TRC 검증에서 거부될 수 있음. 현재 모든 씬 전환이 동기 방식
  1. SceneTransitionManager 클래스 신규 생성 — GameManager에서 분리
  2. OVRScreenFade 또는 커스텀 ScreenFader(Camera 앞 검은 Quad)로 페이드 아웃
  3. LoadSceneAsync + allowSceneActivation=false로 백그라운드 로딩
  4. 로딩 완료 후 allowSceneActivation=true → 페이드 인
3. Core 스크립트 네임스페이스 통일
GameManager, VRRigBuilder, StageCommonSystem 등 핵심 클래스가 글로벌 네임스페이스에 있어 다른 에셋/패키지와 이름 충돌 가능성. DumbWays.Gameplay 하위 시스템과 일관성도 부족
  1. DumbWays.Core 네임스페이스 생성
  2. GameManager, GameBootstrap, VRRigBuilder, VRCanvasAdapter, SaveManager, BGMManager 이동
  3. StageCommonSystem, ResultScreenSystem은 DumbWays.Gameplay로 이동
  4. Assembly Definition 참조 업데이트

VRRigBuilder.cs양호
OnRigReady Sticky Event 패턴이 인상적 — 구독 시점과 무관하게 안전하게 콜백 수신. Instance 싱글톤에 중복 생성 방지(Destroy(gameObject))와 DontDestroyOnLoad 적용. XR Rig 프리팹을 Resources에서 자동 탐색하는 로직도 편리함. 다만 글로벌 네임스페이스 사용이 아쉽고, XROrigin/XRCamera 프로퍼티가 public getter만 있어 외부에서 null 체크 없이 접근 시 NRE 위험.
DeathCameraEffect.cs양호
사망 시퀀스의 핵심 오케스트레이터로 Normal/Knockback 두 경로를 깔끔하게 분기 처리. Time.timeScale 관리(슬로모션 0.2→복원), HeadTrackingController/DeathBodyController/DeathCameraAnimator로 책임 분리가 잘 됨. OnCameraRiseComplete 이벤트로 후속 VFX 타이밍 동기화. OnDestroy에서 timeScale 복원은 안전장치로 좋으나, 208줄로 적절한 크기 유지.
StageHapticPlayer.cs개선필요
333줄로 CLAUDE.md의 300줄 제한 초과. HapticEntry 내부 클래스에 SinglePulse/Pattern/Ramp 3가지 모드의 필드가 모두 포함되어 있어 사용하지 않는 필드가 항상 Inspector에 노출됨. HapticEntry를 모드별 서브클래스로 분리하거나 CustomPropertyDrawer를 작성하는 것이 좋음. pulseInterval이 const로 하드코딩(0.05f)되어 있어 Inspector에서 조정 불가. Play(index)의 경계 검사는 잘 되어 있음.
MicrowaveDeathEffect.cs개선필요
281줄로 300줄에 근접. 3페이즈(충전→섬광→연기) 구조가 체계적이나 ConfigureDeathVolume()이 빈 메서드로 오버라이드됨 — BaseDeathEffect의 템플릿 메서드를 사용하지 않고 자체 Volume을 3개나 생성하여 base 클래스의 의도와 다르게 동작. OnChargingStarted/OnExplosionStarted 구독을 OnEnable/OnDisable에서 관리하는 것은 올바름. 하지만 Phase별 SerializeField가 과도하게 많아(20개+) Inspector 설정이 복잡함.
EatDeathTrigger.cs양호
State enum(Idle→Eating→Done)으로 상태를 명확히 관리. RequireComponent로 EatDebrisSpawner/EatAnimator/EatAudioPlayer를 강제하여 누락 방지. [Inject]로 StageCommonSystem 주입받는 VContainer 패턴 준수. InputActionReference → 키보드 Y → XR A버튼 폴백 체인으로 에디터/디바이스 모두 테스트 가능. [ContextMenu("Debug — Force Death")]로 디버깅 편의성도 고려. 204줄로 적절한 크기.
GasTankDeathEffect.cs개선필요
MicrowaveDeathEffect와 구조가 거의 동일(섬광→해제→연기)하여 코드 중복이 심함. flashAttack, bloomMax, exposureMax, releaseDuration, smokeVignette 등 SerializeField가 양쪽에 반복됨. 공통 폭발 효과를 ExplosionDeathEffectBase 같은 중간 클래스로 추출하면 중복을 줄일 수 있음. ConfigureDeathVolume()이 역시 빈 메서드로 오버라이드.
LazyFollowUI.cs양호
시야 이탈 감지 로직(GetOutOfViewFraction — RectTransform 4코너를 뷰포트 투영)이 정확하고 효율적. outOfViewThreshold=1.0으로 완전히 벗어날 때만 재배치하여 UX 안정성 확보. static readonly Vector3[] _corners로 매 프레임 GC 할당 방지. Canvas World Space 변환과 canvasScale 적용도 깔끔함. 188줄로 적절한 크기. RequireComponent(typeof(Canvas))로 필수 컴포넌트 보장.
ResultScreenSystem.cs개선필요
ShowDeathResult/ShowDeathResultImmediate/ShowAt 3가지 표시 경로가 있어 호출 시점에 따른 동작 예측이 어려움. IsHandledByExternalEffect 플래그로 분기하는 것보다 Strategy 패턴이 더 명확할 수 있음. FindTMP, WireButton 등 유틸리티 메서드가 static으로 정의되어 있어 테스트 용이하나, transform.Find(childName)으로 자식 탐색하는 것은 이름 변경 시 런타임 에러 위험. ZTest Always 머티리얼 오버라이드로 항상 최상위 렌더링은 VR에서 필수적인 좋은 처리.

아키텍처 평가

Singleton(VRRigBuilder, GameManager)+Observer(Static Event)+DI(VContainer StageScope)+Template Method(BaseDeathEffect)를 조합한 실용적 하이브리드 아키텍처. 전역 싱글톤은 DontDestroyOnLoad로 수동 관리하고, 스테이지별 의존성만 VContainer로 주입하는 이중 구조가 XR 프로젝트의 '영속 리그+일회성 스테이지' 특성에 정확히 부합함. 특히 Static Event 기반 통신(OnPlayerDeath, OnRigReady, OnExplosionStarted)으로 시스템 간 결합도가 매우 낮아 새 스테이지/이펙트 추가 시 기존 코드 수정이 불필요함. GameBootstrap의 RuntimeInitializeOnLoadMethod 전략으로 어떤 씬에서든 Play Mode 진입 가능한 것은 개발 생산성 면에서 훌륭한 설계. 다만 BaseDeathEffect의 템플릿 메서드가 일부 서브클래스에서 무시(빈 오버라이드)되고 있어 상속 계층 재설계가 필요하며, Core 클래스의 네임스페이스 부재로 모듈 경계가 코드상 불명확함.

시스템 간 연결이 Static Event 기반으로 잘 설계됨. StageCommonSystem.OnPlayerDeath → DeathCameraEffect, PlayerInteractionBlocker, StageSFXPlayer, StageVFXPlayer, BaseDeathEffect가 독립적으로 구독. GameManager.OnStageResultRecorded → BGMManager, SFX/VFX 플레이어. VRRigBuilder.OnRigReady는 Sticky Event 패턴(구독 시 이미 ready면 즉시 콜백)으로 레이스 컨디션 방지. VContainer StageScope가 매 스테이지 씬마다 DeathTrigger들을 자동 등록하고 StageCommonSystem을 주입. 게임 루프: Title→Lobby(스테이지 선택)→Stage(유혹→사망)→Result(재시도/로비). 씬 전환은 SceneManager.LoadScene 동기 방식으로 구현됨. 비동기 로딩이나 로딩 화면 미구현이 감점 요인.
콘텐츠 13/20점
16개 씬, 30개 프리팹, 52개 머티리얼, 48개 텍스처, 14개 FBX 모델, 4개 커스텀 셰이더(수면 셰이더, 코스틱 셰이더 등)로 에셋 규모가 상당함. 최근 커밋에서 Shark 스테이지 수중 환경(코스틱, 수면 셰이더, 수중 파티클, 뎁스 포그)을 활발히 작업 중. 하지만 오디오 파일이 4개뿐으로 매우 부족하고 애니메이션 파일이 1개뿐. DeathEffect 코드에서 AudioClip을 SerializeField로 참조하지만 실제 할당 여부 불확실. GasTankFlameEffect, FireScreenOverlay 등에서 VFX 프리팹을 코드에서 적극 사용하고 있어 시각 효과는 코드와 연결됨.
폴리싱 10/20점
UI는 LazyFollowUI(시야 이탈 시 재배치), InteractableHighlight(MaterialPropertyBlock 기반 발광 펄스), InteractableTooltip(빌보드 툴팁)으로 기본 완성도 있음. ResultScreenSystem에서 ZTest Always로 항상 최상위 렌더링 처리. SaveManager로 JSON 기반 저장/로드 구현됨. 하지만 네임스페이스 불일치 문제 있음: Core 스크립트(GameManager, VRRigBuilder, StageCommonSystem 등)는 글로벌 네임스페이스, DeathTriggers/DeathEffects/UI는 DumbWays.* 네임스페이스 사용. BaseDeathEffect.ConfigureDeathVolume()이 빈 메서드로 오버라이드되는 케이스 존재(GasTankDeathEffect, MicrowaveDeathEffect). 에러 처리는 Debug.LogWarning 수준으로 기본적이나 빈 catch 블록은 발견되지 않음.
