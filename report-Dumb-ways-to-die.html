프로젝트 현황 요약

이 팀은 Meta Quest 2용 VR 블랙코미디 데스게임 'Dumb Ways to Die'를 만들고 있으며, 현재 코어 아키텍처와 게임 루프(Title→Lobby→Stage→Death→Result)가 완전히 구현된 상태입니다. GameBootstrap/GameManager/VRRigBuilder를 중심으로 한 부트스트래핑 시스템, BaseDeathEffect 템플릿 패턴 기반의 스테이지별 사망 이펙트(Microwave 3페이즈, GasTank 폭발+연기, Mushroom 중독), 7종의 DeathTrigger, 햅틱/SFX/VFX 피드백 시스템, 그리고 JSON 기반 진행 저장까지 갖추어 코드 아키텍처 수준이 학생 팀 프로젝트로서는 매우 높습니다. 그러나 콘텐츠 측면에서 완성된 스테이지는 3개 정도이며, 오디오 에셋(4개)과 애니메이션(1개)이 심각하게 부족하고, Shark 스테이지가 미머지 브랜치에서 작업 중입니다. 가장 시급한 것은 기존 아키텍처를 활용하여 나머지 스테이지의 콘텐츠(에셋+사운드)를 빠르게 채우고, stage/shark 브랜치를 main에 머지하여 팀 전체가 최신 코드베이스에서 작업하는 것입니다.

리스크 알림

스테이지 콘텐츠 부족 — 출시 가능한 스테이지 수 제한
현재 구현된 사망 트리거가 7종이지만 실제 완성된 스테이지는 Microwave, GasTank, Mushroom(Eat) 정도이며, Shark 스테이지는 기초 세팅 단계입니다. VR 게임으로서 최소 5-6개 스테이지가 필요한데, 각 스테이지마다 고유한 사망 이펙트 + 환경 에셋 + 사운드가 필요하여 제작 시간이 상당합니다.
  • 기존 DeathTrigger 재활용하여 신규 스테이지 빠르게 프로토타이핑 (CollisionDeathTrigger는 어떤 씬에든 바로 적용 가능)
  • Shark 스테이지를 이번 주 내 최소 플레이 가능 수준으로 완성 (stage/shark 브랜치에 4커밋 진행 중)
  • 에셋 제작과 로직 구현을 병렬화 — 한 명은 씬 배치, 다른 한 명은 DeathEffect 스크립트 작성
오디오 에셋 심각하게 부족
VR 블랙코미디 데스게임에서 사운드는 몰입감의 핵심인데, 현재 오디오 파일이 4개뿐입니다. 코드에서 explosionSound, fireSound, chargingSound 등을 SerializeField로 참조하고 있지만, 실제 할당 여부가 불확실합니다. null 체크가 있어 크래시는 없지만 무음 사망 연출은 게임 경험을 크게 저해합니다.
  • Freesound.org/Mixkit 등에서 폭발, 전자레인지, 먹기, 불, 비명 등 효과음 10개 이상 즉시 확보
  • 각 DeathEffect/DeathTrigger의 AudioClip SerializeField에 실제 에셋 할당 확인
  • BGM 트랙 최소 2개(로비용, 스테이지 공통용) 추가
커밋 크기 불균형 — 대형 커밋으로 인한 머지 충돌 위험
평균 커밋 398줄, 상위 5개 커밋이 2800-4900줄 변경으로 과도하게 큽니다. 특히 Shark 스테이지 기초 세팅이 49파일/4929줄인데, 이런 대형 커밋은 코드 리뷰가 불가능하고 머지 충돌 시 해결이 극도로 어렵습니다. 현재 unmerged 브랜치가 1개(stage/shark)이며, main이 7커밋 뒤처져 있습니다.
  • 커밋 단위를 기능별로 분리 (예: 환경 에셋 추가 / AI 로직 / 이펙트 각각 별도 커밋)
  • stage/shark 브랜치를 main에 조기 머지하여 분기 격차 확대 방지
  • Unity 씬 파일(.unity)은 머지 충돌이 사실상 해결 불가능하므로, 씬 작업 전 항상 최신 main pull

stage/shark 브랜치가 main보다 4커밋 앞서 있고, main은 7커밋 뒤처져 있습니다. 시간이 지날수록 .unity 씬 파일의 머지 충돌 가능성이 급증합니다. Unity 씬은 YAML 기반이지만 실질적으로 수동 머지가 불가능합니다.

권고사항 (Top 3)

1. 공유 상수 및 설정 ScriptableObject 도입
씬 이름, 타이밍 값, UI 자식 이름 등이 코드 곳곳에 하드코딩되어 있어 변경 시 런타임 에러가 발생합니다. MicrowaveDeathEffect의 충전 시간 불일치 문제도 같은 원인입니다.
  1. GameConstants.cs에 씬 이름 상수 정의 (public const string TitleScene = "Title" 등)
  2. 스테이지별 타이밍 값을 StageTimingConfig ScriptableObject로 추출하여 DeathTrigger와 DeathEffect가 동일 소스 참조
  3. UI 프리팹의 자식 이름을 상수화하거나, [SerializeField]로 직접 참조하도록 변경
2. 오디오/에셋 파이프라인 집중 보강
코드 아키텍처는 우수하나 게임 콘텐츠(사운드, 애니메이션, 스테이지)가 부족합니다. VR 블랙코미디 장르에서 사운드 이펙트와 환경음은 재미의 핵심이며, 현재 4개 오디오로는 턱없이 부족합니다.
  1. 스테이지별 필요 사운드 목록 작성 (최소 각 3개: 환경음, 인터랙션, 사망)
  2. 무료 사운드 라이브러리에서 일괄 확보 후 StageSFXPlayer에 할당
  3. Shark 스테이지 수중 환경음 + 상어 효과음 우선 제작
3. QA 체크리스트 작성 및 빌드 테스트 루틴 확립
Meta Quest 2 실기기 테스트 커밋('fix: meta quest 2 테스트')이 있으나 체계적인 QA 프로세스가 보이지 않습니다. VR은 데스크톱과 동작이 크게 다르므로, 매 기능 완성 시 실기기 테스트가 필수입니다.
  1. 스테이지별 QA 체크리스트 작성 (인터랙션 동작, 사망 연출 재생, 결과 화면 표시, 씬 전환)
  2. 주 2회 Quest 2 빌드 테스트 일정 확보
  3. 핸드 트래킹/컨트롤러 양쪽 모드에서 테스트 (PlayerInteractionBlocker의 상태 복원 검증)

코드 리뷰

VRRigBuilder.cs양호
XR Rig 인스턴스화, EventSystem 관리, 카메라 태깅을 한 곳에서 처리하는 핵심 싱글톤입니다. 특히 씬 EventSystem을 모두 비활성화한 후 XR용 EventSystem을 생성하는 순서가 정확합니다 — XRI의 RegisteredUIInteractorCache가 interactor의 OnEnable에서 EventSystem.current를 참조하기 때문에, 이 순서가 뒤바뀌면 UI 인터랙션이 완전히 깨집니다. OnRigReady의 'sticky event' 패턴(이미 준비됐으면 즉시 콜백)도 레이스 컨디션을 방지하는 좋은 설계입니다. Asset 해상도 폴백(SerializeField → AssetDatabase 검색)도 실용적입니다.
DeathCameraEffect.cs양호
사망 연출의 전체 시퀀스를 오케스트레이션하는 핵심 스크립트입니다. Play() 메서드에서 PhysicsKnockback 유무에 따라 일반/넉백 경로를 분기하며, 넉백 경로에서는 XROrigin 상태를 넉백 적용 전에 미리 저장하는 것이 핵심입니다. ShowPanelFixed()에서 패널 배치 후 HeadTracking을 복원하는 순서도 정확합니다. OnCameraRiseComplete 이벤트로 다른 이펙트(GasTankFlameEffect 등)와 동기화하는 설계가 깔끔합니다. DeathBodyController/DeathCameraAnimator로의 위임 구조도 단일 책임 원칙을 잘 따릅니다.
StageCommonSystem.cs양호
'사망이 곧 클리어'라는 게임 컨셉의 중심 시스템입니다. 유혹 팝업(7초 간격, '저 빨간 버튼... 누르고 싶지 않아?' 등)이 게임 디자인과 잘 맞습니다. OnPlayerDeath 이벤트를 발행하여 모든 이펙트/UI/차단 시스템이 구독하도록 하는 허브 역할을 합니다. VContainer [Inject]로 DeathTrigger들에 주입되는 구조도 적절합니다. 다만 팝업 메시지가 코드에 하드코딩되어 있어, ScriptableObject로 외부화하면 기획자가 직접 수정할 수 있을 것입니다.
EatDeathTrigger.cs개선필요
Idle→Eating→Done 상태 머신이 명확하고, XRGrabInteractable 해제 후 리지드바디 비활성화하는 흐름이 정확합니다. 4개 서브시스템(EatAnimator, EatAudioPlayer, EatDebrisSpawner, EatInputReader)으로의 위임도 깔끔합니다. 그러나 eatButtonAction?.action을 null 체크 없이 .IsPressed 호출하는 부분(입력 미설정 시 NullReferenceException 위험), _stage를 KillAfterDelay에서 null 체크 없이 사용하는 부분이 문제입니다. _triggered 플래그로 재트리거를 방지하지만 씬 재로드 시 명시적 Reset()이 없어, MonoBehaviour 라이프사이클에 의존합니다.
MicrowaveDeathEffect.cs개선필요
3페이즈(충전→섬광→연기) 포스트프로세싱 이펙트가 훌륭하게 구현되었습니다. 각 페이즈별 별도 Volume(priority 102-104)을 런타임 생성하여 충돌을 방지합니다. 이벤트 구독/해제가 OnEnable/OnDisable에서 정확히 대칭됩니다. 다만 OnChargingStarted 코루틴 내 충전 시간이 3f로 하드코딩되어 MicrowaveDeathTrigger.chargeDuration과 불일치할 위험이 있습니다. 주석으로 '// TODO: match with trigger' 표시가 있지만 미해결 상태입니다. chargeDuration을 공유 상수나 ScriptableObject로 추출해야 합니다.
StageHapticPlayer.cs양호
SinglePulse/Pattern/Ramp 3가지 햅틱 모드를 코루틴 기반으로 깔끔하게 구현했습니다. HapticImpulsePlayer를 씬에서 자동 탐색하는 폴백이 실용적이고, 계층 경로를 로그에 출력하여 디버깅을 돕습니다. Play(index)/Stop(index)/StopAll() API가 UnityEvent에서 바로 사용할 수 있도록 int 인덱스 기반으로 설계된 점이 좋습니다. 각 entry의 activeCoroutine을 추적하여 중복 재생을 방지합니다. 333줄로 CLAUDE.md 권장(300줄) 초과이지만, 기능 밀도 대비 적절한 수준입니다.
ResultScreenSystem.cs개선필요
사망 메시지 표시, 재시도/로비 버튼 와이어링, Always-on-top ZTest 처리가 잘 구현되었습니다. ShowAt()으로 DeathCameraEffect에서 명시적 위치 지정이 가능하고, IsHandledByExternalEffect 플래그로 이중 표시를 방지합니다. 그러나 ApplyZTestToTMP()에서 TMP fontMaterial을 매번 new Material()로 복제하는 것은 메모리 누수 위험이 있습니다(폰트가 공유 에셋이므로). FindTMP/WireButton에서 자식 이름을 문자열로 검색하는 패턴('DeathMessageText', 'RetryButton')은 프리팹 구조 변경 시 조용히 실패합니다.
GasTankDeathEffect.cs양호
MicrowaveDeathEffect와 동일한 BaseDeathEffect 템플릿을 따르며, 섬광(Bloom+Exposure 화이트아웃)→연기(Vignette+탈색+차가운 색온도) 2페이즈 구조가 깔끔합니다. OnExplosionStarted/OnCameraRiseComplete 이벤트 기반 동기화가 정확하고, EnsureAudioSource 패턴으로 오디오 컴포넌트를 안전하게 획득합니다. OnResultDimComplete에서 모든 Volume weight를 0으로 초기화하여 다음 씬에 잔여 효과가 남지 않도록 합니다.

아키텍처 평가

VContainer를 활용한 DI(StageScope)와 정적 이벤트(OnPlayerDeath, OnExplosionStarted, OnCameraRiseComplete 등)를 결합한 하이브리드 아키텍처를 사용합니다. 싱글톤 패턴(VRRigBuilder, GameManager, BGMManager)은 DontDestroyOnLoad 영속 객체에 한정하고, 스테이지별 의존성은 VContainer로 관리하여 적절한 균형을 이루었습니다. BaseDeathEffect → MicrowaveDeathEffect/GasTankDeathEffect 등의 Template Method 패턴, InteractableStateBase → InteractableTooltip/InteractableHighlight의 추상화가 확장성을 보장합니다. 이벤트 기반 통신으로 DeathTrigger → StageCommonSystem → BaseDeathEffect/DeathCameraEffect/PlayerInteractionBlocker 간 결합도가 낮아 새 스테이지 추가가 용이합니다. 개선점으로는 이벤트 발행 순서의 문서화, 싱글톤과 DI의 경계 명확화(일부 스크립트가 FindFirstObjectByType으로 직접 검색), 그리고 에러 복구 메커니즘(코루틴 타임아웃 등)이 필요합니다.

심각도파일설명
HIGHMicrowaveDeathEffect.csOnChargingStarted 코루틴 내 충전 시간이 3f로 하드코딩되어 있으며, MicrowaveDeathTrigger.chargeDuration 값과 동기화되지 않습니다. 두 값이 불일치하면 이펙트와 실제 폭발 타이밍이 어긋납니다.
HIGHEatDeathTrigger.cseatButtonAction이 Inspector에서 미할당 시, action 프로퍼티 접근에서 NullReferenceException이 발생합니다. eatButtonAction?.action?.IsPressed() 형태의 null 조건부 접근이 필요합니다.
MEDIUMResultScreenSystem.csApplyZTestToTMP()에서 fontMaterial마다 new Material()을 생성하지만, 이 Material을 캐싱하거나 OnDestroy에서 Destroy하지 않아 메모리 누수 가능성이 있습니다. TMP 폰트가 공유 에셋이므로 원본 머티리얼이 변경될 위험도 있습니다.
MEDIUMGameManager.csLoadStage/LoadLobby/LoadTitle에서 씬 이름이 문자열 하드코딩('Title', 'Lobby')입니다. 씬 이름 변경 시 컴파일 에러 없이 런타임 실패합니다. const 또는 ScriptableObject로 추출 권장.
MEDIUMResultScreenSystem.csFindTMP/WireButton 헬퍼가 자식 오브젝트를 문자열 이름('DeathMessageText', 'RetryButton', 'LobbyButton')으로 검색합니다. 프리팹 구조 변경 시 Debug.LogWarning만 출력하고 조용히 실패하여 버튼이 동작하지 않습니다.
LOWStageHapticPlayer.cs파일이 333줄로 CLAUDE.md 권장 기준(300줄 이하)을 초과합니다. HapticEntry 클래스와 코루틴 로직을 별도 파일로 분리하면 가독성이 향상됩니다.
LOWGasTankFlameEffect.csSpawnFlames()에서 flameCount+1 크기의 배열을 할당하지만, Raycast 실패 시 일부 원소가 null인 채로 남습니다. 이후 null 접근은 없지만 불필요한 메모리 할당입니다.
LOWMicrowaveDeathTrigger.cs_triggered 플래그가 true로 설정된 후 리셋되지 않습니다. MonoBehaviour 라이프사이클(씬 재로드 시 Destroy→재생성)에 의존하므로 Additive 씬 로드 시 문제가 될 수 있습니다.
LOWInteractableTooltip.csUpdateBillboard()이 blend > 0.001f일 때 매 LateUpdate마다 호출됩니다. 툴팁이 화면 밖에 있어도 Billboard 연산이 수행되므로, 가시성 체크를 추가하면 성능이 개선됩니다.
LOWLazyFollowUI.cs_corners 배열이 static readonly로 선언되어 여러 LazyFollowUI 인스턴스가 동시에 Update할 경우 레이스 컨디션이 발생할 수 있습니다. 멀티 인스턴스 환경에서는 인스턴스별 배열이 안전합니다.
