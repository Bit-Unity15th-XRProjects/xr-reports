프로젝트 현황 요약

이 팀은 'Dumb Ways to Die' 컨셉의 VR 블랙코미디 데스게임을 개발하고 있으며, '죽는 것이 클리어 조건'이라는 독특한 게임 디자인을 중심으로 7종의 사망 트리거(충돌, 그랩, 소켓, 먹기, 가스통 폭발, 곰 덫, 전자레인지)와 8종의 사망 이펙트(화상, 감전, 폭발, 중독, 멍 등)를 실제 동작하는 코드로 구현했다. GameBootstrap→GameManager→StageCommonSystem→DeathTrigger→DeathEffect→ResultScreen으로 이어지는 전체 게임 루프가 코드 레벨에서 완성되어 있고, VContainer DI, 물리 넉백 시스템, 슬로모션 사망 카메라, 저장/로드 시스템까지 갖추고 있어 아키텍처 완성도가 학생 프로젝트치고 매우 높다. 현재 Juhyun Son이 상어 스테이지의 수중 환경(코스틱, 수면 셰이더, 파티클)을 집중 개발 중이고, tfarcenim123-afk가 UI/인터랙션 버그를 수정하고 있다. 가장 시급한 것은 오디오 에셋(4개뿐)과 애니메이션(1개뿐)의 확보로, 게임플레이 코드는 훌륭하지만 콘텐츠 레이어가 비어 있어 VR 몰입감을 크게 저하시킬 수 있다.

오디오 에셋 심각한 부족 (4개)
VR 블랙코미디 데스게임에서 사운드는 몰입감의 핵심이다. StageSFXPlayer와 EatAudioPlayer 등 사운드 재생 시스템은 완성되었으나, 실제 오디오 클립이 4개뿐이라 대부분의 스테이지에서 사운드가 비어 있을 가능성이 높다. 폭발음, 비명, 환경음, UI 효과음 등이 모두 필요하다.
  • 스테이지별 필수 사운드 목록 작성 (폭발, 전기, 먹기, 불, 비명 등 최소 15종)
  • freesound.org 등에서 CC0 효과음 확보 후 Assets/_Game/Audio에 배치
  • 각 StageSFXPlayer의 Inspector에서 클립 할당 및 트리거 타이밍 조정
상어 스테이지 브랜치(stage/shark) 미병합 — 12커밋 ahead

  • stage/shark 브랜치를 main 기준으로 rebase 또는 merge하여 3 behind 해소

  • 향후 장기 브랜치는 매일 main을 pull하여 분기 최소화
애니메이션 파일 1개 — 캐릭터/오브젝트 애니메이션 부재
BearTrapDeathTrigger의 덫 닫힘, EatAnimator의 먹기 모션, LighterIgniter의 점화 등 코드에서 시각적 피드백을 주는 시스템이 있지만, .anim 파일이 1개뿐이라 대부분 코드 기반 트윈(scale 변경 등)에 의존한다. VR에서 애니메이션 부재는 몰입 저하로 직결된다.
  • 핵심 인터랙션(곰 덫 닫힘, 전자레인지 문 열림/닫힘, 라이터 점화)에 Animation Clip 추가
  • Animator Controller 설정 후 DeathTrigger 코드에서 Animator.SetTrigger()로 연결
대용량 커밋 패턴 — 씬 파일 포함 4000+줄 커밋 빈발
상위 5개 대용량 커밋 중 4개가 씬 배치+에셋 추가를 한 커밋에 포함(4929줄, 4517줄 등). Unity .unity 파일은 YAML이라 diff가 방대해지고, 충돌 해결이 극히 어렵다. 에셋과 로직 변경이 혼재되면 롤백도 어렵다.
  • 씬 오브젝트 배치와 스크립트 로직을 별도 커밋으로 분리
  • 프리팹 워크플로우 활용 — 씬에 직접 배치 대신 프리팹 편집 후 씬에 참조
  • .gitattributes에서 *.unity를 merge=unityyamlmerge로 설정
static event 구독 해제 누락 가능성
StageCommonSystem.OnPlayerDeath, DeathCameraEffect.OnCameraRiseComplete 등이 static event여서 씬 전환 시 자동 해제되지 않는다. 대부분 OnDisable에서 해제하지만, 하나라도 누락되면 Destroyed된 객체 참조로 MissingReferenceException이 발생한다.
  • 모든 static event 구독자가 OnDisable에서 -= 하는지 일괄 검증
  • 장기적으로 MessageBus 또는 VContainer의 IMessageSubscriber 패턴으로 전환 고려

권고사항 (Top 3)

1. 폭발형 DeathEffect 공통 베이스 클래스 추출
GasTankDeathEffect와 MicrowaveDeathEffect의 섬광→릴리스→연기 시퀀스가 거의 동일한 구조를 80% 이상 공유한다. 현재 코드를 복붙하고 있어 버그 수정 시 양쪽 모두 수정해야 하며, 새로운 폭발 스테이지 추가 시 또 복붙이 필요하다.
  1. BaseExplosionDeathEffect 클래스를 만들어 섬광(flash), 릴리스(release), 연기(smoke) 코루틴을 공통으로 구현
  2. GasTankDeathEffect와 MicrowaveDeathEffect가 이를 상속하고, 충전(charging) 페이즈 등 차이점만 override
  3. 파라미터(flashAttack, bloomMax, smokeVignette 등)는 베이스 클래스의 protected 필드로 이동
2. IDeathTrigger 인터페이스 도입으로 StageScope 확장성 확보
현재 StageScope.Configure()에서 7종 DeathTrigger를 개별 foreach로 등록하므로, 새 트리거 추가 시 이 파일 수정을 빠뜨리면 런타임에 DI 실패가 발생한다. 컴파일 타임에 감지할 수 없는 위험한 패턴이다.
  1. IDeathTrigger 인터페이스(또는 abstract DeathTriggerBase) 정의
  2. 모든 DeathTrigger가 이를 구현하도록 변경
  3. StageScope에서 FindObjectsByType<MonoBehaviour>().OfType<IDeathTrigger>()로 일괄 등록
  4. 새 트리거 추가 시 StageScope 수정 불필요
3. Core 스크립트에 네임스페이스 적용
DeathTrigger(DumbWays.Gameplay.DeathTriggers)와 DeathEffect(DumbWays.Gameplay.DeathEffects)는 네임스페이스를 사용하는데, GameManager/VRRigBuilder/StageCommonSystem 등 Core 스크립트는 전역 네임스페이스에 있어 일관성이 없다. 프로젝트 규모가 커지면 이름 충돌 위험이 있다.
  1. Core 스크립트를 DumbWays.Core 네임스페이스로 이동
  2. UI 스크립트를 DumbWays.UI 네임스페이스로 이동
  3. 기존 참조 코드에 using 문 추가

코드 리뷰

GameManager.cs양호
124줄의 간결한 싱글톤으로, 씬 내비게이션(LoadTitle/LoadLobby/LoadStage), 스테이지 언락 로직(IsStageUnlocked), SaveManager 연동이 깔끔하게 구현되어 있다. InitStageFromScene()으로 에디터 직접 실행을 지원하는 점이 좋다. StageRegistry null 체크와 인덱스 범위 검증도 충실. RecordClear()와 RecordDeath()가 동일 이벤트를 isDead 플래그만 다르게 발행하는 설계가 명확하다. 네임스페이스가 없는 점만 개선하면 좋겠다.
StageCommonSystem.cs양호
146줄로 '사망이 곧 클리어'라는 게임 컨셉에 맞게 TriggerDeath() 하나로 OnPlayerDeath 발행 + RecordClear + RecordDeath를 처리한다. 유혹 팝업 시스템(popupInterval 기반 타이머 + 랜덤 메시지)이 게임 디자인과 잘 맞는다. IsDead 가드로 중복 사망을 방지하고, HidePopup()에서 StopAllCoroutines()로 팝업 코루틴을 정리하는 점도 좋다. 다만 OnPlayerDeath가 static event여서 씬 전환 후에도 이전 씬 구독자가 남을 수 있으므로 주의가 필요하다.
DeathCameraEffect.cs양호
208줄의 멀티페이즈 사망 카메라 시퀀스. Normal 경로(슬로모션→시체고정→헤드트래킹 비활성화→상승+회전→결과패널)와 Knockback 경로(물리 넉백 대기→Origin 수평화→상승+회전)를 분기 처리하며, Time.timeScale 조작과 unscaledDeltaTime 사용이 일관된다. OnCameraRiseComplete static event로 이펙트 타이밍을 외부에 알리는 설계가 좋다. VRRigBuilder.Instance?.XROriginTransform null 체크 후 즉시 fallback(ShowDeathResultImmediate)하는 방어 코딩이 인상적.
StageScope.cs개선필요
VContainer LifetimeScope를 활용한 DI 설정으로, StageCommonSystem을 서비스로 등록하고 7종 DeathTrigger를 소비자로 등록한다. 패턴 자체는 좋으나, 새로운 DeathTrigger를 추가할 때마다 이 파일에 foreach 블록을 수동으로 추가해야 하므로 확장 시 누락 위험이 있다. 인터페이스(예: IDeathTrigger)를 도입하여 FindObjectsByType<IDeathTrigger>로 일괄 등록하는 방식으로 개선하면 OCP(개방-폐쇄 원칙)에 부합한다.
EatDeathTrigger.cs양호
204줄의 복잡한 상태 머신(Idle→Eating→Done). XRGrabInteractable 그립 감지 → 먹기 버튼 홀드 → 타이머 진행 → 사망 딜레이 순서가 잘 구조화되어 있다. EatAnimator/EatAudioPlayer/EatDebrisSpawner로 책임을 위임하는 Composition 패턴이 좋다. 먹기 취소 시 Rigidbody 복원, OnEatingStarted/OnEatingCancelled 이벤트로 외부 이펙트(MushroomHallucinationOverlay)와 연동하는 설계가 세밀하다.
PhysicsKnockback.cs양호
185줄로 폭발/충돌 시 물리 넉백을 시뮬레이션한다. KnockbackHandler에 물리 컴포넌트 생명주기를 위임하고, SimulateUntilSettled()에서 벽 반사(EnforceWallBounds), 최대 이동 반경(ClampDisplacement), 수직 속도 제한(maxVerticalSpeed) 등 안전장치가 꼼꼼하다. 폭발 이벤트(GasTankDeathTrigger.OnExplosionStarted, MicrowaveDeathTrigger.OnExplosionStarted) 구독으로 자동 트리거되는 점도 좋다. 다만 구체 트리거 클래스의 static event를 직접 구독하므로 결합도가 높다.
LazyFollowUI.cs양호
188줄의 VR 친화적 UI 팔로워. 4코너 뷰포트 검사(GetOutOfViewFraction)로 화면 밖 비율을 계산하고, outOfViewThreshold를 넘으면 Lerp/Slerp로 재배치하는 설계가 VR UX에 적합하다. canvasScale 파라미터로 VRCanvasAdapter와 호환되며, SnapToFront()로 초기 배치도 처리한다. static readonly _corners 배열 재사용으로 GC 부하를 줄인 점도 좋다.
GasTankDeathEffect.cs개선필요
230줄의 가스통 폭발 사망 이펙트. 2페이즈(섬광→연기 분위기) 구조로 Bloom/ColorAdjustments/Vignette를 런타임 Volume으로 제어한다. OnExplosionStarted/OnCameraRiseComplete 이벤트에 구독하여 타이밍을 맞추는 점은 좋으나, MicrowaveDeathEffect와 코드 구조(섬광→릴리스→연기)가 거의 동일하여 중복이 상당하다. 공통 로직을 BaseExplosionDeathEffect 등으로 추출하면 유지보수성이 크게 개선될 것이다.

아키텍처 평가

프로젝트는 Singleton(VRRigBuilder/GameManager/BGMManager/VRCanvasAdapter 4개), Observer(static event 기반 — OnPlayerDeath/OnStageResultRecorded/OnCameraRiseComplete/OnExplosionStarted 등 최소 8개 이벤트), DI(VContainer StageScope), Template Method(BaseDeathEffect → ConfigureDeathVolume/OnDeath/OnResultDimComplete), Composition(EatDeathTrigger → EatAnimator/EatAudioPlayer/EatDebrisSpawner 위임) 패턴을 적절히 조합하고 있다. 특히 GameBootstrap의 RuntimeInitializeOnLoadMethod + VRRigBuilder.OnRigReady 스티키 이벤트 패턴은 'Title 씬을 거치지 않고 어떤 스테이지든 직접 테스트 가능'하게 만드는 우수한 설계다. 데이터 흐름은 단방향에 가깝다: DeathTrigger → StageCommonSystem.TriggerDeath() → OnPlayerDeath → 각 이펙트/UI 시스템. 개선점으로는 static event가 과다하여 전체 이벤트 흐름을 추적하기 어려운 점, PhysicsKnockback이 구체 트리거 클래스(GasTankDeathTrigger, MicrowaveDeathTrigger)에 직접 의존하는 점, 그리고 4개 Singleton이 DontDestroyOnLoad로 영구 존속하면서 씬 전환 시 상태 정리 복잡도를 높이는 점이 있다.

심각도파일설명
MEDIUMStageCommonSystem.csOnPlayerDeath가 static event로 선언되어 있어, 씬 전환 시 이전 씬의 구독자가 해제되지 않으면 MissingReferenceException이 발생할 수 있다. 대부분의 구독자가 OnDisable에서 해제하고 있지만, 누락 시 디버깅이 어렵다.
MEDIUMStageScope.cs7종 DeathTrigger를 개별 foreach로 등록하므로, 새 트리거 타입 추가 시 이 파일 수정을 잊으면 [Inject]가 동작하지 않아 NullReferenceException이 런타임에 발생한다. 컴파일 타임에 감지 불가.
MEDIUMGasTankDeathEffect.cs / MicrowaveDeathEffect.cs두 파일의 섬광→릴리스→연기 분위기 시퀀스 코드가 80% 이상 동일하다(flashAttack, bloomMax, exposureMax, releaseDuration, smokeVignette 등 파라미터와 코루틴 구조 복제). DRY 위반으로 버그 수정 시 양쪽 모두 수정해야 한다.
LOWGameManager.cs / VRRigBuilder.cs / StageCommonSystem.cs 등 Core 스크립트Core 스크립트 대부분이 전역 네임스페이스에 선언되어 있다. DeathTrigger/DeathEffect 쪽은 DumbWays.Gameplay.DeathTriggers/DeathEffects 네임스페이스를 사용하는데, Core는 빠져 있어 일관성이 없다.
LOWPhysicsKnockback.csGasTankDeathTrigger.OnExplosionStarted와 MicrowaveDeathTrigger.OnExplosionStarted를 직접 구독하여 구체 클래스에 의존한다. 새로운 폭발형 트리거가 추가되면 이 파일도 수정해야 하므로 OCP 위반.
LOWStageCommonSystem.csHidePopup()에서 StopAllCoroutines()를 호출하는데, 이 컴포넌트에 다른 코루틴이 추가되면 의도치 않게 전부 중단될 수 있다. 특정 코루틴 참조를 저장하고 StopCoroutine()으로 개별 중단하는 것이 안전하다.
LOWVRCanvasAdapter.cs씬 로드 시 FindObjectsByType<Canvas>로 모든 Canvas를 World Space로 변환하는데, 의도적으로 Overlay/ScreenSpace로 남겨야 하는 Canvas(예: 디버그 UI)도 변환될 수 있다. 태그나 레이어 기반 필터링이 없다.
LOWBGMManager.csOnSceneLoaded에서 씬 이름을 하드코딩('Title', 'Lobby')으로 비교한다. GameManager의 titleSceneName/lobbySceneName과 동기화되지 않아, 씬 이름 변경 시 불일치 가능성이 있다.
