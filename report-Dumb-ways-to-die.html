프로젝트 현황 요약

이 팀은 'Dumb Ways to Die' IP를 VR 블랙코미디 데스게임으로 재해석하고 있으며, 2주간의 집중 개발로 핵심 프레임워크를 거의 완성했습니다. VRRigBuilder/GameManager/GameBootstrap으로 구성된 부트스트랩 시스템, BaseDeathEffect/DeathCameraEffect를 중심으로 한 모듈식 사망 연출 시스템, 그리고 7종의 DeathTrigger(독버섯 먹기, 가스통 폭발, 전자레인지, 곰덫 등)가 실제 동작하는 로직으로 구현되어 있습니다. Title→Lobby→Stage→Result 게임 루프가 완성되어 있고, SaveManager로 진행 상황이 저장됩니다. 가장 부족한 부분은 ScoreManager의 부재(점수 시스템 없음), 오디오 에셋의 극도 부족(4개), 그리고 상어 스테이지의 미병합 상태입니다. 가장 시급한 것은 ScoreManager 구현으로, 이것 없이는 플레이어에게 성과 피드백을 줄 수 없어 게임의 리플레이 가치가 크게 떨어집니다. 코드 품질과 아키텍처는 학생 프로젝트로서 매우 우수한 수준이며, 특히 이벤트 기반 설계와 VContainer DI 활용은 프로페셔널한 수준에 근접합니다.

ScoreManager 부재 — 게임의 핵심 피드백 루프 없음
'Dumb Ways to Die'는 점수 기반 게임입니다. 현재 GameManager에 RecordClear/RecordDeath만 있고, 생존 시간/사망 횟수/스테이지별 성적 등 점수 개념이 전혀 없습니다. 플레이어가 '잘했는지 못했는지' 알 수 없어 리플레이 동기가 없습니다.
  • ScoreManager 싱글톤 생성 — 스테이지별 생존시간, 사망원인, 시도횟수 기록
  • ResultScreenSystem에 점수 표시 UI 추가
  • LobbyManager에 스테이지별 최고 점수/등급 표시
상어 스테이지(stage/shark) 미병합 — 팀 작업 분리 리스크
stage/shark 브랜치가 main보다 3커밋 앞서 있고, 4929줄 변경의 대형 커밋(3b44c4f)을 포함합니다. Suimono 에셋, PostProcess 효과 등 대규모 변경이 오래 분리되면 병합 충돌이 기하급수적으로 증가합니다.

오디오 에셋 극도로 부족 — VR 몰입감 저해
오디오 파일이 4개뿐입니다. VR 게임에서 공간 오디오는 몰입감의 핵심이며, '사망 게임' 장르에서 비명/폭발/환경음 등이 없으면 블랙코미디 톤을 전달할 수 없습니다. StageSFXPlayer 시스템은 구현되어 있지만 재생할 에셋이 부족합니다.
  • 스테이지별 최소 3개 SFX 확보 (환경음, 사망효과음, UI피드백)
  • 무료 에셋(freesound.org 등)에서 CC0 오디오 수집
  • BGMManager가 이미 존재하므로 스테이지별 BGM 추가
PhysicsKnockback null 경로에서 OnComplete 미발화
XROriginTransform이 null일 때 Apply()가 조기 반환하면서 OnComplete를 호출하지 않습니다. DeathCameraEffect가 이 이벤트를 기다리는 경우 사망 시퀀스가 영구 정지되어, QA에서 재현하기 어려운 간헐적 버그가 됩니다.
  • Apply() 메서드의 null 반환 경로에 OnComplete?.Invoke() 추가
  • 또는 HasCompleted = true로 설정하여 폴링 경로도 안전하게 처리

평균 398줄, 최대 4929줄(49파일) 변경의 커밋이 있습니다. 이 크기의 커밋은 코드 리뷰가 사실상 불가능하고, 버그가 유입되어도 어떤 변경이 원인인지 추적이 어렵습니다. 중간값(56줄)과 평균(398줄)의 차이가 큰 것은 소수의 대형 커밋이 품질을 낮추고 있다는 의미입니다.

DeathCameraEffect.cs양호
VR 사망 시퀀스의 핵심 오케스트레이터로, Play() 메서드에서 XROrigin null 체크 후 폴백(ShowDeathResultImmediate)을 제공합니다. 일반 경로(SlowMo→Freeze→Rise→Rotate→Result)와 PhysicsKnockback 경로를 분기 처리하며, OnCameraRiseComplete static 이벤트로 다른 이펙트 시스템과 느슨하게 연결됩니다. DeathBodyController/DeathCameraAnimator/HeadTrackingController로 책임이 잘 분리되어 있습니다. timeScale 조작과 WaitForSecondsRealtime 사용이 적절합니다.
VRRigBuilder.cs양호
Singleton 패턴이 모범적입니다 — OnDestroy에서 Instance == this 가드로 중복 인스턴스 파괴 시 Instance가 null되는 버그를 방지합니다. OnRigReady 'sticky' 이벤트는 구독 시점에 관계없이 콜백을 보장하는 영리한 패턴입니다. XR Rig 프리팹을 Resources에서 자동 로드하고, EventSystem 중복 생성을 방지합니다. 166줄로 단일 책임 원칙을 잘 지킵니다.
EatDeathTrigger.cs양호
State enum(Idle/Eating/Done)으로 상태 전이가 명확하고, OnEatingStarted/OnEatingCancelled static 이벤트로 다른 시스템(MushroomDeathEffect 등)과 통신합니다. VContainer [Inject]로 StageCommonSystem을 주입받아 결합도를 낮췄습니다. XRGrabInteractable + RequireComponent로 필수 컴포넌트를 강제합니다. eatDuration(2.5초)/deathDelay 같은 타이밍이 SerializeField로 조절 가능합니다.
StageHapticPlayer.cs개선필요
333줄로 CLAUDE.md 권장 라인 수(300줄)를 초과합니다. HapticEntry 내부 클래스에 SinglePulse/Pattern/Ramp 3가지 모드의 필드가 모두 포함되어 있어, 사용하지 않는 모드의 필드가 Inspector에 노출됩니다. Play(int index)에서 범위 체크 후 Debug.LogWarning을 출력하는 것은 좋지만, StopAll()에서 entry.activeCoroutine null 체크만 하고 StopCoroutine 호출 후 null 할당이 코드에서 보이지 않아 중복 Stop 호출 시 문제가 될 수 있습니다. HapticEntry를 별도 파일로 분리하고, 모드별 데이터를 구조체로 나누는 것을 권장합니다.
MicrowaveDeathEffect.cs개선필요
BaseDeathEffect를 상속하면서 ConfigureDeathVolume()을 빈 메서드로 오버라이드합니다. 이는 Template Method 패턴의 불필요한 강제 구현으로, BaseDeathEffect에서 virtual(abstract 대신)로 선언하는 것이 더 적절합니다. OnEnable/OnDisable에서 MicrowaveDeathTrigger.OnChargingStarted/OnExplosionStarted static 이벤트를 구독/해제하는 패턴은 올바르지만, 281줄로 복잡도가 높습니다. 충전→섬광→연기 3단계 PostProcess Volume 관리를 별도 헬퍼로 추출할 수 있습니다.
ResultScreenSystem.cs개선필요
ShowDeathResult()와 ShowDeathResultImmediate() 두 진입점이 있어 호출자가 혼동할 수 있습니다. IsHandledByExternalEffect 플래그로 외부 이펙트 연동을 제어하는데, 이 상태 관리가 암묵적입니다. FindTMP()와 WireButton()이 string 기반으로 자식 오브젝트를 찾아 리팩토링에 취약합니다. 다만 SetActive() null 체크 헬퍼와 ShowAt(position, rotation)으로 VR 공간 배치를 지원하는 점은 좋습니다.
PhysicsKnockback.cs개선필요
185줄에 13개의 SerializeField가 있어 Inspector 설정이 복잡합니다. wallLayer, maxRadius, maxVerticalSpeed 등 경계 안전장치는 VR에서 필수적이지만, 이 설정값들을 ScriptableObject(PhysicsKnockbackConfig)로 추출하면 스테이지 간 공유가 가능합니다. Apply(Vector3 direction)에서 VRRigBuilder.Instance?.XROriginTransform null 체크 후 즉시 return하는데, 이 경우 OnComplete 이벤트가 발화되지 않아 DeathCameraEffect가 무한 대기할 수 있습니다.
GasTankDeathEffect.cs양호
BaseDeathEffect 상속 구조를 잘 따르며, OnEnable/OnDisable에서 이벤트 구독/해제가 대칭적입니다. _flashVolume과 _smokeVolume 두 PostProcess Volume을 관리하면서 OnResultDimComplete()에서 weight를 0으로 초기화하여 클린업이 확실합니다. GasTankDeathTrigger.OnExplosionStarted와 DeathCameraEffect.OnCameraRiseComplete 이벤트를 조합하여 폭발→섬광→연기 시퀀스를 구현한 설계가 깔끔합니다.

아키텍처 평가

이벤트 드리븐 아키텍처를 핵심으로, Singleton(VRRigBuilder, GameManager, BGMManager), Observer(OnPlayerDeath, OnRigReady, OnStageResultRecorded), Template Method(BaseDeathEffect → 6개 구현체), Dependency Injection(VContainer StageScope)을 조합한 설계입니다. 특히 VRRigBuilder의 'sticky' OnRigReady 이벤트 패턴은 Unity의 비동기 초기화 문제를 우아하게 해결하며, RuntimeInitializeOnLoadMethod 기반 부트스트랩으로 어떤 씬에서든 Play 가능한 구조를 만들었습니다. 클래스 간 결합도가 낮아 새 스테이지 추가 시 기존 코드 수정이 불필요한 OCP(개방-폐쇄 원칙)를 잘 따릅니다. 다만 static event 의존도가 높아 이벤트 구독 해제를 놓치면 씬 전환 후 NullReferenceException이 발생할 수 있고, DeathCameraEffect→DeathBodyController→DeathCameraAnimator→HeadTrackingController→PhysicsKnockback으로 이어지는 사망 시퀀스 체인이 암묵적 순서 의존성을 가져 디버깅이 어려울 수 있습니다.
