프로젝트 현황 요약

이 팀은 VR 환경에서 사무실 물건을 무기로 파괴하는 시뮬레이터를 만들고 있으며, 현재 핵심 게임플레이 루프(무기 스윙 → 오브젝트 파괴 → 점수 획득 → 스테이지 클리어)가 Tutorial부터 Office3까지 4개 스테이지에 걸쳐 동작하는 상태까지 구현되었다. UI 시스템은 MVP 패턴으로 잘 설계되어 스테이지 선택, 볼륨 조절, 팝업 등이 이벤트 기반으로 연결되어 있고, OpenFracture 라이브러리를 활용한 물리 파괴 + FragmentCleaner의 메모리 관리도 갖추어져 있다. 그러나 VR 성능에 치명적인 GameObject.Find() 사용, 매니저 싱글톤의 DontDestroyOnLoad 미보장, 하드코딩된 테스트 데이터('ASLDKSA;FLSKAAGASG')와 부적절한 문자열('FUCK...')이 프로덕션 코드에 남아있는 것이 주요 문제다. 가장 시급한 것은 파괴 시 매 프레임 발생하는 성능 병목(Find, OverlapBox 필터 누락) 수정과, 발표 전 하드코딩 데이터 정리이다. 또한 팀원 3명 중 1명(verpivia-1)이 67%의 커밋을 담당하고 있어, 나머지 팀원의 기여도 확대와 코드 리뷰 도입이 필요하다.

리스크 알림

VR 프레임 드롭으로 인한 멀미 위험
ThrowableBreakable.BreakObject()의 GameObject.Find(), GameManager.BindSceneReferences()의 FindObjectsByType<Transform>, Weapon의 필터 없는 OverlapBox가 VR 90fps 유지를 위협함. VR에서 프레임 드롭은 직접적인 사용자 멀미(VR sickness)로 이어짐.
  • GameObject.Find를 Fracture 콜백 기반 참조로 교체
  • FindObjectsByType<Transform>를 GameObject.Find(name) 또는 SerializeField 직접 참조로 변경
  • OverlapBox에 breakableLayer 마스크 파라미터 추가
매니저 싱글톤 씬 전환 시 소실 가능성
GameManager, UIManager, AudioManager, ScoreManager 4개 싱글톤이 자체적으로 DontDestroyOnLoad를 호출하지 않음. Eternitiy 컴포넌트가 같은 GameObject에 붙어있어야만 유지되는데, 이 의존성이 코드에서 보장되지 않음. 씬 전환 시 매니저가 파괴되면 전체 게임 로직이 중단됨.
  • 각 매니저 Awake()에서 DontDestroyOnLoad(gameObject) 직접 호출 추가
  • Eternitiy 클래스 의존 제거 또는 명시적 문서화
  • 에디터 Play 모드에서 Tutorial→Office1 전환 테스트로 매니저 생존 확인
테스트 데이터가 실제 UI에 노출
StageData.cs의 'ASLDKSA;FLSKAAGASG', GameManager의 'FUCK...How do you find out...' 등 개발용 문자열이 게임 내 UI에 그대로 표시됨. 학교 발표/시연 시 교수나 평가자에게 부적절한 인상을 줄 수 있음.
  • StageData의 _value 리스트를 실제 스테이지 설명으로 교체
  • GameManager.ShowClearMessage()의 문자열을 적절한 클리어 메시지로 변경
  • 전체 코드에서 하드코딩된 한글/영문 문자열을 검색하여 검토
기여도 불균형과 브랜치 관리
verpivia-1이 54커밋 중 36개(67%)를 담당하고, Test/UI_Archetectures 브랜치가 49커밋 뒤처져 있어 사실상 방치됨. yuno3384는 2커밋으로 기여가 거의 없음. 한 사람에 대한 의존도가 높으면 병목 및 코드 리뷰 부재로 이어짐.
  • 방치된 Test/UI_Archetectures 브랜치 정리(삭제 또는 병합)
  • 김연준, yuno3384에게 구체적 태스크 분배(예: StageData JSON 파싱, 저장/로드 시스템)
  • 코드 리뷰 프로세스 도입(최소 PR 1개당 1명 리뷰)
Office3 이후 엔딩 플로우 취약

  • EndingPopup이 없을 때의 폴백 처리 추가

권고사항 (Top 3)

1. 성능 크리티컬 코드 즉시 수정
VR은 프레임 드롭에 극도로 민감하며, 현재 3곳의 비효율적 탐색(GameObject.Find, FindObjectsByType, 필터 없는 OverlapBox)이 파괴 시마다 실행됨. 이는 게임의 핵심 메카닉(물건 파괴)에서 발생하므로 가장 빈번하게 트리거됨.
  1. Weapon.cs:57의 Physics.OverlapBox에 breakableLayer 파라미터 추가: Physics.OverlapBox(worldCenter, boxSize/2f, transform.rotation, breakableLayer)
  2. FractureFinalRunner.cs에서 GameObject.Find 대신 Fracture 콜백 또는 Transform.Find 활용
  3. GameManager.BindSceneReferences()를 SerializeField 참조 또는 태그 기반으로 변경
2. 하드코딩 데이터를 ScriptableObject 또는 JSON으로 분리
현재 SoundData, StageData, BGMEvent, NextStageButton, GotoEvent 등 5개 이상의 파일에 씬 이름과 데이터가 문자열로 분산 하드코딩됨. 새 스테이지 추가 시 최소 6개 파일을 동시에 수정해야 하며, 하나라도 누락하면 런타임 에러 발생.
  1. StageConfig ScriptableObject 생성: 씬 이름, 클리어 점수, BGM 클립, 다음 씬을 하나로 관리
  2. GameManager, BGMEvent, NextStageButton이 이 ScriptableObject를 참조하도록 리팩터링
  3. StageData의 텍스트 데이터도 JSON 또는 ScriptableObject로 이전
3. Namespace 오타 수정 및 코드 정리
'Pressentation' 오타가 12개 이상 파일에 적용되어 있어 코드 검색과 유지보수에 혼란을 줌. 'Eternitiy' 오타, 빈 FractureFinalRunner 클래스, 미사용 breakableLayer 필드 등 작은 이슈들이 코드 품질 인상을 떨어뜨림.
  1. IDE의 전체 리팩터링 기능으로 VRProject.UI.Pressentation → VRProject.UI.Presentation 일괄 변경
  2. 빈 FractureFinalRunner 클래스를 삭제하고 ThrowableBreakable 내부에서 별도 MonoBehaviour 없이 코루틴 실행
  3. Eternitiy → Eternity 이름 수정

코드 리뷰

GameManager.cs개선필요
StageClearData 구조체로 씬별 클리어 조건을 관리하고, Dictionary 빠른 탐색 + OnRealScoreChanged 이벤트 구독으로 실시간 클리어 판정하는 설계가 우수함. 중복 구독 방지(OnRealScoreChanged -= / +=)도 적절. 그러나 BindSceneReferences()에서 FindObjectsByType<Transform>(FindObjectsInactive.Include)로 씬 전체 Transform을 탐색하는 것은 매우 비효율적. 또한 clearMessageText에 'FUCK...How do you find out...'라는 부적절한 문자열이 하드코딩되어 있어 발표/데모 시 문제될 수 있음.
UIManager.cs양호
7종의 C# event를 통한 중앙 이벤트 허브 설계가 깔끔하고, Awake에서 모든 이벤트 객체를 초기화한 뒤 구독하는 패턴이 일관됨. TriggerXxx() 메서드로 외부 접근점을 명확히 제공. Singleton 패턴도 적절. 다만 OnStagePopup 이벤트의 람다에서 3개 메서드를 순차 호출하는 부분은 StagesetPopupEvent 내부에 하나의 메서드로 통합하는 것이 좋겠음.
Weapon.cs개선필요
FixedUpdate에서 수동 속도 계산 후 OnCollisionEnter에서 속도 기반 데미지 적용하는 구조가 적절. BreakableHealth가 없는 객체에 대해 Fracture 직접 호출하는 폴백도 좋음. ReleaseNearbyObjects()의 OverlapBox로 주변 물리 해방 아이디어도 흥미로움. 문제점: OverlapBox의 layerMask 파라미터를 전달하지 않아 모든 레이어의 콜라이더를 탐색함(breakableLayer 필드가 선언만 되고 사용되지 않음). 또한 damage = speed * 10f 매직넘버가 하드코딩됨.
FractureFinalRunner.cs개선필요
ThrowableBreakable 클래스가 실질적 로직 담당. XRGrabInteractable 이벤트(selectEntered/selectExited)로 grab/release 물리 전환, SuperStabilizeRoutine 코루틴으로 fragment 물리 안정화(isKinematic → trigger → damping → destroy) 3단계 처리가 잘 설계됨. 문제점: (1) OnRelease에서 SetPhysics(false)를 호출하여 놓을 때도 isKinematic=false가 되는데, 이는 OnGrab과 동일한 동작이라 의미 불명. (2) GameObject.Find(fragmentsName)은 VR에서 프레임 드롭 원인. (3) FractureFinalRunner 클래스가 빈 껍데기(7줄)로 MonoBehaviour를 낭비.
FragmentCleaner.cs양호
기존 List 기반 InvokeRepeating에서 Queue<KeyValuePair<float, GameObject>> 기반 Update 방식으로 개선된 것이 diff에서 확인됨. MaxFragmentRoots=10 하드캡으로 VR 성능 보호, FIFO 순서로 시간 경과 체크가 O(1)으로 효율적. static 메서드 Register()로 외부 접근이 깔끔함. OnDestroy에서 큐 클리어도 적절.
WeaponScoreAdder.cs양호
HashSet<Fracture>로 중복 콜백 등록을 방지하고, Fracture.callbackOptions.onCompleted에 UnityAction을 등록한 뒤 콜백 내에서 자동 RemoveListener + HashSet 제거하는 자가 정리 패턴이 인상적. 'Real' 태그 검사로 Score/RealScore를 분리하는 게임 디자인도 명확함. callbackOptions.onCompleted null 체크도 포함되어 있어 방어적 프로그래밍이 양호.
BGMEvent.cs양호
Dictionary<string, AudioClip> 캐시로 Resources.Load 반복 호출을 방지하는 최적화가 적절. switch expression으로 씬→인덱스 매핑이 깔끔. null 체크와 범위 검사가 모두 포함됨. 다만 씬 이름이 하드코딩되어 있어 새 씬 추가 시 이 파일도 수정해야 하는 OCP 위반이 있음.
StageData.cs심각
스테이지 데이터가 하드코딩된 리스트('Test game', 'This game', 'ASLDKSA;FLSKAAGASG')로 구성됨. TODO 주석에 'JSON 파싱으로 변경 예정'이라 적혀 있지만 미구현 상태. 이 데이터가 실제 게임 스테이지 선택 화면에 표시되므로 발표 전 반드시 실제 데이터로 교체 필요. 또한 매번 StageButtonInfo.Start()마다 new StageData()로 새 인스턴스를 생성하여 불필요한 할당 발생.

아키텍처 평가

UI 시스템에 MVP(Model-View-Presenter) 패턴을 적용한 것이 이 프로젝트의 가장 큰 설계적 강점. Core 레이어(인터페이스 IInteractable<T>, IGetUIType + 데이터 SoundData/StageData), Model 레이어(이벤트 클래스 + 매니저 싱글톤), Presentation 레이어(ScoreDisplay, BGMPlayer 등 표시 전용), Presenter 레이어(Close, Goto 등 입력 처리)로 명확히 분리됨. UIManager가 Observer 패턴의 이벤트 허브로서 모든 UI 상호작용을 중앙 관리하고, GameManager-ScoreManager 간 C# event 기반 느슨한 결합이 잘 구현됨. 그러나 4개 싱글톤 매니저가 서로의 Instance에 직접 접근하는 점은 Service Locator 안티패턴에 가까우며, 초기화 순서 의존성이 암묵적으로 존재함. Fracture 시스템은 외부 라이브러리(OpenFracture)를 잘 래핑하여 BreakableHealth→Fracture→FragmentCleaner 파이프라인을 구축했으나, 문자열 기반 오브젝트 탐색(GameObject.Find)으로 인해 이 통합 지점이 가장 취약함.

HIGHFractureFinalRunner.cs:71ThrowableBreakable.BreakObject()에서 GameObject.Find(fragmentsName)를 사용하여 씬 전체를 문자열로 탐색. VR 60fps 환경에서 프레임 드롭의 직접적 원인이 될 수 있음. Fracture.CauseFracture()의 콜백을 통해 생성된 fragments 참조를 받아오는 방식으로 변경 필요.
