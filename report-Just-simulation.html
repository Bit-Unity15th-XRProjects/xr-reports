프로젝트 현황 요약

이 팀은 VR 파괴 시뮬레이터 'Just-simulation'을 개발하고 있으며, OpenFracture 기반의 물리 파괴 시스템 위에 MVP 아키텍처의 UI/게임 로직 레이어를 구축했다. 현재 핵심 게임플레이 루프(무기로 오브젝트 파괴→점수 획득→스테이지 클리어→다음 스테이지 진행)가 GameManager, ScoreManager, Weapon, Fracture, NextStageButton을 통해 완성되어 있고, AudioManager가 BGM 자동 전환과 히트/사망 사운드를 처리한다. 가장 시급한 문제는 ThrowableBreakable의 OnGrab/OnRelease 물리 버그로, 이것이 수정되지 않으면 VR에서 물체를 잡는 기본 인터랙션이 불안정해질 수 있다. 또한 최근 추가된 death 사운드가 두 경로에서 중복 재생되는 문제와, Fracture의 태그 필터 로직이 반전되어 있는 버그도 시연 전에 반드시 해결해야 한다. 전반적으로 3명 중 verpivia-1이 시스템 개발의 대부분을 담당하고 있어 팀 기여도 균형이 필요하지만, 코드 품질과 아키텍처 설계는 학생 프로젝트 기준으로 매우 우수한 수준이다.
!

리스크 알림

ThrowableBreakable의 OnGrab/OnRelease 물리 버그
VR 파괴 시뮬레이터에서 물체를 잡아서 던지는 것이 핵심 인터랙션인데, OnGrab과 OnRelease가 동일하게 SetPhysics(false)를 호출하면 잡는 순간 물리가 활성화되어 손에서 떨어질 수 있음. 시연 시 '물체를 잡을 수 없음' 현상으로 프로젝트 전체의 게임플레이가 작동하지 않을 위험.
  • FractureFinalRunner.cs 라인43의 OnGrab을 SetPhysics(true)로 수정
  • OnRelease는 현재대로 SetPhysics(false) 유지
  • XR Device Simulator로 잡기→던지기→파괴 시나리오 테스트
Death 사운드 이중 재생 문제
최근 커밋(da5bc00, fb2da67)에서 ThrowableBreakable.BreakObject()에 PlayDeathSound() 호출을 추가했는데, 이미 Stage3DeathSoundTrigger가 onFracture 이벤트로 같은 함수를 호출하고 있음. 시연 시 매번 사운드가 겹쳐 재생되어 어색한 오디오 경험 발생.
  • ThrowableBreakable.BreakObject()에서 AudioManager.Instance?.PlayDeathSound() 라인 제거
  • Stage3DeathSoundTrigger에 사운드 재생을 일원화
  • Office3 씬에서 오브젝트 파괴 시 사운드 1회만 재생되는지 확인
Fracture filterCollisionsByTag 로직 반전
Fracture.OnCollisionEnter()에서 filterCollisionsByTag가 false이면 파괴가 절대 트리거되지 않음. 이는 직접 CauseFracture()를 호출하는 Weapon.cs 경로에서는 문제없지만, 일반 충돌 기반 파괴 오브젝트는 Inspector에서 filterCollisionsByTag를 반드시 true로 설정하고 태그를 등록해야만 작동함. 설정 누락 시 '부딪쳐도 안 깨짐' 버그 발생.
  • Fracture.OnCollisionEnter() 조건을 (!triggerOptions.filterCollisionsByTag || tagAllowed)로 수정
  • 또는 모든 파괴 오브젝트의 Inspector에서 filterCollisionsByTag=true + 허용 태그 확인
  • 빌드 전 각 스테이지의 파괴 오브젝트 동작 테스트
팀 기여도 불균형 및 마감 리스크
verpivia-1이 전체 78커밋 중 54커밋(69%)을 담당하고, yuno3384는 8일간 비활성. 최근 4주간 커밋 트렌드(1→18→33→26)가 마감 직전 집중 개발 패턴을 보이며, 거대 커밋(4843줄, 4182줄)이 존재하여 코드 리뷰 없이 대량 변경이 이루어지고 있음.
  • yuno3384의 Test/UI_Archetectures 브랜치(74커밋 뒤처짐) 상태 확인 및 머지 결정
  • 남은 작업을 팀원별로 분배하고 일일 체크인 실시
  • verpivia-1의 번아웃 방지를 위해 나머지 팀원에게 테스트/문서화 작업 위임
저장/로드 시스템 미구현
현재 스테이지 진행도가 영속화되지 않아 앱을 종료하면 처음부터 다시 시작해야 함. VR 시연 시 모든 스테이지를 순서대로 플레이해야 하므로, 중간 스테이지부터 시연하려면 직접 씬을 로드해야 하는 불편함.
  • PlayerPrefs로 최소한의 진행도(마지막 클리어 스테이지) 저장 구현
  • 스테이지 선택 UI에서 클리어된 스테이지까지만 선택 가능하도록 제한
  • 시연용으로 모든 스테이지 잠금 해제 옵션 추가 고려

권고사항 (Top 3)

1. ThrowableBreakable 물리 버그 즉시 수정 및 통합 테스트
OnGrab/OnRelease 버그, 사운드 이중 재생, filterCollisionsByTag 로직 반전 등 3개의 런타임 버그가 시연 품질에 직접 영향을 미침. 특히 물리 버그는 게임의 핵심 메카닉을 망가뜨릴 수 있음.
  1. FractureFinalRunner.cs의 OnGrab을 SetPhysics(true)로 수정
  2. ThrowableBreakable.BreakObject()에서 PlayDeathSound() 제거
  3. Fracture.OnCollisionEnter()의 태그 필터 조건 수정
  4. 각 스테이지에서 잡기→던지기→파괴→점수→클리어 플로우 전체 테스트
2. WeaponHitSound에 충돌 필터링 추가
현재 모든 충돌에서 히트 사운드가 재생되어 VR에서 과도한 사운드 피드백이 발생. 프레젠테이션 시 전문성이 떨어져 보일 수 있음.
  1. 충돌 대상에 Fracture 또는 BreakableHealth 컴포넌트가 있는지 확인하는 조건 추가
  2. Weapon.cs와 동일한 최소 속도 임계값(minVelocityThreshold) 체크 추가
  3. 충돌 속도에 비례한 볼륨 조절 고려
3. 시연 안정성을 위한 FragmentCleaner 개선
static 큐와 MonoBehaviour 생명주기의 불일치로 씬 전환 시 프래그먼트 메모리 누수 또는 추적 손실이 발생할 수 있음. VR에서 메모리 부족은 프레임 드롭과 멀미로 직결됨.
  1. FragmentCleaner를 DontDestroyOnLoad 싱글턴으로 변경하거나, OnDestroy에서 큐의 모든 오브젝트를 실제로 Destroy 후 Clear
  2. SuperStabilizeRoutine에서 root null 체크 추가
  3. VR 빌드에서 3스테이지 연속 플레이 시 메모리 사용량 모니터링

코드 리뷰

Fracture/Model/FractureFinalRunner.cs심각
ThrowableBreakable.OnGrab()과 OnRelease()가 모두 SetPhysics(false)를 호출하는 버그가 있음(라인 43-44). OnGrab은 SetPhysics(true)(kinematic=true)여야 잡았을 때 물리를 비활성화하고, OnRelease는 SetPhysics(false)(kinematic=false)여야 놓았을 때 물리를 활성화함. 현재 코드는 잡든 놓든 항상 kinematic=false가 되어 XR Grab 중 물체가 떨어질 수 있음. SuperStabilizeRoutine은 3단계 안정화(키네마틱→트리거→물리 복원)가 잘 설계되었으나, 코루틴 중 root가 외부에서 파괴되면 MissingReferenceException 발생 가능.
UI/Model/Manager/GameManager.cs양호
StageData JSON에서 클리어 조건을 로드하고, ScoreManager 이벤트를 구독하여 스테이지 클리어를 판정하는 핵심 게임 루프가 잘 구현됨. OnSceneLoaded에서 BindSceneReferences()로 씬별 Canvas/Text를 동적 탐색하는 방식이 DontDestroyOnLoad 싱글턴에 적합. _clearedThisStage 플래그로 중복 클리어 방지도 적절. SubscribeToScoreManager()에서 _isSubscribed 플래그를 사용하지만, 한 번 구독 후 ScoreManager가 재생성되면 갱신되지 않는 미묘한 제약이 있음.
UI/Model/Manager/AudioManager.cs양호
BGM/HitSound/DeathSound 3가지 오디오 카테고리를 이벤트 클래스로 분리하고, PlayOneShot으로 동시 재생을 지원하는 설계가 적절함. PlayDeathSound()에서 StageData로 Stage3 씬 이름을 확인하는 것은 하드코딩 방지에 좋으나, Stage3 전용 로직이 AudioManager에 있어 다른 스테이지에 비슷한 기능 추가 시 확장성이 제한됨. UIManager.Instance?.EffectVolume으로의 null-safe 볼륨 참조가 적절.
UI/Presenter/Weapon/WeaponScoreAdder.cs양호
Fracture.callbackOptions.onCompleted에 람다 콜백을 등록하고, 콜백 내에서 자체 RemoveListener + HashSet 제거를 수행하는 일회성 패턴이 정교하게 구현됨. 'Real' 태그로 실제 점수 대상을 구분하는 설계도 적절. 다만 onCompleted가 null인 경우의 방어 로직(라인 20-24)은 CallbackOptions 생성자에서 항상 초기화하므로 Inspector 직렬화 문제 시에만 발생할 수 있는 엣지 케이스.
Fracture/Fracture.cs개선필요
OpenFracture 기반 파괴 시스템의 핵심 컴포넌트로 Collision/Trigger/Keyboard 3가지 트리거 모드를 지원. ComputeFracture()에서 동기/비동기 분기 처리, CreateFragmentTemplate()에서 머티리얼/콜라이더/리지드바디 복사가 체계적. 그러나 동기 모드에서 Fragmenter.Fracture()가 메인 스레드를 블로킹하므로 복잡한 메시에서 프레임 드롭 발생 가능. CopyFractureComponent()에서 callbackOptions를 참조 복사하므로 원본과 프래그먼트가 같은 콜백 인스턴스를 공유하는 문제가 있음.
Fracture/Model/Weapon.cs양호
FixedUpdate에서 수동 속도 계산(위치 변화/deltaTime)으로 XR 컨트롤러의 실제 스윙 속도를 추적하는 방식이 적절함. OverlapBox로 주변 Breakable 객체의 물리 제약을 해제하는 ReleaseNearbyObjects()가 VR 파괴 시뮬레이터에 맞는 연출 제공. OnDrawGizmosSelected로 감지 범위를 시각화한 점도 좋음. 다만 OnCollisionEnter는 빠른 스윙에서 충돌을 놓칠 수 있어 ContinuousDynamic 충돌 감지 모드 설정이 필요할 수 있음.
UI/Core/Stagedata/StageData.cs양호
JSON 파싱으로 스테이지 설정을 데이터 드리븐하게 관리하는 핵심 클래스. RuntimeInitializeOnLoadMethod로 도메인 리로드 시 캐시 초기화가 적절하고, IReadOnlyDictionary/IReadOnlyList로 외부 수정을 방지. 다만 일반 C# 클래스(MonoBehaviour 아님)의 Singleton이라 Resources.Load를 생성자에서 호출하는데, 이는 메인 스레드에서만 안전함.
UI/Presenter/Weapon/WeaponHitSound.cs개선필요
OnCollisionEnter에서 조건 없이 AudioManager.PlayHitSound()를 호출하므로, 바닥/벽/다른 무기 등 모든 충돌에서 히트 사운드가 재생됨. 최소한 충돌 대상이 Fracture 컴포넌트를 가진 파괴 가능 객체인지, 또는 속도가 임계값 이상인지 확인하는 필터링이 필요함. 현재 상태로는 무기를 바닥에 내려놓기만 해도 히트 사운드가 재생되는 문제 발생.

아키텍처 평가

프로젝트는 UI 레이어에 MVP(Model-View-Presenter) 패턴을 성공적으로 적용하고 있음. Model 레이어(Manager 싱글턴 + Event 클래스)가 비즈니스 로직을 캡슐화하고, Presenter 레이어(onClicke/, BGM/, Weapon/ 폴더)가 Unity UI 컴포넌트와 Model을 연결하는 구조가 일관됨. 특히 UIManager가 이벤트 어그리게이터로서 7가지 UI 이벤트를 중앙 관리하고, 각 Event 클래스(GotoEvent, PopupEvent 등)가 실제 로직을 담당하는 위임 패턴이 깔끔함. StageData/SoundData의 JSON 데이터 드리븐 접근법은 학생 프로젝트에서 보기 드문 수준의 설계 성숙도를 보여줌. 그러나 Fracture 시스템이 글로벌 네임스페이스에 있어 UI.Model에서 직접 참조(FractureFinalRunner→AudioManager, WeaponScoreAdder→Fracture)하는 양방향 의존성이 존재하며, 이상적으로는 인터페이스나 이벤트 버스를 통한 간접 참조가 바람직함. Observer 패턴(C# event + UnityEvent)의 일관된 사용으로 시스템 간 결합도가 적절히 관리되고 있음.

심각도파일설명
CRITICALFracture/Model/FractureFinalRunner.csThrowableBreakable의 OnGrab(라인43)과 OnRelease(라인44) 모두 SetPhysics(false)를 호출함. OnGrab은 SetPhysics(true)여야 잡는 동안 kinematic이 활성화됨. 현재 코드는 VR에서 물체를 잡으면 바로 떨어지는 치명적 버그 유발 가능.
HIGHUI/Presenter/Stage3DeathSoundTrigger.cs + Fracture/Model/FractureFinalRunner.csThrowableBreakable.BreakObject()(라인67)과 Stage3DeathSoundTrigger.OnFracture()(라인35) 양쪽에서 AudioManager.PlayDeathSound()를 호출하여 Office3 씬에서 파괴 시 사운드가 2번 재생됨. ThrowableBreakable에서의 호출을 제거하거나 Stage3DeathSoundTrigger에 위임해야 함.
HIGHUI/Presenter/Weapon/WeaponHitSound.csOnCollisionEnter에서 충돌 대상이나 속도 필터링 없이 모든 충돌에 히트 사운드를 재생함. 바닥, 벽, 다른 무기 등과의 무의미한 충돌에도 사운드가 재생되어 VR 몰입감을 해침.
MEDIUMFracture/Fracture.csCopyFractureComponent()(라인258-264)에서 callbackOptions를 참조 복사하므로 리프래크처된 프래그먼트가 원본과 동일한 UnityEvent 인스턴스를 공유함. 프래그먼트 파괴 시 원본의 콜백이 중복 호출될 수 있음.
MEDIUMFracture/FragmentCleaner.cs_fragmentsQueue가 static이지만 OnDestroy()(라인43)에서 Clear()를 호출하여, FragmentCleaner 컴포넌트가 파괴되면 다른 씬에서 등록된 모든 프래그먼트 추적 정보가 사라짐. DontDestroyOnLoad가 아니므로 씬 전환 시 큐가 초기화되면서 이전 프래그먼트가 영구적으로 메모리에 남을 수 있음.
MEDIUMFracture/Model/FractureFinalRunner.csSuperStabilizeRoutine에서 root 오브젝트가 FragmentCleaner에 의해 먼저 파괴되면 코루틴 내부에서 null인 fragments/colliders에 접근하여 MissingReferenceException 발생 가능. FragmentCleaner.Register()와 SuperStabilizeRoutine이 같은 root를 관리하므로 경합 조건이 존재함.
MEDIUMFracture/Fracture.csOnCollisionEnter(라인83-84)에서 filterCollisionsByTag가 true이고 tagAllowed도 true일 때만 파괴되므로, filterCollisionsByTag가 false이면 충돌로는 절대 파괴되지 않음. 일반적으로 필터링이 꺼져 있으면 모든 태그를 허용해야 하지만 현재 로직은 반대로 동작함.
LOWUI/Model/Manager/GameManager.csSubscribeToScoreManager()에서 _isSubscribed를 true로 설정한 후 ScoreManager가 재생성되어도 재구독하지 않음. DontDestroyOnLoad 싱글턴이므로 일반적으로 문제없지만, 초기화 순서가 보장되지 않으면 영구적으로 구독 실패 가능.
LOWFracture/Model/Weapon.csOnCollisionEnter로 충돌 감지 시 빠른 스윙에서 터널링이 발생할 수 있음. Rigidbody의 CollisionDetectionMode를 ContinuousDynamic으로 설정하는 것이 VR 무기에 적합하지만 코드에서 설정하지 않음(Inspector 의존).
