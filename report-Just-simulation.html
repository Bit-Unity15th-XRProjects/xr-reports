프로젝트 현황 요약

이 팀은 VR 파괴 시뮬레이터를 만들고 있으며, 현재 핵심 게임플레이 루프(무기로 오브젝트 타격 → HP 감소/직접 파괴 → 프래그먼트 생성 → 점수 증가 → 클리어 조건 체크 → 다음 스테이지 이동)가 완성되어 있다. OpenFracture 기반의 메시 분할 시스템, XR Grab/Throw 연동의 ThrowableBreakable, 이중 점수 체계(가짜 점수/진짜 점수)라는 독특한 게임 디자인, 4개 스테이지(Tutorial, Office1~3), BGM/효과음 시스템, 볼륨 조절 UI까지 구현되어 프로토타입 시연이 가능한 수준이다. UI 아키텍처가 MVP 패턴으로 체계적이고, 네임스페이스와 Assembly Definition으로 모듈 분리도 시도했다. 가장 시급한 것은 ThrowableBreakable의 OnGrab 물리 버그 수정, 발표용 부적절한 문자열(clearMessage) 교체, 그리고 CallbackOptions.onCompleted null 초기화로 인해 일부 오브젝트 파괴 시 점수가 안 올라갈 수 있는 문제 해결이다.
!

리스크 알림

ThrowableBreakable의 OnGrab/OnRelease 물리 버그
OnGrab과 OnRelease 모두 SetPhysics(false)를 호출하여, VR에서 물건을 잡았을 때 kinematic으로 전환되지 않음. XR Interaction Toolkit의 VelocityTracking 모드에서는 큰 문제가 아닐 수 있지만, 잡은 물체가 다른 물체와 충돌하여 의도치 않은 파괴가 발생할 수 있음.
  • OnGrab의 SetPhysics(false)를 SetPhysics(true)로 수정
  • VR에서 물건 잡기→던지기→파괴 시퀀스를 테스트
발표용 부적절한 문자열
GameManager.cs의 clearMessage와 TextName('....')이 발표/심사에서 노출됨. 'FUCK'이 포함된 클리어 메시지는 학점에 부정적 영향을 줄 수 있음.
  • clearMessage를 한국어 적절한 문구로 변경 (예: '클리어! 숨겨진 점수를 발견했습니다!')
  • TextName 상수도 의미 있는 이름으로 수정
씬 이름 하드코딩 분산
'Office1', 'Office2', 'Office3', 'Tutorial' 문자열이 GameManager, NextStageButton, GotoEvent, BGMEvent 등 최소 5개 파일에 분산 하드코딩. 씬 이름 변경 시 연쇄 버그 발생 위험.
  • static class에 씬 이름 상수를 정의하거나 ScriptableObject로 씬 목록 관리
  • 기존 하드코딩을 상수 참조로 교체
CallbackOptions.onCompleted null 초기화
Fracture의 CallbackOptions 생성자가 onCompleted를 null로 설정. Inspector에서 콜백을 설정하지 않은 Fracture 오브젝트에 WeaponScoreAdder가 점수를 등록하려 하면 경고만 출력되고 점수가 안 올라감. 발표 시 '부수었는데 점수가 안 오르는' 상황 발생 가능.
  • CallbackOptions 생성자에서 onCompleted = new UnityEvent()로 변경
  • 또는 WeaponScoreAdder에서 null일 때 자동 초기화 로직 추가
using Unity; 컴파일 경고/에러 가능성
Popup.cs와 EscapeEvent.cs에서 'using Unity;'를 사용. 일부 Unity 버전에서 이 네임스페이스가 존재하지 않아 컴파일 에러 발생 가능.
  • 해당 줄을 삭제하거나 'using UnityEngine;'으로 교체

권고사항 (Top 3)

1. 씬 이름 및 스테이지 설정 중앙화
현재 씬 이름이 5개 이상의 파일에 분산 하드코딩되어 있어, 씬 추가/이름 변경 시 버그 발생 확률이 높음. 발표 직전 긴급 수정 시 실수 위험.
  1. ScriptableObject 또는 static class로 씬 이름 상수 정의
  2. GameManager의 stageClearConditions를 기준으로 다른 클래스들이 참조하도록 변경
  3. NextStageButton._nextSceneMap과 BGMEvent.GetIndexForScene을 데이터 드리븐으로 전환
2. ThrowableBreakable 물리 버그 수정 및 발표 문자열 정리
두 가지 모두 발표 당일에 눈에 띄는 문제. 물리 버그는 데모 시 예측 불가한 동작을, 부적절한 문자열은 심사위원에게 부정적 인상을 줄 수 있음.
  1. ThrowableBreakable.OnGrab을 SetPhysics(true)로 수정
  2. GameManager.clearMessage를 적절한 한국어로 변경
  3. VR에서 잡기→던지기→파괴→점수 증가 전체 플로우를 테스트
3. StageData/SoundData JSON 파싱 최적화
StageButtonInfo.Start()에서 매번 new StageData()를 호출하여 JSON을 중복 파싱. VR 환경에서 불필요한 GC 압력을 유발하여 프레임 드랍 가능.
  1. StageData와 SoundData를 static 캐싱 패턴으로 변경 (한 번만 로드)
  2. 또는 ScriptableObject로 전환하여 에디터에서 미리 로드

Weapon.cs양호
속도 기반 충돌 감지가 올바르게 구현됨. FixedUpdate에서 velocity를 직접 계산하고, OnCollisionEnter에서 minVelocityThreshold로 필터링. TryGetComponent 패턴 사용이 좋고, BreakableHealth와 Fracture 양쪽 경로를 모두 처리. OverlapBox로 주변 오브젝트 물리 해제하는 ReleaseNearbyObjects()도 잘 설계됨. OnDrawGizmosSelected로 디버깅 지원.
GameManager.cs양호
Singleton + DontDestroyOnLoad 패턴이 올바르게 구현됨. _clearScoreMap 딕셔너리로 씬별 클리어 조건을 효율적으로 관리하고, _defaultClearScores 폴백까지 준비. SceneManager.sceneLoaded 이벤트 구독/해제가 OnEnable/OnDisable에서 올바르게 처리됨. _clearedThisStage로 중복 클리어 방지. 다만 clearMessage = 'FUCK...How do you find out...'은 발표용으로 부적절.
ThrowableBreakable.cs개선필요
XR Grab → 던지기 → 충돌 파괴 흐름이 잘 구현됨. SuperStabilizeRoutine의 3단계 안정화(정지→재활성화→파괴)가 독창적. 그러나 치명적 버그: OnGrab과 OnRelease가 둘 다 SetPhysics(false)를 호출함(41~42줄). OnGrab은 SetPhysics(true)여야 잡을 때 kinematic이 되고, OnRelease에서 false로 물리 활성화되어야 함. 현재는 잡아도 물리가 비활성화되지 않아 의도와 다르게 동작할 수 있음.
WeaponScoreAdder.cs양호
Fracture의 onCompleted 콜백에 동적으로 점수 추가 리스너를 등록하는 패턴이 잘 설계됨. HashSet으로 중복 등록 방지, 콜백 내에서 자기 자신을 RemoveListener하여 메모리 누수 방지. 'Real' 태그로 진짜 점수와 가짜 점수를 구분하는 게임 디자인 반영. onCompleted null 체크도 적절.
ScoreManager.cs양호
Score/RealScore 이중 점수 체계가 깔끔하게 구현됨. ScoreEvent를 내부 데이터 모델로 분리하고, Action<int> 이벤트로 외부 통신. 씬 로드 시 자동 리셋(OnSceneLoaded). 단순하지만 목적에 맞게 잘 동작하는 설계.
BGMEvent.cs개선필요
씬별 BGM 매핑과 Resources.Load 캐싱이 잘 구현됨. 그러나 GetIndexForScene()에서 씬 이름을 switch 하드코딩으로 매핑하고 있어, 새 스테이지 추가 시 코드 수정 필요. SoundData.json의 bgmSounds 배열 순서에 의존하는 암묵적 계약이 위험. 씬 이름과 인덱스의 매핑을 JSON에 포함시키는 것이 더 안전.
StageButtonInfo.cs개선필요
매번 Start()에서 new StageData()를 호출하여 JSON 파싱을 반복 수행. 버튼이 여러 개면 같은 JSON을 여러 번 파싱하는 비효율. 또한 gameObject.tag를 'st1', 'st2' 형식으로 매칭하는 방식은 Unity 태그 시스템의 오용. SerializeField로 stageId를 직접 설정하는 것이 더 명확.
FragmentCleaner.cs양호
static Queue로 전역 프래그먼트 관리 + maxFragmentRoots 하드 캡 + lifeTime 기반 자동 정리가 깔끔하게 구현됨. VR 환경에서 프래그먼트 과다 생성으로 인한 성능 저하를 효과적으로 방지. null 체크도 적절.

아키텍처 평가

UI 시스템에 MVP(Model-View-Presenter) 패턴을 체계적으로 적용한 점이 학생 프로젝트로서 돋보임. Model 레이어(Event 클래스들: ScoreEvent, BGMEvent, VolumeEvent 등)가 순수 C# 클래스로 구현되어 MonoBehaviour 의존성 없이 로직을 캡슐화하고, Presenter 레이어(ScoreDisplay, BGMPlayer, WeaponScoreAdder 등)가 Unity 컴포넌트로서 View와 Model을 연결. Singleton + Observer 패턴으로 매니저 간 통신(ScoreManager → GameManager → NextStageButton). OpenFracture 라이브러리를 적극 활용하되 BreakableHealth, ThrowableBreakable, FragmentCleaner 등 커스텀 래퍼를 추가하여 VR 특화 요구사항에 맞춤. 다만 씬 이름 등 설정값이 여러 클래스에 분산 하드코딩되어 있고, StageData/SoundData를 매번 new 생성하는 비효율이 아키텍처 수준의 개선 포인트.
