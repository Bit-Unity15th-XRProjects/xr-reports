프로젝트 현황 요약

이 팀은 VR 환경에서 오브젝트를 파괴하는 시뮬레이터를 개발하고 있으며, 핵심 게임 루프(무기 타격 → 오브젝트 파쇄 → 점수 획득 → 스테이지 클리어)가 코드 수준에서 완전히 연결되어 있다. Fracture 시스템(OpenFracture 기반)은 메시 슬라이싱, Delaunay 삼각분할, 비동기 파쇄까지 구현되어 기술적 깊이가 있고, Weapon/BreakableHealth/ThrowableBreakable로 다양한 파괴 인터랙션을 제공한다. UI 시스템은 MVP 패턴과 이벤트 기반 아키텍처로 학생 프로젝트 수준을 넘어서는 설계를 보여주나, 현재 가장 시급한 문제는 Fragmenter.cs의 `using UnityEditor`로 인한 빌드 실패와 ThrowableBreakable의 OnGrab/OnRelease 물리 버그다. verpivia-1이 마감 직전 집중적으로 버그를 수정하고 있으며(OOM, JSON 파싱, null 참조 등), 빌드 테스트까지 진행 중이므로 이 두 가지 이슈만 해결되면 프로토타입 시연이 가능한 상태다.
!

리스크 알림

Fragmenter.cs의 using UnityEditor로 빌드 실패
Fragmenter.cs가 Editor 폴더 밖 런타임 경로에 있으면서 UnityEditor 네임스페이스를 참조. standalone 빌드 시 CompileError 발생. 최근 'Test/BildTest' 커밋이 이 문제를 테스트 중인 것으로 보이며, 해결하지 않으면 제출 빌드 자체가 불가능.
  • Fragmenter.cs에서 `using UnityEditor;` 제거 또는 `#if UNITY_EDITOR` / `#endif`로 감싸기
  • UnityEditor를 참조하는 코드가 있다면 해당 부분도 조건부 컴파일 처리
  • 빌드 후 실제 디바이스에서 실행 테스트
ThrowableBreakable OnGrab/OnRelease 동일 동작 버그
VR에서 오브젝트를 잡을 때와 놓을 때의 물리 동작이 구분되지 않음. XRGrabInteractable이 내부적으로 kinematic을 관리하더라도, 코드 의도가 불명확하여 특정 상황(예: Movement Type 변경)에서 예기치 않은 물리 동작 발생 가능.
  • OnGrab에서 SetPhysics(true) 호출로 변경하여 잡는 동안 kinematic 유지
  • XRGrabInteractable의 movementType 설정과 충돌하지 않는지 테스트
  • 던지기 후 파괴 동작이 정상 작동하는지 VR 디바이스에서 검증
팀원 기여도 불균형
verpivia-1이 75개 중 51개(68%) 커밋을 담당하고 최근 버그 수정/최적화를 거의 혼자 수행. yuno3384는 7일간 비활동이며 Test/UI_Archetectures 브랜치가 71 커밋 뒤처짐. 제출 직전 한 명에게 부하가 집중되면 번아웃과 품질 저하 위험.
  • yuno3384의 UI 브랜치를 main에 머지하거나 폐기 결정
  • 남은 작업을 팀원별로 명확히 분배
  • 빌드 에러 수정은 최우선으로 전담자 배정
FragmentCleaner의 static 큐와 인스턴스 수명 불일치
static Queue가 MonoBehaviour의 OnDestroy()에서 Clear되므로 씬 전환 시 파편 추적이 리셋됨. DontDestroyOnLoad 없이 사용하면 이전 씬 파편이 관리 대상에서 빠져 메모리 누수 발생.
  • FragmentCleaner에 DontDestroyOnLoad 적용하거나
  • OnDestroy()에서 _fragmentsQueue.Clear() 제거하고 별도 정리 로직 구현
VR 환경에서 동기 파쇄 시 프레임 드롭
Fracture.ComputeFracture()의 동기 모드에서 복잡한 메시(fragmentCount 높을 때) 파쇄 시 메인 스레드가 블로킹되어 VR 프레임 드롭 발생. VR에서 90fps 미달은 사용자 멀미를 유발.
  • FractureOptions.asynchronous를 기본 true로 설정
  • fragmentCount를 VR 환경에 맞게 5~10으로 제한 권장
  • 프레임 타임 모니터링 후 파쇄 요청 큐잉 고려

권고사항 (Top 3)

1. 빌드 에러 즉시 해결 및 디바이스 테스트
프로젝트 제출의 전제 조건이 '실행 가능한 빌드'임. Fragmenter.cs의 using UnityEditor 문제가 해결되지 않으면 빌드 자체가 불가능하여 모든 구현이 무의미해짐.
  1. Fragmenter.cs에서 using UnityEditor 제거 또는 조건부 컴파일 처리
  2. Unity Build Settings에서 target platform 설정 후 빌드 실행
  3. 빌드된 앱을 VR 디바이스에서 실행하여 전체 게임 루프(튜토리얼 → 스테이지 → 엔딩) 테스트
  4. 빌드 성공 후 즉시 커밋하여 안정 버전 확보
2. ThrowableBreakable 물리 버그 수정 및 VR 인터랙션 검증
던질 수 있는 파괴 오브젝트는 VR 파괴 시뮬레이터의 핵심 인터랙션. OnGrab/OnRelease 버그와 GameObject.Find 의존성이 시연 중 크래시나 예기치 않은 동작을 유발할 수 있음.
  1. OnGrab → SetPhysics(true), OnRelease → SetPhysics(false)로 수정
  2. GameObject.Find 대신 Fracture.ComputeFracture()에서 생성하는 fragmentRoot를 직접 참조하도록 리팩터링
  3. WeaponHitSound에 Breakable 레이어 필터링 추가
3. 미머지 브랜치 정리 및 팀 작업 통합
Test/UI_Archetectures가 71 커밋 뒤처져 있고, Test/New-Map이 28 커밋 뒤처짐. 제출 직전에 대규모 머지는 위험하므로 필요한 변경만 cherry-pick하거나 폐기 결정 필요.
  1. 각 브랜치의 변경 사항을 확인하여 main에 필요한 기능만 식별
  2. 충돌 위험이 낮은 변경만 선별 머지
  3. 불필요한 브랜치는 아카이브 처리
  4. main 브랜치에서 최종 빌드 테스트

코드 리뷰

Fracture.cs개선필요
핵심 파쇄 컴포넌트. 3가지 트리거 타입 지원(Collision/Trigger/Keyboard), ComputeFracture()에서 동기/비동기 분기 처리, CreateFragmentTemplate()에서 MeshCollider(convex), Rigidbody 속성 복사, 재파쇄(refracture) 지원까지 완성도 높음. 그러나 같은 파일에 FragmentDestroyer 클래스가 정의되어 있어 분리 필요. OnCollisionEnter()에서 filterCollisionsByTag가 true일 때만 파쇄가 트리거되므로, 태그 필터링 비활성화 시 충돌로 파쇄가 불가능한 로직 구조가 의도적인지 확인 필요. CauseFracture()가 public이어서 외부에서 직접 호출 가능한 점은 Weapon 연동에 적절함.
Weapon.cs양호
속도 기반 데미지 시스템이 깔끔하게 구현됨. FixedUpdate()에서 프레임 간 위치 차이로 속도 계산, OnCollisionEnter()에서 BreakableHealth/Fracture 순서로 TryGetComponent 사용. ReleaseNearbyObjects()의 OverlapBox 기반 주변 오브젝트 해방 로직이 VR 파괴 시뮬레이터에 적합. LayerMask와 Fracture 컴포넌트 이중 체크로 정확한 대상 필터링. OnDrawGizmosSelected()로 디버그 시각화도 제공. DefaultExecutionOrder(1)로 실행 순서 관리한 점도 좋음.
GameManager.cs양호
스테이지 클리어 로직이 견고하게 구현됨. StageData에서 JSON 설정을 읽어 _clearScoreMap 구성, ScoreManager의 OnRealScoreChanged 이벤트 구독, 씬 전환 시 BindSceneReferences()로 UI 재바인딩. _clearedThisStage 플래그로 중복 클리어 방지. Tutorial 씬 예외 처리도 적절함. FindObjectsByType 사용은 Unity 2023+ API를 활용한 점이 좋으나, SubscribeToScoreManager()에서 _isSubscribed 플래그가 한 번만 true로 설정되어 ScoreManager가 늦게 초기화되면 영구적으로 구독 실패할 수 있음.
WeaponScoreAdder.cs양호
Fracture 완료 시 점수를 추가하는 브릿지 역할을 잘 수행. HashSet으로 중복 콜백 등록 방지, UnityAction callback 내부에서 자가 제거(RemoveListener + HashSet.Remove) 패턴이 메모리 누수를 방지. 'Real' 태그 오브젝트만 RealScore에 반영하는 이중 점수 로직도 게임 디자인 의도에 부합. callbackOptions.onCompleted null 체크가 최근 Fix 커밋으로 추가된 것으로 보이며 안정성 향상에 기여.
ThrowableBreakable (FractureFinalRunner.cs)개선필요
XRGrabInteractable 연동으로 VR 던지기-파괴 메카닉 구현. SuperStabilizeRoutine의 3단계 안정화(전체 정지 → 감쇠 물리 재개 → 삭제)가 VR에서 파편 폭발을 방지하는 좋은 접근. 그러나 OnGrab과 OnRelease가 모두 SetPhysics(false)를 호출하여 동일한 동작을 수행하는 것은 버그로 의심됨 — OnGrab은 SetPhysics(true)여야 잡는 동안 kinematic으로 유지될 것. 또한 GameObject.Find(fragmentsName)은 성능과 신뢰성 면에서 위험하며, 이름 충돌 시 잘못된 오브젝트를 참조할 수 있음.
StageData.cs양호
JSON 기반 스테이지 설정 로더. RuntimeInitializeOnLoadMethod로 에디터 Play/Stop 시 캐시 초기화 처리가 전문적. 다중 Dictionary(SceneNameByKey, ClearScoreByScene, BgmIndexByScene)로 O(1) 조회 제공. IReadOnlyDictionary/IReadOnlyList로 외부 수정 방지. null/empty 체크와 기본값 폴백이 꼼꼼함. 씬 이름 순서 보장을 위한 별도 List(_sceneNames) 유지도 적절한 설계.
FragmentCleaner.cs개선필요
정적 Queue로 파편 루트 관리, Hard Cap(최대 개수)과 Soft Cap(수명) 이중 정리 전략이 효과적. 그러나 _fragmentsQueue가 static이면서 OnDestroy()에서 Clear()를 호출하므로, 씬 전환 시 이 MonoBehaviour가 파괴되면 다른 씬의 파편 큐까지 초기화됨. DontDestroyOnLoad이 설정되어 있지 않으면 씬마다 큐가 리셋되는 문제 발생. 또한 static Queue에 MonoBehaviour의 인스턴스 필드(lifeTime, maxFragmentRoots)를 사용하여 여러 인스턴스 존재 시 설정 충돌 가능.
AudioManager.cs양호
PlayOneShot으로 동시 다중 사운드 재생 지원, 별도 AudioSource 추가하여 BGM과 효과음 분리. BGMEvent의 Dictionary 캐싱으로 Resources.Load 반복 호출 방지. UIManager.EffectVolume 연동으로 볼륨 제어 통합. 씬 로드 시 자동 BGM 전환(OnSceneLoaded → PlayBGMForScene)과 수동 초기 재생(PlayCurrentSceneBGM) 이중 지원이 BGMPlayer의 Start() 타이밍 문제를 해결.

아키텍처 평가

UI 시스템이 MVP(Model-View-Presenter) 패턴으로 설계되어 학생 프로젝트치고 높은 수준의 아키텍처를 보임. Core 레이어(인터페이스, 데이터 클래스), Model 레이어(이벤트 핸들러, 매니저), Presenter 레이어(UI 컴포넌트)가 네임스페이스로 명확히 분리됨. 시스템 간 통신은 C# event(Action delegate)를 통해 이루어지며, UIManager가 중앙 이벤트 버스 역할을 수행. Singleton 패턴이 7개 이상 클래스에 적용되어 전역 접근은 편리하나, 초기화 순서 의존성이 암묵적이며 테스트 가능성이 낮아짐. Fracture 시스템은 OpenFracture 라이브러리를 기반으로 확장했으며, CallbackOptions의 UnityEvent를 통해 파쇄 시스템과 점수 시스템을 느슨하게 연결한 점이 좋은 설계 판단. JSON 기반 외부 설정(StageData, SoundData)으로 하드코딩을 줄인 것도 최근 Fix)HardCoding-Optimization 커밋에서 확인됨.

기반 구축 4/5점
OpenXR 기반 프로젝트 세팅 완료. XR Interaction Toolkit 3.3, XR Hands 1.7.3 샘플 포함. 씬 구조(BasicScene, Maps, SampleScene, Testing)가 잘 분리되어 있고 asmdef 4개로 어셈블리 정의 구성됨. 핵심 시스템 23/25점
핵심 시스템 대부분 구현 완료. (1) Fracture 시스템: Fracture.cs가 3가지 트리거 타입(Collision/Trigger/Keyboard) 지원, Fragmenter가 동기/비동기 파쇄 알고리즘 구현, MeshSlicer가 평면 기반 메시 슬라이싱 수행, Delaunay 삼각분할(Triangulator/ConstrainedTriangulator) 완전 구현. (2) Weapon: 속도 기반 데미지 계산(speed × damageMultiplier), OverlapBox로 주변 오브젝트 해방. (3) BreakableHealth: HP 시스템, 데미지 임계값, 셰이크 이펙트, HP 0에서 파쇄 트리거. (4) ThrowableBreakable: XRGrabInteractable 연동, 충돌 속도 기반 파괴, SuperStabilizeRoutine으로 파편 안정화. (5) ScoreManager: 이중 점수(Score/RealScore), 이벤트 기반 알림. (6) GameManager: 스테이지 클리어 감지, 클리어 메시지 표시. (7) UIManager: 중앙 이벤트 디스패처. (8) AudioManager: 씬별 BGM, 타격 효과음. (9) FragmentCleaner: 시간/개수 기반 파편 정리. BatSwing이라는 명시적 클래스는 없지만 Weapon이 동일 기능을 수행함.

코드 이슈 (10건)

심각도파일설명
CRITICALFragmenter.cs`using UnityEditor;`가 런타임 스크립트(Assets/Scripts/Fracture/Fragment/)에 포함되어 있어 standalone 빌드 시 컴파일 에러 발생. #if UNITY_EDITOR로 감싸거나 제거 필요. 최근 'Test/BildTest' 커밋이 이 빌드 에러와 관련된 것으로 추정됨.
HIGHFractureFinalRunner.csThrowableBreakable의 OnGrab과 OnRelease가 모두 SetPhysics(false)를 호출함. OnGrab은 SetPhysics(true)여야 잡는 동안 kinematic으로 유지되어야 하며, 현재 코드는 잡을 때와 놓을 때 동일한 물리 상태를 설정하는 로직 오류.
HIGHFragmentCleaner.csstatic _fragmentsQueue를 사용하면서 인스턴스 메서드 OnDestroy()에서 Clear()를 호출. DontDestroyOnLoad이 없으면 씬 전환 시 큐가 초기화되어 이전 파편 참조가 유실되고 메모리 누수 발생 가능.
MEDIUMFractureFinalRunner.csThrowableBreakable.BreakObject()에서 GameObject.Find(fragmentsName)을 사용. 동일 이름의 오브젝트가 여러 개 있을 경우 잘못된 오브젝트를 참조할 수 있으며, Find()는 O(n) 탐색으로 성능 저하 유발.
MEDIUMBreakableHealth.csStart()에서 originalPosition = transform.localPosition으로 저장하지만, 오브젝트가 이동하거나 부모가 변경된 후 ShakeObject()가 호출되면 원래 위치가 아닌 Start() 시점 위치로 복귀. 런타임 중 위치가 변하는 오브젝트에서는 셰이크 후 위치 오류 발생.
MEDIUMFracture.csOnCollisionEnter()에서 triggerOptions.filterCollisionsByTag && tagAllowed 조건이 AND로 결합됨. filterCollisionsByTag가 false이면 아무 충돌에서도 파쇄가 트리거되지 않음. filterCollisionsByTag가 false일 때 모든 태그를 허용하려면 조건을 (!filterCollisionsByTag || tagAllowed)로 변경해야 함.
MEDIUMGameManager.csSubscribeToScoreManager()에서 _isSubscribed가 한 번 true로 설정되면 영구적. ScoreManager가 씬 전환 후 재생성되더라도 재구독하지 않음. DontDestroyOnLoad 순서에 따라 문제 가능.
MEDIUMWeaponHitSound.csOnCollisionEnter()에서 모든 충돌에 대해 PlayHitSound() 호출. 바닥/벽 충돌에도 타격음이 재생되어 불필요한 사운드 스팸 발생 가능. Breakable 레이어나 Fracture 컴포넌트 체크 필요.
MEDIUMFracture.csComputeFracture()에서 동기 모드 사용 시 프레임 드롭이 심할 수 있음. fragmentCount가 높을 경우(최대 1024) 메인 스레드 블로킹. VR에서는 프레임 드롭이 멀미를 유발하므로 비동기 모드를 기본값으로 권장.
LOWFracture.csFragmentDestroyer 클래스가 Fracture.cs 파일 끝에 별도 클래스로 정의됨. 단일 책임 원칙 위반이며 파일 검색 시 혼동 유발. 별도 파일로 분리 권장.
