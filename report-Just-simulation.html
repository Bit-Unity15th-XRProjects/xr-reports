프로젝트 현황 요약

이 팀은 VR 환경에서 사무실 오브젝트를 파괴하는 시뮬레이터를 만들고 있으며, OpenFracture 라이브러리 위에 BreakableHealth(체력+흔들림), TestWeapon(속도 기반 데미지), ThrowableBreakable(XR 그랩→던지기→파괴) 파이프라인을 구축하여 핵심 파괴 메카닉을 완성했습니다. UI 시스템은 MVP 패턴 기반으로 타이틀→스테이지 선택→게임 진입 흐름이 구현되어 있고, BGM 자동 전환도 동작합니다. 5개의 커스텀 에디터 도구(SmartFractureAssigner 등)로 맵 세팅 워크플로우를 자동화한 점은 실무적입니다. 그러나 GameManager와 ScoreManager가 없어 '파괴→점수→클리어'의 게임 루프가 존재하지 않고, 데이터가 하드코딩되어 있으며, ThrowableBreakable의 OnRelease 물리 버그가 핵심 메카닉을 위협합니다. 가장 시급한 것은 GameManager/ScoreManager 구현과 OnRelease 버그 수정으로, 이 두 가지가 해결되면 데모 가능한 수준의 게임이 됩니다.
!

리스크 알림

GameManager 부재로 게임 루프가 없음
현재 VR 파괴 시뮬레이터에 시작→플레이→종료→결과의 게임 루프가 전혀 없습니다. 사용자가 씬에 진입하면 물건을 파괴할 수 있지만, 스테이지 클리어 조건, 제한시간, 점수 집계 등이 없어 '게임'이 아닌 '샌드박스 데모' 상태입니다.
  • GameManager 싱글톤을 만들어 GameState enum(Title/Playing/Paused/Result) 관리
  • 스테이지별 클리어 조건(모든 오브젝트 파괴 또는 시간 제한) 구현
  • BreakableHealth.Explode()에서 GameManager에 파괴 이벤트 통지하도록 연동
ThrowableBreakable.OnRelease 물리 버그
OnRelease에서 SetPhysics(false)를 호출하여 던진 물체가 kinematic=false, gravity=true가 되지 않고, 공중에서 물리 시뮬레이션이 비활성화된 상태로 남을 수 있습니다. VR에서 물건을 던지는 것이 핵심 메카닉인데 이 부분이 동작하지 않으면 게임 플레이가 불가능합니다.
  • OnRelease 콜백에서 SetPhysics(false) 대신 rb.isKinematic = false; rb.useGravity = true;로 변경
  • XRGrabInteractable.throwOnDetach가 true이므로 velocity 전달이 정상 작동하는지 테스트
  • VR 헤드셋에서 실제 던지기 테스트 수행
ScoreManager 미구현으로 플레이어 피드백 부재
파괴 시뮬레이터에서 점수는 핵심 피드백입니다. 현재 물건을 파괴해도 아무런 수치적 피드백이 없어 플레이어가 성취감을 느낄 수 없습니다. 데모 발표 시 '게임성'을 보여줄 수 없는 상태입니다.
  • ScoreManager 싱글톤 생성, BreakableHealth.Explode() 시 점수 가산
  • UI에 실시간 점수 표시 (WorldSpace Canvas 권장)
  • 스테이지 종료 시 결과 화면에 총점 표시
하드코딩된 게임 데이터(StageData/SoundData)
StageData._value에 더미 텍스트('ASLDKSA;FLSKAAGASG')가 남아있고, BGM 매핑도 switch문 하드코딩입니다. 새 스테이지 추가 시 C# 코드 3곳을 동시에 수정해야 하며, 빌드 없이는 콘텐츠 변경이 불가능합니다.
  • ScriptableObject로 StageConfig(씬이름, 설명, BGM클립, 클리어조건) 정의
  • BGMEvent의 switch문을 ScriptableObject 딕셔너리로 교체
  • Resources.Load 대신 Addressables 또는 직접 참조로 전환
VR 성능 최적화 미검증
OpenFracture의 실시간 메시 분할은 CPU 집약적이며, FractureOptimizingTool에서 fragmentCount=3으로 낮췄지만 다수 오브젝트 동시 파괴 시 프레임 드롭이 발생할 수 있습니다. VR은 90fps 유지가 필수인데, 성능 프로파일링 흔적이 없습니다.
  • Unity Profiler로 동시 파괴 시 CPU/GPU 부하 측정
  • Object pooling으로 파편 재사용 구현 검토
  • Prefracture(사전 분할)를 더 적극 활용하여 런타임 부하 감소

권고사항 (Top 3)

1. GameManager + ScoreManager 즉시 구현
게임 루프와 점수 시스템 없이는 '시뮬레이터'가 아닌 '기술 데모'입니다. 발표/평가에서 게임성을 보여주려면 최소한의 게임 루프가 필수입니다.
  1. GameManager.cs 생성: GameState(Title/Playing/Result), StartStage(), EndStage(), 타이머
  2. ScoreManager.cs 생성: AddScore(int), GetScore(), ResetScore(), 파괴 시 TestWeapon/BreakableHealth에서 호출
  3. ResultUI: 스테이지 종료 시 총점/파괴 오브젝트 수/소요시간 표시
  4. UIManager에 OnScoreChanged, OnStageComplete 이벤트 추가
2. 하드코딩 데이터를 ScriptableObject로 전환
현재 SoundData/StageData의 하드코딩은 확장을 막고 있으며, 더미 데이터가 남아있어 실제 콘텐츠가 반영되지 않았습니다.
  1. StageConfig ScriptableObject 생성 (씬이름, 표시이름, 설명텍스트, BGM AudioClip 참조, 목표점수)
  2. SoundConfig ScriptableObject 생성 (효과음 AudioClip 딕셔너리)
  3. BGMEvent/GotoEvent가 ScriptableObject를 참조하도록 리팩토링
  4. 에디터에서 드래그앤드롭으로 콘텐츠 관리 가능하게 함
3. ThrowableBreakable OnRelease 버그 수정 및 VR 테스트
던지기가 핵심 메카닉인 VR 파괴 시뮬레이터에서 OnRelease 물리 버그는 게임 플레이를 불가능하게 만들 수 있습니다.
  1. OnRelease에서 SetPhysics(false) → rb.isKinematic = false; rb.useGravity = true;로 수정
  2. GameObject.Find 제거: Fracture.CauseFracture() 후 fragmentRoot를 직접 참조하도록 변경
  3. VR 헤드셋에서 그랩→던지기→충돌→파괴 전체 흐름 테스트

코드 리뷰

Weapon.cs (TestWeapon)양호
속도 기반 데미지 계산(speed * 10f)과 BreakableHealth/Fracture 이중 처리가 잘 구현되어 있습니다. OverlapBox를 이용한 주변 오브젝트 해방(ReleaseNearbyObjects)도 VR 파괴 시뮬레이터에 적합한 설계입니다. OnDrawGizmosSelected로 디버깅도 용이합니다. 다만 매직넘버(10f)를 SerializeField로 노출하면 밸런싱이 쉬워집니다.
BreakableHealth.cs양호
체력 시스템 + 흔들림 코루틴 + 파괴 흐름이 깔끔합니다. TakeDamage()에서 damageThreshold 미만 데미지를 무시하는 저항 시스템도 좋습니다. 단, ShakeObject()에서 originalPosition을 Start()에서 한 번만 저장하므로, 부모 오브젝트가 이동하면 흔들림 후 원래 위치로 돌아가지 않는 버그가 발생할 수 있습니다.
FractureFinalRunner.cs (ThrowableBreakable)개선필요
XRGrabInteractable 이벤트 연동과 SuperStabilizeRoutine(파편 안정화 3단계 코루틴)이 잘 설계되어 있습니다. 그러나 FractureFinalRunner는 빈 MonoBehaviour로 코루틴 러너 역할만 하는데, 이는 안티패턴입니다. BreakObject()에서 GameObject.Find(fragmentsName)를 사용하는 것은 성능과 안정성 모두 문제입니다. 또한 OnGrab과 OnRelease 둘 다 SetPhysics(false)를 호출하는데, OnRelease는 SetPhysics(true)가 되어야 하지 않는지 확인이 필요합니다.
UIManager.cs양호
Singleton 패턴 + C# event/Action 기반 디스패칭이 깔끔합니다. 6종 이벤트(Escape/Close/Goto/Popup/StagePopup/FakeStart)에 대해 각각 전용 Event 클래스를 생성하고 구독하는 구조가 확장성 있습니다. OnStagePopup의 람다에서 3개 함수를 순차 호출하는 부분은 가독성을 위해 별도 메서드로 분리하면 좋겠습니다.
AudioManager.cs + BGMEvent.cs + BGMPlayer.cs개선필요
AudioManager가 SceneManager.sceneLoaded를 구독하여 자동 BGM 전환하고, BGMPlayer가 AudioSource로 재생하는 구조는 좋습니다. 하지만 BGMEvent.GetIndexForScene()에서 씬 이름→인덱스 매핑이 switch문 하드코딩이고, SoundData도 리스트 하드코딩입니다. 새 씬 추가 시 3곳(SoundData 리스트, BGMEvent switch, Resources 폴더)을 동시에 수정해야 하므로 유지보수가 어렵습니다.
StageData.cs + SoundData.cs심각
스테이지/사운드 데이터가 C# 코드에 하드코딩되어 있습니다. StageData의 _value는 'Test game', 'This game', 'ASLDKSA;FLSKAAGASG' 같은 더미 데이터이며, TODO 주석에 'JSON 파싱으로 변경' 의도가 남아있지만 미구현 상태입니다. 이는 콘텐츠 확장의 병목이 됩니다.
FragmentCleaner.cs개선필요
10초 주기로 등록된 파편을 일괄 삭제하는 구조입니다. static List를 사용하여 씬 전환 시에도 참조가 유지되는 점은 좋으나, CleanFragments()가 등록 시간과 무관하게 모든 파편을 즉시 삭제하므로, 방금 생성된 파편도 다음 10초 주기에 삭제될 수 있습니다. 또한 Fracture.ComputeFracture()와 ThrowableBreakable.BreakObject() 양쪽에서 Register를 호출하여 이중 등록 가능성이 있습니다.
Editor Tools (SmartFractureAssigner 등 5개)양호
5개의 에디터 도구(SmartFractureAssigner, FractureHealthAssigner, FractureMaterialAssigner, FractureOptimizingTool, ThrowableSetupTool)가 MenuItem으로 잘 구성되어 있습니다. 특히 SmartFractureAssigner의 볼륨 기반 소품/대형물체 분류 로직과, FractureOptimizingTool의 fragmentCount=3 + detectFloatingFragments=false 최적화 설정은 VR 성능을 고려한 좋은 설계입니다. Undo 지원도 대부분 포함되어 있습니다.

아키텍처 평가

UI 시스템은 MVP(Model-View-Presenter) 패턴을 시도하여 Core(enum/interface) → Model(UIManager/Event 클래스) → Presentation(버튼 클릭 핸들러) 3계층으로 분리한 점이 이 수준의 학생 프로젝트에서는 돋보입니다. UIManager가 C# event/Action으로 이벤트를 디스패치하고, 각 Event 클래스가 실제 로직을 처리하는 Observer 패턴 구조입니다. 그러나 Fracture 시스템은 OpenFracture 라이브러리에 BreakableHealth/TestWeapon/ThrowableBreakable을 직접 결합하는 방식으로, UI 시스템과 완전히 분리되어 있어 점수/진행도 등의 데이터가 흐르지 않습니다. 또한 Singleton(UIManager, AudioManager) + DontDestroyOnLoad(Eternitiy)으로 씬 간 데이터를 유지하려 하지만, 실제 게임 상태를 관리하는 중앙 객체가 없어 두 시스템이 '섬'처럼 존재합니다.

핵심 시스템 12/25점

GotoEvent.OnclickGoto()에서 SceneManager.LoadSceneAsync($"Office{key}")로 씬 전환이 구현되어 있고, AudioManager가 SceneManager.sceneLoaded 이벤트를 구독하여 BGM을 자동 전환합니다. 하지만 게임 루프(시작→플레이→종료→결과)가 없고, 스테이지 간 데이터 전달이나 게임 상태 관리가 전혀 없습니다.

심각도파일설명
CRITICALFractureFinalRunner.csThrowableBreakable.OnRelease()에서 SetPhysics(false)를 호출하는데, 던진 후 물리를 활성화해야 하므로 SetPhysics(true) 또는 isKinematic=false가 의도된 것으로 보입니다. 현재 코드는 던진 물체가 공중에 멈출 수 있습니다.
HIGHFractureFinalRunner.csBreakObject()에서 GameObject.Find(gameObject.name + "Fragments")를 사용합니다. 동일 이름 오브젝트가 여러 개일 때 잘못된 파편 그룹을 참조할 수 있고, 매 프레임이 아니더라도 Find는 VR에서 프레임 드롭을 유발할 수 있습니다.
HIGHCloseEvent.csOnClick()에서 GetComponentInParent<PanelInfo>() 결과를 null 체크 없이 바로 .gameObject.SetActive(false)로 사용합니다. PanelInfo가 없는 부모 계층에서 NullReferenceException이 발생합니다.
HIGHStageData.cs스테이지 데이터(_value)가 'Test game', 'ASLDKSA;FLSKAAGASG' 같은 더미 텍스트로 하드코딩되어 있어 실제 게임 콘텐츠가 반영되지 않았습니다. JSON 또는 ScriptableObject로 전환 필요합니다.
MEDIUMStageButtonInfo.csgameObject.tag를 'st1', 'st2', 'st3' 형식으로 비교하여 스테이지 ID를 결정합니다. Unity 태그는 미리 등록해야 하고, 새 스테이지마다 태그를 추가해야 하므로 확장성이 매우 낮습니다. SerializeField int로 직접 할당하는 것이 낫습니다.
MEDIUMFracture.cs (line 266)FragmentDestroyer 클래스가 Fracture.cs 파일 끝에 같은 파일 내 별도 클래스로 정의되어 있습니다. 별도 파일로 분리하거나, 최소한 namespace로 관리해야 합니다.
MEDIUMPopupEvent.cscanvasdata.ChildPanel[i].name.Split('_')[0]으로 패널 이름을 매칭합니다. 이름 기반 매칭은 오브젝트 이름 변경 시 즉시 깨지며, 디버깅이 어렵습니다.
LOWLinkedButtonInfo.csGetTypedata()에서 Enum.GetValues(typeof(PanelType)).Cast<PanelType>().ToList()를 사용하여 매번 새 리스트를 생성합니다. 단순히 Enum.TryParse(gameObject.name, out PanelType type)로 대체할 수 있습니다.
LOWNamespace 전체Presentation 네임스페이스가 'VRProject.UI.Pressentation'으로 오타가 있습니다(ss가 두 개). 6개 파일(Close, Escape, FakeStart, Goto, Popup, StagePopup, BGMPlayer)에 모두 동일한 오타가 적용되어 있습니다.
LOWFragmentCleaner.csstatic _fragmentsRegistry 리스트에 Fracture.ComputeFracture()와 ThrowableBreakable.BreakObject() 양쪽에서 같은 fragmentRoot를 Register할 수 있어 이중 삭제 시도가 발생합니다. null 체크로 방어하고 있지만 의도치 않은 동작입니다.
