프로젝트 현황 요약

이 팀은 VR 파괴 시뮬레이터를 만들고 있으며, 현재 핵심 게임플레이 루프(무기 잡기→던지기/스윙→오브젝트 파괴→점수 획득→스테이지 클리어→다음 스테이지)가 완성된 상태이다. Fracture 시스템(Delaunay 삼각분할 기반 메시 분해)과 VR 상호작용(ThrowableBreakable, Weapon), 점수 시스템(ScoreManager + WeaponScoreAdder), UI 시스템(MVP 패턴, 볼륨 슬라이더, 스테이지 선택 팝업), 오디오 시스템(BGM 자동 전환, 타격음, 파괴음)이 모두 동작하는 로직으로 구현되어 프로토타입 시연이 가능한 수준이다. 가장 시급한 문제는 CallbackOptions.onFracture 미초기화로 인한 NullReferenceException 크래시 위험과 Stage3 DeathSound 이중 재생 버그로, 이 두 가지를 수정하면 안정적인 발표 시연이 가능하다. 기여도 불균형(verpivia-1 69%, yuno3384 3%)은 팀 평가 시 리스크가 될 수 있으며, 남은 기간에 yuno3384의 기여를 높이는 것이 바람직하다.
!

리스크 알림

Stage3 DeathSound 중복 재생 + onFracture NullReference 위험
ThrowableBreakable.BreakObject()에서 PlayDeathSound() 직접 호출과 Stage3DeathSoundTrigger의 콜백 호출이 동시 발동하여 사운드 이중 재생. 더 심각한 것은 CallbackOptions.onFracture가 Inspector에서 미설정 시 null이라 Stage3DeathSoundTrigger의 AddListener에서 NullReferenceException으로 크래시 가능
  • CallbackOptions 생성자에 this.onFracture = new UnityEvent<Collider, GameObject, Vector3>() 추가
  • ThrowableBreakable.BreakObject()에서 AudioManager.Instance?.PlayDeathSound() 라인 제거 (Stage3DeathSoundTrigger가 이미 처리)
발표 시연 시 프래그먼트 폭발 불안정성
ThrowableBreakable의 SuperStabilizeRoutine이 0.1초 후 Trigger 해제하는데, 이 타이밍에 프래그먼트끼리 겹쳐있으면 물리 엔진이 강한 반발력을 생성하여 파편이 화면 밖으로 날아갈 수 있음. VR에서 이런 물리 불안정은 멀미를 유발할 수 있음
  • SuperStabilizeRoutine의 2단계에서 explosionForceLimit 값을 더 낮추거나 linearVelocity를 완전히 0으로 설정
  • WaitForSeconds(0.1f) 이후 colliders 재활성화 시 각 collider의 overlap을 Physics.ComputePenetration으로 검사하는 안전장치 추가
팀 기여도 불균형
verpivia-1이 54/78 커밋(69%)을 담당하고, yuno3384는 2커밋으로 8일 비활동 상태. Test/UI_Archetectures 브랜치가 74커밋 뒤처져 있어 사실상 방치됨. 발표 시 팀 기여도 질문에 취약
  • yuno3384에게 명확한 마감 태스크 배정 (예: UI 프리팹 정리, 테스트 씬 구성)
  • 방치된 Test/UI_Archetectures 브랜치 삭제 또는 머지
씬 빌드 설정 및 JSON 파일 누락 가능성
StageData.json과 SoundData.json이 Resources 폴더에 있어야 하고, 모든 스테이지 씬이 Build Settings에 등록되어야 함. 빌드 시 누락되면 SceneManager.LoadSceneAsync가 실패하고, JSON 로드 실패 시 모든 매니저가 기본값으로 동작
  • 빌드 전 File→Build Settings에서 모든 씬(Tutorial, Office1, Office2, Office3) 등록 확인
  • StageData.json의 sceneName 값과 실제 씬 파일명 일치 여부 검증

권고사항 (Top 3)

1. CallbackOptions.onFracture 초기화 및 중복 사운드 수정
현재 가장 크래시 가능성이 높은 버그. Stage3에서 시연할 때 NullReferenceException이 발생하면 발표에 치명적
  1. CallbackOptions.cs 생성자에 this.onFracture = new UnityEvent<Collider, GameObject, Vector3>() 추가
  2. FractureFinalRunner.cs의 BreakObject()에서 AudioManager.Instance?.PlayDeathSound() 라인 삭제
  3. Stage3DeathSoundTrigger가 모든 파괴 사운드를 통합 관리하도록 단일 경로 유지
2. 발표용 시연 안정성 강화
VR 시연은 예측 불가능한 물리 상호작용이 많아 크래시/버그가 발표 현장에서 발생할 확률이 높음. 방어적 코드가 필요
  1. 모든 Singleton 접근(AudioManager.Instance, ScoreManager.Instance 등)에 null-conditional(?.) 사용 통일
  2. SuperStabilizeRoutine에서 fragments 배열의 null 체크를 각 단계마다 추가
  3. 빌드 후 Tutorial→Office1→Office2→Office3→엔딩 전체 플로우 최소 3회 통테스트
3. 커밋 크기 정규화 및 .csproj 파일 gitignore 등록
Hot Files 상위 10개가 전부 .csproj 파일로, Unity가 자동 생성하는 파일이 매 커밋마다 변경되어 실질적 코드 변경 추적이 어려움. 평균 커밋 764줄은 에셋 파일 포함 때문이며, 4000줄 이상 커밋이 3건
  1. .gitignore에 *.csproj 추가 (Unity 자동 생성 파일)
  2. 향후 커밋 시 기능 단위로 분리 (예: 'Feat/Stage3-Death,Hit Sound'처럼 복합 기능은 별도 커밋으로)

코드 리뷰

GameManager.cs양호
스테이지 클리어 판정 로직이 깔끔하게 구현됨. StageData.Instance에서 클리어 점수 맵을 복사해 사용하고, ScoreManager.OnRealScoreChanged 이벤트를 구독하여 실시간 점수 감시. BindSceneReferences()에서 FindObjectsByType으로 동적 UI 바인딩하는 방식이 합리적. _clearedThisStage 플래그로 중복 클리어 방지. OnDestroy에서 이벤트 해제도 처리. 다만 _isSubscribed 플래그가 한번 true가 되면 다시 false로 돌아가지 않아, ScoreManager가 재생성되는 경우(이론적) 재구독 불가.
Weapon.cs양호
FixedUpdate에서 position delta로 속도를 직접 계산하는 방식이 VR 환경에서 적합함(XRI가 Rigidbody velocity를 덮어쓸 수 있으므로). TryGetComponent로 BreakableHealth와 Fracture를 분기 처리하며, damageMultiplier로 속도→데미지 변환. ReleaseNearbyObjects()가 Physics.OverlapBox로 주변 고정 오브젝트를 해제하는 것이 VR 파괴 시뮬레이터에 적합한 설계. OnDrawGizmosSelected로 디버그 시각화도 구현.
FractureFinalRunner.cs (ThrowableBreakable)개선필요
XRGrabInteractable 연동과 SuperStabilizeRoutine 3단계 안정화(프리즈→댐핑 적용→파괴)가 잘 설계됨. 그러나 두 가지 문제: (1) OnGrab과 OnRelease 모두 SetPhysics(false)를 호출하여 의도가 불명확 - XRI가 물리를 관리하므로 동작은 하지만 코드 가독성이 낮음. (2) BreakObject()에서 AudioManager.Instance?.PlayDeathSound() 직접 호출과 Stage3DeathSoundTrigger의 onFracture 콜백이 동시에 발동하여 Stage3에서 사망 사운드 이중 재생 버그 존재.
WeaponScoreAdder.cs양호
Fracture.callbackOptions.onCompleted에 동적 콜백을 등록하여 파괴 완료 시 점수를 추가하는 설계가 우수. HashSet<Fracture>로 중복 등록 방지, 콜백 내부에서 RemoveListener + HashSet 제거로 메모리 누수 방지. tag 'Real' 체크로 진짜 점수와 가짜 점수를 분리하는 게임 메커닉도 명확. onCompleted null 체크 후 경고 로그도 적절.
StageData.cs양호
JSON 파싱 기반 데이터 드리븐 설계의 핵심 클래스. RuntimeInitializeOnLoadMethod로 에디터 도메인 리로드 대응, IReadOnlyDictionary/IReadOnlyList로 외부 수정 방지. 하위 호환성을 위해 기존 프로퍼티(_key, _value, _id)와 신규 프로퍼티(_sceneNameByKey, _clearScoreByScene)를 모두 유지하는 점이 리팩터링 과정을 보여줌. null/empty 체크가 꼼꼼함.
AudioManager.cs개선필요
BGM/Hit/Death 사운드 3종을 깔끔하게 관리. PlayOneShot 사용으로 임시 GameObject 생성 없이 효율적. 씬 전환 시 자동 BGM 교체. 그러나 PlayDeathSound()가 StageData.Instance.SceneNameByKey에 직접 접근하는데, StageData.Instance가 null일 수 있는 상황(Resources에서 JSON 로드 실패)에 대한 방어가 없음. 또한 _effectSource 하나로 모든 효과음을 재생하므로, PlayOneShot이 동시 다수 재생을 지원하긴 하지만 볼륨 제어가 클립별로 불가능.
CallbackOptions.cs개선필요
생성자에서 onCompleted만 new UnityEvent()로 초기화하고 onFracture는 초기화하지 않음. Inspector에서 onFracture를 설정하지 않으면 null 상태. CallOnFracture에서 null-conditional(?.)로 안전하게 호출하지만, 일관성이 부족하고 다른 코드에서 onFracture에 직접 접근 시(Stage3DeathSoundTrigger의 f.callbackOptions.onFracture.AddListener) NullReferenceException 위험이 있음.
FragmentCleaner.cs양호
static Queue로 전역 프래그먼트 관리, maxFragmentRoots(10)로 하드캡 적용, lifeTime(10초) 기반 자동 정리. Update에서 큐 순회하며 만료된 프래그먼트부터 제거하는 FIFO 전략이 효율적. Destroy 전 null 체크, OnDestroy에서 큐 클리어. OOM 방지에 실질적으로 기여하는 핵심 컴포넌트. 다만 static Queue가 씬 전환 시 자동 초기화되지 않아 이미 Destroy된 오브젝트 참조가 남을 수 있으나, null 체크로 안전하게 처리됨.

아키텍처 평가

MVP 변형 아키텍처를 3계층으로 구현: Core(StageData, SoundData, Interface, Utils) → Model(Manager 4개 + Event 11개) → Presenter(onClick, BGM, Weapon, Score). UIManager가 Observer 패턴의 이벤트 허브 역할을 하며, 각 Presenter가 UIManager.Trigger*()를 호출하면 등록된 Event 클래스가 실제 로직을 수행하는 구조. Fracture 시스템은 별도 모듈로 Options/Fragment/Model/Support/Utilities 5개 하위 폴더에 관심사가 잘 분리됨. 데이터 흐름이 JSON(Resources) → StageData/SoundData(Singleton) → Manager → Presenter로 단방향에 가까움. 다만 Singleton 간 초기화 순서(StageData→GameManager→ScoreManager)가 암묵적이어서 씬 내 오브젝트 배치 순서에 의존하는 점이 아쉬움. Fracture 시스템과 UI 시스템 간 결합은 WeaponScoreAdder와 Stage3DeathSoundTrigger가 브릿지 역할을 하여 비교적 느슨하게 연결됨.

핵심 시스템 27/25점

(3) FractureSystem: Delaunay 삼각분할 기반 메시 분해(Triangulator→MeshSlicer→Fragmenter) + Prefracture/Refracture 지원. (4) ScoreManager: 이중 점수(Score/RealScore) + 씬 전환 시 자동 리셋. (5) UIManager: 8개 이벤트(Escape/Close/Goto/Popup 등) 중앙 허브. (6) VRInteraction(ThrowableBreakable): XRGrabInteractable 연동, 던지기→충돌→파괴 플로우. (7) StageLoader(GotoEvent+NextStageButton): JSON 기반 씬 전환 + 엔딩 처리. (8) AudioManager: BGM/Hit/Death 사운드 3종 관리. (9) BreakableHealth(DestructionPhysics): HP 시스템 + 흔들림 + 폭발 분기. (10) FragmentCleaner: Queue 기반 프래그먼트 메모리 관리. 대부분 스텁이 아닌 실제 런타임 로직이며, Inspector 설정과 연동됨. -3점은 ThrowableBreakable의 중복 사운드 버그와 일부 시스템 간 연결 불완전성 때문.
