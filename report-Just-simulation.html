프로젝트 현황 요약

이 팀은 VR 환경에서 사무실 물건을 무기로 파괴하는 시뮬레이터를 만들고 있으며, 현재 4개 스테이지(Tutorial, Office1-3)의 기본 게임 루프(무기 휘두르기 → 오브젝트 파괴 → 점수 획득 → 클리어 조건 달성 → 다음 스테이지)가 작동하는 상태까지 구현했다. OpenFracture 라이브러리를 활용한 실시간 메시 파괴와 BreakableHealth 기반 HP 시스템이 핵심 메커닉이며, UI는 MVP 패턴으로 정교하게 설계되어 있다. 그러나 ThrowableBreakable의 OnGrab/OnRelease 물리 설정 버그가 VR 인터랙션의 핵심을 깨뜨릴 수 있고, 매니저 초기화 순서 미보장으로 씬 전환 시 크래시 위험이 있다. 가장 시급한 것은 ThrowableBreakable 물리 버그 수정, DontDestroyOnLoad 매니저 관리 확립, 그리고 하드코딩된 게임 데이터의 ScriptableObject 전환이다. 팀원 기여도 불균형(verpivia-1이 67%)도 프로젝트 리스크 요인이다.

리스크 알림

매니저 Singleton 초기화 순서 미보장
GameManager.Start()에서 ScoreManager.Instance를 참조하고, BGMPlayer.Start()에서 AudioManager.Instance와 UIManager.Instance를 참조. Unity의 MonoBehaviour 실행 순서가 보장되지 않아 씬 로드 시 NullReferenceException 발생 가능. 현재 에러 로그만 출력하고 복구 메커니즘 없음.
  • Script Execution Order에서 매니저 초기화 순서 명시 설정
  • 또는 모든 매니저를 하나의 Bootstrap GameObject에 넣고 Awake/Start 순서 관리
  • Presentation 레이어에서 Instance 접근 전 null 체크 일괄 추가
ThrowableBreakable OnGrab/OnRelease 로직 오류
OnGrab(SetPhysics(false))과 OnRelease(SetPhysics(false))가 동일한 인자를 전달하여 잡았을 때 물리가 비활성화되지 않음. VR에서 오브젝트를 잡아서 던지는 핵심 메커닉이 의도대로 작동하지 않을 수 있음.
  • OnGrab에서 SetPhysics(true) 호출로 수정하여 잡는 동안 키네마틱 활성화
  • XRGrabInteractable의 Movement Type이 VelocityTracking이므로 물리 설정과의 상호작용 테스트
  • 실제 VR 디바이스에서 잡기→던지기→파괴 전체 플로우 QA
하드코딩된 게임 데이터의 확장성 부재
StageData, SoundData, NextStageButton._nextSceneMap, BGMEvent.GetIndexForScene, GameManager.stageClearConditions 모두 하드코딩. 스테이지 추가 시 최소 5개 파일을 동시에 수정해야 하며, 하나라도 누락하면 게임이 깨짐.
  • ScriptableObject로 StageConfig 에셋 생성(씬 이름, 클리어 점수, BGM, 다음 씬)
  • 모든 매니저가 이 에셋을 참조하도록 리팩토링
  • 에디터에서 데이터 수정만으로 스테이지 추가 가능하게 구조 변경
DontDestroyOnLoad 관리 부재
Eternitiy.cs가 DontDestroyOnLoad을 담당하지만, 4개 매니저(GameManager, ScoreManager, AudioManager, UIManager) 자체에는 DontDestroyOnLoad이 없음. 씬 전환 시 매니저가 파괴되면 Singleton Instance가 null이 되어 전체 시스템 붕괴.
  • 각 매니저 Awake()에 DontDestroyOnLoad(gameObject) 추가
  • 또는 하나의 PersistentManagers 프리팹에 모든 매니저를 배치하고 Eternitiy를 부착
  • 씬 전환 후 매니저 Instance null 여부 테스트
커밋 크기 과대 및 팀원 기여도 불균형
verpivia-1이 55개 커밋 중 37개(67%) 담당. 김연준 12개, yuno3384 2개로 극심한 불균형. af595b6 커밋이 4843줄 변경으로 리뷰 불가능한 크기. Test/UI_Archetectures 브랜치가 main 대비 49커밋 뒤처져 있어 머지 충돌 위험.
  • 커밋 단위를 기능별 300줄 이하로 분리하는 규칙 수립
  • yuno3384의 작업 범위를 명확히 할당하고 일일 커밋 목표 설정
  • UI_Archetectures 브랜치를 main에 리베이스하거나 폐기 결정

권고사항 (Top 3)

1. ScriptableObject 기반 데이터 레이어 전환
현재 SoundData, StageData가 하드코딩 클래스이고, 5개 이상 파일에 씬 이름/인덱스가 중복 정의되어 있어 유지보수가 극히 어려움. ScriptableObject로 전환하면 에디터에서 시각적으로 편집 가능하고 코드 변경 없이 콘텐츠 추가 가능.
  1. StageConfig ScriptableObject 생성: 씬 이름, 클리어 점수, BGM 클립, 다음 씬, 설명 텍스트 필드 포함
  2. Resources 폴더에 에셋 배치 후 GameManager, BGMEvent, NextStageButton에서 참조
  3. SoundData/StageData 하드코딩 클래스 제거
2. 매니저 Bootstrap 시스템 구축
4개 Singleton 매니저의 초기화 순서가 보장되지 않고, DontDestroyOnLoad 관리가 불명확하여 씬 전환 시 크래시 위험이 높음.
  1. PersistentManagers 프리팹 생성, 모든 매니저를 자식으로 배치
  2. Bootstrap MonoBehaviour에서 Script Execution Order를 -100으로 설정
  3. Awake()에서 DontDestroyOnLoad 호출 및 중복 인스턴스 방지 로직 통합
3. Fracture 스크립트 네임스페이스 통합 및 ThrowableBreakable 버그 수정
Weapon.cs, BreakableHealth.cs, ThrowableBreakable.cs가 글로벌 네임스페이스에 있어 UI 레이어와 아키텍처 일관성이 없고, ThrowableBreakable의 OnGrab/OnRelease 버그가 핵심 VR 인터랙션을 깨뜨릴 수 있음.
  1. 세 파일을 VRProject.Gameplay 네임스페이스로 이동
  2. OnGrab: SetPhysics(true), OnRelease: SetPhysics(false)로 수정
  3. Weapon.cs의 damage 계수를 SerializeField로 노출
  4. VR 디바이스에서 잡기/던지기/파괴 통합 테스트

코드 리뷰

GameManager.cs개선필요
Singleton 패턴으로 스테이지 클리어 관리가 잘 구현됨. ScoreManager 이벤트 구독/해제, 씬 전환 시 재구독 등 라이프사이클 처리가 꼼꼼함. 그러나 TextName = '....'이 의도가 불명확하고, BindSceneReferences()에서 FindObjectsByType<Transform>으로 모든 Transform을 순회하여 이름 매칭하는 방식은 비효율적. SerializeField나 태그 기반 검색이 더 적절. ShowClearMessage의 'FUCK...How do you find out...' 텍스트는 프로덕션 부적절.
Weapon.cs개선필요
FixedUpdate에서 속도 계산 후 OnCollisionEnter에서 데미지 처리하는 핵심 로직이 정상 동작. BreakableHealth → HP 기반 파괴, Fracture → 즉시 파괴 분기가 명확함. ReleaseNearbyObjects()의 OverlapBox 주변 물체 해방도 적절. 그러나 damage = speed * 10f의 매직 넘버, 글로벌 네임스페이스 사용, 한국어 주석 인코딩 깨짐이 문제. LayerMask.NameToLayer('Breakable')를 매 충돌마다 호출하는 것도 비효율적.
UIManager.cs양호
7종 UI 이벤트를 중앙에서 관리하는 이벤트 버스 패턴이 깔끔하게 구현됨. Awake에서 내부 이벤트 객체 생성 및 구독, Trigger 메서드로 외부 호출 인터페이스 제공. VolumeEvent의 BGMVolume/EffectVolume을 프로퍼티로 노출하는 설계도 적절. OnStagePopup의 복합 람다가 다소 복잡하지만 기능적으로 올바름.
ScoreManager.cs양호
Score와 RealScore 이중 점수 시스템이 이벤트 기반으로 깔끔하게 구현됨. SceneManager.sceneLoaded 구독/해제가 올바르고, 씬 전환 시 자동 리셋. ScoreEvent 객체로 상태 캡슐화. 다만 ScoreEvent.OnAddScore()가 항상 1점만 증가시키므로 가변 점수 시스템으로의 확장이 어려움.
ThrowableBreakable(FractureFinalRunner.cs)개선필요
XRGrabInteractable 연동으로 VR 투척 파괴가 잘 구현됨. SuperStabilizeRoutine의 3단계 안정화(키네마틱 고정 → 감속 재활성화 → 시간 후 삭제)가 파편 폭발 문제를 효과적으로 해결. 그러나 FractureFinalRunner가 빈 클래스로 코루틴 호스트 역할만 하는 것은 코드 스멜. OnRelease에서 SetPhysics(false)를 호출하는데, 놓았을 때 isKinematic=false로 설정하면 의도와 다를 수 있음(OnGrab과 동일한 인자). fragmentsName = gameObject.name + 'Fragments'로 GameObject.Find 사용은 취약함.
WeaponScoreAdder.cs양호
Fracture.callbackOptions.onCompleted에 동적 리스너를 등록하여 파괴 완료 시 점수를 추가하는 영리한 설계. HashSet으로 중복 등록 방지, 콜백 내에서 자가 제거(RemoveListener + HashSet.Remove). 'Real' 태그 기반 RealScore 분기도 게임 컨셉(진짜 물건 vs 가짜 물건)에 부합. null 체크도 적절.
BGMEvent.cs양호
씬 이름 → 인덱스 → Resources 경로 매핑으로 BGM 재생. Dictionary 캐싱으로 중복 로드 방지. 다만 GetIndexForScene의 switch문이 하드코딩이라 씬 추가 시 수정 필요. SoundData 클래스와의 이중 매핑 구조가 불필요하게 복잡.
NextStageButton.cs개선필요
스테이지 진행의 핵심 로직이 잘 구현됨. GameManager.OnStageClear 1순위 + ScoreManager 직접 체크 2순위 폴백 설계가 방어적. Office3 마지막 스테이지 처리(EndingPopup)도 포함. 그러나 _nextSceneMap 하드코딩, FindAnyObjectByType<EndingPopup> 런타임 검색, Tutorial 씬에서의 동작이 정의되지 않음.

아키텍처 평가

UI 시스템은 MVP 패턴을 채택하여 Model(Event 클래스: ScoreEvent, BGMEvent, VolumeEvent 등), View(ButtonInfo, PanelInfo, CanvasInfo 등 MonoBehaviour 데이터 홀더), Presenter(onClick 핸들러: Goto, Close, Escape 등)로 명확히 분리됨. Manager 계층(GameManager, ScoreManager, AudioManager, UIManager)은 Singleton + Observer 패턴으로 이벤트 기반 통신을 구현하여 결합도를 낮춤. 특히 WeaponScoreAdder가 Fracture.callbackOptions.onCompleted 이벤트에 동적으로 리스너를 등록하는 방식은 OpenFracture 라이브러리와의 통합에서 창의적. 그러나 Fracture 계열 스크립트(Weapon, BreakableHealth, ThrowableBreakable)는 글로벌 네임스페이스에 위치하여 UI 레이어와 아키텍처적 일관성이 없고, 에디터 도구(6개)도 별도 네임스페이스 없이 작성됨. 데이터 레이어가 ScriptableObject 대신 순수 C# 클래스 하드코딩으로 구현되어 Unity 에디터의 장점을 활용하지 못함.

심각도파일설명
CRITICALFractureFinalRunner.cs (ThrowableBreakable)OnGrab과 OnRelease 모두 SetPhysics(false)를 호출하여 놓았을 때도 isKinematic=false 설정. OnRelease는 SetPhysics(true)가 의도일 가능성이 높으나, 놓은 후 물리 시뮬레이션이 필요하므로 false가 맞다면 OnGrab은 SetPhysics(true)여야 함. 현재 잡을 때와 놓을 때 동일한 동작이라 그랩 중 물리가 꺼지지 않음.
HIGHScoreDisplay.csStart()에서 ScoreManager.Instance.Score를 null 체크 없이 접근. ScoreManager가 아직 초기화되지 않은 상태에서 NullReferenceException 발생 가능. RealScoreDisplay.cs도 동일한 문제.
HIGHGameManager.csTextName 상수가 '....'로 설정되어 있어 실제 UI 오브젝트 이름과 매칭되지 않을 가능성. BindSceneReferences에서 TMP_Text를 못 찾으면 클리어 메시지가 표시되지 않음. ShowClearMessage 텍스트에 비속어 포함.
HIGHStageData.cs스테이지 설명이 'Test game', 'This game', 'ASLDKSA;FLSKAAGASG'로 플레이스홀더 상태. JSON 파싱 TODO가 남아있으며 실제 게임 데이터가 입력되지 않음.
HIGHEternitiy.cs클래스명 오타(Eternity → Eternitiy). DontDestroyOnLoad만 호출하고 어떤 매니저가 이 컴포넌트를 사용하는지 코드에서 추적 불가. 매니저별 자체 DontDestroyOnLoad가 없어 씬 전환 시 매니저 파괴 위험.
MEDIUMWeapon.cs한국어 주석이 인코딩 깨짐(UTF-8 BOM 없이 저장된 것으로 추정). damage = speed * 10f 매직 넘버가 SerializeField로 노출되지 않아 밸런싱 조정 불가. 글로벌 네임스페이스 사용으로 이름 충돌 위험.
MEDIUMWeaponHitSound.csOnCollisionEnter에서 어떤 오브젝트와 충돌해도 무조건 히트 사운드 재생. 바닥, 벽, 플레이어 손 등 모든 충돌에 반응하여 과도한 사운드 재생 발생 가능.
MEDIUM전체 Presentation 레이어네임스페이스가 'VRProject.UI.Pressentation'으로 오타. 모든 Presentation 파일(ScoreDisplay, BGMPlayer, WeaponScoreAdder 등 8+ 파일)에 일관되게 적용되어 나중에 수정 시 대규모 리팩토링 필요.
LOWSoundData.csSoundData 클래스가 생성될 때마다 동일한 하드코딩 리스트를 새로 할당. BGMEvent와 FakeStartEvent에서 각각 new SoundData()를 호출하여 불필요한 중복 인스턴스 생성. ScriptableObject로 전환 필요.
LOWSmartFractureAssigner.csFindObjectsOfType<MeshRenderer>()로 씬 전체를 순회하며 이름 기반 필터링. 'orenge'(orange 오타) 등 하드코딩된 문자열로 소품 판별. 새 오브젝트 추가 시 코드 수정 필요.
