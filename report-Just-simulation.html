프로젝트 현황 요약

이 팀은 OpenXR 기반의 VR 파괴 시뮬레이터를 만들고 있으며, 현재 GameManager, ScoreManager, AudioManager, UIManager 등 매니저 인프라와 MVP 패턴의 UI 시스템까지 구축한 상태. OpenFracture 라이브러리를 프로젝트에 통합하여 메시 파괴 기능의 기반을 마련했고, 352개의 프리팹과 Office 맵 씬, BGM/효과음 에셋도 상당량 확보됨. 그러나 프로젝트의 핵심인 'VR 컨트롤러로 무기를 잡고 오브젝트를 파괴하는' 게임플레이 루프가 아직 완성되지 않은 것으로 보이며, BatSwing, VRInteraction, DestructionPhysics 등 XR 인터랙션 코드가 확인되지 않음. verpivia-1 한 명에게 개발이 집중되어 있어 팀 협업 측면에서도 리스크가 있으며, .csproj 파일의 git 추적, Fragmenter.cs의 using UnityEditor 문제 등 빌드 안정성 이슈도 해결이 필요함. 가장 시급한 것은 Fracture 시스템과 VR 인터랙션을 연결하여 핵심 게임플레이를 완성하는 것이다.

리스크 알림

VR 핵심 인터랙션 코드 부재
VR 파괴 시뮬레이터의 핵심은 '플레이어가 VR 컨트롤러로 무기를 잡고 오브젝트를 파괴하는 경험'인데, BatSwing, VRInteraction, DestructionPhysics 등 XR Interaction Toolkit 기반의 그랩/스윙/충돌 코드가 확인되지 않음. 매니저와 UI는 있지만 실제 게임플레이가 동작하지 않을 수 있음.
  • XR Interaction Toolkit의 XRGrabInteractable을 무기 프리팹에 설정
  • 배트/무기의 OnCollisionEnter에서 충돌 속도 기반 파괴 트리거 구현
  • Fragmenter.Fracture()를 충돌 이벤트에서 호출하여 ScoreManager와 연동
  • XR Origin + Controller 인풋 바인딩 확인 및 테스트 씬 구성
기여도 불균형으로 인한 버스 팩터 1
verpivia-1이 전체 56커밋 중 38커밋(68%)을 담당하고, yuno3384는 2커밋뿐. verpivia-1이 매니저, UI, 버그 수정을 모두 담당하고 있어 이 팀원이 이탈하면 프로젝트가 멈출 위험이 있음. 김연준은 맵 관련 작업만 수행 중.
  • yuno3384에게 BatSwing 또는 VRInteraction 등 명확한 기능 담당 배정
  • 김연준이 StageLoader 씬 전환 로직을 담당하도록 역할 분배
  • 주 2회 코드 리뷰 세션을 통해 팀원 간 코드 이해도 공유
Fragmenter.cs의 using UnityEditor로 빌드 실패
Fragmenter.cs에 using UnityEditor가 전처리기 없이 포함되어 있어, VR 디바이스(Quest 등) 빌드 시 컴파일 에러가 발생함. OpenFracture 원본 코드를 수정 없이 가져온 결과.
  • Fragmenter.cs에서 using UnityEditor를 #if UNITY_EDITOR로 감싸기
  • EditorApplication 등 에디터 전용 API 호출부도 동일하게 처리
  • Android/Quest 타겟으로 빌드 테스트 실행
매니저 초기화 순서 미보장
BGMPlayer.Start()에서 UIManager.Instance와 AudioManager.Instance가 null이면 이벤트 구독 없이 return함. Unity의 Awake/Start 호출 순서는 보장되지 않으므로, 씬 로드 순서에 따라 BGM이 재생되지 않는 간헐적 버그 발생 가능.
  • Script Execution Order에서 매니저들의 실행 순서를 명시적으로 설정
  • 또는 BGMPlayer에서 코루틴으로 매니저 초기화를 대기하는 패턴 적용
  • 모든 Presenter 클래스에서 매니저 null 시 재시도 로직 추가
OpenFracture 원본 폴더 중복 존재
스크립트를 Assets/Scripts/Fracture/로 복사했지만 OpenFracture-main/ 폴더가 삭제되지 않아 Tests, Samples~, Documentation~ 등이 프로젝트에 남아있음. 같은 클래스가 두 곳에 존재하면 컴파일 에러 또는 혼란 유발.
  • OpenFracture-main/Runtime/Scripts/ 폴더 삭제 (이미 복사 완료)
  • OpenFracture-main/Tests/는 테스트 참조로 유지하되 asmdef로 격리 확인
  • 불필요한 Samples~, Documentation~ 폴더 정리

권고사항 (Top 3)

1. VR 게임플레이 루프 우선 구현
현재 매니저와 UI 인프라는 갖춰졌지만, '무기로 오브젝트를 때리면 파괴되고 점수가 오르는' 핵심 루프가 연결되지 않음. 남은 개발 기간에 이것이 가장 중요.
  1. WeaponController.cs 작성: XRGrabInteractable + Rigidbody로 무기 그랩 구현
  2. DestructionTrigger.cs 작성: OnCollisionEnter에서 충돌 속도 체크 → Fragmenter.Fracture() 호출
  3. ScoreManager와 연동: 파괴 시 ScoreEvent 발생 → GameManager에서 클리어 점수 도달 여부 체크 → OnStageClear 이벤트
  4. 하나의 Office 씬에서 엔드투엔드 테스트 완료 후 나머지 씬에 적용
2. .gitignore 정비 및 커밋 크기 관리
.csproj 파일 21개가 매 커밋마다 변경되어 diff를 오염시키고, 4000줄 이상의 대형 커밋이 5건 발생하여 코드 리뷰와 롤백이 어려움.
  1. .gitignore에 *.csproj, *.sln 추가 후 git rm --cached로 추적 해제
  2. 기능 단위로 커밋을 쪼개는 습관 정착 (한 커밋 = 한 기능/버그 수정)
  3. 커밋 메시지에 '커밋' 같은 무의미한 메시지 지양, 변경 내용을 구체적으로 기술
3. 매니저 간 초기화 안정성 확보
DontDestroyOnLoad 추가로 씬 전환 문제는 해결했지만, 매니저 초기화 순서와 이벤트 구독/해제 관리가 미비하여 런타임 에러 가능성이 있음.
  1. Project Settings > Script Execution Order에서 매니저 Awake 순서 지정: AudioManager → UIManager → ScoreManager → GameManager
  2. 모든 Presenter의 OnDestroy()에서 이벤트 구독 해제 코드 추가
  3. BGMPlayer, BGMVolumeSlider 등에서 매니저 null 시 WaitUntil 코루틴으로 대기 후 구독

코드 리뷰

GameManager.cs개선필요
싱글톤 패턴과 DontDestroyOnLoad가 올바르게 구현됨. stageClearConditions 인스펙터 배열을 _clearScoreMap 딕셔너리로 변환하고, 코드 기본값(_defaultClearScores)으로 폴백하는 방어적 설계가 좋음. 그러나 _defaultClearScores에 'Office1'(20), 'Office2'(1), 'Office3'(2)가 하드코딩되어 있어 씬 이름 변경 시 깨짐. OnStageClear 이벤트가 선언되어 있지만, 실제로 이 이벤트를 발생시키는 점수 체크 로직이 제공된 코드에서 보이지 않아 게임 루프 완성 여부가 불확실.
AudioManager.cs양호
싱글톤 + DontDestroyOnLoad 패턴이 깔끔하게 적용됨. _bgmEvent와 _hitSoundEvent를 Awake에서 초기화하여 다른 컴포넌트의 Start()에서 안전하게 구독 가능. BGM과 타격음을 분리된 이벤트로 관리하는 설계가 적절. OnBGMPlay 이벤트를 통해 BGMPlayer와 느슨하게 연결된 점이 좋음.
BGMPlayer.cs개선필요
최근 리팩토링에서 싱글톤 패턴이 추가되고 DontDestroyOnLoad로 씬 전환 시 유지되도록 개선됨. UIManager.Instance와 AudioManager.Instance의 null 체크가 추가된 점이 좋으나, Start()에서 null이면 return으로 빠져나와 이벤트 구독 자체를 건너뛰므로 이후 볼륨 조절과 BGM 재생이 완전히 동작하지 않게 됨. null일 때 재시도 로직이나 초기화 순서 보장이 필요. OnDestroy에서 이벤트 구독 해제 코드가 diff에 보이지 않아 메모리 누수 가능성 있음.
FragmentData.cs양호
OpenFracture 라이브러리의 핵심 데이터 구조. MeshVertex 리스트, 삼각형 인덱스, IndexMap, Bounds를 관리하며, Mesh API의 VertexAttributeDescriptor를 사용한 ToMesh() 변환이 구현됨. AddCutFaceVertex(), WeldCutFaceVertices() 등 메시 절단면 처리 로직이 완성도 높음. 다만 이 코드는 팀이 직접 작성한 것이 아니라 OpenFracture 오픈소스를 그대로 가져온 것.
Fragmenter.cs개선필요
Fracture(), FractureAsync(), Slice() 세 가지 파괴 모드를 제공하는 OpenFracture의 핵심 클래스. FractureAsync는 코루틴 기반으로 프레임 분산 처리가 가능. 그러나 using UnityEditor가 포함되어 있어 빌드 시 에러 발생함. #if UNITY_EDITOR 전처리기로 감싸야 함. CreateFragment()에서 fragmentTemplate을 복제하여 프래그먼트를 생성하는데, 대량 파괴 시 성능 이슈가 예상됨.
FakeStartEvent.cs개선필요
최근 수정에서 Camera.main null 체크가 추가된 것은 좋으나, Resources에서 AudioClip을 캐싱하는 _cachedClip 패턴이 구현되어 있음. Camera.main은 FindWithTag를 내부적으로 호출하므로 VR 환경에서 XR Origin의 카메라가 'MainCamera' 태그를 가지고 있는지 확인 필요. AudioSource.PlayClipAtPoint는 임시 GameObject를 생성하므로 빈번한 호출 시 GC 압력 발생.
ScoreManager.cs개선필요
싱글톤 패턴과 DontDestroyOnLoad가 적용됨. _scoreEvent를 통해 점수 변경을 이벤트로 전파하는 구조. 그러나 제공된 코드에서 실제 점수 증가/감소 메서드, 현재 점수 프로퍼티, GameManager와의 클리어 판정 연동 로직이 보이지 않아 이벤트 선언만 있고 실질적 게임플레이 로직이 미완성일 가능성이 높음.
UIManager.cs개선필요
싱글톤으로 _escapeEvent, _closeEvent, BGMVolume, EffectVolume을 관리. UI 시스템의 중앙 허브 역할. DontDestroyOnLoad 추가로 씬 전환 시 설정 유지됨. 그러나 모든 UI 이벤트가 하나의 매니저에 집중되어 있어 기능 추가 시 비대해질 우려가 있음. 볼륨 설정의 저장/복원(PlayerPrefs 등) 기능이 보이지 않아 앱 재시작 시 초기화될 것으로 예상됨.

아키텍처 평가

프로젝트는 Singleton 패턴(GameManager, ScoreManager, AudioManager, UIManager)과 Observer 패턴(C# event/delegate)을 핵심 아키텍처로 사용함. UI는 MVP(Model-View-Presenter) 변형으로 설계되어 VRProject.UI.Model(매니저, 이벤트)과 VRProject.UI.Presentation(BGMPlayer, 슬라이더 등 뷰 로직)으로 분리됨. 이는 학생 프로젝트로서 상당히 체계적인 설계. 다만 매니저들이 서로의 Instance를 직접 참조(BGMPlayer → UIManager.Instance, AudioManager.Instance)하는 구조는 순환 의존성과 초기화 순서 문제를 야기할 수 있음. FractureSystem은 OpenFracture 라이브러리를 통째로 가져온 것으로, Delaunay Triangulation 기반의 메시 절단 알고리즘이 포함되어 있으나 이를 VR 인터랙션과 연결하는 커스텀 브릿지 코드가 부족함.

단계별 진행률 (AI 평가)

기반 구축 4/5점
Unity 프로젝트 구조, OpenXR 패키지, 씬 구성(BasicScene, Maps, Testing), asmdef 5개로 어셈블리 분리가 세팅되어 있음. .csproj 파일들이 git에 커밋되어 Hot Files 상위 10개를 모두 차지하고 있어 .gitignore 설정이 미비함. 프로젝트 폴더 구조는 Scripts/Fracture, Scripts/UI/Model, Scripts/UI/Presenter로 정리되어 있으나, OpenFracture 라이브러리 원본 폴더(OpenFracture-main/)가 여전히 남아있는 상태.
핵심 시스템 14/25점
GameManager(스테이지 클리어 조건, _clearScoreMap, OnStageClear 이벤트), ScoreManager(_scoreEvent), AudioManager(_bgmEvent, _hitSoundEvent), UIManager(_escapeEvent, _closeEvent, BGMVolume/EffectVolume)가 실제 동작 로직으로 구현됨. FractureSystem은 OpenFracture 오픈소스 라이브러리를 그대로 가져온 것으로 Fragmenter.Fracture(), Fragmenter.Slice(), MeshSlicer, ConstrainedTriangulator 등이 포함됨. 그러나 BatSwing(배트 스윙 메커닉), Weapon(무기 시스템), VRInteraction(VR 컨트롤러 인터랙션), StageLoader(씬 전환 로더), DestructionPhysics(파괴 물리 커스텀 로직)의 소스 코드가 제공된 코드에서 확인되지 않음. 커밋 메시지에 'Fix/WeaponBugFixed'가 있어 무기 코드가 존재할 가능성이 있으나 실제 로직 확인 불가.
통합 10/25점
매니저들이 C# event를 통해 연결됨: AudioManager.OnBGMPlay → BGMPlayer.PlayBGM, UIManager.OnBGMVolume → BGMPlayer.SetVolume. GameManager는 stageClearConditions로 씬별 클리어 점수를 관리하고 OnStageClear 이벤트를 발생시킴. 모든 매니저에 DontDestroyOnLoad가 추가되어 씬 전환 시 유지됨. 그러나 실제 게임 루프(플레이어가 오브젝트를 파괴 → 점수 증가 → 클리어 판정)의 전체 흐름을 연결하는 코드는 제공된 소스에서 확인되지 않음.
