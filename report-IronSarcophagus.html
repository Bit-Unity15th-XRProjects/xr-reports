프로젝트 현황 요약

IronSarcophagus 팀은 Netcode for GameObjects 기반의 VR 멀티플레이어 탱크 시뮬레이션을 개발하고 있으며, 4인 승무원(포수/조종수/장전수/수리병) 역할 분담 시스템(RoleSync)을 핵심으로 방향별 장갑 관통 판정(TankStatus), 시간 스텝 탄도 시뮬레이션(BallisticCalc), 6상태 FSM 기반 적 AI(EnemyTank), 웨이브 기반 섬멸전 게임 모드(AnnihilationGameMode+WaveManager)까지 핵심 게임플레이 루프를 구현 완료한 상태다. 56개 C# 스크립트, 41개 프리팹, 6개 씬으로 구성된 프로젝트 규모는 학생 팀 프로젝트로서 상당히 야심적이며, 특히 XR 입력→네트워크 동기화→물리 시뮬레이션→피격 판정으로 이어지는 발사 파이프라인의 완성도가 높다. 그러나 오디오 에셋이 전혀 없고(0개), 애니메이션도 부재하여 시연 시 몰입감이 크게 떨어질 수 있으며, 3인 팀 중 namwon-RT가 13일간 비활성이고 미병합 브랜치 4개가 존재하여 마감 전 통합 리스크가 있다. 가장 시급한 것은 기본 사운드 에셋 추가와 미병합 브랜치 정리를 통한 안정 빌드 확보이다.

  • CI에서 컴파일 확인 후 안정 빌드 태그 생성

1. 오디오 에셋 긴급 추가 및 사운드 시스템 구현
VR 탱크 시뮬레이션에서 사운드는 몰입감의 핵심. 현재 오디오 0개는 발표 평가에서 가장 큰 감점 요인이 될 수 있음. 코드 구조상 이벤트 기반으로 연결하면 1-2일 내 기본 사운드 구현 가능.
  1. Unity Asset Store 무료 'War SFX' 패키지에서 탱크 발사/피격/엔진/폭발 사운드 5-8개 확보
  2. FireSystem.OnImpact에 폭발 AudioClip 재생, BarrelLoader 장전 완료 시 장전음 재생
  3. TankDriver에 AudioSource 추가하여 _netCurrentSpeed 값에 비례한 엔진 루프 재생
  4. 03_Game 씬에 앰비언트 BGM AudioSource 배치
2. 미병합 브랜치 정리 및 안정 빌드 확보
발표까지 시간이 촉박한 상황에서 4개 미병합 브랜치가 존재. 특히 chore/scene-prefab-updates(+5 ahead)와 fix/compiler-warnings(+3 ahead)는 최근 활동이 있어 즉시 병합 가능. 안정적인 발표 빌드를 위해 코드 동결 시점을 명확히 해야 함.
  1. fix/compiler-warnings를 develop에 먼저 병합 (컴파일 경고 해결)
  2. chore/scene-prefab-updates를 develop에 병합 (프리팹/씬 업데이트)
  3. develop을 origin/main에 동기화
  4. 병합 후 VR 빌드 테스트하여 안정 확인
  5. 이후 코드 동결하고 버그 수정만 허용
3. VR 프레임레이트 확보를 위한 성능 핫픽스
VR에서 프레임 드롭은 사용자 멀미로 직결. SharedHUDController 마커 누수, BallisticCalc 배열 할당, EnemyTank 5점 Linecast가 후반 웨이브에서 복합적으로 성능을 저하시킬 수 있음.
  1. SharedHUDController: 마커 오브젝트 풀링 적용 (기존 비활성 마커 재활용)
  2. BallisticCalc: SimulateTrajectory() 내 배열을 사전 할당된 버퍼로 교체
  3. EnemyTank: CalculateExposure()를 매 틱이 아닌 0.5초 간격으로 실행하도록 타이머 추가

코드 리뷰

EnemyTank.cs개선필요
736줄로 프로젝트 내 최대 규모 파일. 6상태 FSM(Idle/Alert/Combat/Repositioning/Tracking/Immobilized)과 인식도 시스템(0-100, FOV/거리/노출도 기반), 방향별 장갑 시스템, 탄도 발사, 재배치 AI까지 실동작 로직이 모두 구현되어 있어 기능적으로는 완성도가 높음. 그러나 AILoop(), UpdateAwareness(), AimAndFire() 등 핵심 메서드가 각각 수십 줄을 초과하며 단일 파일에 인식/전투/이동/포탑 회전/피격 판정이 모두 집중되어 있어 유지보수가 어려움. CalculateExposure()의 5점 Linecast가 매 프레임 실행 시 다수 적 탱크에서 성능 부담. EnemyPartHPState를 struct로 깔끔하게 분리한 점은 좋음.
AimController.cs양호
499줄이지만 역할이 명확: XR 입력 읽기→ServerRpc→NetworkVariable 동기화→포탑/포신 회전. NetworkVariable<float> _netYaw/_netPitch를 Server WritePermission으로 설정하고 UpdateAimServerRpc로 델타 전송하는 패턴이 정석적. XRGrabInteractable 기반 Grab→조작 활성화→Release→비활성화 흐름이 VR UX에 적합. ContinuousTurnProvider 비활성화로 Grab 중 의도치 않은 회전 방지한 점 우수. 줌(1x/2x/4x) 및 민감도 조절까지 구현. 다만 민감도 값이 로컬 전용으로 네트워크 동기화되지 않아 시각적 불일치 가능성 있으나, 로컬 UX 설정이므로 실질적 문제는 아님.
TankStatus.cs양호
서버-권위 부위별 HP 관리의 핵심 클래스. PartHPState를 INetworkSerializable struct(6 float=24바이트)로 효율적 직렬화. 방향별 장갑(ArmorProfile)의 관통 판정 로직이 정확: PenetrationMm≤0(HE탄 등 관통 불필요) || EffectivePenetrationMm≥thickness. 부위 파괴 시 spillover 데미지(포탑→차체/포신, 엔진→궤도)로 연쇄 피해 구현. Critical 부위(Hull/AmmoRack) 파괴 시 전차 파괴 이벤트 발생. 수리 기능(RepairPartServerRpc)도 구현. 다만 수리 시 최대 HP 초과 검증이 없는 점, PartHP 프로퍼티가 struct 값 복사를 반환하는 비효율이 소소한 개선점.
RoleSync.cs양호
408줄. 4개 역할 좌석의 NetworkVariable<ulong> 점유 상태 관리. RequestBoardServerRpc/RequestLeaveServerRpc로 서버 검증 후 좌석 배정. 클라이언트 연결 해제 시 OnClientDisconnectCallback에서 자동으로 빈 좌석 처리하는 안전장치 구현. 좌석 점유 시 해당 NetworkObject의 소유권을 이전(ChangeOwnership)하여 AimController 등 하위 컴포넌트의 IsOwner 검증이 자연스럽게 작동. OnLocalRoleChanged/OnStationOccupancyChanged 이벤트로 UI/입력 시스템과 느슨한 결합 유지. 다만 RequestBoardServerRpc에서 clientId가 실제 연결된 클라이언트인지 검증하지 않는 점이 유일한 문제.
FireSystem.cs양호
130줄로 간결하면서 역할이 명확. AimController.OnFireRequested 이벤트 구독→ProjectilePoolManager에서 시각 투사체 획득→BallisticCalc.TryGetImpactData로 착탄점 계산→비행시간 대기 후 SplashDamageUtils.Apply로 데미지 적용. 시각적 투사체와 논리적 데미지를 분리하여 네트워크 일관성 확보. OnImpact 이벤트로 VFX/SFX 핸들러와 디커플링. 다만 시각 투사체는 착탄 계산 실패 시에도 스폰되는 점, 발사 전 탄약 잔량 검증이 BarrelLoader 쪽에서만 이루어져 FireSystem 자체에는 없는 점이 개선 여지.
WaveManager.cs개선필요
213줄. WaveData 기반 데이터 주도 웨이브 설계로 Inspector에서 적 종류/수량 설정 가능. Fisher-Yates 셔플로 스폰 위치 다양성 확보. _aliveCount를 사전 설정하여 스폰 중 적 사망 시 레이스 컨디션 방지. NavMesh.SamplePosition으로 지형 위 정확한 배치. 그러나 SpawnEnemy에서 Instantiate 후 NetworkObject.Spawn() 호출 시 NetworkObject 컴포넌트 존재 여부를 검증하지 않아 프리팹 설정 실수 시 런타임 크래시 위험. 또한 씬에 NavMesh가 없을 때의 처리가 없음.
SharedHUDController.cs개선필요
495줄로 프로젝트 내 두 번째 대형 파일. 부위별 HP 색상 그라데이션, 미니맵(North-up, 차체/포탑 아이콘, 적 마커), 좌석 상태 표시 등 풍부한 UI 구현. 그러나 미니맵 마커 시스템에 메모리 누수 존재: CreateDot()으로 생성된 GameObject가 RebuildMarkers()에서 비활성화만 되고 파괴되지 않아 적 사망이 반복되면 비활성 오브젝트 누적. FindGameObjectsWithTag에서 미등록 태그 시 예외를 try-catch로 무시하여 실제 버그 은폐 가능성. 파일 길이를 줄이려면 미니맵 로직을 별도 클래스로 분리 권장.
BallisticCalc.cs양호
189줄. 시간 스텝 기반 탄도 시뮬레이션으로 중력 적용, Linecast 충돌 감지 구현. SimulateTrajectory()가 시각 투사체와 데미지 판정 양쪽에 동일하게 사용되어 물리 일관성 확보. TimeOfFlight와 TrajectoryTimeStep을 분리하여 클라이언트 예측과 서버 판정의 유연성 제공. AmmoData ScriptableObject로 탄종별 속도/관통력 설정. 다만 궤적 계산 시 points[..(i+1)] 배열 슬라이싱이 매번 새 배열을 할당하여 GC 압박 유발, _currentAmmo null 시 안전 기본값 반환은 하지만 명시적 경고가 없는 점이 개선 여지.

커밋 분석

커밋 메시지 서술성 96%로 매우 우수하며, Conventional Commits(feat/fix/chore/docs) 형식을 일관되게 사용. 최근 4주 커밋 추이(0→50→38→60)는 마감을 향해 가속하는 전형적인 학생 프로젝트 패턴. 이재영이 최근 주요 버그 수정(AimController NetworkVariable 권한, WaveManager 레이스 컨디션, AmmoDispenser 중복 Grab)을 집중 처리하며 프로젝트 안정화를 주도. Nullma는 인프라/에셋/문서 작업에 집중. 다만 커밋 크기의 중앙값 141줄 대비 평균 656줄로 일부 대형 커밋(4867줄 GunnerHUD, 4449줄 EnemyAI)이 평균을 끌어올림. 이런 대형 커밋은 리뷰가 어렵고 롤백이 복잡해지므로 기능 단위로 분할하는 것이 바람직했음.
S

아키텍처 평가

서버-권위(Server-Authoritative) 아키텍처를 일관되게 적용: 모든 게임 상태 변경은 NetworkVariable(WritePermission.Server)로 관리되고, 클라이언트는 ServerRpc를 통해서만 변경 요청 가능. IDamageable 인터페이스로 EnemyBase/EnemyTank/TankStatus가 동일한 데미지 파이프라인(SplashDamageUtils→TakeDamage)을 공유하는 다형성 설계가 우수. RoleSync의 이벤트 기반 역할 변경 알림(Observer 패턴)으로 HUD/입력 시스템과의 결합도를 효과적으로 낮춤. ScriptableObject(AmmoData)를 활용한 데이터 주도 설계로 탄종 밸런싱이 Inspector에서 가능. 다만 EnemyTank이 인식/전투/이동/포탑/피격을 모두 포함하는 God Object 경향이 있고, GameManager의 상태 전이가 Coroutine 기반이라 복잡한 에러 상태 처리가 어려운 점은 아쉬움.

심각도파일설명
HIGHSharedHUDController.cs미니맵 RebuildMarkers()에서 CreateDot()으로 생성된 적 마커 GameObject가 비활성화만 되고 Destroy되지 않음. 적이 반복 사망/생성되면 비활성 오브젝트가 무한 누적되어 메모리 누수 발생.
HIGHWaveManager.csSpawnEnemy()에서 Instantiate 후 GetComponent<NetworkObject>()의 null 체크 없이 Spawn() 호출. 프리팹에 NetworkObject가 누락되면 NullReferenceException으로 웨이브 전체 중단.
MEDIUMEnemyTank.csCalculateExposure()가 5점 Physics.Linecast를 매 AI 틱마다 실행. 적 탱크 10대 동시 활성 시 프레임당 50회 Linecast로 VR 프레임레이트(72/90Hz) 유지에 영향 가능.
MEDIUMRoleSync.csRequestBoardServerRpc(StationRole role, ulong clientId)에서 전달받은 clientId가 NetworkManager.ConnectedClientsIds에 실제 존재하는지 검증하지 않음. 잘못된 clientId로 ChangeOwnership 호출 시 예외 발생 가능.
MEDIUMBallisticCalc.csSimulateTrajectory() 내부 points[..(i+1)] 범위 연산자 사용이 매 호출마다 새 배열을 할당하여 GC(가비지 컬렉션) 부담 유발. VR 환경에서 GC spike는 프레임 드롭으로 직결됨.
MEDIUMEnemyBase.csAwake()의 Physics.IgnoreLayerCollision(gameObject.layer, gameObject.layer, true)가 해당 레이어의 모든 오브젝트 간 충돌을 전역 비활성화. 의도는 적 간 충돌 방지이나, 같은 레이어의 다른 오브젝트(환경 등)도 영향받을 수 있음.
MEDIUMGameManager.cs씬 로딩(LoadScene) 호출 후 타임아웃이 없어 에셋 누락이나 로딩 실패 시 무한 대기 상태에 빠질 수 있음. 특히 발표 시연 중 이 상황 발생 시 복구 불가.
MEDIUMAmmoPackDispenser.cs_pendingInteractor 브리지 패턴에서 Grab→Release→Re-Grab이 빠르게 발생하면 _isRequestPending 플래그와 실제 상태가 불일치할 수 있는 레이스 컨디션 존재.
LOWTankStatus.csRepairPartServerRpc에서 수리 시 현재 HP가 최대 HP를 초과하는지 검증하지 않음. Mathf.Min(current + amount, maxHP) 클램핑 필요.
LOWAmmoPack.cs_ammoData(ScriptableObject) null 체크 없이 접근하는 경로 존재. Inspector에서 미설정 시 NullReferenceException 발생 가능.
