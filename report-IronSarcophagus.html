프로젝트 현황 요약

IronSarcophagus 팀은 Netcode for GameObjects + XR Interaction Toolkit 기반의 VR 4인 협동 전차 시뮬레이션을 개발 중이며, 포수(AimController/FireSystem/BallisticCalc), 운전수(TankDriver), 장전수(BarrelLoader/AmmoPackDispenser), 수리수(TankStatus.RepairPartServerRpc) 4개 역할의 핵심 게임플레이가 구현되어 있다. 적 AI는 EnemyTank의 6단계 상태머신+인지도 시스템, EnemyInfantry의 병과별 행동으로 상당한 깊이를 가지며, WaveManager→AnnihilationGameMode→HUD로 이어지는 게임 루프도 완성되어 프로토타입 시연은 가능한 상태다. 가장 시급한 것은 오디오 에셋 부재(0개)로, VR 체험의 몰입감을 결정적으로 저해하고 있으며, 미병합 브랜치 5개의 통합과 게임 오버 흐름 완성이 발표 전까지 해결해야 할 핵심 과제이다. 코드 품질은 학생 프로젝트 수준을 상회하나, EnemyTank.cs(736줄)의 비대함과 namwon-RT의 12일 비활성이 유지보수와 팀 역학 측면에서 우려된다.

EnemyTank.cs개선필요
736줄로 프로젝트에서 가장 큰 파일. AI 상태머신(Idle→Alert→Combat→Repositioning→Tracking→Immobilized), 인지도 시스템(CalculateExposure 5점 Linecast), 포탑/차체 회전, 발사 로직, 부위별 HP+장갑 관통 계산까지 모두 하나의 클래스에 집중. 각 기능의 구현 품질은 높으나(accuracy 기반 spread, 방향별 장갑 프로파일, 탄약고 유폭 확률 등), 단일 책임 원칙을 크게 위반. EnemyAwareness, EnemyCombat, EnemyArmor 등으로 분리하면 유지보수성이 크게 향상됨. _turretTransform null 시 fallback으로 transform.position + Vector3.up*2f를 사용하는 GetMuzzlePos()는 안전한 처리.

499줄이지만 복잡한 책임(XR 입력, 네트워크 동기화, 줌, 민감도, 발사 요청)을 체계적으로 관리. 최근 diff에서 NetworkVariable 쓰기 권한을 Owner→Server로 변경하고 클라이언트 예측(_localYaw/_localPitch)을 도입한 것은 Netcode 비-호스트 클라이언트 문제를 정확히 해결. 커밋 메시지 품질이 96% 서술적으로 매우 우수하며, conventional commit 형식(feat/fix/chore/refactor)을 일관되게 사용. 특히 이재영의 최근 커밋들(fix(AimController): NetworkVariable 쓰기 권한을 Owner→Server로 변경하여...)은 변경 이유까지 메시지에 포함하여 모범적. 다만 평균 변경 라인이 650줄로 중앙값(144줄) 대비 과도하게 높은데, 이는 GunnerHUD(4867줄), EnemyAI(4449줄) 등 초대형 커밋 때문. 이런 커밋은 리뷰 불가능한 크기이며 Unity 프리팹/씬 변경이 포함된 것으로 추정됨. 주간 커밋 트렌드(0→38→44→60)가 가속되고 있어 마감 직전 집중 개발 패턴이 보이며, 이재영이 최근 모든 버그 수정과 기능 추가를 주도하고 있음.

아키텍처 평가

서버 권위(Server Authority) 아키텍처가 전체 시스템에 일관되게 적용됨 — 데미지(TankStatus.TakeDamageServerRpc), 조준(AimController.UpdateAimServerRpc), 역할 배정(RoleSync.RequestBoardServerRpc), 스폰(WaveManager 서버 전용)이 모두 서버에서 검증됨. Observer 패턴을 적극 활용하여(OnFireRequested, OnLocalRoleChanged, OnAliveCountChanged, OnWaveStarted) 시스템 간 결합도를 효과적으로 낮춤. AnnihilationGameMode의 static event + ClientRpc 브릿지 패턴은 NetworkObject가 씬 UI를 직접 참조할 수 없는 Netcode 제약을 실용적으로 해결한 좋은 판단. 데이터 흐름은 Inspector SerializeField → NetworkVariable → OnValueChanged → UI 업데이트로 일관되며, AmmoData ScriptableObject로 탄종 데이터를 분리한 것도 적절한 SO 활용. 다만 RoleSync가 Singleton 패턴(Instance)인 반면 GameManager는 static 필드 기반이어서 생명주기 관리 방식이 혼재됨.
