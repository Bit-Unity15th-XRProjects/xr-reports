프로젝트 현황 요약

IronSarcophagus 팀은 VR 멀티플레이 탱크 시뮬레이션을 개발 중이며, 핵심 전투 시스템(포물선 탄도, 방향별 장갑 관통, 부위별 데미지)과 적 AI(인지도 기반 상태머신, 수송차량, 보병)가 높은 완성도로 구현되어 있다. Netcode 기반 4인 협동(포수/조종수/장전수/수리) 역할 시스템이 RoleSync를 통해 동작하며, GameManager 상태머신이 Title→Lobby→Game 씬 전환을 관리한다. 그러나 가장 시급한 문제는 HUD 시스템이 3개 feature 브랜치에 분산되어 develop에 미병합된 상태로, 플레이어가 게임 정보를 확인할 수 없다는 점이다. 또한 오디오와 애니메이션이 완전히 부재하여 VR 몰입감이 크게 저하되고, namwon-RT의 6일 비활성과 46커밋 뒤처진 JWT 브랜치가 팀 리스크로 작용한다. 코드 품질은 학생 프로젝트 수준을 넘어서는 설계력을 보이나, RoleSync 이벤트 구독해제 버그와 GameManager 중괄호 누락 같은 네트워크 안정성 이슈를 시연 전에 반드시 수정해야 한다.

코드 리뷰

EnemyTank.cs개선필요
736줄로 프로젝트 가이드라인(300줄)을 크게 초과. 그러나 내부 로직은 매우 인상적 — 6상태 AI 상태머신(Idle/Alert/Combat/Repositioning/Tracking/Immobilized), 5점 레이캐스트 노출도 계산(CalculateExposure), 포물선 탄도 발사(TryCalculateLaunchDir에서 이차방정식 풀이), 방향별 장갑 관통 판정이 모두 실제 동작하는 코드. 문제점: ①AILoop 코루틴이 상태별 분기를 모두 포함해 복잡도가 높음 — CombatBehavior, PatrolBehavior 등으로 분리 권장. ②OnDrawGizmos()가 OnDrawGizmosSelected()가 아니라 항상 실행되어 에디터 성능 저하. ③EnemyPartHPState가 같은 파일에 struct로 정의되어 있어 별도 파일로 분리 필요.
TankStatus.cs양호
부위별 HP(6부위) + 방향별 장갑(Hull/Turret 각각 전면/측면/후면) + 관통/도탄 판정이 완전하게 구현됨. CalculateDamage()에서 hitNormal과 탱크 forward의 dot product로 피격 면 판별(>0.5=전면, <-0.5=후면, else=측면)하는 로직이 정확. 관통 실패 시 5% 도탄 데미지 적용도 게임 디자인적으로 적절. ApplyPartDamage()에서 엔진/트랙 파괴 시 Immobilize, 탄약고 파괴 시 즉사 등 부위별 효과가 잘 분기됨. NetworkVariable로 각 부위 HP를 동기화하는 구조도 올바름.
RoleSync.cs개선필요
4역할 스테이션 관리와 NetworkObject 소유권 이전 로직이 잘 설계됨. RequestBoardServerRpc에서 빈 좌석 확인 후 소유권 변경, roleObjects 토글까지 한 번에 처리하는 플로우가 깔끔. 그러나 심각한 버그: OnNetworkDespawn()에서 '(_, v) => HandleOccupancyChanged(...)' 람다로 -= 구독해제하는데, 이는 OnNetworkSpawn()에서 += 한 람다와 다른 인스턴스이므로 실제로 해제되지 않아 메모리 누수 및 MissingReferenceException 발생 가능. 델리게이트를 필드에 저장하고 재사용해야 함. 또한 Instance 싱글톤 패턴이 탱크 1대만 가정하여 다중 탱크 확장 시 구조 변경 필요.
AimController.cs양호
NetworkVariable(Owner write)로 Yaw/Pitch를 동기화하여 조준 지연을 최소화한 설계가 적절. XRGrabInteractable 기반 'Grab to Aim' 메카닉으로 핸들을 잡아야 조준 가능하고, Grab 시 ContinuousTurnProvider/SnapTurnProvider를 비활성화해 thumbstick 입력 충돌을 방지하는 세심한 처리. OnRoleSyncRoleChanged로 역할 변경 시 입력을 동적으로 Enable/Disable. 에디터 테스트용 키보드 입력 코드가 #if UNITY_EDITOR로 적절히 분리됨. 다만 _cachedOrigin이 Grab 해제 후에도 유지되는데, XROrigin이 파괴되면 dangling reference 가능.
GameManager.cs개선필요
상태 머신 기반 씬 전환 워크플로우가 체계적이고, Additive 씬 로딩으로 XROrigin 영속화를 해결한 최근 리팩토링이 좋은 판단. 그러나 InLobby와 InGamePlaying 상태에서 'if (NetworkManager.Singleton != null)' 다음 줄의 'NetworkManager.Singleton.Shutdown();'이 if 블록 중괄호 없이 작성되어, null일 때도 Shutdown이 호출되는 버그. 또한 모든 상태 전환 로직이 하나의 Workflow() 코루틴에 집중되어 있어 가독성이 떨어지고, 새 상태 추가 시 switch문이 계속 비대해질 구조.
FireSystem.cs양호
BallisticCalc.TryGetImpactData()로 탄도 계산 후 착탄점/비행시간을 받아 시각적 투사체 스폰과 지연 데미지 적용을 분리한 설계가 우수. 직격(DirectHit) 시 즉시 데미지, 범위 피해는 SplashDamageUtils로 위임하여 단일 책임 원칙 준수. AmmoData ScriptableObject에서 탄종별 속도/데미지/폭발반경을 읽어오는 데이터 주도 설계. AimController.OnFireRequested 이벤트 구독으로 결합도를 낮춤.
BarrelLoader.cs개선필요
XRSocketInteractor 기반 물리적 장전 메카닉이 VR 몰입감에 기여. InsertAmmo/ExtractAmmo 시 RPC로 서버 검증 후 AimController.IsLoaded 갱신하는 흐름이 올바름. 그러나 SetLoadedServerRpc에서 _loadedPack이 이미 Despawn된 경우의 null 방어가 부족하고, AmmoPack이 빈 상태에서 1초 후 자동 Despawn하는데 그 1초 사이에 장전 시도가 가능한 레이스 컨디션 존재. OnAmmoInserted/OnAmmoRemoved 이벤트가 선언만 되고 실제 구독자가 현재 코드에서 확인 안 됨(애니메이션 미구현 가능성).
EnemyLightVehicle.cs양호
5단계 AI(탐지대기→접근→하차→퇴각→디스폰)가 명확한 코루틴 체인으로 구현됨. SteerVehicle()에서 NavMesh steeringTarget과 현재 방향의 각도차로 속도를 조절하는 최소 회전반경 시뮬레이션이 인상적. UnloadInfantry()에서 측면 간격을 두고 보병 스폰 후 SetTarget 주입하는 패턴이 깔끔. 정체 감지(0.3초간 30cm 미만 이동 시 즉시 하차)로 AI 루프 막힘 방지. Despawn 시 NetworkObject.Despawn()과 Destroy() 분기 처리도 올바름.

아키텍처 평가

Netcode for GameObjects 기반으로 서버 권위적(Server-Authoritative) 아키텍처를 올바르게 구현. NetworkVariable(Owner write)로 실시간 동기화가 필요한 조준 데이터를, RPC(Server validation)로 발사/장전 같은 게임 액션을 처리하는 이원 구조가 적절. IDamageable 인터페이스 + DamageInfo 구조체 + SplashDamageUtils 유틸리티로 데미지 파이프라인을 추상화하여 플레이어 탱크(TankStatus)와 적(EnemyTank/EnemyBase)이 동일 시스템을 공유하는 설계가 우수. EnemyBase → EnemyTank/EnemyLightVehicle/EnemyInfantry 상속 구조로 공통 로직(HP, Die, TakeDamage)을 재사용. 그러나 RoleSync.Instance 싱글톤 패턴이 탱크 1대를 강제하는 확장성 제약이 있고, GameManager가 씬 전환/상태 관리/네트워크 제어를 모두 담당하여 God Object 경향. Observer 패턴(이벤트)을 적극 활용하여 시스템 간 결합도를 낮춘 점은 좋으나, 일부 시스템(StationMapUI)이 폴링에 의존하는 불일치가 있음.

CRITICALRoleSync.csOnNetworkDespawn()에서 람다 '(_, v) => HandleOccupancyChanged(StationRole.Gunner, v)'로 -= 구독해제하는데, 이는 OnNetworkSpawn()에서 += 한 람다와 다른 인스턴스이므로 실제 해제되지 않음. NetworkVariable.OnValueChanged 콜백이 객체 파괴 후에도 호출되어 MissingReferenceException 발생 가능
CRITICALGameManager.csInLobby 상태(약 line 100)와 InGamePlaying 상태에서 'if (NetworkManager.Singleton != null)' 다음 줄 'NetworkManager.Singleton.Shutdown();'이 중괄호 없이 작성됨. diff에서 확인되는 들여쓰기상 if 블록에 속하지 않아 NullReferenceException 위험
HIGHEnemyTank.cs736줄로 프로젝트 코딩 가이드라인(파일 300줄 이하)의 2.4배 초과. AI 상태머신, 인지 시스템, 전투 로직, 데미지 처리, 포탑 회전이 모두 한 클래스에 집중되어 유지보수 어려움

MEDIUMBarrelLoader.csSetLoadedServerRpc()에서 _loadedPack 참조가 AmmoPack 자동 Despawn(빈 탄창 1초 후 파괴)과 레이스 컨디션 가능. Despawn된 NetworkObject 접근 시 InvalidOperationException
LOWAimController.cs_cachedOrigin(XROrigin)이 Grab 해제 후에도 필드에 유지됨. XROrigin이 씬 전환 등으로 파괴되면 dangling reference. OnLocomotionReleased에서 cachedOrigin 사용 시 null 체크는 있으나, 다음 Grab 시 재탐색 실패하면 stale 캐시 사용 가능

LOWPlayerAvataSpawner.cs파일명에 오타: 'Avata' → 'Avatar'. 클래스명도 동일하게 PlayerAvataSpawner로 오타. 리팩토링 시 참조 추적이 혼란스러울 수 있음
