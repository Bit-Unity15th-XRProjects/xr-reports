프로젝트 현황 요약

IronSarcophagus 팀은 Netcode 기반 VR 멀티플레이 탱크 시뮬레이션을 개발 중이며, 4좌석(포수/운전수/장전수/수리) 역할 분담 시스템(RoleSync)과 물리 기반 탄도(BallisticCalc) + 방향별 장갑 관통(TankStatus) 등 핵심 전투 메커니즘이 실동작 수준으로 구현되어 있다. EnemyTank의 6단계 AI 상태머신과 인식 시스템, WaveManager의 웨이브 스포너, 3종 HUD(Gunner/Driver/Shared)까지 상당한 분량의 코드가 작성되었다. 그러나 가장 시급한 문제는 게임 루프의 부재로, WaveManager와 GameManager/GameResultManager 간 연결이 없어 웨이브 클리어→승리 조건이 동작하지 않는다. 또한 오디오·VFX 에셋이 전무하고, EnemyLightVehicle의 네트워크 동기화 버그, FireSystem에서 탄약 차감 경로 단절 가능성 등 통합 테스트가 필요한 이슈가 남아있다. Nullma가 전체 커밋의 48%를 담당하며 사실상 메인 개발자 역할을 하고 있고, namwon-RT의 비활성(8일)은 팀 리소스 관점에서 리스크이다.

리스크 알림

게임 루프 미완성 — 승리/패배 조건 부재
WaveManager에 OnWaveStarted 이벤트만 있고 웨이브 종료 감지(적 전멸), 다음 웨이브 자동 진행, 최종 승리 조건이 구현되지 않음. GameResultManager는 탱크 파괴(패배)만 감지하고 승리 시 ShowGameClear()를 호출하는 코드가 없음. 현재 상태로는 '게임'이 아닌 '전투 데모' 수준.
  • WaveManager에 적 전멸 감지 로직 추가 (EnemyBase.OnDied 이벤트 구독 + 카운터)
  • 모든 웨이브 클리어 시 GameResultManager.ShowGameClear() 호출 체인 구현
  • StartNextWave() 자동 호출 또는 웨이브 간 대기 UI 추가
EnemyLightVehicle 클라이언트 동기화 불가
경차량이 transform을 직접 수정하지만 NetworkTransform 없이 서버에서만 위치가 갱신됨. 멀티플레이에서 클라이언트는 경차량이 스폰 위치에 멈춰 보이거나 보이지 않음. 보병 하차 연출이 클라이언트에서 깨짐.
  • EnemyLightVehicle 프리팹에 NetworkTransform 컴포넌트 추가
  • NetworkTransform의 Interpolate=true로 부드러운 이동 보장
  • 멀티플레이 환경에서 경차량 스폰→이동→보병 하차→퇴장 전체 시퀀스 테스트
VR 성능 — 적 탱크 레이캐스트 부하
EnemyTank.IsPlayerVisible()가 매 프레임 5회 Linecast를 수행. 적 탱크 4대만 있어도 프레임당 20회 Physics.Linecast. VR은 90fps 유지가 필수이며, 물리 연산 과부하는 멀미 유발.
  • CalculateExposure()를 0.2~0.5초 간격으로 캐싱하여 호출 빈도 감소
  • AI 상태별 레이캐스트 빈도 차등 적용 (Idle=1초, Combat=0.2초)
  • Unity Profiler로 Physics.Linecast 비용 측정 후 Job System 병렬화 검토
발사 후 탄약 차감 경로 단절 가능성
최근 diff에서 FireSystem에서 _barrelLoader 참조와 NotifyFired() 호출이 제거됨. BarrelLoader.NotifyFired()가 AmmoPack.NotifyFired()를 호출해 잔탄을 차감하는 구조인데, 이 체인이 끊어졌을 수 있음. 무한 발사 버그로 이어질 위험.
  • FireSystem→BarrelLoader→AmmoPack 탄약 차감 체인이 다른 경로로 유지되는지 검증
  • 발사 후 AmmoPack._netCurrentRounds 감소 확인 테스트
  • 차감 경로가 없다면 FireSystem에서 AimController.IsLoaded=false 설정 + BarrelLoader 알림 복구
namwon-RT 8일 비활성 — 브랜치 머지 위험
namwon-RT가 23커밋을 기여했지만 3/17 이후 비활성. feature/jwt_0000 브랜치가 main 대비 77커밋 뒤처져 있음. 오래 방치된 브랜치는 충돌이 누적되어 머지 불가능해질 수 있음.
  • namwon-RT의 작업 상태 확인 및 feature/jwt_0000 브랜치 처리 방향 결정
  • 77커밋 뒤처진 브랜치는 rebase 또는 폐기 판단
  • unmerged 4개 브랜치(feat/EnemyAI, feat/DriverHUD 등) 정리 일정 수립

권고사항 (Top 3)

1. 게임 루프 완성 — WaveManager↔GameResultManager 연결
현재 전투 시스템은 개별적으로 잘 작동하지만 '게임'으로 연결되지 않음. 웨이브 클리어→다음 웨이브→최종 승리의 흐름이 없으면 데모 이상의 경험을 제공할 수 없음.
  1. WaveManager에 _aliveEnemyCount 카운터 + EnemyBase.OnDied 구독으로 적 전멸 감지 구현
  2. 전멸 시 OnWaveCleared 이벤트 발행 → 마지막 웨이브면 GameResultManager.ShowGameClear() 호출
  3. 웨이브 간 5초 대기 + SharedHUD에 '다음 웨이브 카운트다운' 표시
  4. WaveData[] 배열을 ScriptableObject로 관리하여 레벨 디자인 분리
2. EnemyTank 클래스 분리 리팩토링
736줄의 God Class가 인식·전투·이동·피해 판정을 모두 담당. 버그 수정과 밸런싱이 극도로 어렵고, 다른 팀원이 이해하기 힘듦. 향후 AI 변종(중전차, 자주포 등) 추가 시 확장 불가능한 구조.
  1. AwarenessSystem 컴포넌트 분리 (UpdateAwareness, CalculateExposure, IsPlayerVisible)
  2. EnemyCombatBehavior 분리 (AimAndFire, RotateTurretToTarget, Fire, TryCalculateLaunchDir)
  3. EnemyPartDamageReceiver 분리 (TakeDamage, CalculateDamage, ApplyPartDamage)
  4. EnemyTank은 상태머신 + 컴포넌트 조합만 담당하도록 경량화
3. 오디오·VFX 에셋 최소 구현
VR 탱크 시뮬레이션에서 소리와 시각 효과는 몰입감의 핵심. 현재 오디오 0개, VFX 1개, 머즐 플래시 주석 처리 상태. 발표/시연 시 '미완성' 인상을 강하게 줌.
  1. 주포 발사음, 착탄 폭발음, 엔진 루프음 3개 SFX 우선 추가
  2. FireSystem의 _muzzleFlashVFX 주석 해제 + 간단한 파티클 프리팹 연결
  3. EnemyDeathHandler의 사망 VFX에 폭발 파티클 + 사운드 추가
  4. SplashDamageUtils.Apply() 착탄 지점에 임팩트 VFX 스폰 로직 추가

코드 리뷰

EnemyTank.cs개선필요
736줄로 프로젝트 최대 파일. 6단계 상태머신(Idle/Alert/Combat/Tracking/Repositioning/Immobilized), 인식 시스템(0-100 awareness + 3단계 증가 속도 + 감쇠), 5점 노출도 계산(CalculateExposure), 탄도 조준(TryCalculateLaunchDir 이차방정식), 부위별 장갑 판정까지 고도화된 AI. 잘 만들었지만 단일 책임 원칙 위반이 심각 — AwarenessSystem, CombatBehavior, PartDamageReceiver로 분리 필요. IsPlayerVisible()이 매 프레임 5회 레이캐스트를 수행해 성능 부담. TakeDamage()에서 TankStatus와 동일한 장갑 판정 로직이 중복됨.
TankStatus.cs양호
339줄의 부위별 HP 시스템이 잘 구현됨. ArmorProfile(Front/Side/Rear mm)로 방향별 장갑, CalculateFinalDamage()에서 관통 판정 + 도탄 처리, ApplySpilloverDamage()로 Track→Hull(20%), Barrel→Turret(50%) 연쇄 피해 구현. NetworkVariable<PartHPState>로 6부위 HP 동기화. 다만 PartHPState.Equals()에서 float == 비교는 Mathf.Approximately로 변경 권장. ApplyDamageEditor()가 #if UNITY_EDITOR 안에만 있어 빌드 테스트 시 사용 불가.
RoleSync.cs양호
379줄의 4좌석 역할 관리 시스템. NetworkVariable 4개로 각 좌석의 clientId를 동기화하고, RequestBoardServerRpc에서 빈 좌석 검증→이전 좌석 이탈→새 좌석 탑승→소유권 이전까지 원자적 처리. OnStationOccupancyChanged/OnLocalRoleChanged 이벤트로 UI·HUD·아바타가 반응. RequireOwnership=false 적용으로 비소유자 RPC 문제 해결 (최근 diff에서 수정). 인라인 문서화도 우수.
SharedHUDController.cs양호
493줄이지만 3개 서브시스템(피해 상태, 미니맵, 좌석 표시)을 통합한 공용 HUD. Gradient 기반 HP 색상(적→황→녹), 태그 기반 적 스캔(1초 간격), 월드→UI 좌표 변환 미니맵이 잘 구현됨. ScanByTag()에서 태그 미등록 예외를 catch로 방어한 것은 실용적. EnemyMarker struct가 readonly로 불변성 보장. 다만 RebuildMarkers()가 매 1초 FindGameObjectsWithTag를 4회 호출해 적이 많을 때 GC 압박 가능.
AimController.cs양호
387줄의 포수 입력 시스템. XR 그립 감지→포탑 yaw/pitch 회전→NetworkVariable 동기화→발사 이벤트 파이프라인 완성. Locomotion 잠금(그립 시 TurnProvider 비활성화)으로 VR 멀미 방지. 에디터 디버그(키보드 조작)도 구현. 최근 diff에서 _debugBallisticCalc 연동 추가로 에디터 테스트 개선. TryFire()의 공백 제거와 같은 코드 정리도 보임.
BallisticCalc.cs양호
189줄의 탄도 계산기. SimulateTrajectory()가 Linecast 기반 충돌 탐지로 착탄점 계산, CalculateTimeOfFlight()로 비행시간 산출. AmmoData의 MuzzleVelocity를 사용한 현실적 물리 시뮬레이션. 최근 diff에서 _currentAmmo == null 시 에디터 디버그 로그 추가로 디버깅 편의성 개선. maxSteps(50) 내 착탄 없는 경우의 로깅도 추가됨.
GameManager.cs개선필요
240줄의 게임 상태머신. Workflow() 코루틴으로 Authenticating→Lobby→InGame 전환 관리, NGO SceneManager로 네트워크 동기 씬 로딩. 구조는 좋으나 MockGameManager와 90% 코드 중복이 문제 — 공통 베이스 클래스로 추출 필요. 씬 로드 실패 핸들링 없고, RestartRoutine에 타임아웃 없는 WaitUntil 사용. WaveManager와의 승리/패배 조건 연결이 코드에서 보이지 않음.
WaveManager.cs개선필요
196줄의 웨이브 스포너. Fisher-Yates 셔플로 스폰포인트 분산, WaveData SO로 보병/경차량/전차 수량 설정. SpawnEnemy()에서 NavMesh 스냅 + 네트워크 스폰 + SetTarget 호출까지 완성. 그러나 OnWaveStarted 이벤트만 있고 웨이브 종료 감지(적 전멸 체크)·승리 조건·다음 웨이브 자동 진행 로직이 없음. _playerTank null 체크 없이 사용(183줄). StartNextWave()를 외부에서 수동 호출해야 하는 구조.

아키텍처 평가

이벤트 기반 Observer 패턴이 핵심 아키텍처로, RoleSync.OnLocalRoleChanged/OnStationOccupancyChanged를 통해 HUD·StationMapUI·XRINetworkPlayer가 느슨하게 결합됨.
