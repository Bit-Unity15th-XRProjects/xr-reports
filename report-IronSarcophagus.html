프로젝트 현황 요약

IronSarcophagus 팀은 Netcode for GameObjects 기반의 VR 멀티플레이 탱크 시뮬레이터를 개발하고 있으며, 현재 핵심 전투 시스템(포물선 탄도 계산, 방향별 장갑 관통, 부위별 HP/전이 데미지, 6-상태 적 AI)과 멀티플레이 인프라(역할 기반 좌석 시스템, XR 입력 동기화)가 실제 동작하는 수준으로 구현되어 있다. SharedHUDController의 미니맵·손상 표시·좌석 아이콘 등 UI도 상당히 완성되었고, World Space HUD 플로팅(HUDFollower)과 AlwaysOnTop 렌더링까지 VR 특화 문제를 해결했다. 그러나 오디오 에셋이 전무하고, 적 사망 후 NetworkObject Despawn이 누락되어 있으며, WaveManager의 승리 조건이 GameManager와 연결되지 않아 게임 루프가 완결되지 않은 상태다. 가장 시급한 것은 적 사망 처리 파이프라인 완성(Despawn + 승리 판정 연결)과 RoleSync의 이벤트 구독 해제 버그 수정이며, 그 다음으로 발사/피격 SFX를 추가하여 VR 몰입감을 확보해야 한다.
!

리스크 알림

적 사망 후 NetworkObject 미정리로 메모리/네트워크 누수
EnemyBase.Die()가 agent 정지만 하고 Despawn/Destroy를 호출하지 않으며, WaveManager도 사망한 적의 정리를 담당하지 않음. 웨이브가 진행될수록 비활성 NetworkObject가 누적되어 네트워크 대역폭과 메모리를 소비하고, 장시간 플레이 시 성능 저하 발생.
  • EnemyBase.Die()에서 서버인 경우 NetworkObject.Despawn(destroy: true) 호출 추가
  • WaveManager.HandleEnemyDied()에서 OnDied 이벤트 구독 해제
  • 사망 연출(폭발 VFX)이 필요하면 Despawn 전 지연 코루틴 추가
RoleSync 이벤트 해제 실패로 인한 멀티플레이 버그
OnNetworkDespawn의 람다 해제가 실제로 동작하지 않아, 재접속이나 씬 재로드 시 이전 핸들러가 남아 HandleOccupancyChanged가 중복 호출됨. 좌석 UI가 잘못 표시되거나 역할이 꼬이는 현상 발생 가능.
  • OnValueChanged 구독용 메서드를 필드에 저장하고 동일 인스턴스로 해제
  • Action<ulong, ulong> 타입 메서드 4개를 OnNetworkSpawn에서 캐싱
오디오/애니메이션 에셋 전무 — 플레이 경험 미완성
SFX/BGM 폴더가 비어있고, 애니메이션 파일 0개. VR 게임에서 발사/피격/이동 사운드와 시각 피드백 없이는 몰입감과 공간감이 크게 떨어짐. 현재 _muzzleFlashVFX도 주석 처리 상태.
  • 발사 SFX + 피격 SFX를 최우선으로 추가 (무료 에셋 활용 가능)
  • FireSystem.OnImpact 이벤트에 AudioSource.PlayOneShot 구독
  • 엔진 루프 사운드를 TankDriver에 연결
GameManager↔WaveManager 승리 조건 미연결
WaveManager.OnAllWavesCleared 이벤트가 정의되어 있지만 GameManager가 이를 구독하지 않음. 모든 웨이브를 클리어해도 게임이 계속 InGamePlaying 상태로 유지되어 승리 화면이 표시되지 않음.
  • GameManager 또는 GameResultManager에서 WaveManager.OnAllWavesCleared 구독
  • 승리 시 GameResultUI 활성화 → RestartGame() 또는 ReturnToTitle() 연결
커밋 크기 불균형으로 코드 리뷰 및 롤백 어려움
평균 커밋 669줄, 최대 4867줄(GunnerHUD). 대형 커밋은 문제 발생 시 원인 추적과 부분 롤백이 어려움. 특히 프리팹 포함 커밋은 바이너리 변경이 대부분이라 실질 코드 변경량 파악 불가.
  • 커밋 당 하나의 논리적 변경만 포함하도록 팀 컨벤션 수립
  • 프리팹/에셋 변경과 스크립트 변경을 별도 커밋으로 분리

권고사항 (Top 3)

1. 적 사망 처리 파이프라인 완성
현재 사망한 적이 씬에 영구 잔류하여 네트워크 리소스를 소비하고, 게임 승리 조건 판정도 불완전함. 이 두 가지를 함께 해결하면 게임 루프가 완성됨.
  1. EnemyBase.Die()에 사망 VFX 재생 후 Despawn 코루틴 추가
  2. WaveManager에서 OnDied 구독 해제 + _aliveCount 로그 추가
  3. GameResultManager에서 OnAllWavesCleared 구독하여 승리 UI 표시
  4. TankStatus.OnDestroyed에서 패배 UI 표시
2. RoleSync 이벤트 구독을 메서드 참조로 리팩토링
현재 람다 기반 구독/해제는 NGO의 NetworkVariable 이벤트에서 흔한 실수로, 재접속 시 역할 동기화 버그를 유발. 수정이 간단하고 효과가 큼.
  1. Action<ulong, ulong> _onGunnerChanged, _onDriverChanged 등 4개 필드 선언
  2. OnNetworkSpawn에서 _onGunnerChanged = (_, v) => HandleOccupancyChanged(StationRole.Gunner, v) 할당 후 구독
  3. OnNetworkDespawn에서 동일 필드로 해제
3. 핵심 SFX 3종 우선 추가 (발사/피격/엔진)
VR 게임에서 청각 피드백은 공간 인지와 몰입에 필수. 3종만 추가해도 플레이 경험이 크게 향상됨.
  1. AudioSource를 FireSystem(발사), TankStatus(피격), TankDriver(엔진)에 추가
  2. FireSystem: HandleFireRequested()에서 PlayOneShot(fireSFX)
  3. TankStatus: OnPartHPChanged 콜백에서 피격 SFX 재생
  4. TankDriver: Update()에서 속도 비례 엔진 루프 볼륨 조절

코드 리뷰

TankStatus.cs양호
부위별 HP, 방향별 장갑 관통 판정(ArmorProfile.GetThickness), 전이 데미지(Track→Hull, Barrel→Turret, Turret→Hull+Barrel)가 깔끔하게 구현됨. PartHPState가 INetworkSerializable을 구현하여 NetworkVariable로 전체 상태를 원자적으로 동기화하는 설계가 우수. CriticalParts 상수로 파괴 조건을 명시한 점도 좋음. ApplySpillover에서 연쇄 파괴 시 OnDestroyed를 호출하는 로직도 정확. 단, RepairPartServerRpc에서 수리량(_repairAmount)이 고정값이라 밸런싱 유연성이 부족.
AimController.cs개선필요
XR 입력→NetworkVariable→Transform 동기화 파이프라인이 잘 구성됨. Locomotion Lock(Grab 시 Turn Provider 비활성화)으로 썸스틱 입력 충돌 해결한 설계가 좋음. 그러나 #if UNITY_EDITOR 블록이 Update() 내부에서 40줄 이상 차지하여 가독성 저하. 에디터 디버그 코드를 별도 partial class나 컴포넌트로 분리 권장. 또한 _netYaw.Value 직접 쓰기(269줄)가 에디터 블록에서 IsOwner 체크 없이 실행되어, NGO 실행 중 에디터에서 키보드 입력 시 예외 발생 가능.
RoleSync.cs개선필요
Ownership 이전 기반 역할 시스템이 잘 설계됨. NetworkVariable 4개로 좌석 점유를 동기화하고, HandleOccupancyChanged에서 로컬 역할과 UI를 자동 전환. 문서 주석(59줄 가이드)도 훌륭. 그러나 OnNetworkDespawn(180~183줄)에서 람다 '-='는 OnNetworkSpawn의 람다 '+='와 다른 인스턴스이므로 실제로 구독 해제되지 않음. 이는 메모리 누수 및 중복 호출 버그를 유발. 메서드 참조로 변경 필요.
EnemyTank.cs양호
736줄로 파일 크기가 크지만 6-상태 AI, 인지도 시스템(4단계 증가/감소 속도), FOV+노출도 계산, 포탑 독립 회전, 재배치 로직 등 탱크 AI에 필요한 요소가 모두 구현됨. EnemyPartHPState 구조체로 부위별 HP를 관리하고, CalculateDamage에서 플레이어 TankStatus와 동일한 장갑 판정 로직을 적용한 점이 일관적. 다만 50줄 권장에 크게 벗어나는 메서드(AILoop 등)가 있고, 300줄 권장을 2배 이상 초과하여 리팩토링 고려 필요.
FireSystem.cs양호
130줄의 간결한 구현으로 단일 책임 원칙을 잘 따름. HandleFireRequested→SpawnVisualProjectile→TryGetImpactData→ApplyDamageAfterFlight 흐름이 명확. SplashDamageUtils로 데미지 적용을 위임하여 결합도 낮음. OnImpact 이벤트로 VFX/SFX 확장 지점도 준비됨. _muzzleFlashVFX가 주석 처리된 것은 아직 VFX 에셋 미완성으로 보임.
GameManager.cs개선필요
Auth→Lobby→Game 전체 흐름을 Workflow() 코루틴 하나로 관리하는 구조는 읽기 쉬우나, while(true)+yield return null 폴링이 매 프레임 switch를 실행하여 비효율적. 씬 이름 하드코딩('01_Title', '02_Lobby', '03_Game')이 6곳에 반복됨 — const string으로 추출 필요. InLobby 상태에서 세션 끊김 시 NetworkManager.Singleton.Shutdown() 호출 후 yield return null 한 프레임만 대기하는데, Shutdown 완료 보장 안 됨. WaveManager.OnAllWavesCleared를 구독하지 않아 승리 조건 처리 경로 부재.
WaveManager.cs양호
Fisher-Yates 셔플, XZ 지터, NavMesh Warp 등 스폰 품질이 높음. 유닛 타입별 heightOffset으로 지면 박힘 방지한 세심함이 돋보임. HandleEnemyDied로 전멸 감지 후 자동 다음 웨이브 진행. 다만 SpawnEnemy에서 EnemyBase.OnDied 이벤트 구독 후 해제하지 않음 — 적 오브젝트 파괴 시 이벤트 참조가 남아 GC 방해 가능. 또한 모든 웨이브 클리어 후의 게임 흐름(OnAllWavesCleared 구독자)이 연결되어 있는지 확인 필요.
SharedHUDController.cs양호
미니맵, 부위별 손상 표시, 좌석 점유 아이콘 등 복합 UI를 하나의 컨트롤러에서 관리. PartDamageUI 배열로 Inspector에서 부위별 UI를 유연하게 설정 가능. Gradient로 HP 비율에 따른 색상 변환, BlinkRoutine으로 수리 인디케이터 점멸 등 세밀한 UX 구현. 다만 493줄로 300줄 권장을 크게 초과하며, 미니맵 로직과 좌석 로직을 별도 컴포넌트로 분리하면 유지보수성 향상.

아키텍처 평가

Observer 패턴을 핵심 통신 메커니즘으로 사용(OnPartHPChanged, OnFireRequested, OnDied, OnStationOccupancyChanged, OnLocalRoleChanged)하여 시스템 간 결합도를 효과적으로 낮춤. NetworkVariable+RPC 조합으로 상태 동기화(Yaw/Pitch/HP)와 명령 전파(Fire)를 명확히 분리한 점이 Netcode 모범 사례에 부합. EnemyTank(736줄)은 God Object 경향이 있어 인지(Perception), 전투(Combat), 이동(Movement) 모듈로 분리하면 테스트와 유지보수가 용이해질 것.

CRITICALRoleSync.csOnNetworkDespawn()에서 람다 함수로 이벤트 해제 시도(180~183줄). '+= (_, v) => ...'와 '-= (_, v) => ...'는 서로 다른 델리게이트 인스턴스이므로 실제 해제가 되지 않아 메모리 누수 및 이벤트 중복 호출 발생. HandleOccupancyChanged를 감싸는 메서드 4개를 필드에 저장하거나, 직접 메서드 참조 방식으로 변경 필요.
HIGHAimController.csUpdate() 내 에디터 전용 코드(245~278줄)에서 _netYaw.Value에 직접 쓰기를 수행하나, NGO가 실행 중일 때 IsOwner가 아닌 클라이언트에서도 이 코드가 실행될 수 있음. #if UNITY_EDITOR는 에디터 빌드 전체에서 활성화되므로, 멀티플레이 테스트 시 Non-Owner 클라이언트에서 NetworkVariable 쓰기 예외(InvalidOperationException) 발생 가능.
HIGHGameManager.csInLobby(108~120줄)와 InGamePlaying(158~169줄)에서 세션 끊김 감지 후 NetworkManager.Singleton.Shutdown() 호출 직후 yield return null 한 프레임만 대기. Shutdown은 비동기적이며 한 프레임에 완료되지 않을 수 있어, 후속 UnloadActive/LoadLocal에서 예기치 않은 상태 발생 가능.
HIGHWaveManager.csSpawnEnemy()에서 enemy.OnDied += HandleEnemyDied 구독 후 해제하지 않음(184줄). 적 사망 후에도 EnemyBase 오브젝트가 씬에 남아있으면(Die()에서 Destroy 미호출) 이벤트 참조가 유지되어 GC 방해. 또한 NetworkObject.Despawn() 호출이 없어 사망한 적이 네트워크에 계속 존재.
HIGHEnemyBase.csDie()에서 _agent.isStopped = true만 설정하고 GameObject 파괴나 NetworkObject.Despawn()을 호출하지 않음(101줄). 사망한 적이 씬과 네트워크에 계속 남아있게 됨.
MEDIUMEnemyTank.cs736줄로 CLAUDE.md의 '파일 300줄 이하 유지 권장' 규칙을 2배 이상 초과. 인지도 시스템, 전투 AI, 이동, 포탑 회전, 장갑/데미지 등이 모두 한 클래스에 집중되어 있어 유지보수 어려움. 최소한 인지도 시스템과 전투 로직을 별도 컴포넌트로 분리 권장.
MEDIUMSharedHUDController.csScanByTag() 메서드에서 catch { return; }로 태그 미등록 예외를 무시(빈 catch 블록). CLAUDE.md에서 '빈 catch 블록 금지'를 명시하고 있으며, 예외 종류를 특정하지 않아 다른 오류도 삼킬 수 있음.
