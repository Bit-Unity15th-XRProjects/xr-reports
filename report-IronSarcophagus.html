프로젝트 현황 요약

IronSarcophagus 팀은 VR 멀티플레이 탱크 시뮬레이션을 개발하고 있으며, 기대된 12개 핵심 시스템이 모두 실동작 로직으로 구현된 상태다. AimController의 클라이언트 측 예측, TankStatus의 방향별 장갑 관통 계산, EnemyTank의 6상태 AI + 인지도 시스템 등 학생 프로젝트 수준을 넘어서는 기술적 깊이를 보여준다. Nullma와 이재영 두 명이 최근 4일간 하루 5-8커밋씩 집중 작업하여 VFX, 사운드 시스템, 클라이언트 동기화 버그를 빠르게 수정하고 있다. 가장 시급한 문제는 GameManager의 씬 이름 불일치(11_Temp_LJY vs 03_Game)로 인한 재시작 실패 위험과, EnemyTank의 발사 코루틴 중복 실행 버그이며, 이 두 가지는 각각 상수 통일과 1줄 가드 추가로 즉시 해결 가능하다. 전체적으로 프로토타입 시연이 가능한 수준의 완성도이며, 남은 기간에 씬 정리와 멀티플레이 통합 테스트에 집중하면 충분히 발표 가능한 상태다.

씬 이름 하드코딩 불일치로 인한 재시작 실패
GameManager.cs line 136에서 '11_Temp_LJY'를 로드하지만 line 241에서 '03_Game'을 언로드하려 한다. 시연 중 재시작 기능을 사용하면 씬이 중첩되거나 언로드 실패로 게임이 멈출 수 있다.
  • GameManager.cs의 씬 이름을 상수 또는 enum으로 통일
  • 실제 사용할 게임 씬 이름(03_Game 또는 11_Temp_LJY)을 확정하고 모든 참조 수정
  • 시연 전 재시작 흐름을 최소 3회 테스트
EnemyTank AimAndFire 코루틴 중복 실행
적 전차가 매 AI 틱마다 새 AimAndFire 코루틴을 시작할 수 있어, 한 적이 동시에 여러 발을 발사하거나 _agent.updateRotation이 교차 설정되어 AI가 비정상 동작할 수 있다. 다수의 적 전차가 등장하는 후반 웨이브에서 특히 위험하다.
  • AimAndFire 시작 전 if (_isFiring) yield break; 또는 if (_isFiring) return; 가드 추가
  • 테스트: 적 3대 이상 동시 전투 상황에서 발사 빈도 확인
namwon-RT 14일 비활동

권고사항 (Top 3)

1. 씬 이름 상수화 및 재시작 흐름 검증
현재 GameManager에서 씬 이름이 2곳에서 서로 다르게 하드코딩되어 있어 시연 중 재시작 실패 위험이 높다. 가장 적은 작업으로 가장 큰 안정성 향상을 얻을 수 있다.
  1. GameManager.cs 상단에 const string GameSceneName = '03_Game'; 선언
  2. LoadLocal(), RestartRoutine() 등 모든 씬 참조를 이 상수로 교체
  3. Host에서 재시작 3회, Client에서 재접속 후 재시작 1회 테스트
2. EnemyTank 발사 코루틴 가드 추가
시연 중 적 전차가 비정상적으로 빠르게 연사하거나 회전이 꼬이면 프로토타입 인상이 크게 저하된다. 1줄 수정으로 해결 가능하다.
  1. EnemyTank.cs AILoop() Combat 분기에서 if (_isFiring) { yield return null; continue; } 추가
  2. 적 전차 3대 동시 교전 상황에서 발사 간격이 _fireCooldown(5초)을 준수하는지 확인
3. LobbiesUI 로비 목록 복원 또는 UI 정리
현재 방 목록이 주석 처리되어 있어 QuickJoin과 코드 입장만 가능하다. 시연 시 평가자가 방 목록을 기대할 수 있으므로, 복원이 어려우면 UI에서 해당 영역을 숨기는 것이 낫다.
  1. RefreshLobbies 복원이 가능하면 주석 해제 후 LobbiesSlotUI 프리팹 연결
  2. 불가능하면 _lobbiesContent 영역과 _refreshLobbies 버튼을 Inspector에서 비활성화
  3. 방 생성→코드 공유→코드 입장 흐름을 시연 시나리오로 확정

코드 리뷰

EnemyTank.cs개선필요
739줄로 프로젝트 권장 300줄의 2.5배를 초과한다. 6상태 FSM(Idle→Alert→Combat→Repositioning→Tracking→Immobilized)과 인지도 시스템(CalculateExposure의 5점 Linecast), 탄도 조준(TryCalculateLaunchDir의 이차방정식 풀이)은 학생 프로젝트로서 매우 인상적이다. 그러나 치명적 버그가 있다: AILoop()의 Combat 상태에서 CanShootTarget() 성공 시 _isFiring 체크 없이 StartCoroutine(AimAndFire())를 호출하여 동시에 여러 발사 코루틴이 실행될 수 있다. 또한 CalculateDamage()에서 HEAT 탄 유폭 확률(35%)과 APFSDS 유폭 확률(15%)이 구분되어 있어 탄종 시스템이 잘 설계되어 있다.
AimController.cs개선필요
502줄. 클라이언트 측 예측(localYaw/localPitch 즉시 반영 후 UpdateAimServerRpc로 서버 전송)이 잘 구현되어 있고, 최근 커밋에서 yawDelta/pitchDelta가 0일 때 RPC 스킵 최적화와 서버 보정 무시 로직(OnYawChanged에서 !IsServer && _isGrabbed 체크)이 추가되었다. 문제점: OnEnable()에서 InputAction.Enable() 없이 이벤트만 구독하고 OnLocomotionGrabbed()에서 Enable하는 구조인데, 그립 해제(OnLocomotionReleased) 시 재동기화(reconciliation) 포인트가 없어 누적 오차가 발생할 수 있다. OnFirePerformed에서 _isGrabbed만 체크하고 InputAction 활성화 상태를 검증하지 않는 점도 개선 필요.
TankStatus.cs양호
348줄. 방향별 장갑(ArmorProfile: 전방/측면/후방) 판정에 dot product를 사용하고, 관통 실패 시 도탄 데미지(RicochetDamageMult=5%), 관통 성공 시 cos 각도 보정까지 적용한 물리 기반 피해 계산이 인상적이다. 연쇄피해(ApplySpilloverDamage)로 파괴된 부위의 잔여 데미지가 인접 부위로 전파되며, Hull/AmmoRack 파괴 시 전차 파괴로 이어진다. OnNetworkSpawn()에서 InitialHPPenalty를 적용하나, 페널티가 maxPartHP 이상일 경우 음수 HP가 되는 클램핑 미비가 있다.
FireSystem.cs양호
159줄. 최근 diff에서 포구 화염 VFX 활성화, 포신 연기 VFX(루프+타이머 StopSmokeAfterDelay), OnImpact 시그니처를 Vector3→DamageInfo로 변경, Instigator 설정으로 자폭 방지까지 추가되어 완성도가 높아졌다. SpawnVisualProjectile()에서 ProjectilePoolManager.Instance?.Get() 호출이 null-conditional로 풀 미초기화 시 조용히 실패하지만, 데모에서는 시각 이슈일 뿐 게임 로직에 영향 없다.

RequestBoardServerRpc에서 InvokePermission=Everyone으로 모든 클라이언트가 호출 가능하되, 서버 측에서 중복 탑승/자리 차지 여부를 검증한다. NetworkManager.Singleton.LocalClientId 접근 시 null 체크 미비가 일부 있다.

XRSocketInteractor의 selectEntered/selectExited 이벤트로 VR 물리 장전을 구현하고, NetworkVariable<ulong>(_netLoadedPackId)로 AmmoPack 참조를 클라이언트에 동기화한다. 최근 커밋에서 추가된 이 NetworkVariable 덕분에 late-join 클라이언트도 현재 장전 상태를 올바르게 복원한다. SetLoadedServerRpc에서 중복 장전 방지(loaded && _netIsLoaded.Value), NotifyFired()에서 잔탄 소진 시 자동 언로드까지 잘 처리되어 있다.

심각도파일설명
CRITICALEnemyTank.csAILoop()의 Combat 상태에서 _isFiring 플래그를 체크하지 않고 StartCoroutine(AimAndFire())를 호출하여, AI 루프 틱마다 발사 코루틴이 중복 생성될 수 있다. AimAndFire 내부에서 _isFiring=true를 설정하지만, 다음 yield return null 전에 이미 새 코루틴이 시작될 수 있다.
HIGHBallisticCalc.csGetImpactPoint()(line 42)에서 SimulateTrajectory가 반환한 배열의 points[^1]을 인덱싱하는데, _currentAmmo가 null이면 Array.Empty<Vector3>()가 반환되어 IndexOutOfRangeException이 발생한다. Length==0 체크 후 null을 반환하지만, 호출부에서 null 체크를 하지 않을 경우 NullReferenceException으로 이어진다.
HIGHGameManager.csRestartRoutine()(line 241)에서 SceneManager.GetSceneByName('03_Game')을 사용하지만, 실제 로드되는 씬은 '11_Temp_LJY'(line 136)이다. 씬 이름 불일치로 재시작 시 기존 씬이 언로드되지 않을 수 있다.
MEDIUMTankStatus.csOnNetworkSpawn()(line 85)에서 _maxPartHP - GameConst.Tank.InitialHPPenalty(20)로 초기 HP를 설정하는데, _maxPartHP가 20 이하로 Inspector에서 설정되면 HP가 0 이하가 되어 즉시 파괴 상태가 된다. Mathf.Max(1f, ...) 클램핑이 필요하다.
MEDIUMTankDriver.csOnLocalDriveStarted/OnLocalDriveStopped가 static event인데, OnNetworkDespawn()에서 이 이벤트의 구독자(TankSoundController 등)가 해제되지 않을 경우 씬 리로드 시 파괴된 객체 참조로 NullReferenceException이 발생할 수 있다.
MEDIUMAimController.csOnYawChanged()/OnPitchChanged()에서 '!IsServer && _isGrabbed' 조건으로 서버 보정을 무시하는데, 그립 해제 시점에 서버 NetworkVariable 값과 로컬 예측값 사이의 reconciliation이 없어 장시간 플레이 시 yaw/pitch가 점진적으로 불일치할 수 있다.
MEDIUMSharedHUDController.csUpdate()에서 매 프레임 UpdateMinimap()을 호출하고, RebuildMarkers()에서 FindGameObjectsWithTag를 4회(태그별) 실행한다. 적 수가 많아지면 GC 압력과 프레임 드롭이 발생할 수 있다.
MEDIUMLobbiesUI.csRefreshLobbies()가 전체 주석 처리되어 있어 로비 목록 표시 기능이 완전히 비활성화되어 있다. 방 생성/퀵조인/코드 입장은 동작하지만 기존 방 목록을 볼 수 없다.
LOWSplashDamageUtils.csApply()(line 27)에서 searchRadius = Mathf.Max(info.BlastRadius, ImpactDetectionRadius)를 사용하는데, BlastRadius가 0보다 크고 ImpactDetectionRadius(0.3m)보다 작은 경우 직격 탐색 범위가 BlastRadius보다 넓어져 의도치 않은 대상이 직격 판정될 수 있다. 단계별 반경을 분리해야 한다.
LOWRoleSync.csIsOccupiedByLocalPlayer()(line 243)와 GetSeatColor()에서 NetworkManager.Singleton.LocalClientId를 null 체크 없이 접근한다. NetworkManager 소멸 후 호출되면 NullReferenceException이 발생한다.
