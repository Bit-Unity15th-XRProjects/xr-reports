프로젝트 현황 요약

IronSarcophagus 팀은 Netcode for GameObjects 기반 4인 협동 VR 탱크 시뮬레이션을 개발 중이며, 포수(AimController+FireSystem+BallisticCalc), 조종수(TankDriver+CameraSwitcher), 장전수(AmmoPack+AmmoPackDispenser+BarrelLoader), 수리(TankRepairUI) 4역할 시스템과 방향별 장갑·관통·도탄·부위 전이 데미지(TankStatus)를 갖춘 핵심 전투 시스템이 실동작 수준으로 구현되어 있다. 적 AI도 보병(소총병/대전차 분기), 경차량(수송→하차→퇴각), 전차(6단계 상태머신+인식도 시스템) 3종이 완성되었고, WaveManager로 웨이브 진행과 게임 종료(GameResultManager)까지 연결되어 있다. 그러나 오디오 에셋이 0개, 애니메이션이 0개로 VR 체험의 멀티미디어 피드백이 전무하며, 이것이 현재 가장 치명적인 부족 요소이다. 또한 EnemyTank.cs(736줄), SharedHUDController.cs(493줄) 등 대형 파일의 리팩토링과, 5개 미병합 브랜치의 통합이 마감 전 반드시 처리되어야 할 과제이다. 팀의 주간 커밋량이 0→24→44→49로 급증하고 있어 개발 모멘텀은 양호하나, Nullma 1인에 대한 의존도(49%)가 높고 namwon-RT의 8일 비활동이 우려됨.

코드 리뷰

EnemyTank.cs개선필요
AILoop(), UpdateAwareness(), AimAndFire() 등 AI 상태머신과 인식도 시스템이 정교하게 구현되어 있고, 5점 노출도 체크(CalculateExposure)와 12방향 재배치(TryFindRepositionPoint) 등 전술적 AI가 인상적임. 그러나 736줄로 프로젝트 가이드라인(300줄)의 2.4배. 인식도 시스템(UpdateAwareness, UpdateStateFromAwareness, IsInFOV, IsPlayerVisible, CalculateExposure), 전투 시스템(AimAndFire, Fire, CanShootTarget), 이동 시스템(RotateBodyToTarget, RotateTurretToTarget, TryFindRepositionPoint), 피격 시스템(TakeDamage, CalculateDamage, ApplyPartDamage)을 각각 별도 컴포넌트로 분리해야 함.
TankStatus.cs양호
ArmorProfile의 GetThickness()로 hitNormal과 tankForward를 비교하여 전면/측면/후면 장갑을 판별하고, EffectivePenetrationMm으로 각도 기반 관통 판정을 수행하는 로직이 탱크 시뮬레이션에 적합함. ApplySpilloverDamage()로 궤도→차체, 포신→포탑 등 부위 전이 데미지 체인도 구현. NetworkVariable<PartHPState>로 6개 부위 HP를 단일 변수로 동기화하여 네트워크 효율도 고려함. RepairPartServerRpc()로 수리 기능까지 서버 권한으로 처리.
AimController.cs개선필요
_netYaw/_netPitch를 Owner 권한 NetworkVariable로, _netIsLoaded를 Server 권한으로 분리한 설계가 올바름. XRGrabInteractable의 selectEntered/selectExited로 조준 그립을 감지하고, ContinuousTurnProvider/SnapTurnProvider를 비활성화하여 VR 조작 충돌을 방지하는 UX 배려가 좋음. 그러나 387줄로 가이드라인 초과. ReadInput(), RotateTurret(), TryFire()의 입력 처리 로직과 OnLocomotionGrabbed/Released()의 XR 상호작용 로직, 에디터 디버그 로직을 분리할 필요 있음.
SharedHUDController.cs개선필요
미니맵(적 마커 태그 스캔, 차체/포탑 방향 아이콘), 부위별 HP 색상(Gradient 기반), 좌석 점유 표시(3색 체계)를 하나의 컴포넌트에서 관리. 493줄로 가이드라인의 1.6배. ScanByTag()의 catch { return; }은 프로젝트 금지 패턴(빈 catch 블록) 위반. 미니맵, 데미지 표시, 좌석 상태를 별도 컴포넌트로 분리하고, 태그 미등록은 Awake()에서 사전 검증하는 것이 바람직.
FireSystem.cs양호
AimController.OnFireRequested 이벤트를 구독하여 발사를 처리하고, BallisticCalc.TryGetImpactData()로 착탄점을 계산한 뒤 ProjectilePoolManager에서 시각 탄체를 스폰하는 흐름이 깔끔. ApplyDamageAfterFlight() 코루틴으로 비행시간만큼 지연 후 SplashDamageUtils.Apply()를 호출하여 시각-데미지 동기화를 구현. OnImpact 이벤트로 VFX/SFX 연동 포인트도 제공. 129줄로 적절한 크기.
WaveManager.cs양호
WaveData 구조체로 웨이브별 적 구성을 Inspector에서 설정 가능하게 하고, Fisher-Yates 셔플로 스폰 포인트를 무작위화하며, NavMesh.SamplePosition()으로 유효 위치 보정 후 Warp()하는 안정적인 스폰 로직. HandleEnemyDied()에서 _aliveCount를 감소시키고 0이면 다음 웨이브를 시작하는 흐름이 명확. 196줄로 적절한 크기.
RoleSync.cs양호
4개 역할을 각각 NetworkVariable<ulong>로 관리하고, RequestBoardServerRpc에서 좌석 빈 상태 검증 후 ChangeOwnership()으로 해당 스테이션의 NetworkObject 소유권을 이전하는 설계가 Netcode 패턴에 적합. OnStationOccupancyChanged/OnLocalRoleChanged 이벤트로 UI 갱신을 디커플링. TransferStationOwnership()에서 roleObjects 배열의 NetworkObject마다 소유권을 이전하여 Gunner의 AimController, Driver의 TankDriver 등이 IsOwner 체크를 통과할 수 있게 함.
GameManager.cs양호
GameState 열거형과 코루틴 기반 Workflow()로 None→Authenticating→Lobby→InGamePlaying 전체 흐름을 제어. Host는 NetworkManager.SceneManager.LoadScene()으로 씬을 로드하여 클라이언트 자동 동기화를 달성. 세션 연결 해제 감지(CurrentSession == null) 시 Title로 복귀하는 방어 로직도 포함. MockGameManger와의 코드 중복이 존재하지만 테스트용으로는 수용 가능.

아키텍처 평가

EnemyBase를 Template Method 패턴으로 설계하여 Infantry/LightVehicle/Tank가 OnServerSpawn()과 AILoop()를 오버라이드하는 상속 구조가 깔끔하며, IDamageable 인터페이스로 플레이어 탱크와 적 모두에 통일된 데미지 시스템을 적용. Observer 패턴(C# event)을 적극 활용하여 TankStatus↔HUD, AimController↔FireSystem, WaveManager↔GameResultManager 간 결합도를 낮춤. RoleSync의 Singleton+NetworkBehaviour 조합으로 4역할 Ownership 이전을 중앙 관리하는 설계는 Netcode 프로젝트에 적합. 다만 SessionBlackboard가 static 필드만 있는 전역 상태 홀더로 테스트 격리가 어렵고, GameManager와 MockGameManger의 상태머신 코드가 중복되어 있어 기반 클래스 추출이 필요함. 전체적으로 네트워크 권한 모델(Server Authority + Owner Write)이 일관되게 적용되어 있어 치팅 방지 기반이 마련됨.

심각도파일설명
HIGHEnemyTank.cs단일 파일 736줄로 AI 상태머신, 인식도 시스템, 전투 로직, 이동 로직, 장갑/피격 시스템을 모두 포함. 프로젝트 가이드라인(300줄 이하) 위반이며, 단위 테스트와 유지보수가 어려움.
HIGHEnemyLightVehicle.csUnloadInfantry()에서 스폰된 보병의 OnDied 이벤트를 WaveManager에 직접 연결하지 않고 자체 처리. WaveManager의 _aliveCount와 불일치할 가능성 있음(웨이브 진행 조건 오류).
MEDIUMSharedHUDController.csScanByTag()에서 catch { return; }으로 모든 예외를 무시하는 빈 catch 블록 사용. 태그 미등록 시 경고 로그 없이 마커 생성을 건너뛰므로 디버깅이 어려움. 프로젝트 CLAUDE.md의 '빈 catch 블록 금지' 규칙 위반.
MEDIUMSharedHUDController.cs493줄의 단일 MonoBehaviour에 미니맵 마커 관리, 부위별 HP 표시, 좌석 점유 표시, 수리 표시기 점멸 등 4가지 독립적 기능이 결합되어 있음. Single Responsibility 원칙 위반.
MEDIUMAmmoPack.cs / AmmoPackDispenser.cs두 파일 모두 namespace가 선언되지 않아 전역 네임스페이스에 위치. 프로젝트의 다른 모든 스크립트가 IronSarcophagus.* 네임스페이스를 사용하는 것과 불일치.
MEDIUMSharedHUDController.csUpdate()에서 매 프레임 UpdateMinimap()→PositionMarkers()를 호출하여 모든 적 마커의 월드→UI 좌표 변환을 수행. 적 수가 많아지면 프레임 드랍 가능성. RebuildMarkers()에서 FindGameObjectsWithTag()를 1초마다 호출하는 것도 GC 부담.
MEDIUMAimController.cs387줄로 VR 입력 처리, 네트워크 동기화, XR 로코모션 잠금, 에디터 디버그 등 4가지 관심사가 혼재. 가이드라인(300줄) 초과.
LOWMockGameManger.cs파일명에 오타 존재(Manger → Manager). 클래스명도 MockGameManger로 동일한 오타. 코드 검색 시 혼란 유발.
LOWEnemyTank.csEnemyPartHPState 구조체가 EnemyTank.cs 하단에 정의되어 있으나 INetworkSerializable을 구현하지 않음. 서버 전용이라 현재는 문제없지만, 클라이언트에 HP를 동기화하려 할 때 직렬화 오류 발생 가능.
LOWGameManager.csWorkflow() 코루틴 내에서 while(true) + yield return null로 매 프레임 상태를 체크. InGamePlaying 상태에서도 계속 폴링하여 불필요한 오버헤드 발생. WaitUntil이나 이벤트 기반으로 전환 권장.
