프로젝트 현황 요약

IronSarcophagus는 Netcode for GameObjects 기반 VR 멀티플레이 탱크 시뮬레이션으로, 4인 협동 크루(포수/조종수/장전수/수리) 역할 분담이 핵심입니다. 현재 개별 시스템의 완성도가 높습니다 — TankStatus의 6부위 장갑 관통 시스템, BallisticCalc의 포물선 탄도 시뮬레이션, EnemyTank의 인식도 기반 6상태 AI, RoleSync의 소유권 이전 역할 관리가 모두 실제 동작하는 로직으로 구현되어 있습니다. 그러나 이 시스템들을 하나의 게임으로 엮는 게임 루프(승리/패배 조건 → 결과 처리 → 복귀)가 완전히 빠져 있어, 현재 상태는 '잘 만든 부품들의 집합'에 가깝습니다. HUD 시스템이 feature 브랜치에만 존재하고, 오디오/VFX 에셋이 전무하며, 팀원 1명이 6일간 비활동인 점을 고려하면, 남은 기간에 '게임 루프 완성 → HUD 머지 → 최소 SFX 적용' 순서로 집중해야 플레이 가능한 빌드를 만들 수 있습니다.

권고사항 (Top 3)

1. 게임 루프 완성을 최우선으로
현재 핵심 메커닉(사격, 데미지, 적 AI)은 잘 구현되어 있지만, 이를 연결하는 게임 루프(시작→전투→승리/패배→결과)가 없어 하나의 '게임'으로 성립하지 않습니다. 남은 개발 기간을 고려하면 이것이 가장 높은 ROI를 가진 작업입니다.
  1. GameManager.Workflow()의 InGamePlaying 상태에서 WaveManager와 TankStatus 이벤트 구독
  2. 승리 조건: OnAllWavesCleared → 결과 UI 표시 → 로비 복귀
  3. 패배 조건: OnDestroyed → 결과 UI 표시 → 로비 복귀
  4. 간단한 WorldSpace Canvas로 결과 패널 구현 (1일 이내 가능)
2. feature 브랜치 즉시 머지 및 정리
7개 미머지 브랜치 중 SharedHUD와 GunnerHUD는 HUD 시스템으로 플레이에 필수적이며, 뒤처진 커밋 수(behind 4)가 늘어날수록 충돌이 심해집니다. feat/EnemyAI는 7커밋 뒤처져 있고, feature/jwt_0000은 43커밋 뒤처져 사실상 사용 불가합니다.
  1. feat/SharedHUD를 develop에 머지 (최신 HUD 코드 확보)
  2. feat/GunnerHUD 머지 또는 SharedHUD에 통합
  3. feat/EnemyAI 내용 확인 후 머지 또는 폐기 결정
  4. feature/jwt_0000 브랜치 폐기 (43커밋 behind로 리베이스 비용 > 재작성)
3. RoleSync 이벤트 해제 버그 수정
핵심 네트워크 컴포넌트의 메모리 누수 버그로, 세션이 길어질수록 성능과 안정성이 저하됩니다. 멀티플레이 게임에서 이런 류의 버그는 디버깅이 매우 어려우므로 조기에 수정해야 합니다.
  1. OnNetworkSpawn()의 람다를 private void 메서드로 교체 (예: OnGunnerChanged, OnDriverChanged 등)
  2. OnNetworkDespawn()에서 동일 메서드 참조로 -= 해제
  3. 역할 전환 10회 이상 반복 테스트로 이벤트 중복 호출 없음 확인

코드 리뷰

EnemyTank.cs개선필요
736줄로 프로젝트 가이드(300줄)의 2.4배입니다. 6상태 AI 상태머신, 인식도 시스템, 5점 노출도 계산, 포탑/차체 회전, 장갑 관통 계산, 파트별 HP 관리가 모두 한 클래스에 있습니다. UpdateAwareness(), UpdateStateFromAwareness(), AimAndFire() 등 핵심 메서드의 로직은 정교하지만, 인식도 계산을 별도 AwarenessSystem으로, 장갑/HP를 EnemyArmorSystem으로 분리하면 유지보수성이 크게 개선됩니다. TryCalculateLaunchDir()의 포물선 발사각 계산은 수학적으로 올바르게 구현되어 있습니다.
RoleSync.cs심각
OnNetworkDespawn()에서 이벤트 해제 시 람다를 새로 생성하고 있습니다(예: `_netGunnerClientId.OnValueChanged -= (_, v) => HandleOccupancyChanged(...)`). 이는 구독 시 생성된 람다와 다른 인스턴스이므로 실제로 해제되지 않아 메모리 누수와 중복 호출이 발생합니다. 핵심 아키텍처 컴포넌트인 만큼 반드시 수정해야 합니다. 그 외 RequestBoard/RequestLeave의 서버 검증, 소유권 이전 로직은 견고합니다.
AimController.cs양호
NetworkVariable<float> Owner 쓰기 패턴으로 포탑 Yaw/Pitch를 동기화하는 것이 NGO 베스트 프랙티스에 부합합니다. XR Grab → Turn Provider 비활성화 → 썸스틱 조준 독점 전환 흐름이 깔끔합니다. OnLocomotionGrabbed()에서 XROrigin을 캐싱하고, Released 시 캐시된 origin을 우선 사용하는 방어 로직도 좋습니다. #if UNITY_EDITOR 키보드 입력 지원으로 VR 기기 없이 테스트 가능합니다.
TankStatus.cs양호
PartHPState가 INetworkSerializable을 구현하여 6개 파트 HP를 단일 NetworkVariable로 동기화하는 설계가 효율적입니다. CalculateFinalDamage()에서 방향 기반 장갑(Front/Side/Rear) 판정 + cos(입사각) 관통력 감쇠 + 도탄 대미지(5%) 계산이 물리적으로 타당합니다. OnPartHPChanged 이벤트로 UI 갱신을 분리한 점도 좋습니다.
FireSystem.cs양호
119줄의 간결한 클래스로 발사 파이프라인을 명확히 구현했습니다. HandleFireRequested() → SpawnVisualProjectile() + TryGetImpactData() → ApplyDamageAfterFlight(TOF 지연) 흐름이 직관적입니다. BallisticCalc와의 분리가 잘 되어 있어 탄도 계산과 데미지 적용이 독립적입니다.
WaveManager.cs개선필요
Fisher-Yates 셔플 스폰 포인트 선택과 NavMesh 스내핑이 잘 구현되어 있습니다. 그러나 OnAllWavesCleared 이벤트가 선언만 되어 있고 GameManager와 연결되지 않아 모든 웨이브 클리어 후 게임이 종료되지 않습니다. WaveData에 난이도 스케일링(적 HP 배율, 수량 증가 등)이 없어 후반 웨이브 밸런싱이 Inspector 수동 설정에 의존합니다.
LobbiesUI.cs개선필요
세션 생성/참가/빠른 참가 기능이 모두 구현되어 있고, Relay 네트워크 설정도 포함되어 있습니다. 그러나 네임스페이스가 'IronSarcophagus.Netowork'로 오타가 있으며(AuthUI도 동일), 이는 RoleSync의 'IronSarcophagus.Network'와 불일치합니다. 또한 세션 쿼리 실패 시 사용자에게 피드백이 Debug.LogError뿐이라 UI 레벨 에러 표시가 필요합니다.
BarrelLoader.cs양호
XRSocketInteractor의 selectEntered/selectExited 이벤트를 활용한 장전 메커닉이 VR 인터랙션으로서 자연스럽습니다. RoleSync.StationRole.Loader 검증으로 역할 권한을 확인하고, SetLoadedServerRpc()로 서버 권한 하에 장전 상태를 동기화합니다. AimController와 BallisticCalc에 탄약 데이터를 전파하는 연동도 완성되어 있습니다.

아키텍처 평가

서버 권한 모델이 모든 게임플레이 시스템에 일관되게 적용된 점이 가장 큰 강점입니다. NetworkVariable(지속 상태)과 RPC(일회성 명령)의 용도 구분이 명확하고, RoleSync를 중심으로 한 역할 기반 접근 제어가 AimController, TankDriver, BarrelLoader에 체계적으로 적용됩니다. 단계별 진행률 (AI 평가)

기반 구축 4/5점
Unity Services 인증(AuthUI), 세션/로비 관리(LobbiesUI), 씬 전환 상태머신(GameManager)이 모두 구현되어 있고 실제 동작하는 로직입니다. GameManager는 10단계 GameState enum으로 Title→Lobby→Game 워크플로우를 관리하며, DontDestroyOnLoad로 씬 간 지속됩니다. SessionBlackboard를 통한 세션 참조도 구현되어 있습니다. 다만 AuthUI/LobbiesUI의 네임스페이스에 'Netowork' 오타가 있고, 에러 복구 로직이 로그 출력에 그치는 부분이 아쉽습니다.
핵심 시스템 18/25점
핵심 시스템 대부분이 실제 동작하는 로직으로 구현되어 있습니다. (1) TankStatus: 6부위(Hull/Turret/Engine/Barrel/AmmoRack/Track) 파트별 HP + 방향 기반 장갑 관통 계산(Front/Side/Rear ArmorProfile), INetworkSerializable로 동기화. (2) AimController: NetworkVariable<float>로 포탑 Yaw/Pitch 동기화, XR 입력 + Grab 기반 조준 독점. (3) BallisticCalc: 포물선 탄도 시뮬레이션(0.05s 스텝, Linecast 충돌), TOF 계산. (4) FireSystem: 발사→시각 투사체→TOF 후 SplashDamageUtils.Apply() 데미지 적용 파이프라인. (5) EnemyTank: 6상태 AI(Idle/Alert/Combat/Repositioning/Tracking/Immobilized) + 인식도 시스템(0-100, 노출도 5점 레이캐스트). (6) WaveManager: 웨이브별 적 스폰 + Fisher-Yates 셔플 + NavMesh 스내핑. (7) RoleSync: 4역할(Gunner/Driver/Loader/Repair) 탑승/하차 + NetworkObject 소유권 이전. HUD 시스템이 feature 브랜치에만 존재하고 미병합 상태라 감점.

CRITICALRoleSync.csOnNetworkDespawn()에서 람다 `(_, v) => HandleOccupancyChanged(...)` 로 이벤트를 해제하지만, 이는 OnNetworkSpawn()에서 등록한 람다와 다른 인스턴스입니다. C# 람다는 참조 동일성이 보장되지 않으므로 -= 연산이 실제로 해제되지 않아 메모리 누수와 이벤트 중복 호출이 발생합니다.
HIGHAuthUI.cs, LobbiesUI.cs네임스페이스가 'IronSarcophagus.Netowork'로 'Network'의 오타입니다. RoleSync 등 다른 네트워크 코드는 'IronSarcophagus.Network'를 사용하므로 using 불일치가 발생하며, 코드베이스 전반의 일관성을 해칩니다.
HIGHWaveManager.csOnAllWavesCleared 이벤트가 선언되어 HandleEnemyDied()에서 호출되지만, GameManager에서 구독하는 코드가 없습니다. 모든 웨이브를 클리어해도 게임 상태가 변하지 않아 승리 조건이 작동하지 않습니다.
HIGHGameManager.csTankStatus.OnDestroyed 이벤트를 구독하는 코드가 GameManager에 없습니다. 플레이어 탱크가 파괴되어도 패배 처리가 되지 않아 게임이 무한히 계속됩니다.
MEDIUMEnemyTank.cs736줄로 단일 책임 원칙을 위반합니다. AI 상태머신, 인식도 시스템, 장갑/HP 관리, 포탑 회전, 탄도 계산이 한 클래스에 혼재되어 있어 수정 시 영향 범위 파악이 어렵습니다.
MEDIUMTankDriver.csSetInputRpc()에서 매 프레임 RPC를 전송합니다. 입력 변화가 없을 때도 전송하면 불필요한 네트워크 트래픽이 발생합니다. 이전 입력과 비교하여 변경 시에만 전송하는 최적화가 필요합니다.
MEDIUMEnemyTank.csFire()에서 TryCalculateLaunchDir() 실패 시(out 파라미터 false) 대체 로직 없이 return되어, 특정 거리/각도에서 적 전차가 영구적으로 발사하지 못하는 상황이 발생할 수 있습니다.
LOWRoleSync.csSingleton 패턴(Instance = this)으로 구현되어 있어 멀티 탱크(다수 팀) 확장 시 구조적 제약이 됩니다. 현재 단일 탱크 설계에서는 문제없으나 장기적 확장성이 제한됩니다.
LOWStationMapUI.csTeleportToSeat()에서 XROrigin.FindAnyObjectByType<XROrigin>()으로 매번 검색합니다. 빈번한 호출 시 성능 이슈가 있을 수 있으며, Awake()에서 캐싱하는 것이 바람직합니다.
LOWEnemyBase.csStart()에서 #if UNITY_EDITOR 분기로 IsSpawned가 아닐 때 AI를 시작하는데, 빌드에서는 이 경로가 없어 NetworkSpawn 이전에 Start()가 호출되면 AI가 시작되지 않습니다. 에디터 전용이므로 실제 빌드에는 영향 없지만 테스트 모드 의존도가 높습니다.
