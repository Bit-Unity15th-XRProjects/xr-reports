프로젝트 현황 요약

IronSarcophagus 팀은 Netcode for GameObjects와 XR Interaction Toolkit 기반의 VR 멀티플레이 탱크 시뮬레이션을 만들고 있으며, 현재 기대된 12개 시스템을 모두 구현한 상태이다. 포수(AimController·FireSystem·BallisticCalc)·운전수(TankDriver)·장전수(BarrelLoader)·정비병 4역할 체계가 RoleSync로 동기화되고, EnemyTank의 6상태 AI와 WaveManager의 웨이브 스폰, TankStatus의 부위별 장갑·관통 시스템까지 핵심 게임플레이가 작동하는 프로토타입 수준에 도달했다. 최근 커밋들이 VFX(포구 연기·폭발 착탄), 사운드(공간 오디오 2D/3D 전환), 버그 수정(AimController 클라이언트 예측, BarrelLoader 네트워크 동기화)에 집중되어 있어 폴리싱 단계에 진입한 것으로 보인다. 가장 시급한 것은 WaveManager의 적 카운트 불일치 버그 수정과 탱크 파괴 후 게임 종료 흐름 연결이며, namwon-RT 팀원의 14일 비활성화 상태도 마감 전 해결이 필요하다.

리스크 알림

  • SpawnEnemy()에서 EnemyBase가 없으면 오브젝트를 Destroy하고 return하도록 수정
  • NavMesh.SamplePosition 실패 시 에이전트를 비활성화하거나 오브젝트를 파괴하는 폴백 추가
  • 프리팹 배열 null 체크를 SpawnWave() 시작부에 추가
탄도 계산 이중 구현으로 인한 시각-데미지 불일치

  • SimulateTrajectory()와 TryGetImpactData()를 하나의 공통 궤적 시뮬레이션 메서드로 통합
  • 시각 투사체의 최종 위치와 TryGetImpactData의 impactPoint가 동일한 값을 사용하도록 보장
namwon-RT 팀원 14일 비활성 상태

  • namwon-RT에게 현재 작업 상태 확인 및 브랜치 최신화 요청

async void 안티패턴으로 인한 예외 미처리 크래시 위험
AuthUI.Authenticate()와 LobbiesUI의 3개 async void 메서드에서 Unity Services 호출 실패 시 예외가 전파되지 않아 앱이 예고 없이 크래시될 수 있다. 특히 네트워크 불안정 환경에서의 시연 시 위험하다.
  • async void를 async Task로 변경하고, 버튼 onClick에서는 UniTask.Forget() 또는 try-catch 래퍼를 사용

  • 각 MonoBehaviour의 Start() 또는 OnEnable()에서 필수 참조를 검증하고 경고 로그 출력
  • 특히 TankSoundController의 _interiorSnapshot/_exteriorSnapshot에 null 체크 추가

권고사항 (Top 3)

1. 시연 안정성을 위한 핵심 버그 수정 우선
WaveManager의 적 카운트 불일치와 FireSystem의 TryGetImpactData 실패 시 시각 투사체 잔류는 시연 중 게임 진행을 막거나 시각적 혼란을 줄 수 있는 가장 영향력 큰 버그이다.
  1. WaveManager.SpawnEnemy()에서 EnemyBase 미존재 시 Destroy(go) 후 return하도록 수정
  2. FireSystem.HandleFireRequested()에서 SpawnVisualProjectile()을 TryGetImpactData() 성공 후로 이동
  3. 수정 후 3웨이브 이상 플레이 테스트로 적 카운트 정상 진행 확인
2. EnemyTank 클래스 분할로 유지보수성 확보
740줄의 EnemyTank가 AI 상태 머신, 인지도 시스템, 전투 로직, 이동, 데미지 처리를 모두 담당하고 있어 버그 수정과 기능 추가가 어렵다. 마감 전 추가 기능 구현 시 실수 위험이 높다.
  1. 인지도 시스템(UpdateAwareness, CalculateExposure, IsInFOV, IsPlayerVisible)을 EnemyPerception 컴포넌트로 분리
  2. 전투 로직(AimAndFire, Fire, TryCalculateLaunchDir)을 EnemyCombat 컴포넌트로 분리
  3. EnemyTank는 상태 머신과 컴포넌트 조율만 담당하도록 축소
3. 탱크 파괴 후 플레이어 경험 처리 추가
현재 TankStatus.OnDestroyed 이벤트는 존재하지만, 탱크 파괴 후 플레이어가 할 수 있는 것이 없다. 시연 시 탱크가 파괴되면 게임이 사실상 멈추는 것처럼 보일 수 있다.
  1. GameResultManager에서 OnDestroyed 구독 → 패배 UI 표시 연결 확인
  2. 최소한 '패배' 텍스트와 '재시작' 버튼이 표시되도록 GameResultUI 연동
  3. AnnihilationGameMode에서 플레이어 탱크 파괴 시 게임 종료 트리거 추가

코드 리뷰

AimController.cs양호
501줄의 핵심 포수 입력 시스템. XR 썸스틱 입력→클라이언트 예측→서버 RPC 동기화 파이프라인이 잘 구현되어 있다. OnYawChanged/OnPitchChanged에서 로컬 Gunner의 서버 보정을 스킵하는 클라이언트 예측 로직(`if (!IsServer && _isGrabbed) return`)이 최근 버그 수정으로 추가되었고, 불필요한 RPC 전송 방지(`yawDelta != 0f || pitchDelta != 0f`)도 적절하다. OnEnable에서 InputAction.Enable을 제거하고 OnLocomotionGrabbed에서만 활성화하도록 수정한 것은 스폰 전 오발사를 방지하는 좋은 개선이다. 다만 WaitForRoleSync() 코루틴에서 RoleSync.Instance가 중간에 생성될 경우 이벤트 이중 구독 위험이 있고, 줌 레벨 배열 상수와 감도 범위에 매직 넘버가 산재해 있다.
EnemyTank.cs양호
740줄의 복합 AI 시스템으로, Idle→Alert→Combat→Repositioning→Tracking→Immobilized 6상태 머신, 인지도(0-100) 기반 탐지 시스템(시야각·노출도·근접 감지 3단계), 12방향 재배치 탐색, 독립 포탑 회전을 모두 구현했다. CalculateExposure()의 5점 레이캐스트 차폐 판정, TryCalculateLaunchDir()의 포물선 발사 방향 계산 등 고급 AI 로직이 실제로 동작한다. 학생 프로젝트로서 매우 높은 수준이다. 문제점으로는 파일이 740줄로 과대하여 AI 상태 머신/전투/인지를 분리하는 것이 바람직하고, _target이 null이 될 경우 AILoop 내 여러 지점에서 NullReferenceException 위험이 있다. 또한 _coverMask가 Inspector에서 미설정 시 모든 레이캐스트가 실패하여 AI가 완전 노출로 판단하는 문제가 있다.
TankStatus.cs양호
347줄의 부위별 HP·장갑 시스템. CalculateFinalDamage()에서 Hull/Turret은 장갑 프로파일(전면/측면/후면) 기반 관통 판정을, 기타 부위는 코사인 각도 기반 감쇠를 적용한다. ApplySpilloverDamage()로 Track→Hull(20%), Barrel→Turret(50%), Turret→Hull(30%)+Barrel(40%) 연쇄 파괴를 구현한 것은 탱크 시뮬레이션으로서 사실적이다. NetworkVariable<PartHPState>와 NetworkVariable<TankPart>로 서버 권위 동기화가 잘 되어 있다. 스필오버 연쇄가 이론적으로 최대 3단계까지만 발생하지만 명시적 깊이 제한이 없는 점, ProjectileDir의 부호 방향이 직관적이지 않은 점이 개선 여지다.
RoleSync.cs양호
407줄의 4좌석 승하차 시스템. NetworkVariable 4개로 각 좌석 점유 상태를 동기화하고, RequestBoardServerRpc/RequestLeaveServerRpc로 서버 권위 탑승 처리를 구현했다. TransferStationOwnership()으로 NetworkObject 소유권을 이전하고, OnClientDisconnected()로 접속 해제 시 자동 하차를 처리한다. RequireOwnership을 InvokePermission으로 교체한 최근 수정이 반영되어 있다. 잠재적 문제로 StationConfig.roleObjects 배열 자체가 null일 경우 foreach에서 NullReferenceException이 발생할 수 있고, TransferStationOwnership()에서 stationNetworkObject가 null이면 소유권 이전을 건너뛰지만 NetworkVariable은 이미 업데이트된 상태라 불일치가 발생할 수 있다.
FireSystem.cs개선필요
158줄의 발사 처리 시스템. AimController.OnFireRequested 이벤트 구독→VFX 재생→시각 투사체 스폰→비행시간 후 SplashDamageUtils.Apply() 호출 파이프라인을 구현했다. 포신 연기 VFX의 연속 발사 시 타이머 리셋 로직이 잘 되어 있다. 그러나 HandleFireRequested()에서 _muzzleFlashVFX.Play()와 _muzzleSmokeVFX 시작이 _aimController.IsLoaded 검증보다 먼저 실행될 수 있어, 장전되지 않았는데 VFX가 재생되는 시각적 버그가 있다(실제로는 IsLoaded 체크가 VFX 코드보다 앞에 있지만, TryGetImpactData 실패 시 시각 투사체는 이미 스폰됨). 또한 _muzzleSmokeVFX != null 체크 후 .isPlaying 접근은 안전하지만, _ballisticCalc이나 _aimController가 null일 때의 방어 코드가 없다.
WaveManager.cs개선필요
213줄의 웨이브 스폰 시스템. Fisher-Yates 셔플로 스폰 포인트를 분산하고, NavMesh.SamplePosition으로 적을 네비메시에 배치한다. 그러나 SpawnEnemy()에서 EnemyBase 컴포넌트가 없는 프리팹을 스폰하면 _aliveCount를 감소시키지만 해당 유닛은 게임 월드에 남아 있어, 플레이어가 이를 파괴해도 카운트가 0에 도달하지 않아 다음 웨이브가 시작되지 않는 치명적 버그가 있다. NavMesh.SamplePosition 실패 시 에이전트가 네비메시 밖에 배치되어 AI가 동작하지 않는 문제도 있다. 프리팹 배열에 null이 포함되면 Instantiate에서 크래시가 발생한다.
SharedHUDController.cs양호
511줄의 공용 HUD 시스템. 미니맵(적 마커 스캔·태그 기반 분류·월드→UI 좌표 변환), 좌석 상태(점유/비점유/로컬 색상), 부위별 손상 표시(Gradient 기반 HP→색상 매핑·파괴 색상·수리 점멸)를 구현했다. RebuildMarkers()에서 태그별(Enemy_Infantry/LightVehicle/Tank/AntiTank) 적을 스캔하여 마커를 생성하고, PositionMarkers()에서 1초 간격으로 위치를 갱신한다. Reset()에서 Gradient 기본값을 자동 설정하는 것은 에디터 편의성 면에서 좋다. FindGameObjectsWithTag 예외를 catch로 처리한 것은 방어적이지만, 미등록 태그를 무시하면 디버깅이 어려울 수 있다.
BallisticCalc.cs개선필요
189줄의 탄도 계산 시스템. 포물선 궤적 시뮬레이션·Physics.Linecast 충돌 감지·비행시간 계산을 구현했다. 문제점으로 SimulateTrajectory()와 TryGetImpactData()가 동일한 물리를 다른 코드로 구현하여 착탄점이 미세하게 달라질 수 있다(시각 투사체와 데미지 적용 위치 불일치 위험). CalculateTimeOfFlight()에서 '목표까지 거리가 증가하면 break'하는 휴리스틱은 급각도 사격이나 장거리 곡사 시 부정확할 수 있다. muzzleDir가 정규화되지 않은 경우에 대한 방어 코드가 없다.

커밋 분석

총 168커밋, 주당 53커밋의 높은 개발 속도를 보이며, 최근 4주간 17→50→48→53으로 가속화되는 추세이다. 커밋 메시지 서술 비율이 97%로 매우 우수하고, 'feat(Sound): 사운드 시스템 개선 — 인테리어 공간 2D/3D 분리, 피격 진동 페이드'처럼 변경 내용과 이유를 함께 기술하는 패턴이 일관적이다. Nullma(74커밋)가 시스템·VFX·사운드·인프라를, 이재영(41커밋)이 프리팹·씬·네트워크 버그 수정을 담당하여 역할 분담이 명확하다. 다만 평균 커밋 크기 638줄(중앙값 136줄)로 일부 커밋이 4000줄 이상의 대형 커밋(GunnerHUD 4867줄, EnemyAI 4449줄)이 있어, 이런 커밋은 리뷰와 롤백이 어렵다. namwon-RT(23커밋)가 14일간 비활성으로 팀 기여도 불균형이 있다.

아키텍처 평가

서버 권위(Server-Authoritative) 모델을 일관되게 적용하여 NetworkVariable(읽기: Everyone, 쓰기: Server)과 ServerRpc로 게임 상태를 동기화한다. Observer 패턴을 C# event로 광범위하게 사용하여 시스템 간 결합도를 효과적으로 낮췄다(AimController→FireSystem→SplashDamageUtils→IDamageable 체인이 모두 이벤트/인터페이스로 연결). DamageInfo 구조체를 데미지 파이프라인의 단일 데이터 객체로 사용하여 발사→비행→착탄→스플래시→부위 판정까지 일관된 정보를 전달하는 설계가 잘 되어 있다. GameConst 정적 클래스로 물리·탄도·탱크·적 상수를 중앙 관리하고, AmmoData ScriptableObject로 탄종별 데이터를 분리한 것도 적절하다. 개선점으로는 GameManager의 씬 이름 하드코딩을 ScriptableObject나 enum으로 대체해야 하고, EnemyTank(740줄)은 단일 책임 원칙 위반으로 인지/전투/이동 컴포넌트 분리가 필요하며, Singleton 패턴(RoleSync, ProjectilePoolManager, SoundManager)의 과도한 사용은 테스트 용이성을 저하시킨다.

MEDIUMAimController.csWaitForRoleSync() 코루틴 실행 중 RoleSync.Instance가 생성되면 OnLocalRoleChanged 이벤트에 이중 구독이 발생하여 역할 변경 콜백이 두 번 호출될 수 있다.
MEDIUMAuthUI.csAuthenticate()가 async void로 선언되어 예외 발생 시 호출자가 캐치할 수 없고, 앱이 비정상 종료될 수 있다. LobbiesUI의 OnQuickJoinAsync, OnCodeJoinAsync, OnConfirmCreateRoom도 동일한 문제.
MEDIUMEnemyTank.csAILoop() 코루틴 실행 중 _target이 파괴되어 null이 되면, RotateTurretToTarget()·RotateBodyToTarget()·UpdateAwareness() 등의 메서드 시작부 null 체크는 있으나, 메서드 내부에서 _target 접근 사이에 파괴될 경우 NullReferenceException이 발생할 수 있다.
MEDIUMRoleSync.csSetRoleObjectsActive()에서 station.roleObjects 배열 자체가 null인 경우 foreach 순회 시 NullReferenceException이 발생한다. 개별 요소의 null 체크는 있지만 배열 자체의 null 체크가 없다.
MEDIUMTankSoundController.cs_interiorSnapshot과 _exteriorSnapshot이 Inspector에서 미설정 시 OnDriveStarted/OnDriveStopped에서 TransitionTo() 호출이 NullReferenceException을 발생시킨다.
LOWAnnihilationGameMode.csHandleAliveCountChanged()에서 NetworkVariable 업데이트와 UpdateEnemyCountRpc()를 모두 호출하여, 클라이언트가 OnValueChanged 콜백과 RPC를 둘 다 수신해 OnEnemyCountChanged 이벤트가 이중 발생할 수 있다.
