프로젝트 현황 요약

IronSarcophagus 팀은 Netcode for GameObjects 기반 VR 멀티플레이 탱크 시뮬레이션을 만들고 있으며, 현재 탱크 조작(포수/조종수/장전수), 적 AI(전차/보병/경장갑차), 웨이브 전투 시스템까지 핵심 게임플레이를 구현한 상태입니다. TankStatus의 부위별 장갑 관통 시스템, BallisticCalc의 포물선 탄도 계산, EnemyTank의 경계도 기반 AI 상태머신 등 시뮬레이션 깊이가 학생 프로젝트 수준을 상회합니다. 그러나 GunnerHUD가 미구현이라 포수가 조준 정보를 확인할 수 없고, 오디오/애니메이션 에셋이 전무하여 VR 몰입감이 크게 부족합니다. RoleSync의 이벤트 구독 해제 버그는 멀티플레이 안정성에 직접 영향을 미치므로 가장 시급하게 수정해야 하며, 이후 GunnerHUD 구현과 최소한의 오디오 통합이 데모 가능한 빌드를 위해 필수적입니다.
!

리스크 알림

RoleSync 이벤트 구독 해제 실패로 인한 메모리 누수
RoleSync는 싱글톤으로 전체 게임 세션 동안 유지되며, 씬 전환이나 재접속 시 OnNetworkDespawn→OnNetworkSpawn이 반복됩니다. 인라인 람다 구독 해제가 작동하지 않으므로 핸들러가 누적되어 HandleOccupancyChanged가 중복 호출되고, 장시간 플레이 시 성능 저하와 예기치 않은 동작(역할 UI 깜빡임 등)이 발생할 수 있습니다.
  • OnNetworkSpawn에서 등록하는 4개 람다를 Action<ulong,ulong> 필드에 저장
  • OnNetworkDespawn에서 저장된 필드 참조로 -= 구독 해제
  • 씬 전환 2-3회 반복 테스트로 중복 호출 없음을 확인
VR 프레임 드롭 유발 가능성 (GC 할당)
VR은 90fps 유지가 필수이며 GC 스파이크는 즉각적인 프레임 드롭과 멀미를 유발합니다. EnemyTank.CalculateExposure()와 BallisticCalc.SimulateTrajectory()가 매 틱/프레임마다 배열을 할당하고, 적 전차가 다수 스폰되면 문제가 심화됩니다.
  • CalculateExposure()의 Vector3[]를 클래스 필드로 캐싱하여 재사용
  • SimulateTrajectory()의 Vector3[]도 풀링 또는 사전 할당으로 전환
  • Unity Profiler에서 GC.Alloc 추적하여 VR 환경에서 90fps 유지 확인
GunnerHUD 미구현으로 포수 역할 플레이 불가
커밋 메시지(feat(GunnerHUD): 포수 HUD 시스템 구현)와 달리 실제 GunnerHUD 스크립트가 존재하지 않습니다. 포수가 현재 포탑 각도, 장전 상태, 조준선을 확인할 수 없어 VR에서 전차 조작이 사실상 불가능합니다.
  • AimController의 CurrentYaw/CurrentPitch/IsLoaded를 읽어 표시하는 GunnerHUD 스크립트 생성
  • World-space Canvas로 포수석 시야에 HUD 배치
  • 장전 상태 색상 표시(이미 AimController._loadIndicatorRenderer로 일부 구현됨)를 HUD에 통합
6개 미병합 브랜치와 main 동기화 문제
feature/jwt_0000이 main 대비 43커밋 뒤처져 있고, feat/EnemyAI도 7커밋 뒤처져 있습니다. develop이 실질적 메인 브랜치인데 main과 4커밋 차이가 있어 병합 충돌 리스크가 높아지고 있습니다. namwon-RT가 6일째 비활성이라 해당 브랜치가 방치될 수 있습니다.
  • develop → main 머지하여 동기화
  • feature/jwt_0000 브랜치 상태 확인 후 머지 또는 폐기 결정
  • namwon-RT에게 연락하여 feat/EnemyAI 브랜치 상태 확인
오디오/애니메이션 에셋 전무
VR 탱크 시뮬레이션에서 엔진 소리, 포격음, 피격 효과음은 몰입감의 핵심입니다. 현재 오디오 파일 0개, 애니메이션 0개로 데모 시 게임 완성도가 크게 저하됩니다.
  • 최소한 포격음, 장전음, 피격음 3개 SFX를 확보하여 FireSystem/BarrelLoader에 연결
  • 엔진 루프 사운드를 TankDriver에 추가
  • 적 전차 폭발 이펙트+사운드를 EnemyBase.Die()에 연결

  1. GunnerHUD.cs 생성: World-space Canvas에 방위각/고각/장전 상태 표시
  2. AimController.OnFireRequested 이벤트에 AudioSource.PlayOneShot 연결
  3. BarrelLoader 장전 완료 시 장전음 재생 추가
3. EnemyTank.cs 분할 리팩토링
736줄 단일 파일은 코드 리뷰, 병합 충돌, 디버깅 모두에서 병목입니다. 특히 Nullma가 단독으로 관리하는 파일이라 다른 팀원이 AI 로직을 이해하기 어렵습니다.
  1. EnemyTankAwareness(경계 시스템 로직)를 별도 컴포넌트로 분리
  2. EnemyTankCombat(사격/조준/재배치 로직)을 별도 컴포넌트로 분리
  3. EnemyTank은 상태머신 전환과 컴포넌트 조율만 담당하도록 축소

코드 리뷰

EnemyTank.cs개선필요
경계 시스템(awareness 0-100), 노출도 계산(5점 Linecast), 상태머신(6개 상태), 장갑 관통 판정까지 가장 복잡한 AI를 잘 구현했습니다. 그러나 736줄로 파일 크기 제한을 크게 초과하고, AILoop()가 101줄의 거대 switch문으로 되어 있어 각 상태 처리를 별도 메서드로 분리해야 합니다. EnemyPartHPState가 서버 전용이라 클라이언트에서 적 전차 상태를 표시할 수 없습니다. CalculateExposure()에서 매 프레임 Vector3 배열을 할당하는 것도 VR 성능에 영향을 줄 수 있습니다.
RoleSync.cs심각
역할 배정→소유권 이전→UI 오브젝트 활성화 흐름이 잘 설계되었고, 서버 검증(IsEmptyOnServer)으로 동시 탑승을 방지합니다. 그러나 OnNetworkDespawn에서 인라인 람다로 이벤트를 해제하려 하는데, 람다는 매번 새 인스턴스를 생성하므로 실제로 구독이 해제되지 않아 메모리 누수가 발생합니다. OnNetworkSpawn에서 사용한 것과 동일한 델리게이트 참조를 필드에 저장하여 해제해야 합니다.
TankStatus.cs양호
부위별 HP(Hull/Turret/Barrel/AmmoRack/Engine/Track), 방향별 장갑(Front/Side/Rear), 코사인 보정 관통 판정, 도탄 데미지(5%) 등 전차 시뮬레이션의 핵심 메커닉을 정확하게 구현했습니다. PartHPState가 INetworkSerializable을 구현하여 클라이언트 동기화가 올바르고, OnNetworkSpawn/Despawn 라이프사이클도 적절합니다. ArmorProfile 구조체로 장갑 데이터를 깔끔하게 분리한 점이 좋습니다.
AimController.cs양호
NetworkVariable<float>로 포탑 Yaw/Pitch를 Owner가 직접 갱신하고 OnValueChanged로 전 클라이언트에 동기화하는 구조가 적절합니다. XRGrabInteractable 그랩 시 ContinuousTurnProvider/SnapTurnProvider를 비활성화하여 썸스틱을 조준에 독점시키는 설계가 VR UX를 잘 고려했습니다. UNITY_EDITOR 전처리기로 에디터 테스트 지원도 실용적입니다.
GameManager.cs개선필요
인증→로비→게임 씬 전환을 단일 코루틴 Workflow()에서 관리하는데, 120줄의 거대 switch문으로 되어 있어 각 상태 핸들러를 별도 메서드로 분리해야 합니다. Unity Services 초기화와 씬 로딩을 while(true) 루프에서 체크하는 방식이 비효율적이며, 씬 로드 완료와 NGO 준비 사이의 동기화가 암시적(타이밍 의존)입니다.
FireSystem.cs양호
119줄로 간결하며 단일 책임 원칙을 잘 준수합니다. AimController.OnFireRequested 이벤트 구독→BarrelLoader.IsLoaded 검증→BallisticCalc.TryGetImpactData()→ProjectilePoolManager.Get()으로 시각 투사체 생성→WaitForSeconds(비행시간)→SplashDamageUtils.Apply() 데미지 적용의 명확한 파이프라인을 구현했습니다.
BarrelLoader.cs양호
XRSocketInteractor 기반 물리적 탄약 장전이 VR 몰입감에 기여합니다. SetLoadedServerRpc에서 역할 검증(Loader만 가능)을 수행하고, AmmoPack.GrabInteractable을 비활성화하여 이중 그랩을 방지합니다. AmmoData를 BallisticCalc에 전달하는 흐름도 깔끔합니다.
WaveManager.cs양호
피셔예이츠 셔플로 스폰 포인트 무작위화, NavMesh.SamplePosition으로 유효 위치 검증, 1.5초 간격 순차 스폰으로 물리 충돌 방지, 전멸 시 다음 웨이브 자동 시작까지 웨이브 매니저의 핵심 기능을 안정적으로 구현했습니다. ±3m 지터로 클러스터링도 방지합니다.

아키텍처 평가

서버 권위(데미지/스폰/역할 배정) + 소유자 권위(입력/포탑 회전) 하이브리드 네트워크 아키텍처가 VR 멀티플레이에 적합하게 설계되었습니다. RoleSync 싱글톤이 소유권 이전의 단일 진입점 역할을 하며, NetworkVariable의 Owner WritePermission으로 입력 권한을 자연스럽게 제어합니다. Observer 패턴(OnFireRequested, OnValueChanged, OnDied)으로 시스템 간 결합도를 낮추고, ProjectilePoolManager의 Object Pool로 VR 성능을 고려했습니다. 다만 GameManager가 모든 상태 전환을 단일 코루틴에서 처리하고, EnemyTank이 경계+전투+데미지+이동을 모두 담당하여 God Object 경향이 있습니다.

기반 구축 4/5점
네임스페이스(IronSarcophagus.Core/Tank/Enemy/Network/UI) 및 어셈블리 정의(asmdef 6개)로 순환 참조 방지 구조 확립. GameConst로 상수 중앙 관리, IDamageable 인터페이스, DamageInfo 구조체 등 핵심 타입 정의 완료. 단, AuthUI/LobbiesUI에서 네임스페이스 오타(Netowork→Network)가 있고, 에러 복구 로직이 전혀 없어 1점 감점.
핵심 시스템 20/25점
탱크 시스템 완성도 높음: TankStatus(부위별 HP+방향별 장갑+관통 판정), AimController(NetworkVariable 기반 포탑 회전+XR 그랩 연동), BallisticCalc(포물선 궤적 시뮬레이션+비행시간 계산), FireSystem(발사→탄착→스플래시 데미지 풀 파이프라인), BarrelLoader(XR 소켓 물리 장전+서버 검증), TankDriver(스로틀/조향 입력+서버 권위 이동) 모두 실동작 로직. 적 AI도 EnemyTank(경계 시스템 0-100+상태머신 6개 상태+장갑 관통), EnemyInfantry(소총병/대전차 이중 역할), EnemyLightVehicle(수송→하차→퇴각 흐름), WaveManager(웨이브 스폰+피셔예이츠 셔플)까지 구현. 단, HUDController가 미구현이고 수리병 역할 로직이 없어 5점 감점.

심각도파일설명
CRITICALRoleSync.csOnNetworkDespawn()에서 인라인 람다 (_, v) => HandleOccupancyChanged(...)로 이벤트를 해제하지만, 람다는 매번 새 인스턴스를 생성하므로 OnNetworkSpawn()에서 등록한 것과 다른 델리게이트입니다. 실제로 구독이 해제되지 않아 메모리 누수 및 중복 호출이 발생합니다.
HIGHAuthUI.cs / LobbiesUI.cs네임스페이스가 IronSarcophagus.Netowork로 오타가 있습니다(Network의 'w' 누락). 다른 네트워크 스크립트(RoleSync, PlayerSpawner)는 IronSarcophagus.Network을 사용하므로 불일치가 발생합니다.
HIGHAuthUI.csAuthenticationService.Instance.SignInAnonymouslyAsync() 호출 시 실패 처리가 없습니다. 네트워크 오류나 서비스 장애 시 사용자에게 아무 피드백 없이 무음 실패합니다.
MEDIUMEnemyTank.csAILoop() 메서드가 101줄로 50줄 가이드라인을 크게 초과합니다. 6개 상태의 switch문이 하나의 메서드에 모두 포함되어 있어 가독성과 유지보수성이 떨어집니다.
MEDIUMEnemyTank.csCalculateExposure()에서 매 호출마다 Vector3 배열(ExposureCheckPointCount=5)을 new로 할당합니다. AI 틱마다 호출되므로 GC 압박이 발생하며, VR의 프레임 버짓에서 GC 스파이크는 치명적입니다.
MEDIUMGameManager.csWorkflow() 코루틴이 120줄의 while(true) + switch문으로 구성되어 있습니다. 상태 전환 로직이 한 메서드에 집중되어 있어 개별 상태 디버깅이 어렵고, 씬 로드 완료와 NGO 준비 간 동기화가 암시적입니다.
MEDIUMAmmoPackDispenser.cs_pendingInteractor를 통한 비동기 그랩 전달 로직에서, RPC 응답 전에 사용자가 다른 상호작용을 할 경우 경쟁 조건이 발생할 수 있습니다. RequestPackRpc와 ReceivePackRpc 사이의 상태가 보호되지 않습니다.
MEDIUMBallisticCalc.csSimulateTrajectory()에서 매 호출마다 new Vector3[MaxPoints]를 할당합니다. 궤적 시각화가 매 프레임 갱신되면 상당한 GC 할당이 발생합니다.
LOWTankStatus.csOnServerSpawn()에서 초기 HP를 _maxPartHP - 20, _maxHullHP - 20으로 설정하는데, 매직 넘버 20이 GameConst.Tank.InitialHPPenalty로 정의되어 있지만 이유가 주석으로 설명되지 않아 의도 파악이 어렵습니다.
LOWLobbiesUI.csRefreshLobbies() 기능이 주석 처리되어 있어 로비 목록 갱신이 불가능합니다. 사용자가 다른 세션을 탐색할 수 없어 멀티플레이 접근성이 제한됩니다.
