  • 병합 완료 후 원격 브랜치 정리: git push origin --delete <branch>

  • 리팩터링 후 기존 테스트/동작 확인 — 한 번에 하나씩 추출하고 매번 테스트

Assembly Definition 미사용
Assembly Definition(.asmdef) 없이는 스크립트 하나를 수정할 때마다 프로젝트 전체가 재컴파일됩니다. 파일이 많아질수록 이터레이션 속도가 급격히 저하됩니다.
  • Assets/Scripts 폴더에 우클릭 → Create → Assembly Definition
  • 이름 설정: IronSarcophagus.Runtime
  • 필요한 경우 Editor 폴더에도 별도 asmdef 생성 (.Editor)
  • 외부 패키지 의존성은 asmdef의 Assembly Definition References에 추가

권고사항 (Top 3)

1. 미구현 핵심 시스템 착수: HUDController, PlayerSpawner
예상 시스템 중 2개가 감지되지 않았습니다. 핵심 시스템이 없으면 게임 루프를 완성할 수 없습니다.
  1. 우선순위 1위: HUDController — 게임 플로우의 핵심이므로 먼저 구현
  2. 최소 기능부터 시작: 빈 클래스 → public 메서드 시그니처 → 기본 로직 → 테스트
  3. 다른 시스템과의 연결점(이벤트, 콜백)을 먼저 정의하면 병렬 작업 가능
  4. MonoBehaviour가 아닌 일반 C# 클래스로 로직을 분리하면 테스트가 쉬움
2. 브랜치 정리 및 main 통합
분산된 브랜치는 통합 비용이 시간이 갈수록 증가합니다. 조기 병합이 핵심입니다.
  1. git fetch --all 로 최신 상태 동기화
  2. git log --oneline --graph --all 로 브랜치 구조 시각화
  3. 가장 변경이 적은 브랜치부터 main에 병합
  4. 충돌 발생 시 팀원과 함께 resolve — 혼자 임의로 해결하지 않기
